🔄 高并发系统设计

一句话：高并发 = 同时处理大量请求。核心挑战：共享资源冲突 + 协调开销 + 数据一致性。

和高性能的区别：高性能关心"单请求多快"，高并发关心"同时处理多少请求"。两者有交集（如 Worker Pool 既提升并发又提升吞吐），但侧重点不同。

🧵 并发 vs 并行：一个厨师 vs 多个厨师

🍳 类比：并发 = 一个厨师同时做 3 个菜——切菜时煲汤在煮，抽空翻一下炒锅。时间片轮转，看起来同时在干。并行 = 三个厨师各做一个菜——真正的同时执行。

并发 (Concurrency)

  • 逻辑上同时处理
  • 时间片轮转
  • 单核也能实现
  • 关注任务调度

并行 (Parallelism)

  • 物理上同时执行
  • 需要多核
  • 真正的"同时"
  • 关注计算加速

高并发的核心矛盾

共享资源 vs 互斥访问。100 个请求同时扣减库存 → 必须排队，但排队就降低并发。

终极目标：尽量减少串行部分，最大化并行部分。（回归阿姆达尔定律）

🔐 Mutex & RWMutex：厕所门锁 vs 阅览室

🚪 类比：Mutex = 厕所门——一次只能一个人用，别人等。RWMutex = 阅览室——多人可以同时看书，但不能有人写黑板（读共享，写互斥）。

Mutex 的正确使用姿势

// ✅ 正确：锁的粒度尽量小
func Update() {
    mu.Lock()
    data.count++
    mu.Unlock()
    // 耗时操作放在锁外面！
    sendNotification()
}

// ✅ 推荐：defer 防忘解锁
func SafeUpdate() {
    mu.Lock()
    defer mu.Unlock()
    data.count++
}

// ❌ 错误：锁内做 IO——其他 goroutine 全在等
func BadUpdate() {
    mu.Lock()
    defer mu.Unlock()
    data.count++
    http.Post(url, body) // 💀 锁里做网络请求
}

⚛️ 原子操作 & CAS：锁的终极替代品

🪑 类比：CAS（Compare-And-Swap） = 抢椅子游戏——你看到一把空椅子（读当前值），走过去想坐（比较），发现被别人抢了（值变了）？没关系，再找下一把（重试）。整个过程不需要换门锁——CPU 指令级原子操作。

何时用原子操作代替 Mutex？

简单的计数器、标志位 → atomic。一条 CPU 指令完成，无上下文切换。

复杂的多步操作 → Mutex。原子操作只能保证单次读写是原子的，不能保证"读+改+写"的原子性（除非用 CAS 循环）。

var counter int64

// ❌ counter++ 不是原子的（读→加→写 三步）
func badIncr() { counter++ }

// ✅ atomic：一条 CPU 指令
func safeIncr() { atomic.AddInt64(&counter, 1) }

// ✅ CAS 循环：无锁更新（乐观锁模式）
func casUpdate() {
    for {
        old := atomic.LoadInt64(&counter)
        if atomic.CompareAndSwapInt64(&counter, old, old+1) {
            break  // 成功！没被人抢
        }
        // 失败：值被别人改了，重试
    }
}

📨 Channel：Go 的传送带

🏭 类比：工厂传送带——工位 A 把零件放上传送带（ch <- data），工位 B 从传送带取走（data := <-ch）。传送带长度有限（buffered channel），满了 A 就等着，空了 B 就等着。天然的生产者-消费者协调。

Channel 类型

类型声明行为场景
无缓冲ch := make(chan int)发送者阻塞直到有人接收（同步）精确同步、保证送达
有缓冲ch := make(chan int, 100)缓冲区满之前不阻塞解耦、削峰、限流
只读/只写<-chan int / chan<- int编译时防止误用函数签名约束

Channel 三大坑

① 向已关闭的 channel 发送 → panic。关闭前确保没有发送者还在用。

② 关闭已关闭的 channel → panic。用 sync.Once 或在生产者端统一关闭。

③ goroutine 泄漏。channel 的接收端退出了，发送端永远阻塞。

⚙️ GMP 调度器：Go 并发的引擎

GMP 调度模型：G（goroutine）→ P（本地队列）→ M（OS 线程执行）+ Work Stealing

GMP 核心规则

  1. P 的数量 = GOMAXPROCS（默认 CPU 核数），决定最大并行度
  2. M 必须绑定 P 才能执行 G。没有 P 的 M 休眠
  3. G 阻塞在系统调用时 → M 和 P 解绑 → P 找/新建 M 继续执行其他 G
  4. work stealing：P 的本地队列空了 → 从全局队列偷一半 → 还不够就从其他 P 偷

🔀 select：多路复用——同时等好几个 channel

📞 类比：热线接线员——同时守着 3 部电话（3 个 channel），哪部响了接哪部。都响？随机接一部。select = Go 的接线员。

// select 经典模板：超时控制 + 多 channel 监听
select {
case result := <-workCh:
    handle(result)                      // 工作完成
case <-ctx.Done():
    return ctx.Err()                    // 超时/取消
case <-ticker.C:
    log.Println("心跳")              // 定时任务
default:
    // 非阻塞：哪个都没准备好就做别的
}

select 关键特性

特性说明
多个 case 同时就绪随机选择一个执行（伪随机，防止饿死）
有 default非阻塞——没有就绪的 case 就走 default
无 default阻塞——等到至少一个 case 就绪
nil channel对 nil channel 的读写永远阻塞——可以用来动态禁用某个 case
select 多路复用：同时监听多个 channel，哪个先到就处理哪个——Go 并发的一等公民

🧰 sync 包全家桶

工具用途一句话
sync.Mutex互斥锁一次一个人
sync.RWMutex读写锁多人读，一人写
sync.WaitGroup等待一组 goroutine 完成Add + Done + Wait
sync.Once只执行一次单例初始化
sync.Pool对象池复用临时对象，减少 GC
sync.Cond条件变量广播通知等待者
sync.Map并发安全 Map读多写少场景专用

sync.Map vs map+mutex——什么时候用 sync.Map？

推荐 sync.Map：① key 只写一次读很多次 ② 多个 goroutine 读写不同的 key

推荐 map+Mutex：写操作频繁、需要批量操作（range）

经验：不确定就用 map + sync.RWMutex。sync.Map 是特定场景优化，不是通用替代。

🎯 四大并发模式

1. Worker Pool（控制并发数）

func WorkerPool(tasks []Task, workers int) {
    ch := make(chan Task, len(tasks))
    var wg sync.WaitGroup
    for i := 0; i < workers; i++ {
        wg.Add(1)
        go func() { defer wg.Done(); for t := range ch { t.Do() } }()
    }
    for _, t := range tasks { ch <- t }
    close(ch); wg.Wait()
}
Worker Pool 架构：Task Queue → N 个 Worker goroutine → Fan-in 汇聚结果

2. Pipeline（流水线）

// Stage1 → Stage2 → Stage3
gen := func(nums ...int) <-chan int {
    out := make(chan int)
    go func() { for _, n := range nums { out <- n }; close(out) }()
    return out
}
sq := func(in <-chan int) <-chan int {
    out := make(chan int)
    go func() { for n := range in { out <- n*n }; close(out) }()
    return out
}
for n := range sq(gen(2,3,4)) { fmt.Println(n) }  // 4, 9, 16

3. Fan-Out / Fan-In（分发 + 汇聚）

// Fan-Out: 一个输入 channel → 多个 worker
func fanOut(in <-chan Task, workers int) []<-chan Result {
    outs := make([]<-chan Result, workers)
    for i := 0; i < workers; i++ { outs[i] = worker(in) }
    return outs
}
// Fan-In: 多个 channel → 一个
func fanIn(chs ...<-chan Result) <-chan Result {
    out := make(chan Result)
    var wg sync.WaitGroup
    for _, ch := range chs {
        wg.Add(1)
        go func(c <-chan Result) { defer wg.Done(); for r := range c { out <- r } }(ch)
    }
    go func() { wg.Wait(); close(out) }()
    return out
}

4. Pub/Sub（发布订阅）

type Broker struct {
    mu  sync.RWMutex
    subs map[string][]chan Event
}
func (b *Broker) Publish(topic string, evt Event) {
    b.mu.RLock(); defer b.mu.RUnlock()
    for _, ch := range b.subs[topic] {
        select { case ch <- evt: default: }  // 非阻塞发送
    }
}

🔒 高并发下的数据一致性

乐观锁 vs 悲观锁

🎫 类比：悲观锁 = 买票先锁座位，付款后解锁——假设别人会抢。乐观锁 = 直接下单，提交时检查座位还在不在——假设没人抢。

悲观锁

SELECT * FROM stock
WHERE id=1 FOR UPDATE
-- 锁住这一行，别人等
UPDATE stock SET qty=qty-1
WHERE id=1

适合：冲突概率高

性能低（排队等锁）

乐观锁

UPDATE stock SET qty=qty-1,
  version=version+1
WHERE id=1 AND version=5
-- rowsAffected=0 = 冲突，重试

适合：冲突概率低

性能高（不加锁）

Go 中的分布式锁（Redis 实现）

func AcquireLock(ctx context.Context, key string, ttl time.Duration) bool {
    // SET NX + EXPIRE：原子操作，拿到锁 + 设置过期
    ok, err := rdb.SetNX(ctx, key, "locked", ttl).Result()
    return err == nil && ok
}
func ReleaseLock(ctx context.Context, key string) {
    // Lua 脚本：判断 value 是否是自己，避免误删别人的锁
    script := `if redis.call("get",KEYS[1])==ARGV[1] then
               return redis.call("del",KEYS[1]) else return 0 end`
    rdb.Eval(ctx, script, []string{key}, "locked")
}

❓ 高频考题

Q1: goroutine 和 OS 线程的区别？
大小：goroutine 初始 2KB 栈（动态扩缩），线程 ~1MB（固定）。
切换：goroutine 用户态切换 ~几百 ns，线程内核态切换 ~几 μs。
调度：goroutine 由 Go 运行时调度（GMP），线程由 OS 调度。
数量：百万 goroutine 没问题，几千线程就内存炸了。
Q2: channel 和 Mutex 怎么选？
传递数据所有权 → channel："这个数据给你了，我不管了"。数据从一处流向另一处。
保护共享状态 → Mutex：多个 goroutine 都要读/写同一块内存。
性能敏感 → atomic/Mutex：channel 底层有锁+内存分配，比 Mutex 慢。
Go 谚语："Don't communicate by sharing memory; share memory by communicating." 但不是绝对的。
Q3: 用 channel 实现一个无锁的并发安全队列？
buffered channel 天然就是并发安全的队列——make(chan T, 100)。多生产者安全写入，多消费者安全读取。但注意：① 容量固定 ② 满了会阻塞 ③ 关闭后再发会 panic。
Q4: sync.WaitGroup 的坑？
① Add 必须在 goroutine 外部调用——如果在 goroutine 内部 Add，可能 Wait 先执行完。
② 不能复制 WaitGroup——必须传指针。
③ 计数器不能为负——Done 多于 Add 会 panic。
Q5: 怎么限制 goroutine 的并发数量？
① buffered channel 做信号量：sem := make(chan struct{}, 100)，获取信号量再起 goroutine。
② Worker Pool：固定 N 个 goroutine 消费 channel。
③ errgroup.SetLimit：Go 1.20+ 的 errgroup.SetLimit(n)。
Q6: GMP 调度中，如果一个 goroutine 执行时间太长怎么办？
Go 1.14+ 支持基于信号的异步抢占。运行超过 10ms 的 goroutine 会被 SIGURG 信号打断，调度器可以把它切走。但要注意：① 在 CGO 或系统调用中的 goroutine 不可抢占 ② 紧循环（for { i++ }）在 1.14 前会饿死其他 goroutine。

💬 苏格拉底式提问 —— 检验真理解

以下问题用来逼你追问本质。先自己想 30 秒，再点开验证。

goroutine 是越多越好吗？Q1
不是。虽然轻量（初始栈 2KB），仍有成本：
① 调度开销：百万 goroutine → 调度器消耗大量 CPU。
② 内存：2KB × 1M = 2GB，仅栈空间。
③ GC 压力：goroutine 持有的对象越多，GC 扫描越慢。
④ 真正瓶颈：是背后的资源——DB 连接数、文件描述符、网络带宽。
原则：用 Worker Pool 控制上限。不要无脑 go func()。
channel 一定比 Mutex 好？Q2
"share memory by communicating" 是指导原则，不是真理。
Mutex：保护数据结构（map、计数器）；性能敏感（低竞争时极快）；缓存管理。
Channel：传递数据所有权；协调多个 goroutine。
Go 官方："选择最适合的。不明显时用 Mutex（更简单）。"
GMP 中 P 的数量为什么限制为 GOMAXPROCS？Q3
P = 逻辑处理器，决定最大并行度。
① 物理上限：每个 P 绑定 OS 线程。M 超过 CPU → 上下文切换。
② 伪并行：P 比 CPU 多 → cache locality 变差。
③ work stealing 效率下降。
调大：IO 密集型 > CPU 核数。
调小：容器环境 → 匹配容器 CPU limit。

🎙️ 口播总结

这篇文章是 Go 并发的完全指南。

并发和并行的区别：并发是逻辑上同时处理、时间片轮转、单核也能做；并行是物理上同时执行、需要多核。Go 的 GMP 调度器是核心——G 是 goroutine，M 是 OS 线程，P 是逻辑处理器。P 的数量等于 GOMAXPROCS，M 必须绑定 P 才能执行 G。work stealing 让空闲的 P 从其他 P 偷任务。

同步三件套：Mutex 是厕所门一次一人、RWMutex 是阅览室多人读一人写、atomic 是 CPU 指令级原子操作比 Mutex 快 7 倍。Channel 就像传送带，无缓冲的是同步交接、有缓冲的是队列解耦。

Go 的 select 是多路复用，同时监听多个 channel，哪个先到处理哪个。sync 包全家桶里 WaitGroup 和 Once 最常用。四大并发模式：Worker Pool 控制并发数、Pipeline 流水线、Fan-Out/Fan-In 分发汇聚、Pub/Sub 发布订阅。

高并发下的数据一致性靠乐观锁和悲观锁——乐观锁用版本号冲突重试，悲观锁用 SELECT FOR UPDATE 排队。

如果只能记住一句话，那就是"goroutine+channel+select=Go 并发的三驾马车"。
