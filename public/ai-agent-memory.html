🧠 AI Agent Memory — 记忆与上下文管理

深入理解 Agent 记忆系统的三层架构、上下文窗口策略、持久化方案与一致性保证

1. 为什么需要记忆系统

没有记忆的 AI Agent 就像一位健忘的厨师：每次你走进厨房，他都不记得你刚刚点过的菜，也不记得冰箱里还剩什么食材。让我们用一个费曼式的厨房类比来拆解记忆的必要性。

如果把 Agent 比作厨师，工作台就是短期记忆——上面摆着正在处理的食材和菜谱，空间有限，一次只能放 6～8 样东西。冰箱是长期记忆——储存了大量食材（知识），取用稍慢但容量很大。笔记本是持久记忆——记录顾客偏好、历史订单，跨会话保留。

缺少任何一个环节：没有工作台 → 无法即时操作；没有冰箱 → 知识匮乏；没有笔记本 → 每次从零开始。

大规模语言模型（LLM）的上下文窗口天然有限——GPT-4 的 128K tokens 看似很大，但在多轮对话、代码仓库级上下文中仍然捉襟见肘。记忆系统正是为了弥补这个缺口而设计的。

2. 短期记忆——上下文窗口管理

上下文窗口（Context Window）是 Agent 当前能"看到"的所有信息。它直接决定了模型的理解质量和响应连贯性。以下是三种主流策略：

三种上下文管理策略

策略原理优点缺点适用场景
截断 超出窗口时丢弃最早的消息 实现简单，开销为零 丢失早期上下文，对话断裂 短对话、一次性任务
摘要压缩 定期将旧消息压缩为摘要 保留要点，节省 token 丢失细节，摘要可能偏差 长对话、知识密集型
混合策略 滑动窗口 + 摘要 + 检索 兼顾即时性和完整性 实现复杂，延迟略高 生产级 Agent 系统

Go 实现：滑动窗口 + 摘要混合存储

package memory

import (
  "context"
  "fmt"
  "sync"
  "time"
)

// Message 表示单条对话消息
type Message struct {
  Role     string    // "user" | "assistant" | "system"
  Content  string
  Tokens   int
  Time     time.Time
}

// Summary 表示压缩后的对话摘要
type Summary struct {
  Content  string
  Tokens   int
  Scope    string  // "session" | "topic"
}

// HybridStore 混合存储：滑动窗口 + 摘要
type HybridStore struct {
  mu          sync.RWMutex
  window      []Message
  summaries   []Summary
  maxWindow   int          // 窗口最大消息数
  maxTokens   int          // 窗口最大 token 数
  summaryFn   func(ctx context.Context, msgs []Message) (string, error)
}

func NewHybridStore(maxWindow, maxTokens int) *HybridStore {
  return &HybridStore{
    window:    make([]Message, 0, maxWindow),
    maxWindow: maxWindow,
    maxTokens: maxTokens,
  }
}

// Add 添加新消息，必要时触发摘要压缩
func (s *HybridStore) Add(ctx context.Context, msg Message) error {
  s.mu.Lock()
  defer s.mu.Unlock()

  s.window = append(s.window, msg)
  totalTokens := 0
  for _, m := range s.window {
    totalTokens += m.Tokens
  }

  // 触发压缩条件：消息数或 token 数超限
  if len(s.window) > s.maxWindow || totalTokens > s.maxTokens {
    return s.compress(ctx)
  }
  return nil
}

// compress 将窗口前半部分压缩为摘要
func (s *HybridStore) compress(ctx context.Context) error {
  mid := len(s.window) / 2
  oldPart := s.window[:mid]
  newPart := s.window[mid:]

  summaryText, err := s.summaryFn(ctx, oldPart)
  if err != nil {
    return fmt.Errorf("compress failed: %w", err)
  }

  s.summaries = append(s.summaries, Summary{
    Content: summaryText,
    Tokens:  len(summaryText) / 4,  // 近似估算
    Scope:   "session",
  })
  s.window = newPart
  return nil
}

// BuildContext 构建最终发送给 LLM 的上下文
func (s *HybridStore) BuildContext() []Message {
  s.mu.RLock()
  defer s.mu.RUnlock()

  // 摘要在前，窗口在后
  result := make([]Message, 0)
  for _, sum := range s.summaries {
    result = append(result, Message{
      Role:    "system",
      Content: "[摘要] " + sum.Content,
    })
  }
  result = append(result, s.window...)
  return result
}

3. 长期记忆——外部存储

当信息超出上下文窗口的承载能力时，我们需要外部存储系统来"记住"它们。长期记忆的核心挑战在于：如何在海量信息中快速找到相关的部分。

存储方案对比

方案原理查询方式延迟场景
🔷 向量数据库嵌入向量 + 近似最近邻语义相似度10-50msRAG、知识检索
🔶 键值存储Key-Value 映射精确匹配1-5ms用户配置、会话状态
📁 文件系统JSON/YAML 文件全文搜索可变日志、调试、小规模

RAG vs Agent 原生记忆

🔍 RAG (检索增强生成)

流程：用户查询 → 检索外部知识库 → 将检索结果拼入 Prompt → LLM 生成

特点：

  • 知识源静态（文档、数据库）
  • 每次查询独立检索
  • 无对话状态追踪
  • 适用于问答、知识库场景

读多写少

🧠 Agent 原生记忆

流程：Agent 运行时自动记录对话 → 内部记忆模块管理 → 决策时自发调用

特点：

  • 记忆随对话动态更新
  • 包含状态、偏好、计划
  • Agent 自主决定何时写入/读取
  • 适用于任务执行、多步推理

读写均衡

Go 实现：Chroma-go 向量记忆存储

package memory

import (
  "context"
  "fmt"
  "log/slog"

  "github.com/amikos-tech/chroma-go"
  "github.com/amikos-tech/chroma-go/openai"
)

// ChromaMemory 使用 chroma-go 实现长期对话记忆
type ChromaMemory struct {
  client     *chroma.Client
  collection *chroma.Collection
  ef         *openai.OpenAIEmbeddingFunction
  sessionID  string
}

func NewChromaMemory(ctx context.Context, sessionID string) (*ChromaMemory, error) {
  client, err := chroma.NewClient("http://localhost:8000")
  if err != nil {
    return nil, fmt.Errorf("chroma client: %w", err)
  }

  ef, err := openai.NewOpenAIEmbeddingFunction(
    openai.WithModel("text-embedding-3-small"),
  )
  if err != nil {
    return nil, fmt.Errorf("embedding fn: %w", err)
  }

  coll, err := client.GetOrCreateCollection(ctx, "agent_memory",
    chroma.WithEmbeddingFunction(ef),
  )
  if err != nil {
    return nil, fmt.Errorf("collection: %w", err)
  }

  return &ChromaMemory{
    client:     client,
    collection: coll,
    ef:         ef,
    sessionID:  sessionID,
  }, nil
}

// Store 存储一段对话记忆
func (m *ChromaMemory) Store(ctx context.Context, id, content string) error {
  metadata := map[string]interface{}{}
  if m.sessionID != "" {
    metadata["session_id"] = m.sessionID
  }

  _, err := m.collection.Add(ctx,
    chroma.WithIDs(id),
    chroma.WithDocuments(content),
    chroma.WithMetadatas(metadata),
  )
  if err != nil {
    return fmt.Errorf("store: %w", err)
  }
  slog.InfoContext(ctx, "stored memory", "id", id)
  return nil
}

// Retrieve 语义检索相关记忆
func (m *ChromaMemory) Retrieve(ctx context.Context, query string, topK int) ([]string, error) {
  results, err := m.collection.Query(ctx,
    chroma.WithQueryTexts(query),
    chroma.WithNResults(topK),
  )
  if err != nil {
    return nil, fmt.Errorf("query: %w", err)
  }

  memories := make([]string, 0)
  for _, doc := range results.Documents {
    memories = append(memories, doc)
  }
  return memories, nil
}

// SummarizeAndStore 将窗口摘要写入长期记忆
func (m *ChromaMemory) SummarizeAndStore(
  ctx context.Context,
  msgs []Message,
) error {
  summary := ""
  for _, msg := range msgs {
    summary += msg.Role + ": " + msg.Content + "\n"
  }
  return m.Store(ctx,
    fmt.Sprintf("sum_%d", len(msgs)),
    summary,
  )
}

4. 持久记忆——跨会话状态

持久记忆让 Agent 能够在不同会话之间"记住"用户。想象一下，如果每次打开聊天都要自我介绍——这正是没有持久记忆的 Agent 的困境。

三层记忆范围

持久化策略：异步保存与冲突解决

跨会话持久化的核心挑战在于一致性和冲突。以下是常用的解决方案：

  • 异步批量写入：将记忆变更缓冲后批量写入，减少 I/O 压力
  • 版本号乐观锁：每条记忆记录带有版本号，写入时检查版本冲突
  • 时间戳合并：以最后写入为准，配合冲突日志
  • 脏写回滚：检测到冲突时回滚并重试
// MemoryRecord 持久化记忆记录
type MemoryRecord struct {
  ID        string    `json:"id"`
  UserID    string    `json:"user_id"`
  Key       string    `json:"key"`       // 记忆键（如 "preference"）
  Value     string    `json:"value"`     // 记忆值（JSON）
  Version   int       `json:"version"`   // 乐观锁版本
  UpdatedAt time.Time `json:"updated_at"`
}

// AsyncPersist 异步持久化存储器
type AsyncPersist struct {
  db      *sql.DB
  buffer  chan MemoryRecord
  batchSz int
  wg      sync.WaitGroup
}

func NewAsyncPersist(db *sql.DB) *AsyncPersist {
  ap := &AsyncPersist{
    db:      db,
    buffer:  make(chan MemoryRecord, 1000),
    batchSz: 50,
  }
  go ap.loop()  // 后台协程批量写入
  return ap
}

func (ap *AsyncPersist) Save(rec MemoryRecord) {
  ap.buffer <- rec
}

func (ap *AsyncPersist) loop() {
  batch := make([]MemoryRecord, 0, ap.batchSz)
  ticker := time.NewTicker(5 * time.Second)
  defer ticker.Stop()
  for {
    select {
    case rec := <-ap.buffer:
      batch = append(batch, rec)
      if len(batch) >= ap.batchSz {
        ap.flush(batch)
        batch = batch[:0]
      }
    case <-ticker.C:
      if len(batch) > 0 {
        ap.flush(batch)
        batch = batch[:0]
      }
    }
  }
}

// flush 批量写入数据库，使用乐观锁避免冲突
func (ap *AsyncPersist) flush(records []MemoryRecord) {
  tx, _ := ap.db.Begin()
  for _, r := range records {
    query := `INSERT INTO memories (id, user_id, key, value, version, updated_at)
                VALUES ($1,$2,$3,$4,$5,$6)
                ON CONFLICT (user_id, key) DO UPDATE
                SET value = $4, version = memories.version + 1,
                    updated_at = $6
                WHERE memories.version = $5`
    _, err := tx.Exec(query, r.ID, r.UserID, r.Key, r.Value, r.Version, r.UpdatedAt)
    if err != nil {
      slog.Error("flush conflict", "err", err)
      tx.Rollback()
      return
    }
  }
  tx.Commit()
}

5. 消息历史管理策略

在有限的 Token 预算下，如何分配资源是一个关键问题。一个合理的分配方案能显著提升 Agent 的表现。

Token 预算分配

Go 实现：TokenBudget 分配器

package token

import "fmt"

// Budget 表示各部分的 Token 预算
type Budget struct {
  System       int  // 系统指令
  Conversation int  // 对话历史
  Retrieval    int  // 检索内容
  Reserved     int  // 预留缓冲
  Total        int  // 总计
}

// BudgetConfig 预算配置
type BudgetConfig struct {
  TotalTokens    int    // 总预算（如 128K）
  SystemRatio    float64 // 系统指令占比
  ConvRatio      float64 // 对话历史占比
  RetrievalRatio float64 // 检索内容占比
  ReservedRatio  float64 // 预留缓冲占比
}

// DefaultConfig 默认配置（20/50/20/10）
func DefaultConfig(totalTokens int) BudgetConfig {
  return BudgetConfig{
    TotalTokens:    totalTokens,
    SystemRatio:    0.20,
    ConvRatio:      0.50,
    RetrievalRatio: 0.20,
    ReservedRatio:  0.10,
  }
}

// TokenBudget 预算分配器
type TokenBudget struct {
  cfg BudgetConfig
}

func NewTokenBudget(cfg BudgetConfig) *TokenBudget {
  return &TokenBudget{cfg: cfg}
}

// Allocate 计算各部分预算
func (tb *TokenBudget) Allocate() Budget {
  total := tb.cfg.TotalTokens
  system := int(float64(total) * tb.cfg.SystemRatio)
  conv := int(float64(total) * tb.cfg.ConvRatio)
  retrieval := int(float64(total) * tb.cfg.RetrievalRatio)
  reserved := int(float64(total) * tb.cfg.ReservedRatio)

  // 调整四舍五入误差
  used := system + conv + retrieval + reserved
  if used < total {
    reserved += total - used
  }

  return Budget{
    System:       system,
    Conversation: conv,
    Retrieval:    retrieval,
    Reserved:     reserved,
    Total:        total,
  }
}

// Fits 检查给定内容是否能在预算内
func (tb *TokenBudget) Fits(category string, tokens int) bool {
  budget := tb.Allocate()
  switch category {
  case "system":
    return tokens <= budget.System
  case "conversation":
    return tokens <= budget.Conversation
  case "retrieval":
    return tokens <= budget.Retrieval
  case "reserved":
    return tokens <= budget.Reserved
  default:
    return false
  }
}

// Summary 返回预算摘要
func (b Budget) Summary() string {
  return fmt.Sprintf(
    "Token Budget [Total: %d] System: %d (%.0f%%) | Conv: %d (%.0f%%) | " +
    "Retrieval: %d (%.0f%%) | Reserved: %d (%.0f%%)",
    b.Total,
    b.System, float64(b.System)/float64(b.Total)*100,
    b.Conversation, float64(b.Conversation)/float64(b.Total)*100,
    b.Retrieval, float64(b.Retrieval)/float64(b.Total)*100,
    b.Reserved, float64(b.Reserved)/float64(b.Total)*100,
  )
}

6. 记忆一致性

记忆一致性确保 Agent 在不同时间点、不同对话中给出不矛盾的回应。两个核心挑战是状态漂移和指代消解。

想象你和朋友聊装修：第一周你说"想要日式风格"，第二周你说"喜欢浅色木纹"，第三周你问"之前说的日式风格，地板用什么材料"。如果朋友忘了一致性，他可能推荐深色地砖——但你明明说的是浅色木纹。

状态漂移就是 Agent 的记忆随时间渐渐走样。指代消解则是正确理解"之前说的"、"那个方案"这类指向性表述。

多轮对话指代链

一致性保证策略

  • 快照校验：每次记忆更新后，对关键状态做一致性断言
  • 差异检测：新记忆与已有记忆冲突时，触发人工确认
  • 版本化记忆：保留历史版本，支持回滚
  • 上下文锚点：在系统提示中嵌入"事实清单"，每次生成时引用

💡 MemGPT/Letta 为什么特别？

MemGPT（现更名为 Letta）提出了"虚拟上下文管理"的概念——它把 LLM 的上下文窗口看作"工作记忆"，通过操作系统式的分页机制，在对话过程中动态换入换出记忆。核心创新点：

  • 记忆分页：将长期记忆分块，需要时再加载
  • 自检机制：Agent 自己判断何时需要查询记忆
  • 归档策略：不常用的记忆自动归档

这使得 Agent 理论上可以处理无限长的对话，而不会超出上下文窗口限制。

8. 实战：装修助手对话记忆

让我们把前面的理论应用到一个具体的场景——装修助手 Agent。这个助手需要：记住用户偏好、跟踪项目进度、保持风格一致性。

装修助手记忆实战代码

package main

import (
  "context"
  "encoding/json"
  "fmt"
  "log/slog"
  "os"
)

// RenovationMemory 装修助手专属记忆
type RenovationMemory struct {
  UserID      string            `json:"user_id"`
  Style       string            `json:"style"`        // 装修风格
  Preferences map[string]string `json:"preferences"` // 偏好键值对
  Budget      float64           `json:"budget"`       // 预算（万元）
  Rooms       []string           `json:"rooms"`        // 房间列表
  Progress    string            `json:"progress"`     // 项目阶段
  Version     int               `json:"version"`      // 版本号
}

// MemoryStore 文件存储实现
type MemoryStore struct {
  basePath string
}

func NewMemoryStore(basePath string) *MemoryStore {
  os.MkdirAll(basePath, 0755)
  return &MemoryStore{basePath: basePath}
}

func (ms *MemoryStore) Load(userID string) (*RenovationMemory, error) {
  path := fmt.Sprintf("%s/%s.json", ms.basePath, userID)
  data, err := os.ReadFile(path)
  if err != nil {
    if os.IsNotExist(err) {
      return &RenovationMemory{UserID: userID, Version: 1}, nil
    }
    return nil, err
  }
  var mem RenovationMemory
  if err := json.Unmarshal(data, &mem); err != nil {
    return nil, err
  }
  return &mem, nil
}

func (ms *MemoryStore) Save(mem *RenovationMemory) error {
  mem.Version++
  data, _ := json.MarshalIndent(mem, "", "  ")
  path := fmt.Sprintf("%s/%s.json", ms.basePath, mem.UserID)
  return os.WriteFile(path, data, 0644)
}

// RenovationAgent 装修助手 Agent
type RenovationAgent struct {
  store     *MemoryStore
  shortTerm *HybridStore  // 短期记忆
  longTerm  *ChromaMemory  // 长期记忆
}

func (a *RenovationAgent) Process(ctx context.Context, userID, input string) string {
  // 1. 加载用户偏好
  userMem, _ := a.store.Load(userID)

  // 2. 短期记忆记录当前输入
  a.shortTerm.Add(ctx, Message{Role: "user", Content: input})

  // 3. 从长期记忆检索相关知识
  knowledge, _ := a.longTerm.Retrieve(ctx, input, 3)

  // 4. 构建完整上下文
  context := a.shortTerm.BuildContext()
  // ... 拼接 system prompt、knowledge、userMem ...

  // 5. 调用 LLM 生成回复（省略）
  response := "根据您偏好的日式风格和浅色木纹，建议..."

  // 6. 更新短期记忆
  a.shortTerm.Add(ctx, Message{Role: "assistant", Content: response})

  // 7. 异步持久化用户偏好
  go func() {
    userMem.Style = "日式"
    userMem.Preferences["material"] = "浅色木纹"
    if err := a.store.Save(userMem); err != nil {
      slog.Error("persist failed", "err", err)
    }
  }()

  return response
}

9. 常见问题 QA

Agent 记忆和 RAG 有什么区别？可以只用 RAG 吗？

RAG 是"查资料"，Agent Memory 是"记事情"。RAG 从外部知识库检索静态信息，适合问答；Agent Memory 动态追踪对话状态、用户偏好和任务进度。如果只用 RAG，Agent 会在多轮对话中"失忆"——它不知道你刚才说了什么，也不知道你叫什么名字。两者互补，生产系统通常同时使用。

上下文窗口越大约好吗？为什么还要记忆管理？

不一定。更大的窗口意味着：1) 计算成本平方级增长（Attention 是与序列长度平方相关的）；2) 模型在超长上下文中容易"迷失中间"（Lost in the Middle），有效利用的信息反而变少。记忆管理的意义不在于扩展窗口，而在于选择性聚焦——把最相关的信息放在窗口里。

摘要压缩会不会丢失重要信息？

会。摘要本质上是有损压缩。缓解方案包括：1) 分层摘要——对摘要再摘要，保留不同粒度；2) 关键信息提取——用结构化格式（JSON）保留精确值（如预算数字、日期）；3) 混合策略——最近几轮原文保留，较早的才压缩。实际应用中，建议对事实性信息（数字、名字、偏好）做精确存储，对对话流动做摘要。

如何选择合适的向量数据库？

选择取决于规模和场景：
Chroma 轻量级，适合原型和小规模（<1M 向量）
Qdrant 性能优秀，Rust 实现，支持过滤
Milvus 分布式，适合大规模生产（>10M 向量）
Pgvector 如果已有 PostgreSQL，最省运维
装修助手场景推荐 Chroma 或 Pgvector。

记忆一致性怎么保证？Agent 说错话了怎么办？

一致性需要多层防护：1) 写入前校验——新的记忆与旧记忆冲突时，触发确认流程；2) 版本化——每条记忆保留历史版本，支持回滚；3) 审计日志——记录每次记忆变更的轨迹；4) 用户反馈环——让用户能纠正 Agent 的错误记忆（"不，我上次说的是浅色木纹，不是深色"）。

Token 预算怎么动态调整？

可以基于运行时反馈调整：1) 如果模型回复多次截断 → 增大对话预算；2) 如果用户频繁纠正事实 → 增大系统指令预算加入更多约束；3) 如果检索结果经常不相关 → 调整检索预算或优化嵌入质量。动态调整可以用简单的 PID 控制器或基于规则的自适应策略。

💡 记忆系统的 Cold Start 问题怎么解决？

Cold Start（冷启动）指新用户没有任何历史记忆时，Agent 如何表现。方案：

  • 默认模板：为新用户提供预设的"默认偏好"（如中性风格、标准预算）
  • 主动探测：在第一轮对话中主动提问收集信息（"您想要什么风格？预算大概多少？"）
  • 渐进填充：不追求一次填满，随着对话自然积累
  • 知识迁移：从同类用户中推断可能的偏好（基于用户画像分组）

在长期记忆部分，我们对比了向量数据库、键值存储和文件系统的适用场景，深入分析了 RAG 与 Agent 原生记忆的本质区别——一个是查资料，一个是记事情。通过 chroma-go 的代码示例，展示了如何将对话记忆持久化到向量数据库中。

持久记忆让我们看到了会话级、用户级和系统级的三层记忆范围，以及异步保存和乐观锁冲突解决的工程实践。Token 预算管理则教会我们如何在有限的上下文窗口中合理分配资源——系统指令 20%、对话 50%、检索 20%、预留 10%。

我们还探讨了记忆一致性的挑战，通过多轮对话指代链的 SVG 图示，展示了 Agent 如何追踪"之前说的日式风格"这样的指代关系。在开源方案对比中，分析了 MemGPT/Letta 的虚拟上下文管理、LangChain 的可插拔记忆组件，以及轻量定制方案的取舍。

核心收获：好的记忆系统不是简单地把所有东西塞进上下文窗口，而是像人类一样——知道该记住什么、该忘记什么、以及何时去回忆。这正是让 Agent 从"健忘工具"进化为"智能伙伴"的关键一步。
