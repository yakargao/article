📖 开源 Agent 项目源码分析

AutoGPT · BabyAGI · Dify · Superagent — 四种主流架构的源码级拆解

1. 为什么读开源 Agent 源码

🧑‍🍳 费曼类比：读文档就像拿着菜谱学做菜——知道要放多少克盐、煎几分钟；读源码就像站在大厨身后看他颠勺——看到手法如何变通、火候如何判断、翻车了怎么补救。菜谱告诉你「做什么」，源码告诉你「为什么这样做」。真正理解一个 Agent 系统，必须打开它的引擎盖。

截至 2026 年，GitHub 上 Agent 类开源项目已超过 2000 个，累计 Star 数突破 40 万。AutoGPT、BabyAGI、Dify、Superagent 是其中最具有代表性的四个项目，分别代表了自主 Agent、极简框架、平台级编排、API 化服务四种截然不同的设计哲学。

通过源码分析，你可以学到：

  • 循环控制：Agent 如何决定"下一步做什么"——是 while(true) 还是事件驱动？
  • 记忆管理：短期记忆（上下文窗口）和长期记忆（向量存储）如何协同工作？
  • 工具调用：Function Calling 的注册、匹配、执行链路如何设计？
  • 扩展性：插件系统、Provider 抽象、多租户隔离如何从源码层面实现？

2. AutoGPT 架构分析 — 最经典的自主 Agent

AutoGPT 发布于 2023 年 3 月，巅峰时期 16 万 Star，是第一个让大众真正感受到"AI 自主完成任务"的项目。它的核心设计理念是：给定一个高层的 Goal，Agent 自主拆解、执行、评估，直到目标达成。

GitHub: Significant-Gravitas/AutoGPT 语言: Python 最新: v0.5.x 核心概念: AgentLoop

2.1 核心循环 — AgentLoop

AutoGPT 最核心的代码位于 autogpt/core/agent_loop.py，其本质是一个有限状态机驱动的 while 循环：

class AgentLoop:
    def __init__(self, goal, storage, plugins):
        self.goal = goal
        self.storage = storage
        self.plugins = plugins
        self.task_queue = []

    async def run(self):
        while not self._goal_achieved():
            task = await self._get_next_task()
            result = await self._execute_task(task)
            evaluation = await self._evaluate(result)
            await self._update_memory(task, result, evaluation)
        return self._summarize()

这个循环的五个阶段构成了 AutoGPT 的骨架：

  1. Goal Parsing — 将用户输入的 Goal 分解为可执行的子任务
  2. Task Prioritization — 从任务队列中选出"最重要/最相关"的下一个任务
  3. Execution — 调用 LLM + 工具执行具体任务
  4. Evaluation — 检查执行结果是否符合预期，是否需要重新尝试
  5. Memory Update — 将结果存入短期（上下文）和长期（向量数据库）存储

2.2 存储系统

AutoGPT 的存储系统分为三层：

  • Short-term Memory — 当前对话的上下文窗口（LLM 的 prompt 内），只保留最近的 N 条交互
  • Long-term Memory — 使用 Pinecone 或 Weaviate 等向量数据库，存储 embedding 后的任务结果
  • File-based Memory — JSON 文件备份，包含 task_history.json、goal_progress.json 等
📁 关键文件结构：
autogpt/core/agent_loop.py — 主循环
autogpt/core/memory/ — 存储抽象基类及实现
autogpt/core/plugin/ — 插件系统
autogpt/llm/prompt_strategies/ — Prompt 模板策略

2.3 插件机制

AutoGPT 的插件系统设计模式是典型的 Chain of Responsibility（职责链模式）。每个插件可以 Hook 到 Agent 循环的各个生命周期点：

# autogpt/core/plugin/base.py (简化)
class AgentPlugin(ABC):
    @abstractmethod
    def can_handle(self, event: AgentEvent) -> bool: ...

    @abstractmethod
    def handle(self, event: AgentEvent) -> AgentEvent: ...

# 注册点示例：执行前修改 prompt
async def _build_execution_prompt(self, context):
    event = BeforeExecuteEvent(context)
    for plugin in self.plugins:
        if plugin.can_handle(event):
            event = await plugin.handle(event)

2.4 Prompt 模板策略

AutoGPT 的一大特色是其丰富的 Prompt 模板体系：

  • goal_creation — 从用户输入解析 Goal
  • task_creation — 根据 Goal 生成子任务
  • prioritization — 对任务列表排序
  • execution — 实际执行任务
  • evaluation — 检查执行结果
  • summarization — 生成的最终汇总

每个策略都可以通过插件热替换，这也是 AutoGPT 灵活性最强的设计点之一。

3. BabyAGI 架构分析 — 最简洁的 Agent 框架

🧠 费曼说："我不能理解的东西，说明我没有真正学会。" BabyAGI 的核心代码不到 300 行，但实现了完整的 Agent 循环。它用最少的代码展示了 Agent 系统的本质——如果你能理解 BabyAGI，你就能理解所有 Agent 框架。

BabyAGI 由 Yohei Nakajima 创作，最初只是一个 200 行的 Python 脚本。它的设计哲学是 KISS（Keep It Simple, Stupid），使用三个 Agent 角色构成一个三角循环：

GitHub: yoheinakajima/babyagi 语言: Python 核心文件: babyagi.py 3 个 Agent · 1 个 Vector Store

3.1 三 Agent 三角

# babyagi.py 核心逻辑 （简化）
def run_babyagi(objective):
    task_list = [Task("Develop a task list")]
    while task_list:
        # Agent 1: Task Execution Agent
        task = task_list.pop(0)
        result = await llm_execute(objective, task)

        # Agent 2: Task Creation Agent
        new_tasks = await llm_create_tasks(objective, result)

        # Agent 3: Task Prioritization Agent
        task_list = await llm_prioritize(task_list + new_tasks)

3.2 为什么只用 Vector Store 就能工作？

BabyAGI 的巧妙之处在于：它不需要复杂的记忆管理，因为 Vector Store 本身就是记忆。具体来说：

  • 每个任务执行的结果被 embedding 后存入向量数据库
  • Task Creation Agent 会根据"与 Objective 最相似的结果"来生成新任务
  • Task Prioritization Agent 会根据任务的"紧急度 + 相关性"得分排序
  • 整个过程中，没有显式的状态管理 — 状态隐含在 Vector Store 的相似度搜索中
✨ 核心洞察：BabyAGI 证明了 Agent = LLM + Task Queue + Vector Store 这个最小公式。所有更复杂的框架（AutoGPT、Dify 等）都是在"如何更好地管理任务队列"和"如何更有效地使用向量存储"上做文章。

3.3 与 AutoGPT 的关键差异

维度AutoGPTBabyAGI
代码量~20000 行~300 行
循环控制有限状态机简单的 while 循环
记忆系统三层：短期+长期+文件单层：Vector Store
插件系统插件链（职责链模式）无
可扩展性高（插件+策略模式）低（直接修改代码）
最适合复杂、长期任务学习、快速原型

4. Dify 架构分析 — 最完整的开源 Agent 平台

Dify 是目前最活跃的开源 LLMOps 平台之一，GitHub 上超过 70k Star。与 AutoGPT 和 BabyAGI 不同，Dify 不是一个"运行 Agent"的库，而是一个打造 Agent 应用的平台。从源码角度看，Dify 的核心是三个引擎的组合：

GitHub: langgenius/dify 语言: TypeScript/Python 前端: Next.js + React 后端: Flask + Celery 数据库: PostgreSQL + Redis + Weaviate

4.1 Workflow Engine — 可视化的 Agent 编排

Dify 的 Workflow Engine 是其核心创新。在源码层面，它定义了一套 DAG（有向无环图）节点系统：

// api/core/workflow/nodes/base.py （简化）
class WorkflowNode(ABC):
    node_id: str
    node_type: NodeType  # LLM, Code, Tool, Condition, etc.
    inputs: dict
    config: NodeConfig

    @abstractmethod
    async def execute(self, context: ExecutionContext) -> NodeResult: ...

class WorkflowEngine:
    def __init__(self, dag: DAG[WorkflowNode]):
        self.dag = dag

    async def run(self, inputs: dict):
        return await self.dag.topological_execute(inputs)

Dify 支持的节点类型包括：

  • LLM Node — 调用大模型
  • Code Node — 执行 Python/JS 代码片段
  • Tool Node — 调用内置或第三方工具
  • Knowledge Retrieval Node — 从知识库检索
  • Condition Node — 分支判断
  • Variable Aggregation Node — 多路结果合并

4.2 RAG Pipeline

Dify 的 RAG（Retrieval-Augmented Generation）管道是企业级强度：

  1. Document Ingestion — 支持 PDF/DOCX/MD/HTML/TXT 等格式解析
  2. Chunking — 可配置的分块策略（按段落/按 token/按句子）
  3. Embedding — 支持 OpenAI/BAAI/M3E 等多种 embedding 模型
  4. Indexing — 存入 Weaviate/Qdrant/MySQL 等多种向量存储
  5. Retrieval — 混合搜索（向量 + 关键词 BM25）

4.3 Plugin System

Dify 的插件系统与 AutoGPT 不同：它采用 Provider + Tool 双层抽象：

# api/core/plugin/provider/base.py
class PluginProvider(ABC):
    name: str
    tools: list[ToolMeta]

    @abstractmethod
    def validate_credentials(self, config: dict) -> bool: ...

class Tool(ABC):
    name: str
    parameters: list[ParameterDef]

    @abstractmethod
    async def invoke(self, params: dict) -> Any: ...

5. Superagent 架构分析 — 最实用的 Agent API

Superagent 的定位与前三者完全不同：它不是让你自己跑 Agent，而是把 Agent 能力做成 API 卖给你。它最值得学习的源码设计是 Multi-tenant（多租户）隔离和 Provider 抽象层。

GitHub: homanp/superagent 语言: TypeScript 后端: Next.js API Routes 数据库: Prisma + PostgreSQL 核心概念: Agent-as-a-Service

5.1 Agent-as-a-Service 模式

Superagent 的 API 设计遵循 CRUD for Agents 的思路：

// lib/api/agent.ts （简化）
export async function createAgent(params: {
  name: string;
  description?: string;
  llmProvider: LLMProviderType;
  tools: ToolDef[];
  promptTemplate?: string;
  sessionId: string;       // 多租户隔离
}): Promise<Agent> {
  const agent = await prisma.agent.create({
    data: { ...params, userId: getCurrentUser() }
  });
  return agent;
}

export async function runAgent(agentId: string, input: string) {
  const agent = await prisma.agent.findUniqueOrThrow({
    where: { id: agentId }
  });
  const executor = await AgentExecutor.fromAgent(agent);
  return await executor.run(input);
}

5.2 Multi-tenant 隔离

Superagent 的 Multi-tenant 实现非常优雅：

  • 每个 Agent 实例绑定一个 userId 和 sessionId
  • LLM API Key 存储在 Agent 级别（每个用户可以配置不同的 Provider 和 Key）
  • 工具访问权限通过 Agent 级别的 ToolDef 白名单控制
  • 向量存储也按 userId 分库/分 collection

5.4 Superagent 的 Tool Registry 模式

Superagent 的工具注册模式也很值得学习：

// lib/tools/registry.ts
class ToolRegistry {
  private tools = new Map<string, Tool>();

  register(tool: Tool) {
    this.tools.set(tool.name, tool);
  }

  getToolForAgent(agentId: string): Tool[] {
    return this.tools.filterByMetadata({ allowedAgents: [agentId, '*'] });
  }
}

7. 从开源项目能学到什么 — 可复用的设计模式

阅读这四个项目的源码，可以提炼出多个可复用的设计模式。以下是最有价值的五个：

7.1 插件系统设计模式 — Chain of Responsibility

AutoGPT 和 Dify 都使用职责链模式实现插件系统：

# 通用插件接口模式
class AgentPlugin(ABC):
    @property
    def priority(self) -> int: ...  # 执行顺序控制

    @abstractmethod
    def pre_execute(self, prompt: str) -> str: ...

    @abstractmethod
    def post_execute(self, result: str) -> str: ...

关键设计决策：使用 priority 排序而非注册顺序，确保插件之间的依赖关系可控。

7.2 Agent 状态持久化 — Event Sourcing 模式

所有四个项目都面临同一个问题：Agent 运行到一半挂了怎么办？ 最佳实践是将 Agent 的每一步都作为事件持久化：

# 状态持久化的事件模式
class AgentEvent:
    event_id: str
    agent_id: str
    event_type: str  # task_created, task_executed, error, etc.
    payload: dict
    timestamp: datetime
    parent_event_id: str | None  # 形成事件链

这样，即使 Agent 崩溃，也可以通过重放事件链恢复状态。

7.3 Tool Registry 模式 — Strategy + Factory

Superagent 和 Dify 的 Tool Registry 本质是 Strategy Pattern + Factory Pattern 的组合：

# 工具注册 + 工厂模式
class ToolRegistry:
    def __init__(self):
        self._builders = {}

    def register(self, name, builder: Callable[[dict], Tool]):
        self._builders[name] = builder

    def create_tool(self, name, config):
        if name not in self._builders:
            raise ToolNotFoundError(name)
        return self._builders[name](config)

7.4 错误处理与重试策略

从 AutoGPT 和 Dify 的源码中，可以看到三种级别的错误处理：

  1. LLM Level — 模型返回空/格式错误 → 重试 + 降级 prompt
  2. Tool Level — API 调用超时/失败 → 指数退避重试
  3. Agent Loop Level — 连续失败超过阈值 → 发送告警/暂停 Agent
# 指数退避重试（AutoGPT 源码风格）
async def execute_with_retry(fn, max_retries=3, base_delay=1.0):
    for attempt in range(max_retries):
        try:
            return await fn()
        except ToolExecutionError as e:
            if attempt == max_retries - 1:
                raise
            await asyncio.sleep(base_delay * (2 ** attempt))

7.5 记忆压缩与摘要技术

长期运行的 Agent 面临上下文窗口溢出的问题。AutoGPT 和 Dify 都使用了 记忆压缩 策略：

  • Summary-based — 定期将历史对话摘要为一段文字，替换原始内容
  • Sliding Window — 只保留最近的 N 条交互
  • Hybrid — 保留最近交互 + 从向量存储中检索相关历史

9. QA & 口播总结

常见问题

🤔 Q1：AutoGPT 还活跃吗？现在学它还有意义吗？

AutoGPT 的 Star 增长虽然放缓，但它的架构设计理念至今仍然是 Agent 系统设计的参照标准。它的有限状态机循环、三层记忆体系、插件链模式，被后续几乎所有 Agent 框架借鉴。学 AutoGPT 源码，学的是"设计思路"而非"项目本身"。

🤔 Q2：BabyAGI 才 300 行，真的能解决实际问题吗？

BabyAGI 本身确实不是一个生产级工具，但它的教学价值无可替代。它用最少的代码展示了 Agent 系统的本质：任务队列 + LLM 调用 + 向量存储。实际上，很多生产级 Agent 系统（包括 AutoGPT 的早期版本）都是基于 BabyAGI 的架构扩展而来。

🤔 Q3：Dify 和 LangFlow 有什么区别？

两者都是可视化 Agent 编排平台，但设计理念不同：Dify 更偏向产品化，开箱即用，内置 RAG、Prompt 管理、监控面板；LangFlow 更偏向开发，底层直接操作 LangChain 组件，灵活性更高。从源码角度，Dify 的 Workflow Engine 是自研的 DAG 引擎，而 LangFlow 直接依赖 LangChain 的 LCEL。

🤔 Q4：我想自己造一个 Agent 框架，应该参考哪个？

分场景：

  • 如果你想做一个教学/实验框架 → 参考 BabyAGI（最简架构）
  • 如果你想做一个自主 Agent 产品 → 参考 AutoGPT 的循环 + 插件设计
  • 如果你想做一个企业 Agent 平台 → 参考 Dify 的 Workflow Engine + RAG Pipeline
  • 如果你想做一个 Agent API 服务 → 参考 Superagent 的多租户 + Provider 抽象
🤔 Q5：这些项目都用什么 LLM？能接入国产大模型吗？

四个项目都通过 Provider 抽象层支持多模型：

  • AutoGPT / BabyAGI — 通过 LangChain 或直接 API 调用，可接文心、通义千问
  • Dify — 官方支持 OpenAI、Claude、Llama、文心、通义、百川等 20+ 模型
  • Superagent — Provider 接口统一，只需要实现 LLMProvider 接口即可接入国产模型
🤔 Q6：Agent 系统如何处理 Token 消耗过大的问题？

从源码中可以看到三种主要策略：

  1. Prompt 压缩 — AutoGPT/Dify 都会在每次迭代中截断非关键上下文
  2. 记忆摘要 — 定期用 LLM 将历史对话生成摘要，丢弃原始内容
  3. Map-Reduce 模式 — Dify 对长文档使用 Map-Reduce 分片处理，避免一次性塞满上下文
🤔 Q7：Agent 开源项目的前景如何？

2026 年 Agent 开源生态正在快速分化：

  • 框架层 趋于收敛，LangChain/AutoGPT 确立主流范式
  • 平台层 Dify 等补齐企业级能力（审计、监控、权限）
  • 应用层 垂直领域 Agent（代码、客服、数据）成为新热点

建议：平台层选 Dify，应用层关注 AutoGPT 的新插件生态，API 层关注 Superagent 的发展方向。
