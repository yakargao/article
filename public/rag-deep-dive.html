Chunking / Embedding / 混合检索 / Reranking / RAGAS — 从原理到 Go 落地的完整指南

1. RAG 为什么需要

大语言模型存在固有缺陷：知识截止日期、幻觉、缺乏领域专精。Retrieval-Augmented Generation（RAG）通过引入外部知识检索来弥补这些不足。

想象你是一个图书馆管理员。读者问："2025 年诺贝尔物理学奖颁给了谁？"

你不需要自己记住答案——你只需要知道去哪本书里找（检索），然后把那段话读给读者听（生成）。这就是 RAG 的本质：先查资料，再回答问题。

LLM 是阅读者，向量数据库是书架，Embedding 是图书分类标签。

传统微调（Fine-tuning）成本高、周期长、每次更新需要重新训练。RAG 的核心理念是让 LLM 学会"查资料"，而不是"背资料"。它在不改变模型权重的前提下，通过检索增强实现：

  • ✅ 实时知识更新
  • ✅ 减少幻觉
  • ✅ 领域定制无训练成本
  • ✅ 可追溯信息来源

2. Chunking 策略

Chunking（分块）是 RAG Pipeline 的第一步，也是最被低估的一步。块的大小和切分方式直接决定了检索质量。

2.1 四种主流分块策略

策略原理优点缺点适用场景
固定大小按固定 token 数切分，可重叠实现简单、速度快语义割裂严重日志、结构化文本
递归字符按分隔符层级回退切分保留语义边界中文分隔符需要自定义通用文档、代码
语义分块利用 Embedding 相似度检测语义边界语义完整度高耗时长、需要额外模型学术论文、技术文档
特殊文档按 Markdown/HTML/代码结构切分结构感知格式依赖强Markdown、PDF、代码库

2.2 Go 实现：递归字符分割器

package chunker

import "strings"

// RecursiveCharacterSplitter 递归字符分割器
type RecursiveCharacterSplitter struct {
    ChunkSize    int
    ChunkOverlap int
    Separators   []string
}

// NewSplitter 默认使用常见分隔符层级
func NewSplitter(chunkSize, overlap int) *RecursiveCharacterSplitter {
    return &RecursiveCharacterSplitter{
        ChunkSize:    chunkSize,
        ChunkOverlap: overlap,
        Separators:   []string{"\n\n", "\n", ".", " ", ""},
    }
}

// Split 递归分割文本
func (s *RecursiveCharacterSplitter) Split(text string) []string {
    return s.splitWithSep(text, s.Separators)
}

func (s *RecursiveCharacterSplitter) splitWithSep(text string, seps []string) []string {
    chunks := []string{}
    sep := seps[0]
    parts := strings.Split(text, sep)

    for _, part := range parts {
        if len(part) <= s.ChunkSize {
            chunks = append(chunks, part)
        } else if len(seps) > 1 {
            // 递归更深层级分隔符
            sub := s.splitWithSep(part, seps[1:])
            chunks = append(chunks, sub...)
        }
    }
    return s.applyOverlap(chunks)
}

func (s *RecursiveCharacterSplitter) applyOverlap(chunks []string) []string {
    if s.ChunkOverlap == 0 {
        return chunks
    }
    result := []string{chunks[0]}
    for i := 1; i < len(chunks); i++ {
        prev := chunks[i-1]
        overlap := prev
        if len(prev) > s.ChunkOverlap {
            overlap = prev[len(prev)-s.ChunkOverlap:]
        }
        result = append(result, overlap+chunks[i])
    }
    return result
}

递归字符分割器的核心思想：按分隔符优先级从高到低尝试，先按段落（\n\n）切，不行再按句子（。），最后按字符。这保证了语义完整性优先。

3. Embedding 选型

Embedding 是将文本映射到向量空间的"翻译器"。选型核心考量：中文理解能力、维度、成本、延迟。

3.1 选型建议

  • 纯中文场景：bge-large-zh-v1.5（1024 维，检索精度最高）
  • 本地低成本：M3E 或 text2vec（768 维，推理快）
  • 英文为主：OpenAI text-embedding-3-large（3072 维，但贵）
  • 多语言企业：Cohere embed-multilingual-v3

维度越高越好？ 不一定。高维度意味着更丰富的表达能力，但也需要更多的存储和计算。1024 维对于大多数 RAG 系统已经足够。OpenAI 的 3072 维主要是为了适配其生态。

4. 向量检索

将文档转换为向量后，查询向量与库中向量进行相似度匹配。这里涉及两个核心选择：相似度度量和索引结构。

4.1 相似度度量

度量公式范围特点
余弦距离1 - cos(A,B)[0,2]关注方向而非模长，最常用
内积（IP）A·B(-∞,+∞)适合归一化向量，性能略优
欧氏距离（L2）||A-B||²[0,+∞)对模长敏感

4.2 索引结构

暴力搜索（Flat）在小数据集可行，当数据量超过百万级，必须使用近似最近邻（ANN）索引。

🔷 HNSW（Hierarchical NSW）

  • 基于分层可导航小世界图
  • 搜索速度极快（ms 级）
  • 构建慢，内存占用高
  • 适合高召回在线场景

🔶 IVF（Inverted File）

  • 基于聚类+倒排
  • 搜索速度中等
  • 内存占用低
  • 适合大容量离线场景

5. 混合检索

向量检索擅长语义匹配，BM25 擅长关键词精确匹配。混合检索将两者结合，取长补短。

为什么需要混合检索？ 想象搜索"苹果手机保修政策"——向量检索能理解"保修期""售后"等语义相近的词，BM25 能精确命中"iPhone 15 Pro 保修"这样的精确短语。两者互补，效果更好。

5.1 RRF（Reciprocal Rank Fusion）合并

RRF 是一种无参数的排序融合方法，对每个文档在两个排序中的排名取倒数后求和：

score(d) = 1/(k + rank₁(d)) + 1/(k + rank₂(d))

其中 k 通常取 60。这种方法不需要分数归一化，对排名敏感度适中。

5.2 Go 实现：混合检索器

package retriever

import (
    "math"
    "sort"
)

// SearchResult 搜索结果
type SearchResult struct {
    DocID string
    Score float64
}

// HybridRetriever 混合检索器
type HybridRetriever struct {
    VectorRetriever func(query string, k int) []SearchResult
    BM25Retriever   func(query string, k int) []SearchResult
    K               int // 各通道返回数量
    KRRF            int // RRF 常数
}

// RRF 融合排序
func (h *HybridRetriever) RRF(query string) []SearchResult {
    vecResults := h.VectorRetriever(query, h.K)
    bm25Results := h.BM25Retriever(query, h.K)

    // 构建排名映射
    scoreMap := make(map[string]float64)

    for i, r := range vecResults {
        scoreMap[r.DocID] += 1.0 / float64(h.KRRF+i)
    }
    for i, r := range bm25Results {
        scoreMap[r.DocID] += 1.0 / float64(h.KRRF+i)
    }

    // 转换为切片并排序
    merged := make([]SearchResult, 0, len(scoreMap))
    for id, score := range scoreMap {
        merged = append(merged, SearchResult{DocID: id, Score: score})
    }
    sort.Slice(merged, func(i, j int) bool {
        return merged[i].Score > merged[j].Score
    })

    return merged
}

// NormalizedHybrid 归一化加权融合（另一种策略）
func (h *HybridRetriever) NormalizedHybrid(query string, alpha float64) []SearchResult {
    vecResults := h.VectorRetriever(query, h.K)
    bm25Results := h.BM25Retriever(query, h.K)

    // Min-max 归一化
    normalize := func(results []SearchResult) {
        if len(results) == 0 { return }
        min, max := results[0].Score, results[0].Score
        for _, r := range results {
            if r.Score < min { min = r.Score }
            if r.Score > max { max = r.Score }
        }
        diff := max - min
        if diff == 0 { diff = 1 }
        for i := range results {
            results[i].Score = (results[i].Score - min) / diff
        }
    }
    normalize(vecResults)
    normalize(bm25Results)

    scoreMap := make(map[string]float64)
    for _, r := range vecResults {
        scoreMap[r.DocID] += alpha * r.Score
    }
    for _, r := range bm25Results {
        scoreMap[r.DocID] += (1 - alpha) * r.Score
    }

    merged := make([]SearchResult, 0, len(scoreMap))
    for id, score := range scoreMap {
        merged = append(merged, SearchResult{DocID: id, Score: math.Round(score*1000) / 1000})
    }
    sort.Slice(merged, func(i, j int) bool {
        return merged[i].Score > merged[j].Score
    })
    return merged
}

6. Reranking

初次检索（Retrieval）通常返回 Top-K 个候选文档。Reranking 使用更精确但更慢的模型对候选集重新排序，将最相关的文档提到最前面。

6.1 Cross-Encoder vs Bi-Encoder

特性Bi-EncoderCross-Encoder
编码方式Query 和 Doc 分别编码Query 和 Doc 拼接后编码
速度快（可预计算 Doc 向量）慢（必须在线计算）
精度中等高（可交互注意力）
适用阶段初筛检索精排 Reranking

6.2 常用 Reranker 模型

  • Cohere Rerank：商业 API，支持多语言，效果好但有限额
  • bge-reranker-v2-m3：开源，对中文友好，基于 Cross-Encoder
  • ms-marco-MiniLM：轻量级英文 Reranker，4 层 Transformer
  • self-implemented：可用 BERT 微调自己的 Reranker

8. 高级策略

基础的 RAG（Naive RAG）在大规模或复杂场景下效果有限。以下是四种常用的高级 RAG 策略。

HyDE 和 Query Rewriting 有什么本质区别？

Query Rewriting：改写用户 Query（如"它"→"iPhone 16"），让检索更精准。属于查询变换。

HyDE：不改变 Query，而是生成一个假设的文档。核心洞察是："文档-文档"的相似度比"Query-文档"的相似度更可靠。因为 Query 和文档通常不在同一个语义空间（Query 短、文档长），而文档之间的语义空间是统一的。

简单说：Query Rewriting 改问法，HyDE 改答案。

9. Go 实现完整 Pipeline

以下是一个完整的 Go 语言 RAG Pipeline 实现，整合了 Embedding、向量检索、BM25 混合检索、Reranking 和 LLM 生成。

package rag

import (
    "context"
    "encoding/json"
    "fmt"
    "sort"
    "strings"
)

// Config RAG 配置
type Config struct {
    EmbedURL  string // Embedding API 地址
    LLMURL    string // LLM API 地址
    TopK      int    // 检索 Top-K
    RerankTop int    // Rerank 后保留数
    ChunkSize int
}

// Document 文档片段
type Document struct {
    ID      string
    Content string
    Vector  []float64
    Meta    map[string]string
}

// QueryResult 查询结果
type QueryResult struct {
    Query      string
    Documents  []Document
    Context    string
    Answer     string
    Evaluation map[string]float64
}

// RAGPipeline 完整 Pipeline
type RAGPipeline struct {
    cfg       Config
    documents []Document
    bm25      *BM25Index
}

// NewRAGPipeline 创建 Pipeline
func NewRAGPipeline(cfg Config) *RAGPipeline {
    return &RAGPipeline{
        cfg:  cfg,
        bm25: NewBM25Index(),
    }
}

// Ingestion 数据入库流程
func (p *RAGPipeline) Ingestion(ctx context.Context, docs []Document) error {
    for i := range docs {
        // 1. Chunking（此处简化，使用递归分割）
        chunks := simpleChunk(docs[i].Content, p.cfg.ChunkSize)

        for j, chunk := range chunks {
            // 2. Embedding
            vec, err := p.embed(ctx, chunk)
            if err != nil {
                return fmt.Errorf("embedding failed: %w", err)
            }

            doc := Document{
                ID:      fmt.Sprintf("%s_%d", docs[i].ID, j),
                Content: chunk,
                Vector:  vec,
                Meta:    docs[i].Meta,
            }
            p.documents = append(p.documents, doc)

            // 3. BM25 索引构建
            p.bm25.Add(doc.ID, chunk)
        }
    }
    return nil
}

// Query 查询流程
func (p *RAGPipeline) Query(ctx context.Context, query string) (*QueryResult, error) {
    // 1. Embed Query
    qVec, err := p.embed(ctx, query)
    if err != nil {
        return nil, err
    }

    // 2. 向量检索
    vecResults := p.vectorSearch(qVec, p.cfg.TopK*2)

    // 3. BM25 检索
    bm25Results := p.bm25.Search(query, p.cfg.TopK*2)

    // 4. RRF 混合融合
    merged := p.rrfMerge(vecResults, bm25Results)

    // 5. Reranking（使用 Cross-Encoder 精排）
    reranked := p.rerank(ctx, query, merged, p.cfg.RerankTop)

    // 6. 构建上下文
    context := p.buildContext(reranked)

    // 7. LLM 生成
    answer, err := p.generate(ctx, query, context)
    if err != nil {
        return nil, err
    }

    return &QueryResult{
        Query:     query,
        Documents: reranked,
        Context:   context,
        Answer:    answer,
    }, nil
}

// vectorSearch 余弦相似度检索
func (p *RAGPipeline) vectorSearch(qVec []float64, k int) []Document {
    type scoredDoc struct {
        doc   Document
        score float64
    }
    scored := make([]scoredDoc, 0, len(p.documents))

    for _, doc := range p.documents {
        score := cosineSimilarity(qVec, doc.Vector)
        scored = append(scored, scoredDoc{doc, score})
    }

    sort.Slice(scored, func(i, j int) bool {
        return scored[i].score > scored[j].score
    })

    if k > len(scored) {
        k = len(scored)
    }
    result := make([]Document, k)
    for i := 0; i < k; i++ {
        result[i] = scored[i].doc
    }
    return result
}

// cosineSimilarity 余弦相似度
func cosineSimilarity(a, b []float64) float64 {
    var dot, normA, normB float64
    for i := range a {
        dot += a[i] * b[i]
        normA += a[i] * a[i]
        normB += b[i] * b[i]
    }
    if normA == 0 || normB == 0 {
        return 0
    }
    return dot / (sqrt(normA) * sqrt(normB))
}

// buildContext 构建 LLM 上下文
func (p *RAGPipeline) buildContext(docs []Document) string {
    var sb strings.Builder
    sb.WriteString("请根据以下参考资料回答问题：\n\n")
    for i, doc := range docs {
        sb.WriteString(fmt.Sprintf("[%d] %s\n", i+1, doc.Content))
    }
    return sb.String()
}

// generate 调用 LLM 生成回答
func (p *RAGPipeline) generate(ctx context.Context, query, context string) (string, error) {
    prompt := fmt.Sprintf(`%s
用户问题：%s
请基于参考资料回答，如果资料中没有相关信息，请说明无法回答。`, context, query)
    // 这里调用 LLM API（伪代码）
    return p.callLLM(ctx, prompt)
}

// 简化函数：sqrt
func sqrt(x float64) float64 {
    if x == 0 { return 0 }
    z := x / 2
    for i := 0; i < 100; i++ {
        z -= (z*z - x) / (2 * z)
    }
    return z
}

10. QA & 总结

常见问题

Q1: Chunk Size 应该如何选择？

没有唯一答案。经验值：256-512 tokens 是通用推荐。需要实验确定：在验证集上调整 Chunk Size，监控 RAGAS 的 Context Precision 指标。小 Chunk（128）适合 Q&A，大 Chunk（1024）适合摘要和长文分析。

Q2: Embedding 模型需要定期更新吗？

是的。Embedding 模型本身不更新，但文档库需要定期重新 Embedding（re-index），尤其是当新文档引入新领域词汇时。建议每周或每月增量更新索引。

Q3: 混合检索中 RRF 和加权融合哪个更好？

RRF 更鲁棒——因为不需要调权重 α，对分数尺度不敏感。加权融合理论上更灵活（通过 α 调节语义 vs 关键词的偏好），但需要实验确定 α。生产环境推荐 RRF 作为基线，再尝试加权调优。

Q4: Reranking 一定要用吗？成本会不会太高？

如果 Top-K 结果的前 3 个已经很准，可以不强制加 Reranking。但当 K 较大（≥20）或精度要求高时，Reranking 是性价比很高的提升手段。可以用轻量级 Reranker（如 MiniLM）控制成本，单次推理仅需几毫秒。

Q5: RAGAS 评估需要人工标注吗？

RAGAS 的核心理念是使用 LLM 自动评估（LLM as Judge）。你只需要准备好 (Query, Ground Truth Answer, Context) 三元组。但为确保评估质量，建议人工抽查 10-20% 的样本做校正。Faithfulness 维度用 GPT-4 评估效果最好。

Q6: Go 生态下有没有成熟的向量数据库客户端？

主流向量数据库都提供 Go SDK：Milvus（milvus-sdk-go）、Qdrant（qdrant-go）、Weaviate（weaviate-client-go）。如果数据量小于 100 万，也可以用 hnsw-go 或 faiss-go 在内存中构建索引，省去外部依赖。

Q7: HyDE 生成的假设文档质量不高怎么办？

HyDE 对 LLM 的生成质量敏感。改进方法：1）使用更强大的 LLM（GPT-4 优于开源模型）；2）在 Prompt 中要求生成多视角假设文档（3-5 个不同角度），取它们的 Embedding 均值。3）将 HyDE 与 Query Rewriting 组合使用。
