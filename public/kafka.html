📨 Kafka

一句话：Kafka = 高性能的分布式消息引擎。核心三板斧：顺序写磁盘 + Page Cache + 零拷贝。

和传统 MQ（RabbitMQ）的本质区别：Kafka 是日志系统，不是"消息队列"。消息不删除、可回放——像 Git 的 commit log。

🏗️ 整体架构：一个分布式的 commit log

📝 类比：Kafka 像一个超大规模的公告栏系统——

① Producer（公告张贴者）：往公告栏贴消息。

② Broker（公告栏管理员）：管理公告栏，负责接收、存储、分发消息。

③ Consumer（公告阅读者）：来公告栏读消息。读完不会把公告撕掉，下一个人还能读。

④ ZooKeeper / KRaft（协调中心）：告诉张贴者和阅读者"哪个公告栏是可用的"。

Kafka 架构全景：Producer → Broker 集群（Partition + Replica）→ Consumer Group

📊 消息模型：Topic → Partition → Segment

📚 类比：Topic = 一本书。Partition = 书的章节（并行读写）。Segment = 每章拆成若干小节（物理文件，方便清理）。

Topic → Partition → Segment：三层嵌套的物理存储模型

Partition 数量怎么定？

太多：① 文件句柄爆炸（每个 Segment 约 3 个文件）；② Leader 选举时间长；③ 端到端延迟增加。

太少：① 并行度不够，Consumer 浪费；② 单 Partition 吞吐上限 ~ 几十 MB/s（受 Leader 单点限制）。

经验公式：max(目标吞吐量 / 单分区吞吐量, 消费者数) × 1.5（留 buffer）。单 Broker 总分区数建议 < 4000。

✍️ 生产者：怎么把消息发对、发快、发不丢

📮 类比：发快递——① 分区策略 = 按什么规则决定发哪个仓库（默认 key 的 hash）；② 批量发送 = 攒一批包裹一起发，省运费（batch.size + linger.ms）；③ 压缩 = 包裹塞紧再装箱（gzip/snappy/lz4/zstd）。

发送流程——攒批 + 异步

// Producer 关键参数
batch.size = 16384          // 一个 batch 的字节数上限（16KB 默认）
linger.ms = 0               // batch 未满时的等待时间。0=立即发送，>0=攒满或超时再发
buffer.memory = 33554432    // Producer 总缓冲区（32MB）。满了会 block max.block.ms
max.in.flight.requests.per.connection = 5  // 同时能发几个未确认的请求
compression.type = lz4      // 压缩算法：lz4 > snappy > gzip（速度快 → 压缩率高）

🛡️ acks 参数——三种可靠性级别

acks含义丢消息？延迟场景
0Producer 不等待确认，发完就走✅ 可能丢最低日志、监控（丢几条无所谓）
1Leader 写入成功即确认⚠️ Leader 宕机且 Follower 未同步 → 丢中等默认值，一般业务
all / -1所有 ISR 副本都写入成功才确认❌ 不丢（配合 min.insync.replicas ≥ 2）最高金融、订单等不能丢的场景

⚠️ acks=all 也不一定 100% 不丢

需要同时设置 min.insync.replicas ≥ 2。如果 ISR 里只剩 Leader 自己（其他副本挂了），acks=all 会退化成 acks=1。设置 min.insync.replicas=2 → ISR 不足 2 时 Producer 直接报错，宁可不可用也不丢消息。

💾 日志存储：顺序写为什么能百万 QPS？

📝 类比：Kafka 写消息 = 往笔记本后面追加一行。和随机写（翻到第 53 页改一个字、翻到第 207 页再改一个字）相比，顺序写是机械磁盘的物理优势——磁头不需要来回寻道。

Kafka 高性能的秘密：Page Cache + sendfile + 顺序写

📡 ISR：什么叫"跟上了"？

🏃 类比：马拉松比赛——Leader = 领跑者。ISR（In-Sync Replicas）= 跟得上领跑者的选手（落后不超过一圈）。跟不上的被踢出 ISR，追上来的加回来。min.insync.replicas = 最少要有几个人跟得上才继续跑。

ISR 的动态维护

参数含义默认
replica.lag.time.max.msFollower 多久没同步算"掉队"30s
min.insync.replicas最少同步副本数，低于此值 Producer 写入报错1
unclean.leader.election.enable是否允许 ISR 外的副本当 Leaderfalse（生产切勿改 true）

如果 unclean.leader.election.enable=true → ISR 全挂时选个落后的副本当 Leader → 必然丢消息。宁可不可用，也不丢数据。

👥 消费者组与再均衡

🍽️ 类比：消费者组 = 一桌人吃火锅。每人负责夹几种菜（Partition），有人离席（Consumer 挂了）→ 剩下的重新分菜（Rebalance）。

enable.auto.commit = false
// 处理完消息后手动 commitSync()（同步）或 commitAsync()（异步）

🎯 精确一次语义 Exactly-Once

🏧 类比：ATM 取钱——你按了取 100 元，机器出钞了但断网了。银行绝对不能重复扣你两次 100 元。Exactly-Once = 要么成功一次，要么失败，不会出现"成功了但你以为失败了又做一次"。

三种语义对比

语义含义Kafka 默认？
At most once最多一次——可能丢，但绝不重复acks=0 时
At least once至少一次——绝不丢，但可能重复默认（acks=1 + 重试）
Exactly once精确一次——不丢不重需要幂等 Producer + 事务

实现 Exactly-Once 的三板斧

① 幂等 Producer：enable.idempotence=true。Broker 为每个 Producer 分配 PID，Producer 给每条消息打上 。Broker 按 SeqNum 去重——SeqNum 比上次大 1 才写入。

② 事务 Producer：跨分区、跨 Topic 的原子写入。initTransactions() → beginTransaction() → send() → commitTransaction()。Consumer 设置 isolation.level=read_committed 只读已提交事务的消息。

③ 消费者端去重：手动维护已处理的消息 ID 集合（Redis/DB）。消费前查重 → 消费后记 ID。或者利用业务主键幂等（INSERT ... ON DUPLICATE KEY UPDATE）。

事务的代价

事务 Producer 比普通 Producer 吞吐下降约 30-50%。因为需要额外的网络往返（AddPartitionsToTxn、EndTxn 等）。不要一上来就事务——先评估是否真的需要 Exactly-Once，大部分场景 At-least-once + 业务幂等就够了。

❓ 高频考题

Q1: Kafka 为什么快？
三板斧：① 顺序写磁盘（机械磁盘顺序写 600MB/s vs 随机写 100KB/s）；② Page Cache（消费者读热数据时命中内存，不走磁盘）；③ sendfile() 零拷贝（数据从 Page Cache DMA 到网卡，不经过用户态）。
Q2: 怎么保证消息不丢？
三端配合：Producer → acks=all + retries>0；Broker → replication.factor≥3 + min.insync.replicas≥2 + unclean.leader.election.enable=false；Consumer → 手动提交 offset（处理完再提交，不是读完就提交）。
Q3: 怎么保证消息不重复？
Producer：enable.idempotence=true（单分区单会话幂等）+ 事务（跨分区幂等）。Consumer：业务幂等（唯一键去重 INSERT ON DUPLICATE KEY / Redis SETNX）+ 手动维护消费记录。
Q4: 消息积压了怎么办？
① 临时增加 Consumer 实例（但不能超过分区数）；② 增加分区数（注意 key 的 hash 会变）；③ 降级——不重要的消息先跳过或丢弃；④ 提升单 Consumer 性能（批量拉取、异步处理）。
Q5: 怎么保证消息有序？
Kafka 只保证分区内有序，不保证跨分区有序。方案：① 相同 key 的消息发到同一分区 → 局部有序；② 如果必须全局有序 → 只用 1 个 Partition（牺牲并行度）。
Q6: Rebalance 有什么危害？怎么优化？
Rebalance 期间消费者组 Stop-The-World——所有 Consumer 暂停消费，重新分配分区。大状态消费者（如本地缓存了很多数据）恢复很慢。优化：① 使用 Cooperative Sticky 策略（增量分配，不动现有分配）；② 调大 session.timeout.ms 和 heartbeat.interval.ms 减少误判；③ 减少分区数变动。

💬 苏格拉底式提问 —— 检验真理解

以下问题用来逼你追问本质。先自己想 30 秒，再点开验证。

Kafka 为什么不像 RabbitMQ 那样"消费完就删消息"？Q1
因为 Kafka 本质上是一个日志系统，不是消息队列。
RabbitMQ 是"消息分发器"——消息被消费后删除，队列变空。
Kafka 是"commit log"——消息写入后永久保留（按时间或大小清理），多个 Consumer 可以独立消费同一条消息，还可以回溯历史。
这个设计差异决定了一切：RabbitMQ 适合"一条消息一个消费者"的 RPC 模式；Kafka 适合"一条消息多个消费者 + 事件溯源 + 流处理"的模式。
acks=all 真的 100% 不丢消息吗？Q2
不是。acks=all 只能保证"写入时所有 ISR 副本确认"，但两个场景仍然可能丢：
① ISR 里只剩 Leader 自己（其他副本挂了）→ acks=all 退化成 acks=1 → Leader 宕机 → 丢。需要 min.insync.replicas≥2 来兜底。
② Leader 宕机且 unclean.leader.election.enable=true → 选了一个落后很多的副本当 Leader → 新 Leader 没有旧 Leader 上的最新消息 → 丢。必须设 false。
结论：acks=all + min.insync.replicas≥2 + unclean=false 三者缺一不可。
为什么分区数不能随便改大？Q3
Kafka 不支持减少分区数（只能增不能减）。原因是底层 Segment 文件的 offset 是连续递增的，减分区意味着要把部分 offset 区间删掉——这不兼容 Kafka 的"日志不可变"模型。
但增加分区也有代价：① 有 key 的消息 hash 会变 → 同一 key 可能落到不同分区 → 局部有序被破坏；② 分区数增加 → 文件句柄增加 → 选举时间变长；③ 已消费的 offset 在新分区上无效（新分区从 0 开始）→ 消费者需要处理。

🎙️ 口播总结

这篇文章把 Kafka 从入门到讲透了。

Kafka 的本质是分布式日志系统——顺序写磁盘 + Page Cache + 零拷贝三大性能支柱让它做到百万 QPS。架构上记住 Topic→Partition→Segment 三层模型。可靠性由 acks、ISR、min.insync.replicas 三者配合。消费端关注 Consumer Group 的 Rebalance 和 Offset 管理。Exactly-Once 需要幂等 Producer + 事务，但不是所有场景都需要。Kafka 不是万能的——它是"日志"，不是"队列"。选型时看你的场景是"消息分发"还是"事件流"。
