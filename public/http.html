📡 HTTP 协议

一句话：HTTP 是 Web 的"普通话"——浏览器和服务器之间的通信语言。它无状态、基于请求-响应模型，从 1991 年的 0.9 版本进化到今天的 HTTP/3，每一步都在解决"更快、更安全、更高效"的问题。

这篇文章不讲 TCP 握手（看 计算机网络），不讲 TLS 细节（同看网络篇），只聚焦 HTTP 协议本身——实战中 80% 的 HTTP 考点都在这里。

🔬 请求 & 响应解剖：HTTP 报文的"骨架"

✉️ 类比：HTTP 报文 = 快递包裹。起始行 = 快递单（发件人/收件人/物品），Header = 包裹标签（易碎/冷链/加急），空行 = 标签和物品之间的分隔线，Body = 包裹里的东西。

请求报文结构

┌─ 起始行（Request Line）─────────────────────┐
POST /api/users HTTP/1.1
┌─ 请求头（Headers）──────────────────────────┐
Host: api.example.com
Content-Type: application/json
Authorization: Bearer eyJhbGciOi...
Accept: application/json
Accept-Encoding: gzip, br
User-Agent: Mozilla/5.0
┌─ 空行（CRLF）── 头部和Body的隔离符 ─────────┐

┌─ 请求体（Body）── 可选，GET 请求通常没有 ───┐
{"username": "zhangsan", "email": "zhang@example.com"}

响应报文结构

┌─ 状态行（Status Line）──────────────────────┐
HTTP/1.1 200 OK
┌─ 响应头（Headers）──────────────────────────┐
Content-Type: application/json; charset=utf-8
Content-Length: 156
Cache-Control: max-age=3600
Set-Cookie: session_id=abc123; HttpOnly; Secure
┌─ 空行（CRLF）───────────────────────────────┐

┌─ 响应体（Body）────────────────────────────┐
{"id": 42, "username": "zhangsan", "created_at": "2025-01-01"}

🔑 核心约束

① 起始行和每个 Header 以 CRLF（\r\n）结尾。② 头部结束后必须有一个空行（CRLF+CRLF）。③ Content-Length 告诉接收方 Body 有多少字节——没有它，接收方不知道读到哪算完。 HTTP/1.1 的 Transfer-Encoding: chunked 是这个问题的另一种解法：分块传输，每块前标注长度，最后一块长度为 0。

📬 HTTP 方法：不是每个方法都只是"读"和"写"

🏦 类比：GET = 查看存折余额（不改变状态）。POST = 填写转账单提交（创建新资源）。PUT = 拿着房产证去登记处"整个替换"信息。PATCH = 去登记处说"只改电话号码这一项"。DELETE = 注销账户。

8 种方法全解

方法语义特点核心考点
GET获取资源幂等、安全（不改变服务器状态）、可缓存、参数在 URL GET 请求可以带 Body 吗？RFC 不禁止，但绝大多数服务器忽略。
POST创建资源 / 提交数据不幂等、不安全、不可缓存（除非显式设置）、参数在 Body POST 和 PUT 的区别？POST 非幂等（多次=多个资源），PUT 幂等。
PUT完整替换资源幂等（多次请求结果一致）、不安全 PUT 要求提供完整资源表示，缺少字段会设为默认值/null。
PATCH部分更新资源不保证幂等、不安全 PATCH vs PUT？PATCH 是增量修改，PUT 是全量替换。
DELETE删除资源幂等（删两次结果一样——资源不存在）、不安全 第一次返回 204，第二次返回 404——这叫幂等。
HEAD只获取响应头和 GET 一样但不返回 Body。用于检查资源是否存在、获取 Content-Length 文件下载前先用 HEAD 拿文件大小，决定是否要断点续传。
OPTIONS查询支持的方法CORS 预检请求的核心。浏览器在跨域 POST 前自动发 OPTIONS。 响应头 Allow: GET, POST, PUT 告诉客户端支持哪些方法。
TRACE回显请求（调试）用于查看中间代理对请求做了什么修改。生产环境通常禁用（安全风险）。 可能泄露认证头 → XST 攻击 → 生产禁用。

幂等性 —— 最爱考的概念

幂等 = 执行 1 次和执行 N 次，结果一样

幂等方法：GET、HEAD、PUT、DELETE（因为删两次结果都是"不存在"）

非幂等方法：POST（创建两次=两个资源）、PATCH（取决于具体实现）

为什么重要？网络不可靠——超时重试时，幂等方法可以安全重试，POST 可能造成重复创建。这就是为什么支付接口通常要求客户端生成唯一 Idempotency-Key。

301 vs 302 —— 高频必问

301 永久重定向

浏览器会缓存这个重定向。下次访问旧 URL，浏览器直接用缓存的地址跳转，不发请求。

POST → GET 可能被改（浏览器行为）

SEO：权重从旧 URL 转移到新 URL

302 临时重定向

浏览器不缓存。每次访问旧 URL 都会请求服务器。

POST → GET 可能被改（浏览器行为）

SEO：权重保留在旧 URL

💡 307/308 是来解决"方法被改"问题的：301/302 在历史上被浏览器错误实现（POST 变成 GET），307 = 临时 + 不改变方法，308 = 永久 + 不改变方法。

📦 内容协商 & 编码：客户端和服务器"商量着来"

🗣️ 类比：内容协商 = 你去外国餐厅点菜。你说"我要中文菜单"，服务员说"有中日英三种"（Accept-Language 协商）。你说"给我压缩版少占地方"，服务员说"gzip 和 brotli 都行"（Accept-Encoding 协商）。

服务端驱动协商（最常用）

客户端在请求头里声明自己能接受什么，服务器选一个最合适的返回。三个关键头：

  • Accept：text/html, application/json;q=0.9, */*;q=0.8——q 值代表优先级（0-1）
  • Accept-Encoding：gzip, deflate, br——优先 Brotli（压缩率更高）
  • Accept-Language：zh-CN, zh;q=0.9, en;q=0.5

服务器响应时在 Content-Type / Content-Encoding / Content-Language / Vary 中声明它实际选了什么。Vary 头是关键——它告诉 CDN/缓存"我这个响应会因为 XX 头不同而不同"，CDN 会根据 Vary 指定的头做缓存分片。

🍪 Cookie · Session · Token：HTTP 的"记忆"机制

🎫 类比：HTTP 是无状态的——服务器像金鱼，转身就忘。Cookie = 游乐园的纸质手环（每次进门出示）。Session = 手环上只有编号，信息存在游乐园的电脑里。JWT Token = 电子门票（自带加密信息，不用查电脑）。

Cookie 的属性 —— 每一项都是安全考量

属性作用不设置的风险
HttpOnly禁止 JavaScript 读取（document.cookie 拿不到）XSS 攻击可偷走 Token
Secure只在 HTTPS 连接中传输中间人攻击（MITM）可截获 Cookie
SameSite控制跨站请求是否带 Cookie：Strict/Lax/NoneCSRF 攻击（SameSite=Lax 基本防住）
DomainCookie 的作用域名（包括子域）不设则只对当前域有效
PathCookie 的作用路径不设则对所有路径有效
Expires / Max-Age过期时间不设 = 浏览器关闭即失效（会话 Cookie）

Cookie vs Session vs Token vs JWT

四种状态管理方案对比：没有银弹，只有场景适配

🎯 选型决策

服务端渲染的传统 Web → Cookie + Session。 SPA + REST API → Bearer Token（存 localStorage/memory）。微服务 + 无状态要求 → JWT（但要接受无法主动撤销的代价，用短期 Access Token + Refresh Token 缓解）。

🌐 CORS 跨域：浏览器的"门禁系统"

🏢 类比：CORS = 办公楼门禁。同一栋楼的员工（同源）随便走动。外楼的人（跨域）来访需要前台（OPTIONS 预检）确认"这人能进吗？"。简单的事情（穿制服的外卖员 = 简单请求）不用预检直接放行。

什么是同源？

协议 + 域名 + 端口，三者完全相同才算同源。https://a.com:443 和 http://a.com:80 ——不同源！

简单请求 vs 预检请求

类型条件流程
简单请求 方法为 GET/HEAD/POST，且 Content-Type 为 text/plain / multipart/form-data / application/x-www-form-urlencoded，且无自定义头 浏览器直接发请求，检查响应头里的 Access-Control-Allow-Origin
预检请求 任何不满足简单请求条件的：PUT/DELETE、application/json、自定义头（如 Authorization） 浏览器先发 OPTIONS 请求（预检），服务器回复允许的方法和头，然后才发实际请求

关键响应头

Header示例含义
Access-Control-Allow-Originhttps://example.com 或 *允许的源（带凭据时不能是 *）
Access-Control-Allow-MethodsGET, POST, PUT, DELETE允许的方法
Access-Control-Allow-HeadersContent-Type, Authorization允许的请求头
Access-Control-Allow-Credentialstrue允许携带 Cookie/Authorization
Access-Control-Max-Age86400（秒）预检结果缓存时间——避免每次请求都 OPTIONS

⚠️ 常见踩坑

① 带 Authorization 头的请求一定触发预检。② 设置了 Credentials: true 时，Allow-Origin 不能是 *，必须是具体域名。③ 预检请求不携带认证信息。④ CDN/反向代理的 CORS 配置通常是故障重灾区。

⚡ HTTP 缓存体系：最快的请求是不发请求

📚 类比：强缓存 = 图书馆说"这本书至少一个月不会更新，你直接看复印本就行"——压根不问图书馆。协商缓存 = "你每次看之前问一下我这本书有没有新版，没有就看复印本"——发请求但不传 Body。

强缓存（不发请求）

Header示例行为
Cache-Control: max-agemax-age=36001 小时内直接用缓存，不发请求。优先级最高。
Cache-Control: no-cache—可以用缓存，但每次必须验证（走协商缓存）
Cache-Control: no-store—完全不缓存。银行交易/敏感数据用这个。
Cache-Control: public—允许 CDN/代理缓存
Cache-Control: private—只允许浏览器缓存（CDN 不缓存）
ExpiresThu, 01 Dec 2025 16:00:00 GMTHTTP/1.0 的绝对时间，被 max-age 覆盖

协商缓存（发请求但不传 Body）

机制请求头响应头行为
ETag/If-None-Match If-None-Match: "abc123" ETag: "abc123" 服务器比较 ETag，相同返回 304，不同返回 200+新内容+新ETag
Last-Modified/If-Modified-Since If-Modified-Since: Wed, 21 Oct 2025 Last-Modified: Wed, 21 Oct 2025 比 ETag 精度低（秒级），且文件内容没变但修改时间变了也会失效

🎯 缓存决策树（参考解析）

① Cache-Control 检查 → 没过期？直接用（200 from disk cache）

② 过期了但有 ETag？→ 发 If-None-Match → 304 用缓存 / 200 更新

③ 没有 ETag 但有 Last-Modified？→ 发 If-Modified-Since → 同上

④ 什么都没有 → 重新请求全部内容

实战：静态资源的缓存策略

# 带哈希的静态资源：永远缓存
/index-a1b2c3.js  → Cache-Control: max-age=31536000, immutable

# HTML 入口文件：每次都验证
/index.html       → Cache-Control: no-cache  # 走协商缓存

# 用户数据接口：不缓存
/api/user         → Cache-Control: no-store

🔗 连接管理：从短连接到多路复用

🚕 类比：短连接 = 每次出门打一辆出租车，到了就结账走人（一次 TCP 握手+挥手）。长连接 = 包车一整天，司机会等（Keep-Alive）。HTTP/2 多路复用 = 大巴车，所有人都坐一辆车同时出发（一个 TCP 连接承载所有请求）。

模式HTTP 版本行为缺点
短连接HTTP/0.9, 1.0 每请求 → TCP握手 → 响应 → TCP挥手 大量握手开销，慢启动拖慢每个请求
持久连接HTTP/1.1（默认） TCP 连接复用，请求串行：req1→res1→req2→res2 队头阻塞（一个慢全等）
并发连接HTTP/1.1 技巧 同时开 6-8 个 TCP 连接（域名分片） 连接数爆炸，每个都要握手+慢启动
多路复用HTTP/2 1 个 TCP 连接承载多个并发 Stream TCP 层面仍有队头阻塞
QUICHTTP/3 基于 UDP，每个 Stream 独立，无队头阻塞 CPU 开销较高，部分网络 QoS 限制 UDP

🚀 HTTP 进化史：三十年三次革命

HTTP 三十年进化：从"纯文本传 HTML"到"UDP 多路复用"

⚡ HTTP/2 详解：二进制分帧 + 多路复用

📦 类比：HTTP/1.1 = 一个个独立寄快递，每个快递等上一个签收才能寄下一个。HTTP/2 = 把所有东西装进一个集装箱，里面分小格子（Frame），每个格子上贴标签（Stream ID），到了目的地按标签分别取出来。

四个核心特性

特性原理解决什么问题
二进制分帧 HTTP/2 不再用文本协议，而是将请求/响应拆成 HEADERS frame 和 DATA frame，二进制编码 解析更快，错误更少，不再需要复杂的文本解析器
多路复用 一个 TCP 连接上同时跑多个 Stream（每个 Stream 有唯一 ID），Frame 可以交错传输 解决 HTTP/1.1 的队头阻塞（HTTP 层面）
头部压缩 (HPACK) 客户端和服务端各维护一张"字典表"，重复的 Header（如 Cookie、User-Agent）只发索引 减少 85%+ 的 Header 传输量（HTTP/1.1 每次都发完整 Header）
Server Push 服务器可以主动推送资源（如 HTML 里引用的 CSS），无需客户端再请求 减少一次往返（但实际使用中容易推送已缓存的资源，Chrome 已移除支持）

HTTP/2 的遗留问题：TCP 队头阻塞

🚛 类比：HTTP/2 的多路复用像把所有货物装一辆卡车。但卡车本身如果在高速公路上爆胎了（TCP 丢包），车上的所有货物（所有 Stream）都得等轮胎换好——这就是 TCP 层面的队头阻塞。

因为 HTTP/2 只用一个 TCP 连接承载所有 Stream，TCP 的可靠传输特性意味着：丢一个包 → TCP 重传 → 所有 Stream 阻塞等待。这在弱网环境下比 HTTP/1.1（6 个连接分担风险）更差。HTTP/3 正是为了解决这个问题而生的。

🌩️ HTTP/3 & QUIC：换掉 TCP 这条"地基"

✈️ 类比：HTTP/1.1→2 = 把马路修得更宽（优化 TCP 上的协议）。HTTP/3 = 不用马路了，直接开飞机（基于 UDP 的 QUIC）。每个 Stream 就像独立航班，一架延误不影响其他航班。

QUIC 的三大杀手特性

特性原理对比 TCP
0-RTT 握手 曾经连过的服务器，客户端直接用缓存的密钥加密数据发送，跳过握手 TLS 1.3 需要 1-RTT，传统 TLS 1.2 需要 2-RTT
无队头阻塞 QUIC 在 UDP 上实现可靠传输，每个 Stream 独立——Stream1 丢包只阻塞 Stream1 TCP 丢包阻塞整个连接的所有 Stream
连接迁移 连接标识是 Connection ID 而不是 IP+端口。WiFi→4G 切换时，连接不中断 TCP 用四元组标识连接，IP 一变连接就断

QUIC 握手时序

// 首次连接（1-RTT）
Client → Server:  ClientHello (QUIC 密钥交换)
Server → Client:  ServerHello + 加密的 Finished
// 之后的数据都在加密通道中传输

// 恢复连接（0-RTT）
Client → Server:  0-RTT ClientHello + 加密的应用数据 🔥
// 服务器直接处理应用数据，不需要等握手完成！

⚠️ HTTP/3 的现状

① 主流浏览器和 CDN 已广泛支持。② 企业网络/防火墙可能限制 UDP → 需要 fallback 到 HTTP/2。③ QUIC 在用户态实现，CPU 开销比内核态 TCP 高（但随着优化差距在缩小）。④ gRPC 已支持 HTTP/3 transport。

🏗️ RESTful 设计：把 HTTP 用对

📐 类比：REST 不是协议，是设计风格——就像"装修风格"，你可以装成北欧风、日式风。RESTful = 以资源为中心，用 HTTP 方法的语义（GET/POST/PUT/DELETE）操作资源，用状态码和 Header 传递元信息。

RESTful 6 条核心约束

  1. 客户端-服务器分离：前端和后端独立演进。客户端不关心数据怎么存的，服务器不关心 UI。
  2. 无状态：每个请求包含所有必要信息。服务器不保存客户端会话状态——这是 REST 最核心的约束。
  3. 可缓存：响应必须标记是否可缓存（Cache-Control）。
  4. 统一接口：用 URL 标识资源，用 HTTP 方法操作资源，用状态码表示结果。
  5. 分层系统：客户端不知道自己是和真实服务器还是代理/CDN 通信。
  6. 按需代码（可选）：服务器可以发送可执行代码（如 JavaScript）给客户端。

URL 设计原则

// ✅ 好的设计：资源用名词复数，嵌套表示层级
GET    /api/users              # 用户列表
POST   /api/users              # 创建用户 → 返回 201 + Location: /api/users/42
GET    /api/users/42          # 单个用户
PUT    /api/users/42          # 完整替换
PATCH  /api/users/42          # 部分更新
DELETE /api/users/42          # 删除
GET    /api/users/42/orders  # 用户的订单（嵌套资源）

// ❌ 坏的设计：URL 里带动词
POST /api/createUser        # 动词应该用 HTTP 方法表达
GET  /api/getUserById?id=42 # 资源标识应该用路径而不是查询参数

REST 成熟度模型（Richardson Maturity Model）

级别描述例子
Level 0一个 URI，一个方法（通常是 POST），类似 RPCPOST /api，Body 里写操作类型
Level 1多个 URI，每个资源有独立端点/users, /orders
Level 2用 HTTP 方法 + 状态码（大多数 API 的目标）GET/POST/PUT/DELETE + 200/201/404
Level 3HATEOAS：响应中包含下一步操作的链接返回 {"links": {"next": "/users?page=2"}}

❓ 高频考题

Q1: 浏览器输入 URL 到页面展示的完整过程
① URL 解析 → ② DNS 查询（缓存→hosts→递归查询）→ ③ TCP 三次握手 → ④ TLS 握手（HTTPS）→ ⑤ 发送 HTTP 请求 → ⑥ 服务器处理 → ⑦ 返回 HTTP 响应 → ⑧ 浏览器解析 HTML → ⑨ 构建 DOM 树 + CSSOM 树 → ⑩ 布局 + 绘制 → ⑪ 执行 JS → ⑫ 加载子资源（CSS/JS/图片）按第⑤步循环。这是最经典的"一条线"题，考察从网络到渲染的全链路理解。
Q2: POST 和 PUT 的真正区别
核心区别是幂等性。PUT 多次请求结果相同（完整的资源替换），POST 多次请求创建多个资源。
PUT 需要客户端提供完整资源表示（缺少字段会被重置），PATCH 只需要提供要修改的字段。
PUT 的 URI 是具体的（/users/42），POST 的 URI 通常是集合（/users）。
Q3: 301 vs 302 vs 307 vs 308 的精确区别
301 永久 + POST→GET 可能被改。
302 临时 + POST→GET 可能被改（浏览器历史实现问题）。
307 临时严格不改变方法。
308 永久严格不改变方法。
用 307/308 代替 302/301 当需要保留 POST 请求时。但实践中大多数场景 302 就够用了。
Q4: HTTP/2 多路复用为什么还有队头阻塞？
HTTP/2 解决的是HTTP 层面的队头阻塞（一个请求不阻塞另一个）。但 HTTP/2 底层只用一个 TCP 连接，TCP 的可靠传输机制要求数据按序到达——如果某个 TCP 包丢了，所有 Stream都要等这个包重传。这在弱网下比 HTTP/1.1 的 6 个连接表现更差。HTTP/3 用 QUIC（UDP）彻底解决：每个 Stream 独立的可靠传输，一个丢包只影响它自己。
Q5: Cookie 怎么防 XSS 和 CSRF？
防 XSS：HttpOnly（禁止 JS 读取）+ Secure（仅 HTTPS 传输）。
防 CSRF：SameSite=Lax/Strict（跨站请求不带 Cookie）+ 验证 Referer/Origin + CSRF Token（表单中的隐藏字段） + 关键操作二次确认。
SameSite 三种模式：Strict（最严，任何跨站都不带）→ Lax（允许 GET 导航带，表单 POST 不带）→ None（必须配合 Secure）。

💬 苏格拉底式提问 —— 检验真理解

为什么 HTTP 设计成无状态的？这是一种设计缺陷还是有意为之？Q1
是有意为之的架构选择，不是缺陷。无状态意味着每个请求都是独立的，服务器不需要记住"上次这个客户端干了什么"。

好处：① 扩展性——任何服务器实例都能处理任何请求，负载均衡不需要"粘性会话"；② 可靠性——服务器挂了不丢失会话状态；③ 简单性——协议本身极简。

代价：无法直接表示"登录状态""购物车"等需要跨请求保持的信息——所以有了 Cookie/Session/Token 这些"补丁"。

对比：IM 协议（如 XMPP）就是有状态的——长连接+心跳，设计目标完全不同。HTTP 的无状态是 Web 能扩展到今天的核心原因之一。
如果让你重新设计 HTTP，你会保留什么？改掉什么？Q2
从 HTTP 的进化史就能看出设计者对这个问题的回答：
保留的：请求-响应模型、资源标识（URI）、MIME 类型、状态码体系、无状态核心。
改掉的：文本协议→二进制（HTTP/2）、单一请求→多路复用（HTTP/2）、TCP 队头阻塞→QUIC（HTTP/3）、明文传输→TLS 加密（HTTPS 成为事实标准）。

HTTP 的成功恰恰在于它的向后兼容——HTTP/1.1 的服务器仍然能和 HTTP/2 客户端通信（通过 ALPN 协商降级），URL 格式 30 年没变。这是协议设计的最高境界：演进而不断裂。
缓存是 HTTP 性能优化的核心。Cache-Control、ETag、CDN 三者是什么关系？Q3
这是一个从"单机"到"全网"的缓存层级：

① Cache-Control（强缓存）——浏览器本地缓存，不发请求，最快但最"笨"（不看内容是否真的变了）。
② ETag/Last-Modified（协商缓存）——浏览器有缓存但需要问服务器"还能用吗？"，精确但多一次网络往返。
③ CDN——把缓存推到离用户最近的边缘节点，减少物理距离带来的延迟。CDN 依赖 Cache-Control 的 public/private 和 max-age 来决定缓存策略。

实战中是三者配合：HTML 用协商缓存保证实时性，带哈希的 CSS/JS 用强缓存永不过期，CDN 做一层中间缓存。考官想听的是你对缓存层级的理解，不是孤立的知识点。
HTTP/3 把地基从 TCP 换成了 QUIC(UDP)，这是一个激进的决定。你觉得值得吗？Q4
值得。但代价不小。

为什么必须换：TCP 在操作系统内核实现，升级 TCP 协议栈需要所有操作系统配合——这几乎不可能。QUIC 在用户态实现，可以随应用快速迭代。

得了什么：① 彻底解决队头阻塞；② 0-RTT 握手减少延迟（尤其对移动端弱网）；③ 连接迁移（WiFi↔4G 不断线）。

代价：① UDP 被某些网络 QoS 限制或阻止；② 用户态实现 CPU 开销更高；③ 生态迁移成本（CDN、LB、防火墙都要支持）。

Google 的数据显示，YouTube 的重新缓冲率下降了 18%，搜索延迟下降了 8%——在亿级规模下，这个决定已经被验证是正确的。
