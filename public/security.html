📋 总览：安全攻防实战全景

安全攻防是软件开发中最硬核也最容易被忽视的环节。本文从认证授权出发，深度剖析 OAuth2/OIDC、JWT、HTTPS/TLS 的核心原理，再深入到Web 安全攻防——CSRF、XSS、SQL 注入、SSRF——最后给出实战最佳实践。

想象你经营一栋写字楼：认证（Authentication）就是门口的保安检查工牌——"你是谁？"；授权（Authorization）是楼层门禁——"你能进哪几层？"

JWT 就像一张带有防伪芯片的访客卡，里面写着你叫什么、能去哪、什么时候过期。HTTPS/TLS 是包裹信件在传输途中的加密信封——偷看了也没用，撕开就发现被篡改。

而 CSRF、XSS、SQL 注入这些攻击，就像有人从窗户爬进来、冒充你发邮件、或者假装成清洁工进入数据中心。我们要做的是给每扇窗装上报警器，并在进门时多重核验身份。

多层纵深防御体系 —— 每层失效时，下层还能兜底

🔐 认证 vs 授权

这两个概念是最容易被混淆的安全基础，但它们的职责截然不同：

🔐 认证 (Authentication)

问：你是谁？

验证用户身份的过程

  • 登录输入用户名+密码
  • 指纹/人脸识别
  • OAuth 登录（用 Google 登录）
  • 单点登录 (SSO)

结果：确认身份后颁发凭证

✅ 授权 (Authorization)

问：你能做什么？

确定已认证用户的权限范围

  • 普通用户可读，管理员可写
  • API 访问范围 (scope)
  • RBAC 角色权限控制
  • ACL 访问控制列表

结果：决定允许/拒绝访问资源

想象你去一家高级酒店：

认证 = 你站在前台出示身份证，前台确认你是房客张三（"确认你是谁"）。

授权 = 前台给你房卡，这张卡只能刷开 808 房间门，刷不开总统套房和机房（"确认你能去哪"）。

关键区分：认证在"门前"，授权在"门内"。认证前置于授权——必须先知道你是谁，才能判断你能做什么。

常见认证协议对比

协议场景方式特点
HTTP Basic Auth老旧系统Base64(username:password)极简单，需搭配 HTTPS
Session-Cookie传统 Web服务端存 session ID有状态，服务端内存开销
JWT (Bearer Token)现代/分布式自包含 Token无状态，适合微服务
OAuth2第三方授权四种授权码流程委托授权，不暴露密码
OIDC身份认证OAuth2 + ID TokenOAuth2 的认证扩展层

认证和授权是安全的"门"与"墙"——门决定了谁能进，墙决定了能走到哪。很多安全漏洞（比如越权访问）本质上就是授权没做好——门开了，墙却形同虚设。最经典的教训：永远不要仅靠认证来做授权。

🔄 OAuth2 四种授权码流程 + OIDC

OAuth2 是一个授权框架，解决的是"你允许第三方应用访问你在某服务上的资源，但不必把你的密码给它"的问题。OAuth2 定义了四种授权码流程（Grant Types）。

想象你住的酒店有酒廊服务。你要朋友去酒廊帮你取一杯酒：

方式① 授权码模式（最安全）：你给前台打电话授权，前台给朋友一张代金券，朋友用房卡+代金券去酒廊取酒。代金券只能用一次。

方式② 隐式模式（已废弃）：你直接让朋友用房卡去。不安全——房卡就是你的身份。

方式③ 密码模式：你把房卡密码告诉朋友。完全不推荐！

方式④ 客户端凭证模式：内部服务员（API to API）不需要客人授权，直接用自己的工牌取酒。

四种授权码模式速览

模式适用场景安全等级是否需要 client_secret备注
授权码 (Authorization Code) 服务端渲染 Web 应用 ⭐⭐⭐⭐⭐ 是 最推荐，配合 PKCE 更安全
隐式 (Implicit) 废弃 SPA (已不推荐) ⭐⭐ 否 RFC 9700 宣布废弃，用 PKCE 代替
密码 (Resource Owner Password) 信任的一方应用 ⭐ 是 不推荐，暴露密码
客户端凭证 (Client Credentials) M2M (微服务间) ⭐⭐⭐⭐ 是 无用户参与，纯后端

📜 JWT 结构 / 签名 / 刷新

JWT（JSON Web Token）是目前最流行的无状态认证令牌格式。它本身不是安全协议，而是一种令牌格式——使用不当会有严重安全隐患。

JWT 就像一张防伪信用卡：

Header = 信用卡的材质和制式——银联/VISA、芯片类型。

Payload = 卡面信息——卡号、持卡人姓名、有效期。注意：这些信息是Base64 编码而非加密，任何人解码就能看到！

Signature = 卡背面的磁条 + PIN 码的加密校验——银行用密钥验证卡片没有被伪造或篡改。

所以：🟡 敏感数据不要放在 Payload！🔵 永远验证签名！🔴 永远检查过期时间！

JWT 签名算法选择

算法对称/非对称优势劣势推荐
HS256对称性能好密钥分发困难，双方都得知道密钥内部使用
RS256非对称 (RSA)公钥分发，只有签发者知道私钥密钥对较大★ 推荐
ES256非对称 (ECDSA)密钥小，性能好实现复杂度稍高★ 推荐
alg: "none"-无完全无安全保护，攻击者可随意伪造绝对禁用！

🔒 HTTPS / TLS 握手

HTTPS = HTTP + TLS（传输层安全协议）。TLS 的核心目标：加密（防窃听）、完整性（防篡改）、身份验证（防冒充）。

想象你在教室传纸条给同桌——但纸条可能被老师拦截查看（窃听）、被同学改写内容（篡改）、甚至有人冒充你同桌传回假消息（冒充）：

你（客户端）和同桌（服务器）先商量一种加密暗号——这就是 TLS 握手。

① Client Hello：你亮出你会的暗号方法（TLS 版本、密码套件）。

② Server Hello + 证书：同桌选择一种方法，并出示学生证（数字证书）证明自己确实是同桌本人。

③ 证书验证：你检查学生证上的学校公章（CA 签名）是否真实有效——就像浏览器检查 HTTPS 证书。

④ 密钥交换：你们用暗号约定一个只有你俩知道的秘密词（会话密钥），之后所有纸条都用这个秘密词加密。

从此老师只能看到一堆乱码 🎭

证书链验证流程

步骤：

1. 服务端发送证书链（叶子证书 → 中间 CA 证书 → 根 CA 证书）

2. 客户端（浏览器）检查证书链：每个上级签发下级，逐级验证签名

3. 根证书必须在浏览器的信任存储区中（预装的操作系统/Mozilla 根证书包）

4. 验证域名匹配：证书的 CN/SAN 必须等于访问的域名

5. 验证有效期、是否被吊销（CRL/OCSP）

⚠️ 任何一步失败，浏览器显示"不安全"红色警告

前向安全性 (Perfect Forward Secrecy)

TLS 使用临时 Diffie-Hellman 密钥交换（ECDHE）保证 PFS：即使用户的长期私钥将来泄露，攻击者也无法解密之前记录的加密流量。因为每次连接使用独立的临时会话密钥，该密钥用完即销毁。

🛡️ Web 安全攻防：CSRF · XSS · SQL注入 · SSRF

这四个漏洞常年占据 OWASP Top 10，是每位开发者必须掌握的"安全基本功"。

1️⃣ CSRF（跨站请求伪造）

想象你正在银行官网登录着。突然有人给你发了一封带图片的邮件，图片的 URL 其实是 bank.com/transfer?to=hacker&amount=10000。因为浏览器会自动携带银行网站的 Cookie，这个请求就像是你本人发起的！

这就是 CSRF —— 攻击者利用你已经登录的身份，在你不知情的情况下伪造请求。

⚔️ 攻击流程

  1. 用户登录 bank.com（获得 Cookie）
  2. 用户访问 malicious.com（攻击者站点）
  3. 恶意站点在页面中嵌入 <img src="bank.com/transfer?to=hacker">
  4. 浏览器自动带 Cookie 发送 GET 请求（或通过表单自动提交 POST）
  5. bank.com 看到有效 Cookie，执行转账！

🛡️ 防御措施

  • CSRF Token：表单中嵌入随机 token，服务器验证（最有效）
  • SameSite Cookie：设置 SameSite=Strict 或 SameSite=Lax
  • Referer/Origin 校验：检查请求来源
  • 关键操作二次验证：转账前短信/验证码确认
  • RESTful 规范：GET 请求不做写操作

2️⃣ XSS（跨站脚本攻击）

想象一个留言板。正常用户写"今天天气真好"——但攻击者写 <script>alert('你被黑了')</script>。

如果网站没有对用户输入做转义，这个脚本就会被当作正常代码执行，偷走你的 Cookie、跳转到钓鱼网站、甚至在你电脑上装键盘记录器！

XSS 就是攻击者把你的网页变成了他的代码执行器——本质是"数据被当作了代码执行"。

📂 三种类型

存储型 （最危险）

恶意代码被持久化在服务器（如数据库的评论、用户头像）。每个访问该页面的人都会被攻击。

反射型

恶意代码包含在 URL 参数中，服务器将参数直接回显在页面。需要用户点击恶意链接。

DOM 型

纯前端攻击，恶意代码通过 DOM API 注入页面（如 innerHTML），不经过服务器。

🛡️ 防御措施

  • 输出编码：HTML 实体编码 < → &lt;
  • Content-Security-Policy (CSP)：限制脚本加载来源
  • HttpOnly Cookie：禁止 JavaScript 读取 Cookie
  • 避免危险 API：不用 innerHTML，用 textContent
  • 输入验证：白名单限制允许的字符
  • React/Vue 的自动转义：现代框架默认防御 XSS

3️⃣ SQL 注入

你开了一家书店，告诉店员"给我找作者叫'鲁迅'的书"——店员老老实实地走去找。但攻击者过来说"给我找作者叫'鲁迅'; DROP TABLE books;--'的书"。

如果店员（数据库）没有做任何检查，就把整个命令念出来——"找作者叫'鲁迅'；并且把书表删了"——你的书店就完蛋了！

SQL 注入的根源就是：用户输入被拼接到了 SQL 语句中，改变了 SQL 的语法结构。

⚔️ 典型攻击

// ❌ 危险！拼接 SQL
String sql = "SELECT * FROM users " +
  "WHERE name = '" + input + "'";

// 攻击者输入: ' OR '1'='1
// SQL 变成:
SELECT * FROM users
WHERE name = '' OR '1'='1'
// → 返回所有用户！

🛡️ 防御措施

  • ★ 参数化查询 (PreparedStatement)
  • ORM 的默认防注入（但小心原生查询）
  • 输入验证：类型/长度/格式校验
  • 最小权限原则：数据库用户只给必要的表权限
  • 关闭错误详情回显
// ✅ 安全！参数化查询
String sql = "SELECT * FROM users " +
  "WHERE name = ?";
PreparedStatement ps = conn.prepareStatement(sql);
ps.setString(1, input);
// → 用户输入永远是"值"，不会被解析为 SQL

4️⃣ SSRF（服务端请求伪造）

你的网站有一个"根据 URL 获取图片"的功能。攻击者不说"给我张猫的图片"，而是说"给我这个 URL 的内容：http://localhost:6379"。

你的服务器老老实实地去请求内网的 Redis 服务（127.0.0.1:6379），把结果返回给攻击者。攻击者就这样通过你的服务器做跳板，攻击了你本不该暴露的内网服务！

SSRF 的可怕之处：攻击者利用你的服务器作为跳板，攻击你的内网——你的服务器对内部网络有信任关系。

⚔️ 攻击场景

  • 扫描内网 192.168.x.x 网段
  • 攻击 localhost:6379（Redis）
  • 攻击 localhost:3306（MySQL）
  • 攻击云服务元数据 API（AWS 169.254.169.254）
  • 读取本地文件（file:///etc/passwd）
  • 攻击内部管理后台

🛡️ 防御措施

  • URL 白名单：只允许请求预设的 URL/域名
  • 禁用内网/回环地址：禁止 127.0.0.1, 169.254.x.x, 10.x.x.x, 192.168.x.x
  • 使用 AllowList 协议：仅允许 https://，禁用 file://, dict://, gopher://
  • 网络隔离：服务不应有内网连通权限
  • 限制 DNS 解析：禁用 DNS rebinding 攻击

✅ 实战最佳实践

以下是经过实践检验的安全最佳实践清单——每一条背后都有血的教训。

🔐 认证 + 令牌

  • 使用 OAuth2 授权码 + PKCE 流程（不要再用 Implicit 流）
  • Access Token 有效期设为 15-30 分钟；Refresh Token 做 Rotation
  • JWT 必须验证：签名、过期时间（exp）、受众（aud）、签发者（iss）
  • 禁用 alg: none 和弱签名算法
  • 勿在 JWT Payload 中存放敏感信息（Base64 不是加密！）

🔒 传输安全

  • 全站 HTTPS（HSTS 头部：Strict-Transport-Security: max-age=31536000）
  • TLS 1.3 优先，禁用 TLS 1.0/1.1
  • 使用现代化密码套件，禁用 RC4、3DES、CBC 模式
  • OCSP Stapling 提升证书状态查询性能

🛡️ Web 安全

  • 设置 Content-Security-Policy 头限制脚本/样式来源
  • Cookie 设置 HttpOnly、Secure、SameSite=Lax/Strict
  • 所有用户输入 参数化查询（PreparedStatement）
  • 禁止内网 IP 的 URL 请求（SSRF 防御）
  • CSRF Token 绑定 Session（双提交 Cookie 模式也可）
  • 文件上传校验：类型、大小、内容、重命名并存储到非 Web 可访问路径

📋 开发运维

  • 最小权限原则：服务仅申请所需的最小 scope 和权限
  • 密钥管理：使用 Vault/AWS Secrets Manager，不要硬编码在代码中
  • 日志不记录密码、Token、信用卡等敏感信息
  • 依赖扫描：定期检查已知漏洞（npm audit, trivy, snyk）
  • Web 安全扫描：OWASP ZAP、Burp Suite 定期扫描
  • 速率限制：防止暴力破解和 DDoS

安全的第一性原理：不信任（Zero Trust）。永远假设攻击者已经知道了你的代码、你的架构、你的防御措施。设计安全策略时，问自己一个问题："如果攻击者知道了所有信息除了密钥，系统还安全吗？"——答案应该是肯定的。

没有绝对的安全，只有足够好的安全——让攻击者攻击你的成本远大于他获得的收益，他就不会来。

❓ 高频实战问答

以下8道问题覆盖安全方向的核心实战考点，点击题目展开详解。

Q1：认证 (Authentication) 和授权 (Authorization) 的区别？请用现实例子说明。

核心思路：

认证是"你是谁？"——验证身份。如：前台查身份证确认你是房客。

授权是"你能做什么？"——确定权限。如：房卡只能刷开你房间的门。

加分点：

指出两者顺序：认证 先于 授权。提到常见的授权模型：RBAC（基于角色）、ABAC（基于属性）、ACL（访问控制列表）。还提到一个经典漏洞——越权访问（IDOR），本质就是认证正确但授权检查缺失。

Q2：OAuth2 四种授权码模式的适用场景？为什么 Implicit 模式被废弃？

核心思路：

① 授权码模式：有后端的 Web 应用，最安全，配合 PKCE 用于 SPA 和移动端。

② 隐式模式（Implicit）：曾用于 SPA，直接在 URL 返回 Access Token。被 RFC 9700 废弃。

③ 密码模式：仅限高度信任的 first-party 应用，不推荐。

④ 客户端凭证模式：微服务间 M2M 通信，无用户参与。

被废弃原因：

Access Token 暴露在浏览器 URL 中（URL 会被日志记录、Referer 泄露、浏览器历史保存）；无法使用 Refresh Token；不支持 PKCE 绑定。现在所有场景都用授权码 + PKCE 代替。

Q3：JWT 的结构是什么样的？为什么 payload 不能放敏感信息？

核心思路：

JWT 由三部分用点号连接：header.payload.signature。

Header：算法类型（alg）和 token 类型（typ）。

Payload：声明（claims）——sub、exp、iat、自定义数据。

Signature：对头部+载荷的签名，防止篡改。

为什么不能放敏感信息？

因为 Payload 只是 Base64URL 编码而非加密。任何人都可以用 atob() 解码看内容！签名只防止篡改，不提供机密性。敏感数据（密码、信用卡）需要额外加密或放在服务端存储。

加分点：

提到 JWT 的 alg: none 算法攻击——如果服务端不校验算法，攻击者可以修改 alg 为 none 伪造任意身份。还提到 JWT 无状态化的优缺点：无状态利于扩展，但无法主动吊销（除非用黑名单/短时效）。

Q4：TLS 1.3 握手比 TLS 1.2 改进了什么？什么是前向安全性？

核心思路：

TLS 1.3 的握手从 2-RTT 降到 1-RTT（一次往返完成握手）。客户端在 ClientHello 中就携带了密钥共享信息（KeyShare），服务端直接返回 ServerHello + 证书 + 握手完成消息。

另外 TLS 1.3 支持 0-RTT（会话恢复）——对于之前连接过的客户端可以直接发送数据。

前向安全性 (PFS)：

即使服务器私钥将来泄露，攻击者也无法解密过去记录的通信内容。因为 TLS 使用 ECDHE（临时椭圆曲线 Diffie-Hellman）每次连接生成临时密钥，用完即销毁。TLS 1.3 移除了所有不支持 PFS 的密钥交换算法（如 RSA 密钥交换）。

Q5：CSRF 和 XSS 有什么区别？

核心思路：

CSRF（跨站请求伪造）：利用用户的已登录身份，在用户不知情的情况下发送请求。攻击者 不能看到响应内容（盲打）。本质是利用了 Cookie 自动携带的特性。

XSS（跨站脚本攻击）：在目标网站中执行恶意脚本。攻击者 可以读取页面内容、窃取 Cookie、操控 DOM。

关键区别：

CSRF 不需要在目标网站执行代码——只需要用户已登录；XSS 需要目标网站有输入输出漏洞来执行代码。但 XSS 可以用于绕过 CSRF 防御——比如用 XSS 偷 CSRF Token。

防御对比：

CSRF 用 CSRF Token / SameSite Cookie / 验证 Referer；XSS 用 输出编码 / CSP / HttpOnly Cookie。

Q6：SQL 注入的原理是什么？参数化查询为什么能防注入？

原理：

当用户输入被拼接到 SQL 语句中时，输入中的特殊字符（如单引号 '）改变了 SQL 的语法结构，使得攻击者可以执行任意 SQL 命令。

为什么参数化查询有效：

参数化查询（PreparedStatement）将 SQL 语句的结构与数据分离。先编译 SQL 结构（如 SELECT * FROM users WHERE name = ?），然后再绑定参数。绑定参数时，数据库会自动转义参数值，确保用户输入永远被当作数据值而非 SQL 代码来处理。

加分点：

提到即使使用了 ORM（如 JPA/MyBatis），也要注意 ${}（字符串拼接）和 #{}（参数化绑定）的区别——前者仍然有注入风险。另外提到 NoSQL 注入（如 MongoDB 的 $where 注入）。

Q7：SSRF 攻击如何利用？云环境中的 SSRF 为什么特别危险？

核心思路：

SSRF（Server-Side Request Forgery）是攻击者让服务器替自己发送请求到内网/本地。常见场景：URL 预览、Webhook 回调、图片抓取、代理功能。

云环境中的危险性：

云服务器（AWS、GCP、Azure）都有一个元数据服务（如 AWS http://169.254.169.254/latest/meta-data/），返回实例的临时凭据（IAM Role 的 Access Key）。如果能用 SSRF 请求到这个地址，攻击者就能窃取云服务权限，这被称为"元数据攻击"——2019 年 Capital One 数据泄露就是通过 SSRF + 元数据服务实现的。

防御重点：

URL 白名单（只允许特定可信域名）、禁用回环地址和内网 IP、限制协议（仅允许 https）、网络隔离（服务应无内网访问权限）。

Q8：请列举 10 条你日常开发中遵循的安全最佳实践。

核心思路（任选 10 条）：

  1. 全站启用 HTTPS + HSTS，禁用 HTTP 明文传输
  2. 所有数据库操作使用参数化查询（PreparedStatement）
  3. 用户输入/输出做 HTML 实体编码（防 XSS）
  4. 设置 Content-Security-Policy 头限制脚本来源
  5. Cookie 设置 HttpOnly + Secure + SameSite=Lax
  6. 所有表单添加 CSRF Token（或使用 SameSite Cookie）
  7. API 端点实施频率限制（Rate Limiting）
  8. 密码使用 bcrypt/argon2 哈希，禁止明文存储
  9. 密钥、密码、Token 存在 Vault/Secret Manager，不写死在代码中
  10. 日志不记录敏感信息（密码、Token、信用卡、PII）
  11. 最小权限原则：每个微服务/用户仅分配所需的最小权限
  12. 文件上传限制类型、大小，重命名后存非公开路径
  13. 依赖库定期安全审计（npm audit, trivy, snyk）
  14. 错误页面不泄露堆栈信息、版本号、框架细节
  15. JWT 使用 RS256/ES256 签名，验证 exp/aud/iss，禁用 alg: none

加分点：

提到"你不是在写功能，你是在构建一个可能被攻击的系统"——安全是思维方式，不是 checklist。推荐阅读 OWASP Top 10 和参与 CTF 提升实战能力。
