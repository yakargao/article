🗃️ 存储系统·Cache

一句话：Cache 是计算机体系结构中最伟大的性能作弊器——它在 CPU 和主存之间加一层极快但极小的 SRAM，利用程序的局部性，让 CPU 大部分时间都在纳秒级访问数据，而不必等那几百纳秒的内存。没有 Cache，现代 CPU 的 5GHz 主频毫无意义，因为内存根本喂不饱它。

🍳 类比：咖啡师的台面 —— 咖啡师（CPU）做咖啡需要咖啡豆、牛奶、杯子。如果每做一杯都要去仓库拿，一天做不了几杯。台面（L1 Cache） 放最常用的工具和原料，几秒钟取到；吧台柜（L2 Cache） 放备用的；仓库（主存） 放所有原料，但拿一趟要几分钟。聪明的咖啡师会把接下来要用的原料提前放到台面上——这就是预取。

层次核心问题一句话解答
① 层次结构为什么 CPU 不能直接连内存？速度差太大（~0.3ns vs ~100ns），Cache 做缓冲桥梁
② 局部性Cache 为什么能工作？时间局部性（刚用的很快再用）+ 空间局部性（附近的很快会用）
③ 映射方式内存数据怎么放到 Cache 里？直接映射/全相联/组相联——三种放法，各有取舍
④ 替换算法Cache 满了踢谁出去？LRU / Clock / 随机——踢最不常用的那个
⑤ 写策略写数据时 Cache 和内存怎么同步？写直达（同步写）vs 写回（先写 Cache，淘汰时再写回）
⑥ MESI多核 Cache 之间怎么保持一致？MESI 四种状态 + 总线嗅探，核之间协商谁拥有最新数据

🗼 存储器层次结构

🏢 类比：图书馆的取书速度

想象你去图书馆找一本书：

  • 口袋里的便签（寄存器）—— 零延迟，但只能放几张小纸条
  • 桌面上的书（L1 Cache）—— 伸手就拿到，~1ns，32KB
  • 旁边的书架（L2 Cache）—— 转身取，~5ns，256KB
  • 同一楼层的书柜（L3 Cache）—— 走几步，~15ns，8MB+
  • 楼下书库（主存）—— 坐电梯去取，~100ns，16GB+
  • 远程大书库（磁盘/SSD）—— 快递寄送，~10⁶ns（毫秒级）

每一层都是上一层的Cache：L1 Cache 是 L2 的 Cache，L2 是 L3 的 Cache，L3 是主存的 Cache，主存是磁盘的 Cache。每往下走一层，时间增加 5-10 倍，容量增加 50-100 倍。

存储器层次结构：每一层都是下一层的 Cache，速度差达 10⁵ 倍

📊 关键数字：为什么 Cache 不可或缺？

假设 CPU 每周期执行 1 条指令（实际超标量更高）：

  • 如果只连主存（100ns），CPU 每取一个数据要等 ~500 个周期——这就是 Memory Wall（存储墙）
  • L1 Cache 命中时只要 2-4 个周期，现代 CPU 的 Cache 命中率 > 95%
  • 没有 Cache，5GHz CPU 实际性能会降到跟 10MHz 差不多——99.8% 的时间都在等内存

🎯 局部性原理——Cache 能工作的根本原因

📺 类比：你的追剧习惯

时间局部性：你昨晚看了《庆余年》第 10 集，今晚大概率会看第 11 集而不是突然去看《甄嬛传》。刚访问过的数据很快会被再次访问。

空间局部性：你从书架上拿了一本书，顺手也拿了旁边的书——因为它们挨着，你省得多跑一趟。访问了一个地址，附近的地址很快也会被访问。

时间局部性（反复用同一变量）vs 空间局部性（顺序访问连续数据）

💡 Cache Line: Cache 的最小传输单位

Cache 不是按字节搬运的——一次加载 64 字节（一个 Cache Line）。这意味着：

  • 你访问 arr[0]，CPU 会把 arr[0]~arr[15] 全部搬进 Cache
  • 遍历数组时，每 16 个整数只有第一次会 Miss，后面 15 次全 Hit
  • False Sharing（伪共享）：多核修改同一 Cache Line 中的不同变量 → 互相踢 → 性能暴跌

🔄 三种映射方式——数据怎么放进 Cache？

📚 类比：图书馆的存包柜

图书馆有 1000 个存包柜（Cache 行），但每天有 5000 个读者（内存块）要存包。怎么决定每个读者的包放哪个柜子？

  • 直接映射：根据读者编号取模——1号读者→1号柜，1001号读者→1号柜（冲突）。简单但容易打架。
  • 全相联：不限制柜子——任何读者可以存任何空闲柜。最灵活但找柜子需要翻遍所有柜子（速度慢）。
  • 组相联：折中——把柜子分成 100 组，每组 10 个。根据读者编号决定去哪个组，然后在组内随便选一个。既快又灵活。
三种映射方式：直接映射（简单但冲突高）→ 组相联（折中，实际使用）→ 全相联（灵活但慢）

放置位置唯一固定任意位置特定组内任意
查找方式Index 索引遍历所有行Index 索引组 + 遍历组内 N 路
冲突率高低中（N↑→↓）

♻️ 替换算法——Cache 满了踢谁？

🚪 类比：大学宿舍的座位

自习室只有 50 个座位，但有 200 个学生。座位满了，新来的学生必须把某人赶走。赶谁？

  • FIFO：赶来得最早的那个——不管他现在学得多认真。可能把学霸赶走了。
  • LRU：赶最久没翻书的那个人——学霸一直在学习不会被赶，学霸旁边的学渣半小时没翻书了，被赶。合理！
  • 随机：闭着眼赶一个——简单粗暴，运气好赶走学渣，运气不好赶走学霸。
  • Clock：管理员每隔 10 分钟巡查一圈，看到正在学习的就标记一下，看到没标记的就赶走——LRU 的低成本版本。
四种替换算法对比：LRU 命中率最高但代价最大，Clock 是工程最优解

🔧 实际 CPU 用什么？

现代 CPU 的 Cache 替换算法是 pseudo-LRU（伪 LRU）：

  • 不维护完整的访问序链表（太贵）
  • 而是用 Tree-based PLRU：一棵二叉树，每个节点记录"最近访问了左子树还是右子树"，只需 N-1 bit
  • 对于 4 路组相联，只需要 3 bit 就能近似 LRU
  • 4 路组相联用 PLRU，命中率只比精确 LRU 低 1-2%，但硬件成本降低 90%

⚠️ LRU 的致命缺陷：如果程序的工作集刚好比 Cache 大一点点，会发生 Cache Thrashing（缓存抖动）——每访问一个新数据都把旧数据踢出去，下一个循环又需要旧数据，来回踢，命中率趋近于 0%。这就是为什么 Cache 大小必须 > 程序工作集，否则再好的替换算法也救不了。

📝 写策略——写数据时怎么保证一致？

📋 类比：两种记账方式

你在笔记本（Cache）上记了一笔账，但正式的账本（主存）还没更新：

  • 写直达（Write-Through）：每次写笔记的同时，立马把账本也写上。数据绝对一致，但每次写都要等账本更新完——慢但安全。
  • 写回（Write-Back）：先只写笔记，笔记本上画个圈（Dirty Bit）表示"这本账还没抄到正式账本"。等笔记本要换页时，再把画了圈的内容抄到账本上——快但复杂。
写直达（写 Cache 同时写内存）vs 写回（写 Cache 标记 Dirty，替换时写回）
特性写直达 (Write-Through)写回 (Write-Back)
写 Cache 同时写内存？✅ 是❌ 仅写 Cache
内存一致性始终一致可能不一致（Dirty）
写入速度慢（等内存）快（仅 Cache）
Dirty Bit 需要？❌ 不需要✅ 需要
替换时开销低（直接覆盖）可能高（需写回）
写缓冲优化✅ 常用不适用

🤝 MESI 缓存一致性协议

🏠 类比：合租公寓的冰箱

你和室友合租，共用厨房冰箱。你买了一瓶牛奶放进去。如果室友不知道这件事，他也去买了一瓶——冰箱里就有两瓶牛奶，但你们两人都以为牛奶是自己的，互不知道对方的。

MESI 协议就是冰箱上的状态标签：

  • M (Modified) = "我独有的，且我修改过" —— 你冰箱里的牛奶被你喝了一半，标签上写着"我独享，已开封"
  • E (Exclusive) = "只有我有" —— 你放了一瓶新牛奶，标签上写"仅此一份"
  • S (Shared) = "大家都有" —— 你和室友各买了一瓶，标签上写"共享"
  • I (Invalid) = "过期了/被拿走了" —— 你的牛奶被室友喝完了，标签撕掉

每当你要拿牛奶，先看一眼标签（总线嗅探）。如果别人标签是 M，先等他放回冰箱（写回内存），你再拿。

MESI 四状态及转换：M/E/S/I 通过总线嗅探协同，保证多核数据一致性

💣 False Sharing（伪共享）

核 A 写变量 x，核 B 写变量 y——按理互不影响。但如果 x 和 y 在同一个 Cache Line（64B） 里：

核 A 写 x → 整行变 M → 核 B 的该行变 I → 核 B 写 y → 整行变 M → 核 A 的该行变 I → ... 来回踢！

本应各自独立的两个变量，因为共享了一个 Cache Line 而互相竞争，性能暴跌。

🛡️ 解决方案

Cache Line Padding（缓存行填充）：

type Counter struct {
    val int64
    // 填充 56 字节，确保 val 独占一个 Cache Line
    _ [56]byte
}

Go 的 sync.Pool、Java 的 @Contended、Linux 内核的 ____cacheline_aligned 都是此原理。

🧪 MESI 状态转换示例

场景：双核系统，初始都未加载地址 A 的数据：

1. 核0 读 A  → 核0: I→E（独占，内存有副本）
2. 核1 读 A  → 核1: I→S，核0: E→S（共享）
3. 核0 写 A  → 核0: S→M（修改），核1: S→I（无效）
4. 核1 读 A  → 核1: I→S，核0 Flush 写回，M→S（共享且内存已更新）

⚠️ MESI 的局限：当核数增多（>8核），总线嗅探的广播流量成为瓶颈——每个核的每次 Miss 都要广播到所有核，总线带宽迅速耗尽。现代 CPU（如 Intel Mesh/AMD CCX）使用 目录协议（Directory-Based Coherence），用一个中心目录记录每行的共享者列表，只有相关核才收到通知，大幅降低广播开销。

❓ 实战问答

Q1: 为什么现代 CPU 需要三级 Cache？直接一个大 Cache 不行吗？

A: 这是速度 × 容量 × 延迟 的不可能三角：

  • 大 Cache 一定慢：SRAM 越大，访问延迟就越高（寻址路径更长，驱动电流更大）。32KB 的 L1 可以 2 周期访问，8MB 的 L3 需要 30-50 周期。
  • 大 Cache 一定贵：SRAM 6 晶体管/bit vs DRAM 1 晶体管/bit。8MB SRAM 的芯片面积 ≈ 一个 CPU 核心。
  • 三级结构完美折中：L1 极小但极快（处理 90%+ 的访问），L2 中等（处理剩下的 9%），L3 大而共享（兜底最后 1%）。

Q2: 什么是 Cache Thrashing（缓存抖动）？怎么解决？

A: Thrashing 指程序的工作集刚好比 Cache 大一点，导致频繁的 Cache Miss。例如一个 4 路组相联 Cache，程序反复访问 5 个映射到同一组的地址 → 每次访问都替换 → 命中率趋近 0%。

解决方案：

  • 增大 Cache 关联度：8路组相联比4路抗抖动能力强
  • Cache 分块（Cache Blocking）：将大数组切成 Cache 能容纳的小块处理
  • 数据对齐：避免多个频繁访问的数据落在同一组
  • 硬件预取：提前预测下一步需要的数据并加载

Q3: 写直达 + 写缓冲 和 写回，哪个更优？

A: 没有绝对优劣，取决于写模式：

  • 写直达 + 写缓冲适合写频繁且写入位置分散的场景（如 L1 Cache 的写操作）。写缓冲可以合并多次写，CPU 无需等待内存。
  • 写回适合写频繁且写入位置集中的场景（如 L2/L3 Cache）。多次修改同一行只需一次写回，效率极高。
  • 实际 CPU 是混合策略：L1 用写直达（配合写缓冲），L2/L3 用写回。这兼顾了 L1 的速度要求和 L2/L3 的带宽效率。

Q4: 什么是 TLB？和 Cache 有什么关系？

A: TLB（Translation Lookaside Buffer）是 页表的 Cache：

  • 每次 CPU 访问内存，MMU 需要把虚拟地址翻译成物理地址——需要查页表（在内存中）
  • 如果每次都去内存查页表，那每次内存访问都多了一次或多次内存访问（四级页表需要 4 次）
  • TLB 就是页表项的 Cache——它缓存最近用过的虚拟地址→物理地址的映射
  • TLB 命中 → 一次翻译只要 1 个周期。TLB Miss → 需要遍历页表（4 次内存访问）
  • TLB 通常是 全相联 或 组相联（路数少，因为 TLB 项数少）

Q5: 伪共享（False Sharing）怎么检测和修复？

A: 检测：

  • 性能分析：perf stat -e cache-misses 观察异常高的 Miss 率
  • 核心特征：多核程序加速比不升反降，CPU 利用率高但实际计算量小

A: 修复：

  • Cache Line Padding：在每个频繁修改的变量后填充 56 字节（64-8），确保独占 Cache Line
  • Go 示例：
type PaddedCounter struct {
    Val   int64
    Pad   [7]int64  // 7 × 8 = 56 字节填充
}
  • 数据局部化：将不同线程的数据分散到不同内存区域
  • 原子操作拆分：避免多个核同时修改同一 Cache Line 中的不同字段

Q6: 为什么 MESI 在大规模多核系统中不够用？有什么替代方案？

A: MESI 使用总线嗅探（Snooping）——每个核广播所有请求，所有核监听。随着核数增加（>8-16核）：

  • 总线带宽瓶颈：广播请求淹没总线，O(N²) 的通信复杂度
  • 监听开销：每个核都要检查每个请求，功耗飙升

替代方案：目录协议（Directory-Based Coherence）

  • 用中心化目录记录每个 Cache Line 的共享者列表
  • 只有相关的核收到通知（点对点，而非广播）
  • 复杂度 O(N) 而非 O(N²)
  • Intel Mesh 架构、AMD CCX/CCD 都使用目录协议

🎙️ 口播总结（1分钟版）：

「计算机组成原理里存储系统·Cache 这一块，我用一个框架帮你串起来：

首先，为什么需要 Cache？因为 CPU 太快（0.2ns/cycle）、主存太慢（100ns），差了 500 倍。没有 Cache，5GHz 的 CPU 实际性能和 10MHz 差不多。所以有了金字塔式的存储层次——寄存器 → L1（32KB，1ns）→ L2（256KB，5ns）→ L3（8MB+，15ns）→ 主存（100ns）→ 磁盘。

Cache 为什么能工作？局部性。时间局部性（刚用的很快再用）和空间局部性（附近的很快会用）。一次加载 64 字节的 Cache Line，顺序遍历数组时每 16 个 int 只有第一次 Miss。

数据怎么放进去？三种映射方式。直接映射（固定位置但冲突高）、全相联（任意位置但慢）、组相联（折中——现代 CPU 用 8-16 路）。

满了怎么办？替换算法。严格 LRU 最好但太贵，实际用伪 LRU（PLRU）或 Clock，只需 1 个 bit 就能接近 LRU 的效果。

写数据时呢？写直达（同步写内存，配合写缓冲）和写回（只写 Cache，替换时再写回——大部分场景更快）。

多核怎么办？MESI 协议。Modified/Exclusive/Shared/Invalid 四种状态，总线嗅探保证核间一致性。但核多了总线撑不住，大规模系统用目录协议替代。

实战时特别注意伪共享（False Sharing）——两个变量在同一 Cache Line 被不同核写，来回踢，性能暴跌。解决方案就是 Cache Line Padding。」
