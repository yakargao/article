📦 TCP vs UDP：可靠 vs 快速

📞 类比：TCP 是打电话——先拨号接通（三次握手），通话中确认对方听到（ACK），说完挂断（四次挥手）。UDP 是发短信——直接扔过去，对方收没收到不管。

TCP（传输控制协议）

  • 面向连接（三次握手建立连接）
  • 可靠交付：确认+重传+校验
  • 有序：序列号保证顺序
  • 流量控制：滑动窗口
  • 拥塞控制：慢启动/拥塞避免/快速重传
  • 适用：HTTP、文件传输、数据库

UDP（用户数据报协议）

  • 无连接（直接发）
  • 不可靠：发出去就不管了
  • 不保证顺序
  • 没有流控和拥塞控制
  • 头部只有 8 字节
  • 适用：视频直播、DNS、游戏、QUIC
维度TCPUDP
头部大小20-60 字节8 字节
连接面向连接（SYN→SYN-ACK→ACK）无连接
可靠性确认+重传+校验和仅校验和
顺序序列号保证不保证
速度慢（建立连接+确认开销）快（无连接建立）
应用HTTP/1-2、SSH、FTP、SMTPDNS、QUIC(HTTP/3)、VoIP、游戏

🤝 TCP 三次握手 & 四次挥手——高频必考

🤝 类比：三次握手是"你在吗？——在，你也在吗？——在，开始聊"。四次挥手是"我说完了——知道了——我也说完了——知道了——再见"。一方说完了不代表另一方也说完了，所以挥手比握手多一次。

为什么三次握手不是两次？防止已失效的连接请求到达服务端。如果只有两次，服务端收到旧SYN建立连接浪费资源。 SYN 泛洪攻击：攻击者伪造IP发大量SYN，服务端回复SYN-ACK后等待ACK→半连接队列满→拒绝正常请求。防御：SYN Cookie、tcp_syncookies。 TIME_WAIT 为什么 2MSL？① 保证最后ACK能到达对方（丢了对方会重发FIN）② 让旧连接的所有报文在网络中消失，不干扰新连接。 TIME_WAIT 过多怎么办？tcp_tw_reuse（复用）、调整MSL、长连接代替短连接。大量TIME_WAIT表明短连接过多。 CLOSE_WAIT 过多？服务端收到FIN后没调用 close() → 应用层bug，连接泄漏。
三次握手（建立连接）+ 四次挥手（断开连接）+ 关键点

🌊 流量控制：滑动窗口——不要让接收方溢出

📬 类比：滑动窗口是快递站的"收件箱"——接收方告诉发送方"我还能收 5 个包裹"，发送方就只发 5 个，等收到确认说"收了 2 个，还能收 3 个"再继续发。窗口大小由接收方决定（rwnd）。

rwnd (接收窗口)：接收方在 ACK 中通告剩余缓冲区大小→发送方据此限制发送量，防止接收方溢出。 零窗口探测：接收方窗口=0→发送方停止发送→定期发ZeroWindowProbe探测→窗口打开后继续。 糊涂窗口综合征：接收方每次只释放一点点窗口→发送方每次只发一点点→效率极低。解决：Nagle算法 + Clark方案。

🚦 拥塞控制：不要让网络溢出

🚗 类比：流量控制是"对方停车场满了没"（接收方能力），拥塞控制是"路上堵不堵"（网络能力）。TCP 的拥塞控制就像自适应巡航——路况好就加速（窗口倍增），开始堵了就减速（线性增），撞车了就猛刹（窗口减半再慢启动）。

TCP 拥塞控制状态机：慢启动(指数)→拥塞避免(线性)→超时(重置)/3 dup ACK(快速恢复)

BBR vs CUBIC：现代拥塞控制

算法机制优点应用
CUBIC基于丢包的窗口调节（Linux 默认）公平性好、广泛部署大多数 Linux 服务器
BBR基于带宽+RTT 的模型（不依赖丢包）高吞吐低延迟、弱网表现好Google/YouTube、跨国传输

📡 HTTP 协议族：从 1.1 到 3 的进化

🏢 类比：HTTP/1.1 是"一人一窗口"的银行（一个连接一次只能处理一个请求，队头阻塞）。HTTP/2 是"所有窗口随时服务"（多路复用）。HTTP/3 直接换了地基——不用 TCP，用基于 UDP 的 QUIC，连握手都不用等。

HTTP 三代进化：1.1(队头阻塞) → 2(多路复用) → 3(QUIC彻底消除TCP队头阻塞)

🔐 HTTPS & TLS：互联网的安全基石

🔑 类比：HTTPS = HTTP + TLS 加密通道。TLS 握手像特工接头——先确认身份（证书验证），再用非对称加密交换一把临时钥匙（Session Key），之后全程用这把钥匙对称加密通信。TLS 1.3 把两轮握手压到一轮，快了一倍。

非对称加密：公钥加密→私钥解密(RSA/ECDHE)。握手时用，交换对称密钥。 对称加密：同一个密钥加解密(AES-GCM/ChaCha20)。握手后数据通信用，性能高。 数字证书：CA签发→证明"这个公钥确实属于这个域名"。证书链验证Root CA→中间CA→网站证书。 ECDHE 密钥交换：椭圆曲线Diffie-Hellman，双方各自生成临时密钥对，通过交换公钥算出相同的共享密钥而不传输密钥本身。 核心金句：HTTPS = HTTP over TLS。TLS握手=验证证书+密钥交换。ECDHE提供前向安全性(Forward Secrecy)。
TLS 1.3 握手（1-RTT）+ TLS 1.2 对比 + 加密体系三件套

📇 DNS：从域名到 IP 的寻址之路

📞 类比：DNS 是互联网的电话簿——你输入名字（域名），它帮你查出号码（IP）。查过的号码会记在本地（缓存），不用每次翻电话簿。

DNS解析链路：缓存链(浏览器→OS→本地DNS) → 递归查询(Root→TLD→权威DNS)

🌍 IP 协议与路由：数据包如何跨越山海

🚚 类比：IP 协议是"快递公司的分拣系统"——每个包裹（数据包）贴上收件地址（目标IP）和发件地址（源IP），每到一个分拣中心（路由器）查路由表决定下一站，直到送达。

概念说明核心要点
IP 地址网络层标识。IPv4 32位(4.3B地址)→NAT续命。IPv6 128位(几乎无限)公网IP vs 内网IP、CIDR
NAT网络地址转换：内网多设备共享一个公网IPSNAT/DNAT、NAT穿透(STUN/TURN)
ARP地址解析协议：IP→MAC。广播"谁是192.168.1.1?"ARP缓存、ARP欺骗攻击
ICMP网络诊断：ping(echo request/reply)、traceroute(TTL超时)TTL、MTU Path Discovery
路由协议OSPF(内部网关)、BGP(外部网关，AS之间)距离矢量vs链路状态、自治系统AS

❓ 高频考题

Q1: 浏览器输入 URL 后发生了什么？（终极核心题）
① DNS解析域名→IP ② TCP三次握手(80/443端口) ③ 如果是HTTPS→TLS握手 ④ 发送HTTP请求 ⑤ 服务端处理→返回响应 ⑥ 浏览器解析HTML→构建DOM→CSSOM→Render Tree→Layout→Paint ⑦ 四次挥手断开连接。
Q2: TCP 为什么是四次挥手而不是三次？
因为TCP是全双工——一方说"我说完了"(FIN)不代表另一方也说完了。被动方先ACK（表示收到你的FIN），等自己数据发完再发FIN。所以是 FIN→ACK+FIN→ACK 四步。
Q3: HTTP/2 的多路复用解决了什么问题？怎么解决的？
解决了HTTP/1.1的队头阻塞——一个请求慢后面全等。HTTP/2 用二进制分帧，一个TCP连接上同时跑多个Stream，帧交错传输。但TCP层面的队头阻塞（丢包）仍未解决→HTTP/3用QUIC(UDP)彻底解决。
Q4: HTTPS 如何保证安全？中间人攻击怎么防？
三步：① 证书链验证身份（CA签发，信任链）② 非对称加密交换会话密钥（ECDHE）③ 对称加密传输数据（AES-GCM）。中间人攻击通过证书验证防御——攻击者无法伪造CA签名的证书。
Q5: TCP 的拥塞控制和流量控制有什么区别？
流量控制是端到端的——接收方告诉发送方"我还能收多少"（rwnd），防止接收方溢出。拥塞控制是全局的——发送方探测网络是否拥堵（cwnd/ssthresh），防止网络过载。前者看对方能力，后者看网络状态。
Q6: TIME_WAIT 状态的作用？为什么是 2MSL？
两个作用：① 确保最后的 ACK 能到达对方（丢了对方会重发 FIN）② 让旧连接的延迟报文在网络中消失，不会干扰同端口的新连接。2MSL = 报文最大存活时间的2倍——一来一回刚好清空。
