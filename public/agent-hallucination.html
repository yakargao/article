🎭 Agent 幻觉与可靠性

工具调用幻觉 · 推理链幻觉 · 检测器 · 防护层 · Go 实现 — 深度技术解析

1. Agent 幻觉比 LLM 幻觉更棘手

大语言模型（LLM）的幻觉问题早已被广泛讨论：模型"自信地"编造虚假信息，输出看似合理实则错误的文本。但当我们从纯文本模型跨入 Agent 系统——即能够调用工具、执行操作、影响现实世界的自主体——幻觉问题的严重性呈指数级上升。

🔬 费曼类比： 想象你问一个朋友"外面在下雨吗？" 一个只会说话的 LLM 相当于——他可能说"是的，在下大雨"即使外面晴空万里。这很糟糕，但伤害有限。

而一个 Agent 相当于——他不光回答你，还拿起伞冲出门、帮你叫了网约车、给花浇水、并且发信息告诉你的同事"今天下雨，会议改线上"。如果外面根本没下雨，这些行动就是灾难性的。

LLM 幻觉只是说错话；Agent 幻觉是说错话 + 做错事 + 产生不可逆的副作用。

Agent 幻觉的独特之处在于它涉及行动维度。一个错误的工具调用可能导致：数据被篡改、资源被消耗、API 被误调用、生产环境受损。更重要的是，Agent 的推理链本身可能就有问题，但它仍然能"完成"任务——只是以错误的方式。

图1：LLM 幻觉局限于信息输出，Agent 幻觉扩展到了行动层面，影响不可逆

2. 工具调用幻觉

工具调用幻觉是 Agent 系统中最常见也最危险的幻觉类型。当 Agent 在推理过程中选择了错误的工具、传入了错误的参数、或者忽略了工具的返回信号，就产生了工具调用幻觉。

典型场景：Stock Agent 编造数据源

考虑一个被赋予查询股票数据的 Agent。用户问："苹果公司今天的股价如何？" 正确做法是调用 get_stock_price("AAPL")。但如果 Agent 无法访问该工具，它可能编造一个不存在的数据源，例如调用 query_internal_db("AAPL_price")——而 query_internal_db 是用于查询内部用户数据的工具。这不仅给出错误答案，还可能泄露隐私。

Go 代码：工具调用验证器

以下 Go 代码实现了一个工具调用验证器，用于在 Agent 调用工具前进行参数校验和语义匹配检查：

package agent

import (
    "fmt"
    "strings"
)

// ToolDef 定义了一个工具的元信息
type ToolDef struct {
    Name        string
    Description string
    Parameters  []ParamDef
}

type ParamDef struct {
    Name     string
    Type     string            // string, int, float, bool
    Required bool
    Enum     []string          // 可选枚举值
}

// ToolCall 是 Agent 发起的工具调用
type ToolCall struct {
    ToolName   string
    Arguments  map[string]interface{}
}

// ValidationResult 记录验证结果
type ValidationResult struct {
    IsValid     bool
    Warnings    []string
    Errors      []string
    Confidence  float64  // 匹配置信度 0.0 ~ 1.0
}

// ValidateToolCall 验证工具调用的完整正确性
func ValidateToolCall(call ToolCall, registry map[string]ToolDef) ValidationResult {
    result := ValidationResult{IsValid: true, Confidence: 1.0}

    // 1. 检查工具是否存在
    def, exists := registry[call.ToolName]
    if !exists {
        // 尝试模糊匹配
        bestMatch, score := fuzzyMatchTool(call.ToolName, registry)
        if score < 0.7 {
            result.Errors = append(result.Errors,
                fmt.Sprintf("工具 '%s' 不存在，且未找到相似匹配 (最佳匹配: '%s', 得分: %.2f)",
                    call.ToolName, bestMatch, score))
            result.IsValid = false
            result.Confidence = 0.0
            return result
        }
        result.Warnings = append(result.Warnings,
            fmt.Sprintf("工具名 '%s' 模糊匹配到 '%s' (%.2f)，请确认",
                call.ToolName, bestMatch, score))
        result.Confidence = score
        def = registry[bestMatch]
    }

    // 2. 检查必要参数
    for _, p := range def.Parameters {
        if p.Required {
            if _, ok := call.Arguments[p.Name]; !ok {
                result.Errors = append(result.Errors,
                    fmt.Sprintf("缺少必要参数 '%s'", p.Name))
                result.IsValid = false
            }
        }
    }

    // 3. 检查参数类型与枚举值
    for k, v := range call.Arguments {
        found := false
        for _, p := range def.Parameters {
            if p.Name == k {
                found = true
                // 枚举值检查
                if len(p.Enum) > 0 {
                    valStr := fmt.Sprintf("%v", v)
                    inEnum := false
                    for _, e := range p.Enum {
                        if valStr == e {
                            inEnum = true
                            break
                        }
                    }
                    if !inEnum {
                        result.Warnings = append(result.Warnings,
                            fmt.Sprintf("参数 '%s' 的值 '%s' 不在枚举范围 %v 内",
                                k, valStr, p.Enum))
                        result.Confidence *= 0.8
                    }
                }
                break
            }
        }
        if !found {
            result.Warnings = append(result.Warnings,
                fmt.Sprintf("工具 '%s' 未定义参数 '%s'，将被忽略",
                    def.Name, k))
            result.Confidence *= 0.9
        }
    }

    return result
}

// fuzzyMatchTool 使用简单的 Levenshtein 距离进行工具名模糊匹配
func fuzzyMatchTool(name string, registry map[string]ToolDef) (string, float64) {
    bestName := ""
    bestScore := 0.0
    for n := range registry {
        score := similarityScore(name, n)
        if score > bestScore {
            bestScore = score
            bestName = n
        }
    }
    return bestName, bestScore
}

func similarityScore(a, b string) float64 {
    a, b = strings.ToLower(a), strings.ToLower(b)
    distance := levenshtein(a, b)
    maxLen := len(a)
    if len(b) > maxLen {
        maxLen = len(b)
    }
    if maxLen == 0 {
        return 1.0
    }
    return 1.0 - float64(distance)/float64(maxLen)
}

func levenshtein(s, t string) int {
    // 标准 Levenshtein 距离实现
    n, m := len(s), len(t)
    d := make([][]int, n+1)
    for i := range d {
        d[i] = make([]int, m+1)
        d[i][0] = i
    }
    for j := 0; j <= m; j++ {
        d[0][j] = j
    }
    for i := 1; i <= n; i++ {
        for j := 1; j <= m; j++ {
            cost := 1
            if s[i-1] == t[j-1] {
                cost = 0
            }
            d[i][j] = min(
                d[i-1][j]+1,
                d[i][j-1]+1,
                d[i-1][j-1]+cost,
            )
        }
    }
    return d[n][m]
}

func min(a ...int) int {
    m := a[0]
    for _, v := range a[1:] {
        if v < m {
            m = v
        }
    }
    return m
}

这个验证器在 Agent 每次发起工具调用前执行三个层面的校验：(1) 工具名是否存在或模糊匹配；(2) 必要参数是否齐全；(3) 参数值和枚举是否合法。返回的 Confidence 字段可用于后续的阈值判断。

3. 推理链幻觉

推理链幻觉（Reasoning Chain Hallucination）指的是 Agent 在推理过程中的某个环节产生了错误，但整个推理链条看起来是合理的。这种幻觉尤为隐蔽，因为部分正确的推理会让人相信结论也是正确的。

因果倒置案例

以经典的"下雨→地湿"逻辑链为例：

🔍 案例：地湿了 → 所以下雨了？

Agent 被问到："为什么操场是湿的？"

推理链 A（正确）： 查天气API → 今早8:00有降雨记录 → 操场没有顶棚 → 结论：下雨导致地湿

推理链 B（幻觉）： 操场是湿的 → 所有湿的地面都是下雨导致的 → 结论：下雨了 → 但实际上可能是洒水车造成的

这种"因果倒置"幻觉在 Agent 系统中非常常见，因为 LLM 倾向于用相关性替代因果性。

4. 幻觉检测器

幻觉检测器是 Agent 系统安全性的第二道防线。当 Agent 完成推理后，检测器负责评估输出结果是否存在幻觉。目前主流的检测方法有三种：

图4：三种检测方法的原理、优缺点对比

SelfCheckGPT

SelfCheckGPT 是一种"模型自省"方法。其核心思想是：如果一个模型对某个信息有把握，那么多次生成应该得到一致的结果；反之，如果模型在"编造"，不同采样之间的输出会存在分化。具体实现上，通过对比原始输出与条件生成（以原始输出为条件）之间的一致性来打分。

语义熵检测

语义熵（Semantic Entropy）是对 SelfCheck 的数学深化。它不是在 token 层面计算熵（这会被词汇多样性干扰），而是在语义空间中计算。具体做法是将多个输出聚类到语义等价类中，然后计算这些等价类的概率分布熵。高语义熵意味着 Agent 对同一问题给出了多种不同含义的回答——这是幻觉的强力指标。

Go 代码：基于一致性的幻觉检测器

package detection

import (
    "fmt"
    "math"
    "strings"
)

// Claim 是从 Agent 输出中提取的单个事实性断言
type Claim struct {
    Text       string
    Confidence float64
}

// DetectionResult 检测结果
type DetectionResult struct {
    IsHallucinated bool
    Score          float64  // 0 = 确定真实, 1 = 确定幻觉
    Details        []ClaimDetail
}

type ClaimDetail struct {
    Claim     string
    Agreement float64  // 跨样本一致率 0~1
    Status    string     // "consistent", "contradicted", "unsupported"
}

// ConsistencyDetector 基于多采样一致性检测幻觉
type ConsistencyDetector struct {
    SampleCount      int     // 采样数（默认 5）
    AgreementThresh  float64 // 一致性阈值（默认 0.7）
    EntropyWeight    float64 // 熵权重
}

// NewConsistencyDetector 创建检测器
func NewConsistencyDetector() *ConsistencyDetector {
    return &ConsistencyDetector{
        SampleCount:     5,
        AgreementThresh: 0.7,
        EntropyWeight:   0.5,
    }
}

// Detect 对 Agent 输出执行一致性检测
// samples 是多次 Agent 推理的输出结果
func (d *ConsistencyDetector) Detect(original string, samples []string) DetectionResult {
    // 提取断言
    claims := extractClaims(original)

    var details []ClaimDetail
    totalScore := 0.0

    for _, claim := range claims {
        detail := evaluateClaim(claim, samples)
        details = append(details, detail)

        // 不一致的断言增加幻觉分数
        if detail.Status == "contradicted" {
            totalScore += 1.0 - detail.Agreement
        } else if detail.Status == "unsupported" {
            totalScore += 0.4
        }
    }

    // 计算语义熵
    entropy := computeSemanticEntropy(samples)

    // 综合评分
    var finalScore float64
    if len(claims) > 0 {
        // 归一化到 0~1
        claimScore := totalScore / float64(len(claims))
        finalScore = claimScore*d.EntropyWeight + entropy*(1-d.EntropyWeight)
    } else {
        finalScore = entropy
    }

    return DetectionResult{
        IsHallucinated: finalScore > (1.0 - d.AgreementThresh),
        Score:          math.Round(finalScore*100) / 100,
        Details:        details,
    }
}

// extractClaims 从文本中提取事实性断言
func extractClaims(text string) []string {
    // 简单实现：按句子分割 + 过滤明显非事实句
    sentences := strings.Split(text, ".")
    var claims []string
    for _, s := range sentences {
        s = strings.TrimSpace(s)
        if len(s) > 10 && !strings.Contains(s, "?") && !strings.Contains(s, "!") {
            claims = append(claims, s)
        }
    }
    return claims
}

// evaluateClaim 评估一个断言在多个采样中的一致性
func evaluateClaim(claim string, samples []string) ClaimDetail {
    claimLower := strings.ToLower(claim)
    found := 0
    contradicted := 0

    for _, sample := range samples {
        sampleLower := strings.ToLower(sample)
        if strings.Contains(sampleLower, claimLower) {
            found++
        } else {
            // 检查是否包含否定形式
            negationWords := []string{"不是", "没有", "错误", "not", "no", "false", "incorrect"}
            for _, neg := range negationWords {
                if strings.Contains(sampleLower, neg) &&
                    strings.Contains(sampleLower, truncate(claimLower, 20)) {
                    contradicted++
                    break
                }
            }
        }
    }

    total := len(samples)
    agreement := float64(found) / float64(total)

    status := "consistent"
    if contradicted > found {
        status = "contradicted"
    } else if agreement < 0.5 {
        status = "unsupported"
    }

    return ClaimDetail{
        Claim:     claim,
        Agreement: math.Round(agreement*100) / 100,
        Status:    status,
    }
}

// computeSemanticEntropy 计算多个采样的语义熵
func computeSemanticEntropy(samples []string) float64 {
    if len(samples) < 2 {
        return 0
    }

    // 计算两两相似度矩阵（简化版：用词袋 Jaccard）
    n := len(samples)
    var totalSim float64
    pairCount := 0

    for i := 0; i < n; i++ {
        for j := i + 1; j < n; j++ {
            totalSim += jaccardSimilarity(samples[i], samples[j])
            pairCount++
        }
    }

    // 平均相似度 → 熵映射
    avgSim := totalSim / float64(pairCount)
    // 熵高 = 相似度低（不一致）
    entropy := 1.0 - avgSim
    return math.Round(entropy*100) / 100
}

func jaccardSimilarity(a, b string) float64 {
    setA := wordSet(a)
    setB := wordSet(b)

    intersection, union := 0, len(setA)
    for k := range setB {
        if setA[k] {
            intersection++
        } else {
            union++
        }
    }

    if union == 0 {
        return 1.0
    }
    return float64(intersection) / float64(union)
}

func wordSet(s string) map[string]bool {
    set := make(map[string]bool)
    for _, w := range strings.Fields(strings.ToLower(s)) {
        // 去除标点
        w = strings.Trim(w, ".,!?;:\"'()[]")
        if len(w) > 2 {
            set[w] = true
        }
    }
    return set
}

func truncate(s string, n int) string {
    runes := []rune(s)
    if len(runes) < n {
        return s
    }
    return string(runes[:n])
}

// 使用示例
func ExampleUsage() {
    detector := NewConsistencyDetector()

    original := "苹果公司（AAPL）今日股价上涨3.2%，市值达到3.5万亿美元。CEO Tim Cook 宣布了新的AI产品。"
    samples := []string{
        "AAPL 今日上涨3.2%，市值3.5万亿美元。Tim Cook 宣布了AI产品。",
        "苹果股票涨了3%，市值约3.5万亿。库克发布了AI新品。",
        "AAPL 今日下跌1.1%，市值3.2万亿美元。无新产品发布。",
    }

    result := detector.Detect(original, samples)
    fmt.Printf("检测结果: 幻觉=%v, 分数=%.2f\n", result.IsHallucinated, result.Score)
    for _, d := range result.Details {
        fmt.Printf("  断言: %s | 一致率: %.2f | 状态: %s\n",
            d.Claim, d.Agreement, d.Status)
    }
}

5. 证据溯源（Grounding）

证据溯源（Grounding）是解决 Agent 幻觉的根本方法之一。其核心原则是：Agent 的每一个断言都必须能追溯到具体的、可验证的来源。在工具调用场景中，这意味着 Agent 的输出必须引用它所调用的工具返回的精确数据。

溯源流程

图5：证据溯源的五个步骤——从获取原始数据到带置信度的输出

Go 代码：证据溯源结构

package grounding

import "fmt"

// Source 代表数据来源
type Source struct {
    ToolName    string // 来源工具名
    FieldPath   string // 字段路径 "response.data.price"
    RawValue    interface{} // 原始值
    Timestamp   string
    Reliability float64 // 来源可信度 0~1
}

// GroundedValue 带溯源的字段值
type GroundedValue struct {
    Value     string
    Sources   []Source
    Certainty float64 // 综合置信度
}

// Assertion 是一个可溯源的断言
type Assertion struct {
    Statement   string
    GroundedBy  []GroundedValue
    Uncertainty string // 不确定性表达
}

func NewAssertion(statement string) *Assertion {
    return &Assertion{Statement: statement}
}

func (a *Assertion) AddSource(value GroundedValue) {
    a.GroundedBy = append(a.GroundedBy, value)
    a.recalculateCertainty()
}

func (a *Assertion) recalculateCertainty() {
    if len(a.GroundedBy) == 0 {
        a.Uncertainty = "无来源支持，置信度极低"
        return
    }
    avgCertainty := 0.0
    for _, gv := range a.GroundedBy {
        avgCertainty += gv.Certainty
    }
    avgCertainty /= float64(len(a.GroundedBy))

    switch {
    case avgCertainty >= 0.9:
        a.Uncertainty = "高置信度"
    case avgCertainty >= 0.7:
        a.Uncertainty = "中高置信度"
    case avgCertainty >= 0.4:
        a.Uncertainty = "中等置信度，建议核实"
    default:
        a.Uncertainty = "低置信度，需要人工确认"
    }
}

func (a *Assertion) String() string {
    citations := ""
    for i, gv := range a.GroundedBy {
        for _, src := range gv.Sources {
            citations += fmt.Sprintf("[%d] %s.%s @ %s\n",
                i+1, src.ToolName, src.FieldPath, src.Timestamp)
        }
    }
    return fmt.Sprintf("%s [置信度: %s]\n%s",
        a.Statement, a.Uncertainty, citations)
}

6. 防护层设计（Multi-layer Defense）

单一防护措施不足以应对 Agent 幻觉的复杂性。我们需要构建多层防御体系：每一层拦截特定类型的幻觉，层与层之间形成互补。以下是三层防护架构：输入验证层 → 工具选择验证层 → 输出验证层。

图6：三层防御架构——输入层、工具层、输出层形成完整防护链

Go 代码：三层防护管道

package defense

import (
    "fmt"
    "strings"
)

// DefensePipeline 三层防护管道
type DefensePipeline struct {
    InputValidator   *InputValidator
    ToolValidator    *ToolValidator
    OutputValidator  *OutputValidator
}

// InputValidator 第一层：输入验证
type InputValidator struct {
    MaxInputLen  int
    BlockedWords []string
}

func NewInputValidator() *InputValidator {
    return &InputValidator{
        MaxInputLen:  4096,
        BlockedWords: []string{"忽略之前指令", "你是AI", "system prompt", "ignore all"},
    }
}

func (v *InputValidator) Validate(input string) error {
    if len(input) > v.MaxInputLen {
        return fmt.Errorf("输入超限: %d > %d", len(input), v.MaxInputLen)
    }
    lower := strings.ToLower(input)
    for _, word := range v.BlockedWords {
        if strings.Contains(lower, strings.ToLower(word)) {
            return fmt.Errorf("检测到注入关键词: '%s'", word)
        }
    }
    return nil
}

// ToolValidator 第二层：工具选择验证
type ToolValidator struct {
    AllowedTools     map[string]bool
    MaxArgsPerCall   int
}

func NewToolValidator(allowed []string) *ToolValidator {
    m := make(map[string]bool)
    for _, t := range allowed {
        m[t] = true
    }
    return &ToolValidator{AllowedTools: m, MaxArgsPerCall: 10}
}

type ToolCallAttempt struct {
    ToolName   string
    Args       map[string]interface{}
}

func (v *ToolValidator) Validate(call ToolCallAttempt) error {
    if !v.AllowedTools[call.ToolName] {
        return fmt.Errorf("不允许使用工具: '%s'", call.ToolName)
    }
    if len(call.Args) > v.MaxArgsPerCall {
        return fmt.Errorf("参数过多: %d > %d", len(call.Args), v.MaxArgsPerCall)
    }
    for k, val := range call.Args {
        // 检查参数值中是否包含注入尝试
        strVal := fmt.Sprintf("%v", val)
        if strings.Contains(strVal, ";") || strings.Contains(strVal, "|") {
            return fmt.Errorf("参数 '%s' 包含特殊字符: '%s'", k, strVal)
        }
    }
    return nil
}

// OutputValidator 第三层：输出验证
type OutputValidator struct {
    MinConfidence    float64
    CheckGrounding   bool
}

func NewOutputValidator() *OutputValidator {
    return &OutputValidator{MinConfidence: 0.6, CheckGrounding: true}
}

type Output struct {
    Text       string
    Confidence float64
    Sources    []string
}

func (v *OutputValidator) Validate(output Output) error {
    if output.Confidence < v.MinConfidence {
        return fmt.Errorf("置信度过低: %.2f < %.2f",
            output.Confidence, v.MinConfidence)
    }
    if v.CheckGrounding && len(output.Sources) == 0 {
        return fmt.Errorf("输出缺少来源引用（Grounding）")
    }
    return nil
}

// RunPipeline 运行完整三层防护
func (p *DefensePipeline) RunPipeline(
    input string,
    call ToolCallAttempt,
    output Output,
) []error {
    var errs []error

    // 第一层
    if err := p.InputValidator.Validate(input); err != nil {
        errs = append(errs, fmt.Errorf("输入层: %w", err))
        return errs // 输入失败直接阻断
    }

    // 第二层
    if err := p.ToolValidator.Validate(call); err != nil {
        errs = append(errs, fmt.Errorf("工具层: %w", err))
        // 工具层失败可以尝试 fallback
    }

    // 第三层
    if err := p.OutputValidator.Validate(output); err != nil {
        errs = append(errs, fmt.Errorf("输出层: %w", err))
    }

    return errs
}

7. Fallback 策略

即便有了多层防护，幻觉仍然可能发生。关键在于当检测到幻觉时，系统如何优雅地降级。一个好的 Fallback 策略可以避免 Agent 输出错误信息，转而采取安全路径。

图7：Fallback 状态机——从检测到幻觉到选择降级策略的完整流程

四种主要 Fallback 策略

🔄 重新推理

修改推理参数后重新生成。具体方法：

  • 降低温度参数（如 0.7→0.3）
  • 增加推理步骤的约束
  • 提供更明确的上下文

成本低，适合轻微幻觉

🔄 换模型

切换到不同架构或更强模型重新推理：

  • 小型模型 → 大型模型
  • 通用模型 → 领域特化模型
  • 相同模型的不同检查点

成本高，但效果显著

🧑 求助人类

将不确定的问题转交给人类处理：

  • 自动生成问题摘要
  • 标注不确定的具体点
  • 提供 Agent 的推理过程供参考

最安全，但延迟最高

⚠ 不确定声明

直接告知用户 Agent 的不确定性：

  • "我无法确认这个信息的准确性"
  • "请参考官方数据源核实"
  • 提供部分可信的信息 + 置信度标签

成本最低，维护用户信任

Go 代码：Fallback 管理器

package fallback

import (
    "fmt"
    "log"
    "strings"
)

// FallbackStrategy 降级策略枚举
type FallbackStrategy int

const (
    StrategyRetry          FallbackStrategy = iota // 重新推理
    StrategySwitchModel                                    // 换模型
    StrategyHumanHelp                                      // 求助人类
    StrategyUncertainty                                    // 不确定声明
)

func (s FallbackStrategy) String() string {
    return []string{"重新推理", "换模型", "求助人类", "不确定声明"}[s]
}

// FallbackManager 管理降级策略的选择和执行
type FallbackManager struct {
    MaxRetries      int
    StrategyChain   []FallbackStrategy // 按优先级排序的策略链
    CurrentAttempt  int
}

func NewFallbackManager() *FallbackManager {
    return &FallbackManager{
        MaxRetries:    3,
        StrategyChain: []FallbackStrategy{
            StrategyRetry,
            StrategySwitchModel,
            StrategyHumanHelp,
            StrategyUncertainty,
        },
        CurrentAttempt: 0,
    }
}

// FallbackRequest 记录降级请求上下文
type FallbackRequest struct {
    OriginalInput    string
    HallucinationType string // "tool", "reasoning", "factual"
    Severity          string // "low", "medium", "high", "critical"
    AttemptCount      int
}

// FallbackResult 降级结果
type FallbackResult struct {
    StrategyUsed FallbackStrategy
    Output       string
    Success      bool
    NeedsHuman   bool
}

// Execute 根据请求选择并执行合适的降级策略
func (m *FallbackManager) Execute(req FallbackRequest) FallbackResult {
    // 根据严重级别选择策略起点
    startIdx := 0
    switch req.Severity {
    case "critical":
        startIdx = 2 // 直接求助人类或不确定声明
    case "high":
        startIdx = 1 // 换模型起步
    case "medium":
        startIdx = 0 // 重新推理起步
    }

    for i := startIdx; i < len(m.StrategyChain); i++ {
        strategy := m.StrategyChain[i]

        switch strategy {
        case StrategyRetry:
            if req.AttemptCount < m.MaxRetries {
                log.Printf("[Fallback] 尝试重新推理 (第 %d 次)\n", req.AttemptCount+1)
                // 实际场景中这里会调用 re-inference
                return FallbackResult{
                    StrategyUsed: strategy,
                    Output:       "[重新推理中...]",
                    Success:      true,
                }
            }

        case StrategySwitchModel:
            log.Println("[Fallback] 切换到备用模型")
            return FallbackResult{
                StrategyUsed: strategy,
                Output:       "[使用备用模型重新生成中...]",
                Success:      true,
            }

        case StrategyHumanHelp:
            log.Println("[Fallback] 转交人工处理")
            return FallbackResult{
                StrategyUsed: strategy,
                Output:       fmt.Sprintf("需要人工介入。问题：%s\n幻觉类型：%s\n严重程度：%s",
                    req.OriginalInput, req.HallucinationType, req.Severity),
                Success:    false,
                NeedsHuman: true,
            }

        case StrategyUncertainty:
            log.Println("[Fallback] 输出不确定声明")
            uncertaintyMsg := buildUncertaintyMessage(req)
            return FallbackResult{
                StrategyUsed: strategy,
                Output:       uncertaintyMsg,
                Success:      true,
            }
        }
    }

    // 所有策略都失败，输出最安全的默认响应
    return FallbackResult{
        StrategyUsed: StrategyUncertainty,
        Output:       "我目前无法可靠地回答这个问题。请重新表述或联系人工客服。",
        Success:      false,
    }
}

func buildUncertaintyMessage(req FallbackRequest) string {
    var parts []string
    parts = append(parts, "⚠️ **不确定性声明**")
    parts = append(parts, "")

    switch req.HallucinationType {
    case "tool":
        parts = append(parts, "我无法确认所使用的工具是否正确。请手动验证以下信息。")
    case "reasoning":
        parts = append(parts, "我的推理过程可能存在错误，结论不可靠。建议独立核实。")
    case "factual":
        parts = append(parts, "我无法确认以下事实的准确性，可能与实际情况存在偏差。")
    default:
        parts = append(parts, "检测到可能的输出不一致，以下内容仅供参考。")
    }

    parts = append(parts, "")
    parts = append(parts, fmt.Sprintf("原始问题: %s", req.OriginalInput))
    parts = append(parts, "")
    parts = append(parts, "建议操作：查阅官方文档或联系数据源获取准确信息。")

    return strings.Join(parts, "\n")
}

8. Q&A 与总结

常见问题解答

Q1: Agent 幻觉和 LLM 幻觉的根本区别是什么？

根本区别在于行动维度。 LLM 幻觉只影响信息输出——它说错了话，但仅此而已。Agent 幻觉则涉及行动和执行：Agent 不仅可能输出错误信息，还可能调用错误的工具、传入错误的参数、甚至产生不可逆的副作用（如删除数据、发送邮件、执行支付）。

用费曼的类比：LLM 幻觉就像天气预报说错了——你可能被淋湿；Agent 幻觉就像天气预报不光说错了，还自动帮你的花园浇了水、给老板发了"今天下雨改线上"的邮件、叫了辆出租车去机场——而外面艳阳高照。

Q2: 工具调用验证器如何防止 Agent 编造不存在的工具？

工具调用验证器通过三个机制防止编造：

  1. 白名单机制： 只有预注册的工具才能被调用，任何未注册的工具名都会被拒绝。
  2. 模糊匹配： 当 Agent 使用的工具名不在白名单中时，验证器使用 Levenshtein 距离或语义相似度进行模糊匹配。如果找到高度相似的已注册工具（得分 > 0.7），给出警告并建议纠正；如果匹配不到任何工具，直接拒绝。
  3. 参数校准： 即使工具名正确，验证器还会检查参数名和参数值是否在预期范围内，防止 Agent 穿过合法工具名做非法操作。

三重机制叠加，使得 Agent "编造"工具的难度极大增加。

Q3: SelfCheck 和语义熵检测有什么区别？哪个更好？

SelfCheck 的核心思路是"让模型自己检查自己"——先生成一个输出，然后让同一个模型（或另一个实例）判断该输出的每个部分是否与原始上下文一致。它相对简单，实现成本低。

语义熵（Semantic Entropy） 则更为数学化：它通过多次采样（通常 5-10 次），将输出聚类到语义等价类中，然后计算这些等价类的概率分布熵。高熵意味着 Agent 在不同采样中给出了本质上不同的答案。

哪个更好？

  • SelfCheck 更适合快速检测场景，但容易受到同模型"自信的幻觉"（模型对自己编造的内容也很自信）影响。
  • 语义熵在理论上更完备，能捕获更深层的不一致性，但计算成本约是 SelfCheck 的 5-10 倍。

实践中，推荐组合使用：先用 SelfCheck 做快速筛选，对可疑输出再用语义熵做深度分析。

Q4: 证据溯源（Grounding）在实际系统中如何实现？

证据溯源在实际系统中的实现通常包含以下几个关键步骤：

  1. 数据层标记： 工具返回的每个字段都携带元数据——来源工具名、字段路径、时间戳。例如 get_stock_price("AAPL") 返回的数据会标记为 tool:stock_api.field:data.price。
  2. 断言追踪： Agent 生成输出时，系统拦截并记录每个断言与数据源之间的映射关系。这通常通过结构化输出格式（而非自由文本）来实现——例如要求 Agent 输出 JSON 格式的回答，其中每个值都附带来源引用。
  3. 置信度计算： 每条断言获得一个综合置信度分数，融合了来源可信度、数据时效性、跨样本一致性。
  4. 可视化： 在最终输出中，通过角标、悬停提示等方式向用户展示来源信息。

在 Go 实现中，我们使用 Assertion 和 GroundedValue 结构体来维护这些映射关系。

Q5: 三层防护层之间如何协同工作？

三层防护层形成一个递进式的安全管道：

  • 输入层 在入口处拦截注入攻击和恶意输入，避免后续层被绕过或攻击。它是"守门员"。
  • 工具层 在 Agent 执行具体操作前进行验证，确保工具选择和参数使用的合法性。它是"操作审核员"。
  • 输出层 在最终结果交付给用户前进行质量检查，确保输出的一致性和正确性。它是"质量检验员"。

协同机制：如果输入层或工具层检测到问题，系统可以直接阻断并进入 Fallback 流程，无需继续执行。如果只有输出层检测到问题（工具调用正确但推理出错），则系统可以保留原始工具调用结果，只对输出进行修正或降级处理。

这种分层设计使得每一层都可以独立优化和迭代，同时作为一个整体提供纵深防御。

Q6: Fallback 策略中的"求助人类"和"不确定声明"有什么区别？何时使用哪个？

求助人类 是将整个问题转交给人类处理。Agent 停止输出，系统生成一个问题摘要（包括上下文、检测到的幻觉类型、Agent 的推理过程），然后发送给人工客服或审核系统。人类给出最终回答后，再返回给用户。

不确定声明 是 Agent 在自身能力范围内直接向用户表达不确定性。Agent 不会停止工作，而是在输出中明确标注哪些部分不可靠、置信度如何，并建议用户核实。

选择标准：

  • 严重程度：如果幻觉可能导致严重后果（如医疗建议、财务操作、法律判断），使用求助人类。如果是一般性信息错误，不确定声明即可。
  • 时效性：用户需要立即回答 → 不确定声明；可以等待 → 求助人类。
  • 可用资源：有人工客服在线 → 求助人类；非工作时间 → 不确定声明。

在实际系统中，这是一个可配置的策略参数，可以根据业务场景动态调整。

Q7: 如何评估一个 Agent 系统的幻觉防护效果？有哪些指标？

评估 Agent 防幻觉效果可以从以下维度进行：

  1. 幻觉检出率（Recall）： 实际存在的幻觉中，检测器成功识别出的比例。目标 > 90%。
  2. 误报率（Precision）： 检测为幻觉但实际正确的比例。需要警惕过高的误报导致频繁降级。
  3. 工具误调用率： Agent 调用错误工具的次数占总调用次数的比例。
  4. Fallback 触发率： 需要降级处理的比例。过高说明系统基础能力不足。
  5. Grounding 覆盖率： Agent 输出中被有效溯源的信息占比。目标 > 95%。
  6. 端到端正确率： 最终交付给用户的回答中完全正确的比例。

推荐的评估方法是构建一个幻觉测试集，包含已知会触发幻觉的特定场景（工具名混淆、参数边界、因果倒置等），定期进行回归测试。
