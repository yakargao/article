💬 案例一：即时通讯系统

📩 生活类比：邮局 vs 电话

即时通讯就像"对讲机"——你按下说话，对方立刻听到。传统的 HTTP 像"寄信"：每封信都要骑车送到邮局（建立连接）、盖上邮戳（HTTP 头）、送到对方手里再等回执。而 WebSocket 就像你和对讲机之间拉了一根专线，一直通着电，随时可以说话。

即时通讯架构 — WebSocket 长连接保持在线，消息队列异步削峰，Redis 缓存会话状态
📊 核心数据流
  1. 用户 A 与接入网关建立 WebSocket 长连接（一次握手，全双工通信）
  2. 消息发送：A 发送内容 → 接入网关解析协议（Protobuf/JSON）→ 逻辑服务器
  3. 逻辑服务器通过 一致性哈希 找到 B 所在连接节点，转发消息
  4. 消息同时写入 MySQL（历史记录）和 Redis（最近会话缓存）
  5. 如果 B 不在线 → 消息写入离线队列 → 通过 APNs/FCM 推送通知

🔑 关键设计决策

🌐
WebSocket vs 轮询

HTTP 轮询每次请求都有几百字节的 Header 开销，30秒轮询一次 × 1亿用户 = 巨大的带宽浪费。WebSocket 建立连接后 Header 仅 2 字节，实时性从秒级降到毫秒级。

📦
消息模型：推 vs 拉

"推模式"低延迟但服务端压力大；"拉模式"客户端可控但消息有延迟。微信采用推送通知 + 拉取内容：先推一条通知，客户端再拉取具体消息内容。

🔒
消息可靠性（ACK机制）

每条消息有唯一 ID，客户端收到后回 ACK。超时未 ACK 则重传，最多 3 次。类似 TCP 的确认重传机制，但应用层实现更灵活。

🗄️
消息存储策略

使用 Timeline 模型——每个会话维护一个有序消息列表。读扩散（拉模式）走 MySQL 分表，写扩散（推模式）写 Redis。离线消息存 7 天后自动清理。

⚠️ 常见坑点

坑1：WebSocket 连接风暴 — 客户端断线重连时，所有用户同时发起连接，瞬间打崩网关。解法：指数退避重连（1s→2s→4s→8s），加随机抖动。

坑2：消息乱序 — 消息走不同线程发送，抵达顺序可能乱。解法：每条消息带 sequence_id，客户端按 seq 排序显示，丢弃过期消息。

坑3："已读"标记放大 — 群聊中 1000 人，一条消息要写 1000 条已读记录。解法：只存"最新已读消息 ID"，用户维度聚合。

💳 案例二：支付系统

🏦 生活类比：自助售货机

你往售货机投币，它确认收到后才掉饮料。但万一它收了钱却没掉饮料呢？这就需要对账——每天关门后数钱，看看投币数和出货数能不能对上。幂等就像"按一下按钮出货一次"——你再怎么狂按按钮，它也只出一瓶饮料。资金安全就是售货机内部的所有记录都不能篡改，每一分钱都要有迹可循。

支付系统架构 — 幂等校验是资金安全的第一道防线，T+1 对账是最后兜底
📊 核心数据流
  1. 用户发起支付 → API 网关校验签名 + 限流
  2. 幂等校验：根据 order_id + amount 生成幂等 Key，查 Redis 是否已处理。已处理直接返回结果，防止重复扣款
  3. 订单服务创建订单（状态：待支付）→ 支付服务调外部渠道 → 渠道返回结果
  4. 支付成功：更新订单状态 + 写入流水表（事务性确保一致性）
  5. T+1 对账：每天凌晨拉取渠道对账单 vs 内部流水，逐笔勾对。不符的进入差错处理流程

🔑 关键设计决策

🔄
幂等设计

支付接口天然非幂等（一次扣款和两次扣款结果不同）。使用 全局唯一幂等 Key（用户ID+订单号+时间戳），Redis SETNX 加分布式锁，确保同一请求只处理一次。

💰
资金安全：T+1对账

内部流水与外部渠道逐笔勾对，分为：一致（正常）、长款（渠道有、我方无）、短款（我方有、渠道无）。短款自动发起补单，长款人工审核退款。

📋
分布式事务方案

采用 TCC（Try-Confirm-Cancel） 模式：Try 阶段冻结资金，Confirm 阶段实际扣款，Cancel 阶段解冻。配合事务消息（半消息+回调）实现最终一致性。

🔐
敏感数据保护

支付密码 bcrypt 加盐存储，银行卡号 AES-256 加密+掩码展示。所有操作审计日志留存 3 年以上，支持追溯每一笔资金的完整链路。

⚠️ 常见坑点

坑1：重复回调 — 支付渠道可能因网络问题回调多次。解法：回调处理接口必须幂等，以最终结果为准。

坑2：掉单 — 用户支付成功但系统未收到回调。解法：主动查询 + 定时任务补偿（每分钟扫一次"支付中"状态的订单，主动问渠道）。

坑3：对账差异 — 时间差导致跨天对账不平。解法：对账窗口加宽到 T+1，允许 24 小时内的时间差，同时保留"在途"标记。

🔗 案例三：短链接系统

📎 生活类比：快递取件码

快递柜给你一个长长的地址"XX省XX市XX区XX小区3号楼快递柜第24格"，但你只需要一个 6位取件码 "A3B2C9" 就能拿到包裹。短链接就是这个道理——把一长串 URL 映射到一个短字符串，Base62 就是取件码的字母表（0-9 a-z A-Z = 62个字符），6位 Base62 能编码 568 亿个链接，够用很久。

短链接系统架构 — 发号器生成唯一 ID → Base62 编码 → Redis 缓存加速 → 布隆过滤器防遍历攻击
📊 核心数据流
  1. 生成短链接：用户传入长链接 → 发号器生成全局唯一 ID（雪花算法/Redis INCR）→ Base62 编码得到 6-7 位短码 → 存 MySQL（短码→长链）+ Redis 缓存
  2. 访问短链接：用户点击短链接 → DNS 解析到服务器 → 查 Redis 缓存（命中返回）→ 查 MySQL（未命中）→ 返回 302 重定向到原 URL
  3. 统计：每次访问异步记录日志（点击时间、IP、User-Agent、Referer）→ 写入 Kafka → 离线聚合分析（PV/UV/IP 分布）

🔑 关键设计决策

🔤
Base62 vs MD5 截取

MD5 截取有碰撞风险（4位 MD5 的生日攻击碰撞概率约 1%）。Base62 编码采用发号器模式，ID 递增天然唯一，无碰撞。6位 Base62 = 62⁶ ≈ 568 亿，够用了。

⚡
302 vs 301 重定向

301 永久重定向会被浏览器缓存，后续请求不再经过服务器，无法统计。使用 302 临时重定向，每次访问都经过服务器，可做点击统计和风险控制。

🛡️
防恶意攻击

布隆过滤器快速判断短码是否存在（不存在直接 404，不用查库）。限制单 IP 生成频率（每分钟最多 100 个），加验证码防刷。

📊
统计设计

访问日志走 Kafka 异步写入，不阻塞主流程。离线用 Spark/Flink 做小时级聚合，实时展示 PV/UV。存原始日志 30 天，聚合结果永久保存。

⚠️ 常见坑点

坑1：ID 冲突 — 多机部署发号器可能生成重复 ID。解法：雪花算法（机器ID+时间戳+序列号），或 Redis 集群的 INCR 命令。

坑2：热点短链接 — 一条短链接被微博大 V 转发，百万 QPS 打过来。解法：Redis 缓存 + 本地缓存（Caffeine/Guava Cache）多级缓存，避免直穿 DB。

坑3：恶意遍历 — 遍历短码猜链接内容。解法：布隆过滤器挡无效请求，对可疑 IP 限流，敏感链接加访问密码。

⚡ 案例四：秒杀系统

🎪 生活类比：限量版球鞋发售

1000 双限量球鞋，10 万人抢购——这就是秒杀。如果让 10 万人同时涌进实体店，店门都会被挤爆。所以你要：静态化（提前把商品信息贴门口，不用进店也能看）→ 削峰（设排队区，一波一波放人进去）→ 防超卖（POS 机每扫一次减 1 件库存，数字绝对不准多）。

秒杀系统架构 — CDN 静态化 → 限流削峰 → Redis 预减库存 → 异步落单，层层过滤保 DB
📊 核心数据流
  1. 秒杀前：商品详情页 静态化 到 CDN，用户不经过源站就能看详情，避免把服务器打爆
  2. 用户点击"抢购" → 限流网关 做第一层过滤（令牌桶算法，每秒放行 1000 个请求）
  3. 通过限流的请求进入 削峰队列（Redis List / Kafka），排队等待处理
  4. 库存服务从队列消费：Redis 预减库存（DECR 原子操作），库存 > 0 才继续
  5. 预减成功 → 异步生成订单（写入订单表），库存不足则返回"已售罄"
  6. 最终异步将 Redis 库存同步回 MySQL（定时批量更新，避免行锁争抢）

🔑 关键设计决策

🚫
防超卖：Redis 原子减

库存扣减用 Redis DECR 或 Lua 脚本，保证原子性。判断 DECR 返回值 ≥ 0 才算成功。不用 MySQL 行锁（UPDATE ... WHERE stock > 0），因为 InnoDB 行锁在万级并发下性能急剧下降。

⏳
削峰填谷：请求队列化

10 万 QPS 压过来，直接处理肯定打崩。用消息队列或 Redis List 先排队再处理，服务端以可控速率消费。用户体验上展示"排队中"，实际处理可能延迟 1-2 秒。

📄
动静分离：页面静态化

秒杀页面的商品信息（名称、价格、图片）很少变化，提前渲染成 HTML/CSS/JS 推到 CDN。动态部分（剩余库存）走独立的 JS 接口异步拉取，大幅降低源站压力。

🎫
分层限流策略

三层限流：网关层（Nginx 限流模块，按 IP/UserID 限流）→ 应用层（令牌桶，限制总 QPS）→ 服务层（信号量，限制同时处理数）。超过限制直接返回"繁忙，请稍后再试"。

⚠️ 常见坑点

坑1：库存超卖 — 不加锁的 "先查后扣" 模式，两个请求读到同一库存量。解法：Redis 原子 DECR + Lua 脚本保证操作不可分割。

坑2：重复下单 — 用户疯狂点击导致多次扣库存。解法：前端防抖 + 后端接口幂等（用户ID+活动ID 做唯一约束）。

坑3：库存 DB 不一致 — Redis 库存扣了但 MySQL 事务回滚。解法：Redis 预减只是"占位"，最终以 MySQL 订单生成为准，定时对账补偿（Redis 与 MySQL 库存差异 ≤ 偏差阈值时自动回滚 Redis）。

📰 案例五：信息流系统

📬 生活类比：班级通知黑板 vs 私信

推模式就像班主任在教室黑板上写通知——写一次，全班人都能看到（适合大 V 发微博）。拉模式就像你看信箱——每封信只属于你一个人，别人看不了（适合朋友圈）。推拉结合就是班主任写黑板通知，但你在外地看不到，于是班长单独用微信告诉你——大 V 写一次推给所有在线粉丝，不在线的粉丝上线后再拉取。微博的 Feed 就是这样做的。

信息流架构 — 推拉结合策略：粉丝数 ≤ 10万 写扩散（推），粉丝 > 10万 读扩散（拉）
📊 核心数据流
  1. 内容发布：用户发微博/帖子 → 内容写入 DB → 触发 Feed 分发流程
  2. 推拉策略决策：获取粉丝数。≤ 10万用写扩散（推模式）：把内容写入每个粉丝的 Feed 收件箱（Redis ZSET）。> 10万用读扩散（拉模式）：仅存内容，粉丝刷新时拉取关注列表的最新内容
  3. Feed 排序：用户拉取 Feed 时，排序引擎综合时间 + 热度 + 亲密度计算得分，返回 Top N 条内容
  4. Feed 刷新：支持下拉刷新（拉最新）和上滑加载更多（拉历史）。走缓存优先，Cache Miss 才查 DB

🔑 关键设计决策

📨
推 vs 拉的取舍

纯推：发一条微博写百万次收件箱，僵尸粉丝浪费大量存储。纯拉：百万粉丝的大 V 发帖，粉丝同时刷新打崩查询。推拉结合：大部分普通用户推，大 V 粉丝拉，平衡写放大和读放大。

📊
Feed 排序算法

不按时间倒序简单排列，采用 Score = α×新鲜度 + β×热度 + γ×亲密度。新鲜度 = 时间衰减函数，热度 = 点赞/评论/转发数，亲密度 = 历史互动频率。参数 α/β/γ 通过 AB 测试调优。

⚡
冷启动加速

新用户没有关注列表 → 根据注册时选的兴趣标签（科技/体育/娱乐），自动推送各领域 Top 内容。同时收集用户前 10 次交互行为，快速建模兴趣偏好。

🗑️
Feed 去重

用户可能被不同好友转发同一条内容（重复展示）。为每条内容生成唯一 ID，用户维度维护已读 ID 集合（布隆过滤器），排序时过滤掉已读内容。

⚠️ 常见坑点

坑1：写扩散风暴（100万粉丝大V发帖） — 如果小看大 V 粉丝数，给所有粉丝都推，瞬间写入量爆炸。解法：阈值判断，超过 10 万粉丝切换为拉模式。

坑2：僵尸粉丝刷量 — 大量僵尸号关注大 V，推模式下写无效数据。解法：记录粉丝活跃度，低活跃粉丝不推送，上线时再拉取。

坑3：Feed 空洞 — 用户刷新看到的是 2 小时前的内容（因为最近关注的人没发帖）。解法：插入补全策略——当前关注列表内容不足时，填充热门内容和推荐内容。

❓ 高频实战问答

Q1：WebSocket 和 HTTP 长轮询的区别？什么场景选哪个？

WebSocket：建立连接后全双工通信，Header 小（2-10 字节），实时性高（毫秒级），服务端可主动推送。

HTTP 长轮询：客户端发起请求，服务端挂起直到有新数据或超时返回。每次请求都有几百字节 Header 开销，实时性受轮询间隔限制。

选型：即时通讯、在线协作、实时行情 → WebSocket。简单的消息通知、兼容老旧设备 → 长轮询。Facebook 早期用长轮询，后来切到 WebSocket。

Q2：消息队列在系统设计中的作用？Kafka vs RabbitMQ 怎么选？

作用：异步解耦（把强依赖变弱依赖）、削峰填谷（平摊突发流量）、广播（一对多消费）、可靠通信（消息持久化 + ACK）。

Kafka：高吞吐（百万条/秒）、持久化到磁盘、分区顺序、适合日志收集/流处理/大数据场景。

RabbitMQ：低延迟、灵活的路由（Topic/Direct/Fanout）、消息确认机制完善、适合业务消息/任务队列。

秒杀系统用 Kafka 削峰，支付系统用 RabbitMQ 做可靠分布式事务通知。

Q3：Redis 在系统设计中一般承担哪些角色？

① 缓存层：加速热点数据访问，保护 MySQL（短链接查询、用户 Session）

② 分布式锁：SETNX + Redlock，保证多机互斥操作（秒杀减库存、幂等校验）

③ 计数器：INCR/DECR 原子操作（秒杀库存、PV 统计、限流计数）

④ 有序集合：ZSET 做排行榜、Feed 收件箱时间线、延迟队列

⑤ 发布订阅：轻量级消息通知、集群节点心跳

Redis 故障要能降级（本地缓存兜底），不能因为 Redis 挂了导致整个系统瘫痪。

Q4：分布式 ID 生成方案有哪些？各有什么优劣？

① UUID：128位，全球唯一，不依赖中心。缺点：太长（索引性能差）、无序、不適合做主键。

② 数据库自增：实现简单，严格递增。缺点：DB 单点瓶颈，并发低。

③ Redis INCR：高并发自增。缺点：Redis 持久化问题可能导致 ID 重复。

④ 雪花算法：64位（1bit 不用 + 41bit 时间戳 + 10bit 机器ID + 12bit 序列号）。趋势递增、高性能、无中心。缺点：时钟回拨可能导致 ID 重复（需要回拨检测+等待）。

短链接用雪花算法，秒杀订单用 Redis INCR，消息 ID 用雪花算法居多。

Q5：如何保证系统的高可用？有哪些常用手段？

① 冗余部署：多机房 + 多副本，单点故障自动切换。

② 限流降级：令牌桶/漏桶限流，超过阈值降级返回默认值（秒杀"繁忙"提示）。

③ 熔断：Hystrix/Sentinel，下游故障率超标时快速失败，防止雪崩。

④ 超时重试：合理设置超时时间（500ms-3s），重试用指数退避。

⑤ 健康检查：心跳检测 + 自动摘除故障节点。

⑥ 容量规划：压测确定系统瓶颈，预留 30%-50% 余量。

高可用不是某一个手段搞定的，是一整套故障预防 + 故障发现 + 故障恢复的体系。

Q6：秒杀系统为什么不用 MySQL 的行锁来防超卖？

MySQL UPDATE stock SET count = count - 1 WHERE id = ? AND count > 0 确实能防超卖，但问题是：

① 性能差：万级并发同时更新同一行，InnoDB 行锁排队，TPS 急剧下降（实测单行并发更新 > 5000 QPS 时延飙升到秒级）。

② 连接池耗尽：每个请求持有 DB 连接等待锁释放，连接池（通常 200-500）很快耗尽。

③ 死锁风险：多索引 + 不同顺序更新可能触发死锁。

所以秒杀的标准做法是 Redis 预减（O(1)，单线程无锁争抢）→ 异步写入 MySQL，把 MySQL 的写压力从 10万 QPS 降到 100 QPS 级别。

Q7：支付系统的最终一致性如何实现？

采用 TCC + 事务消息 组合方案：

Try 阶段：冻结用户资金（不实际扣款），状态为"处理中"。

Confirm 阶段：收到支付渠道成功回调后，执行实际扣款。如果 Confirm 失败（网络/异常），通过定时任务重试。

Cancel 阶段：超过一定时间（通常 30 分钟）未收到回调，解冻资金。

事务消息：MQ 的半消息机制——先发一条半消息，本地事务执行成功后确认消息，确认前消费者不可见。如果事务超时未确认，MQ 回查事务状态。

最终状态由对账系统兜底：T+1 对账，发现不一致走差错处理流程。三步保证：事务保障→补偿重试→对账兜底。

Q8：设计一个系统时，读者最看重什么能力？怎么准备？

最看重的是：

① 结构化思维：先讲需求分析 → 容量估算 → 核心流程 → 架构图 → 关键设计 → 坑点。有条理地讲，不要跳来跳去。

② Trade-off 意识：为什么选 Kafka 而不是 RabbitMQ？为什么用 Redis 而不是直接写 MySQL？能说出每个选择的利弊。

③ 深度：提到 WebSocket 要能说出帧格式、心跳机制、断线重连策略。提到 Redis 要能说出持久化方案、集群模式、内存淘汰策略。

④ 实战坑点：知道"理论"和"实际"的差距——"理论上线上的分布式锁听起来很美，但实际会遇到 GC 停顿导致锁超时释放"这种级别的认知。

怎么准备：每个案例找张白纸画 3 遍架构图，直到可以不用查资料默画出来。然后对着镜子讲 2 遍，让自己能 5 分钟内说清核心逻辑。
