🔗 分布式数据一致性

一句话：分布式系统里，数据一致性不是"有"和"没有"的二元选择——它是一个从"随时正确"到"最终差不多正确"的连续谱。你的工作不是追求"完全一致"，而是在一致性、可用性和性能之间找到业务能接受的平衡点。

这篇文章从 CAP 定理出发，覆盖 2PC、XA、Saga、TCC、Outbox、幂等设计 6 种主流方案，最后给你一套决策框架——"什么场景用什么方案"。

🎯 什么是一致性？——先搞清楚你在说什么

🏦 类比：一致性 = 你去银行查余额。柜台 A 说 1000，柜台 B 也说 1000，ATM 也显示 1000——这就叫一致。如果柜台 A 说 1000，ATM 说 500，你就懵了——这就是不一致。

在分布式系统中，"一致性"这个词在不同的上下文中含义完全不同。必须先把术语锚定清楚：

一致性类型核心问题关心什么典型场景
ACID 中的 C（Consistency） 事务前后数据满足所有约束 "转账后总金额不变" 单机数据库事务
CAP 中的 C（Consistency） 所有节点在同一时刻看到相同数据 "读任何副本都返回最新写入" 分布式存储、共识系统
最终一致性 如果没有新写入，最终所有副本会一致 "等一会儿就会一致" DNS、CDN、用户资料
因果一致性 有因果关系的操作按顺序看到 "先发帖再评论，看到评论时一定能看到帖" 社交网络 Feed

本文讨论的是 CAP 意义下的一致性——多个副本之间如何保持数据一致。这是分布式系统的核心难题：网络分区 + 节点故障 + 并发写入 = 一致性地狱。

📐 CAP 定理的真相：不是"三选二"那么简单

🌩️ 类比：三个朋友在不同城市打电话做决定。C = 所有人都得出相同结论；A = 任何时候都有人接电话；P = 电话线可能断。当电话线断了（P 发生），你只能在"即使联系不上也硬做决定（牺牲 C）"和"不接新电话等线路恢复（牺牲 A）"之间选择。

CAP 不是三选二，是"当 P 发生时你选 C 还是 A"

大多数人背"CAP 只能同时满足两个"，这个说法容易误导。更准确的表述：

  • 没有网络分区（P 未发生）：系统可以同时拥有 C 和 A
  • 网络分区发生（P 发生）：你必须在 C 和 A 之间选择——要么拒绝写入（选 C 牺牲 A），要么接受可能读到旧数据（选 A 牺牲 C）
CAP 定理 + PACELC 扩展：真相比"三选二"更精细

📖 更详细的 CAP 分析和分布式锁中 CAP 的实战应用，参见 分布式锁 和 DDIA 第9章。

🌈 一致性谱系：不是非黑即白

🎚️ 类比：一致性像音响的音量旋钮——不是"静音"和"最大声"两个档，而是一个连续调节。你可以根据场景把旋钮拧到你需要的精确位置。

一致性谱系：从强一致到最终一致，每一档都有对应的工程方案

🔐 两阶段提交 (2PC) 与三阶段提交 (3PC)

💒 类比：2PC = 婚礼上的"你愿意吗？"——牧师先问双方（Prepare 阶段），双方都说愿意（Vote Yes），牧师宣布"现在你们是夫妻了"（Commit）。有一个人说"不"，婚礼取消。

2PC 的两个阶段

2PC 流程：一次提交 = 两次网络往返 + 协调者单点阻塞风险

2PC vs 3PC

特性2PC3PC
阶段数Prepare → CommitCanCommit → PreCommit → DoCommit
优点逻辑简单，实现广泛引入超时机制，减少阻塞
致命缺陷协调者单点故障 → 参与者无限阻塞多一次网络往返 + 网络分区场景仍有问题
实际使用XA 事务、部分 MQ 事务消息理论方案，几乎无人实际部署
能否容忍网络分区不能不能

📦 XA 分布式事务：2PC 的工业实现

📋 类比：XA = 银行的跨行转账协议。你从工行转 1000 到建行，需要工行扣款 + 建行入账同时成功或同时失败。XA 定义了银行间通信的标准流程（2PC 的标准化实现），由交易管理器（TM）协调各银行的资源管理器（RM）。

角色干什么例子
TM 事务管理器协调整个分布式事务应用服务器的事务管理器、Seata TC
RM 资源管理器管理具体资源（通常是 DB）MySQL InnoDB、PostgreSQL、MQ Broker

XA 的核心问题——为什么互联网公司几乎不用？

✅ XA 能做到的

  • 跨数据库的原子提交
  • ACID 的严格保证
  • 标准接口（JDBC、JTA）

❌ XA 做不到的

  • 性能极差：Prepare 阶段锁住资源，事务期间所有操作阻塞
  • 单点故障：TM 宕机 → 所有参与者锁死直到 TM 恢复
  • 不支持水平扩展：数据量大时锁范围线性增长

🎯 XA 的现实定位

传统行业（银行核心系统、ERP）还在用 XA——因为他们需要绝对正确，数据量可控，可以接受停机维护。互联网公司几乎不用 XA——用 Saga、TCC、本地消息表等最终一致性方案替代。唯一例外是部分数据库中间件（如 ShardingSphere）在 XA 模式下处理简单的跨库事务。

共识算法解决的是"多个节点如何就一个值达成一致"——这是实现强一致性的基础。详情见：

📖 → 共识算法：Paxos · Raft · ZAB · Multi-Paxos 完整讲解

共识算法用于：分布式锁（etcd/ZooKeeper）、配置中心、Leader 选举、分布式数据库的元数据管理。它和本文其他方案的关系是：共识提供强一致性基础设施，Saga/TCC/Outbox 在其上构建业务级一致性。

🔄 幂等性设计：最终一致性的基石

🔘 类比：电梯按钮——按 1 次和按 100 次，结果都是去 5 楼。这就是幂等。不是所有操作天然幂等：扣款不是幂等（扣两次就双倍了），但"设置余额=100"是幂等（设多少次都是 100）。

为什么幂等性是一致性的前提？分布式系统中，网络超时意味着"不确定是否成功"——你必须重试。如果操作不幂等，重试 = 重复执行 = 数据不一致。

五种实现幂等的方式

方式原理适用场景Go 示例
唯一索引 DB 唯一约束天然去重 创建类操作（注册、下单） INSERT ... ON DUPLICATE KEY UPDATE
Token / 幂等键 客户端生成唯一 ID，服务端处理前先查是否已处理 支付、转账、任何写操作 INSERT INTO idempotent_keys(key,result)
版本号（乐观锁） 数据带 version 字段，更新时 CAS 更新操作 UPDATE SET x=?, version=version+1 WHERE id=? AND version=?
状态机 操作只能从特定状态迁移，重复执行不改变已迁移的状态 订单、工单等状态流转 UPDATE SET status='paid' WHERE id=? AND status='pending'
去重表 / Redis 请求 ID 写入 Redis（SETNX），已存在则跳过 MQ 消费、API 回调 redis.SetNX(msgID, "1", ttl)

⚠️ 幂等键的常见坑

① 幂等键本身要持久化（别存在内存里丢了）。② 幂等键要设置合理的过期时间（否则无限增长）。③ "先查幂等键再执行业务"不是原子操作——需要唯一索引兜底。④ 幂等键生成要包含业务语义（如 orderID+operation），不能是随机 UUID。

📬 Outbox 模式 + CDC：可靠消息的终极方案

📮 类比：Outbox = 邮局的发件箱。你把信投进发件箱（写入 outbox 表）就算寄出了，不用关心邮递员什么时候来取（异步发送）。CDC = 邮局的自动分拣机——它监听发件箱的变化，自动把每封信送到正确的管道（MQ Topic）。

核心问题：你要在 MySQL 里创建订单 + 发一条 Kafka 消息通知下游。如果"写 MySQL"和"发 MQ"分开做，任何一个失败都会导致不一致。Outbox 把这两个操作变成一个原子操作。

Outbox 模式流程图

Outbox + CDC：让"写DB"和"发MQ"变成一个原子操作

Outbox 模式的三种实现方式

方式原理优点缺点
轮询 Outbox 定时任务扫 outbox 表，发送后标记为已发送 简单，不依赖外部组件 延迟大（轮询间隔），扫全表压力
事务消息 RocketMQ 等支持事务消息：先发半消息（prepare），业务成功后再提交（commit/rollback） 延迟低，不用单独部署 CDC 依赖 MQ 的事务消息特性（Kafka 不支持）

📖 Saga 模式：长事务的救星

✈️ 类比：旅行社帮你订"机票+酒店+租车"套餐。订机票成功→订酒店成功→租车失败！Saga 不是回滚数据库——而是自动取消酒店（调用取消接口）和退票（调用退票接口）。这就是"补偿事务"：正向操作有对应的反向操作。

Saga 的两种实现方式

🎻 编排式 (Choreography)

每个服务做完自己的事后，发一条事件到 MQ，下一个服务监听该事件自动触发。

✅ 松耦合，无中心节点

❌ 流程散落在各服务中，难以追踪全局状态

适合：简单流程（3-5 步），服务间关系清晰

🎼 编排式 (Orchestration)

引入一个 Saga 协调器（Orchestrator），由它按顺序调用各服务，并处理补偿。

✅ 流程集中管理，易追踪和重试

❌ 协调器自身可能成为瓶颈或单点

适合：复杂流程（5+ 步），需要全局状态管理

Saga 补偿示例

// 创建订单 Saga：正向操作 + 补偿操作
Step1:  订单服务.createOrder()         → 补偿: 订单服务.cancelOrder()
Step2:  库存服务.reserveStock()        → 补偿: 库存服务.releaseStock()
Step3:  支付服务.processPayment()      → 补偿: 支付服务.refund()
Step4:  物流服务.createShipment()      → 补偿: 物流服务.cancelShipment()

// 如果 Step3 失败，按逆序执行补偿：refund() → releaseStock() → cancelOrder()

⚠️ Saga 的三大陷阱

① 补偿也可能失败：取消酒店时酒店系统挂了？需要重试机制 + 人工介入。

② 隔离性缺失：Saga 不是隔离的——Step1 完成后、Step3 失败前，Step1 的效果对外可见（订了机票，但最终可能被取消）。这在某些场景不可接受。

③ 补偿幂等：补偿操作可能被重复调用，必须保证幂等。

✂️ TCC 模式：每步都是一次"尝试-确认-取消"

🛒 类比：电商下单的"预占库存"。你下单时不直接扣库存（那会导致超卖），而是"锁定"一件库存。支付成功后才真正扣掉，支付超时或取消则释放锁。这就是 TCC 的思想——先用 Try 占位，再决定 Confirm 还是 Cancel。

TCC 的三阶段

阶段干什么关键约束库存例子
Try预留资源、检查可行性必须是"软状态"，不能直接生效 UPDATE stock SET frozen=frozen+1 WHERE available>=1
Confirm真正执行，把预留转为实际必须幂等，不能依赖 Try 的上下文 UPDATE stock SET available=available-1, frozen=frozen-1
Cancel释放预留资源，回滚 Try必须幂等，不能假设 Confirm 没执行过 UPDATE stock SET frozen=frozen-1 WHERE frozen>0

TCC vs Saga vs 2PC

特性TCCSaga2PC
资源锁定Try 阶段锁定不锁定，直接执行Prepare 阶段锁定
隔离性较好（Try 锁定资源）差（中间状态对外可见）好（Prepare 锁定到 Commit）
性能中（Try 短暂锁定）高（无锁）低（全程锁）
实现复杂度高（3 个接口 + 幂等）中（2 个接口 + 幂等）低（标准接口）
适用场景资金/库存等需要"预留"的场景长流程、跨服务协调强一致性要求、数据量小

💡 TCC 的核心价值不是"比 Saga 更一致"，而是"用 Try 的短暂锁换来更高的业务正确性"——关键是 Try 的锁定时间要短（秒级），否则退化成 2PC。

❓ 高频考题

Q1: 转账场景（A 减 100，B 加 100），在分布式环境下如何保证一致性？
分层回答：
① 如果 A 和 B 在同一个 DB → 本地事务 ACID 保证。
② 如果 A 在 MySQL，B 在 Redis → 需要最终一致性方案。
③ 推荐方案：Outbox + Saga。A 服务：扣款 + 写入 outbox 表（同一事务）。CDC 捕获 outbox → 发送"转账成功"事件 → B 服务消费事件 → 入账（幂等处理）。如果 B 入账失败 → 补偿：A 退款（调用退款接口 + 写入新的 outbox 事件）。
④ 不可用 2PC：Redis 不支持 XA 协议。
Q2: Outbox 模式中，CDC 组件挂了怎么办？消息会丢吗？
不会丢。CDC（如 Debezium）的核心设计是从 binlog 的某个 offset 开始读取。即使 CDC 挂了重启，它只需要从上次记录的 binlog offset 继续读——binlog 本身就是持久化的 WAL。唯一的问题是延迟：CDC 挂了 → 消息暂时不投递 → 下游看不到新数据，但恢复后所有积压的消息都会投递。真正的风险是 outbox 表的数据无限增长——需要定期归档已投递的记录。
Q3: TCC 的 Try 阶段锁定库存太久怎么办？
这是 TCC 的核心难点。解法：① 设置 Try 超时——超时未 Confirm 则自动 Cancel，释放预留。② Try 不做真正的锁定——只是插入一条"预占记录"（不实际修改 available 字段），Confirm 时才校验并扣减。③ 分层设计——Try 只做基础检查（库存 > 0），Confirm 时再做精确扣减 + 乐观锁 CAS。④ 如果业务允许，不用 TCC 用 Saga——下单时不锁库存，发货时才校验+扣减，超卖的发货失败退款。
Q4: 分布式事务和本地事务在性能上差多少？
数量级差异：本地事务 ~0.1ms。2PC 涉及 2 次网络往返 + 日志刷盘 → 10-100ms，差 100-1000 倍。Saga/Outbox 的延迟取决于 MQ 延迟 + CDC 延迟 → 通常 10-500ms。这也是互联网公司不用 XA 的根本原因——不是做不出来，是做了就活不下去。

💬 苏格拉底式提问 —— 检验真理解

既然最终一致性已经能覆盖 90% 的场景，为什么还要学 2PC 和强一致性？Q1
因为剩下的 10% 是"不能错"的场景。

金融行业的转账、证券交易、医疗系统的处方——这些场景的错误成本远高于性能成本。一次错误的转账可能引发法律诉讼，一次错误的处方可能致命。

但关键是区分"强一致性的范围"：不是整个系统都需要 2PC，只是核心交易链路需要。你可以在核心转账用 2PC（几十 QPS），非核心的查询、报表、通知全用最终一致性。

学 2PC 的另一个原因是：理解为什么 Internet 公司不用它。只有理解了锁阻塞 + 单点故障 + 网络分区的致命组合，你才能真正理解 TCC 的 Try 阶段为什么必须"秒级超时"。
幂等设计最大的坑不是技术上的，而是思维上的——你觉得是什么？Q2
把"幂等"当成"去重"。

很多人理解的幂等：收到重复请求 → 返回上次的结果。但真正的问题是：怎么判断两次请求是"同一个"？
简单场景可以用请求 ID 去重，但复杂场景下：
① 用户连续点了两次"下单"——两次是不同的操作（要下两个单），还是同一个操作的重复提交？
② 同样的扣款请求，第一次余额 1000 扣 500 成功，第二次余额 300 扣 500——应该返回"上次的结果（成功）"还是"余额不足"？

幂等的本质不是技术去重，而是 业务语义的清晰定义。技术实现只是最后一步。
Saga 的补偿逻辑失败后，系统会进入什么状态？这是 Saga 的阿喀琉斯之踵吗？Q3
会进入"部分完成"的中间状态——这就是 Saga 的致命弱点。

机票订了、酒店订了、租车失败 → 退酒店也失败（酒店系统挂了）→ 用户看到"机票已订 + 酒店已订"但实际套餐没下成功。

应对策略是分层防御：
① 自动重试——补偿操作失败？指数退避重试（1s→2s→4s→8s→放弃）。
② 异步补偿——重试耗尽后写入"待补偿任务"表，后台 Job 持续重试直到成功。因为补偿操作通常不依赖时效性（退票晚几分钟无所谓）。
③ 人工介入——真正死都补偿不了的，进入工单系统，人工处理。
④ 设计层面——把最容易失败的操作放在 Saga 最前面。如果租车最常失败，就先订租车，失败直接终止 Saga，避免后续的补偿地狱。
