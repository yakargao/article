目录槽（Page Directory）的二分查找流程
🔍 类比：
一本 400 页的书，你要找"锁"这个关键词。不用一页页翻——目录告诉你"锁"在第 300-308 页（槽），你翻到 300 页，在 8 页内找到。目录槽就是这个"书的目录"。
流程：
① 二分查找目录槽数组 → 定位到目标槽（最多 2B/槽 × 几百槽）→ ② 槽指向一组记录的"最大那条" → ③ 在组内顺序遍历（最多 4~8 条）→ ④ 找到目标行。
💡 页内二分查找 + 组内顺序扫描 = InnoDB 的精妙平衡。槽太少 → 组太大 → 顺序扫描慢。槽太多 → 槽数组占用空间。MySQL 自动维护这个平衡。
Page Directory 二分查找：槽数组 O(logN) + 组内 O(8) = 页内极速定位
📌 索引：图书馆怎么快速找到一本书？
📚 类比：
你去图书馆找《三体》。没有索引 → 你从第一排书架一本一本翻，翻到天黑。有索引 → 电脑告诉你"科幻区 → 刘慈欣 → 第三排第四本"，30秒搞定。

Page Directory
页内槽 + 二分查找
页内记录按主键排序，槽（Slot）是稀疏索引，每 4-8 条记录一个槽指针。页内查找时：二分定位到槽 → 顺序扫描槽内记录。

顺序插入优化
最右叶子延迟分裂
自增主键插入时，数据永远追加到最右侧叶子页。InnoDB 检测到这一模式后，不做严格 50/50 切分，而是等页真正满到阈值（默认 93.75%）才分裂。
页利用率达 ~95%，减少不必要分裂
自适应哈希索引
Adaptive Hash Index
热点数据页被频繁等值查询时，InnoDB 自动为该页构建哈希索引。后续查询直接 O(1) 定位，完全绕过 B+Tree 查找。
热点查询性能提升 5-10 倍
Change Buffer
写缓冲
当要更新的二级索引页不在 Buffer Pool 中时，不立即读盘，而是把修改记录到 Change Buffer。等下次该页被读入时再合并。
减少随机读 IO，写操作延迟大幅降低
碎片整理

回收空间，提升扫描性能

READ UNCOMMITTED
脏读 ✅ · 不可重复读 ✅ · 幻读 ✅
几乎没人用。
但它的存在不是没有道理：
📊
只读报表/数据分析：
跑一份"大概"的报表，脏读带来的微小误差可接受，但省去了加锁开销。如每日大盘概览、非关键趋势统计。
🔍

DBA 排查问题时临时

⚠️
绝对不要用在：
任何涉及金额、库存、状态的业务。看到未提交的余额 = 灾难。
一句话：
能接受"看走眼"的场景才用。
READ COMMITTED
脏读 ❌ · 不可重复读 ✅ · 幻读 ✅
PostgreSQL 和 Oracle 的默认级别。
为什么 MySQL 不用但 PG 用？
🔗
高并发 OLTP：
RC 没有 Gap Lock → 锁范围小 →
死锁概率远低于 RR
。电商下单、社交 Feed、IM 消息这类高并发写入场景，RC 比 RR 吞吐更高。
📰
不需要"事务内一致性快照"的场景：
如新闻列表翻页——第 1 页和第 2 页之间有人发了新文章？完全可以接受。
🚀
配合 Statement Binlog 时不能用 RC：
RC 下 binlog 的 STATEMENT 格式会导致主从数据不一致！所以 MySQL 默认 RR。但如果 binlog 用 ROW 格式（现代标配），RC 完全安全。
⚠️
不能用 RC 的场景：
需要在一个事务里多次读取同一行并保证不变——如"先查余额→扣款→再查余额确认"。
一句话：
追求并发性能，能接受"两次读到不同值"就用 RC。
REPEATABLE READ（MySQL 默认）
脏读 ❌ · 不可重复读 ❌ · 幻读 基本 ❌
MySQL 的默认级别，也是最"中庸"的选择：
💰
金融/交易/对账：
"事务内看到的数据必须一致"——转账时检查余额、扣款、生成流水必须在同一个快照下完成。任何涉及资金流动的系统，RR 是底线。
📋
报表/对账/批量任务：
一个事务里需要 JOIN 多张表、多条件筛选，如果用 RC，每条 SELECT 可能看到不同时间点的数据——结果可能"对不上"。RR 的快照读保证一致性。
🏦
库存扣减：
"查库存→判断够不够→减库存"——如果查库存是快照读但减库存是当前读，中间被插了一行（幻读）就超卖了。RR 的 Next-Key Lock 在你做当前读时自动锁住间隙。
⚠️
RR 的代价：
① Gap Lock 导致死锁概率上升（两个事务的间隙重叠→互相等）；② undo log 膨胀（长事务持有旧版本，导致 purge 线程无法清理）。
一句话：
需要事务内数据一致性、能接受一定死锁风险就用 RR。
SERIALIZABLE
脏读 ❌ · 不可重复读 ❌ · 幻读 ❌
最安全也最慢。MySQL 中 SERIALIZABLE = 所有 SELECT 自动变 SELECT FOR SHARE。

•
Record Lock：

•
Gap Lock：
锁两本书之间的空隙（如 5-10 之间），不准往缝里塞新书。
•

❓ 高频考题（费曼版）
Q1: 为什么用 B+Tree 而不是 Hash？
Hash 只能精确匹配，不能范围查（WHERE id > 100）。B+Tree 叶子节点有链表，范围查询一路扫过去就行。
Q2: 自增主键 vs UUID？
自增 = 新数据永远追加在最后，不会页分裂。UUID 随机插入 → 页分裂频繁 → 性能下降 + 空间碎片。另外自增主键 8 字节，UUID 字符串 36 字节，二级索引叶子节点存主键，越小越好。
Q3: 联合索引 (a,b,c)，WHERE b=2 AND c=3 AND a=1 走索引吗？
走。
优化器会自动把条件顺序调整成 a=1 AND b=2 AND c=3。关键是 a 在不在条件里，书写顺序无所谓。
Q4: VARCHAR 字段上的索引为什么可能失效？
隐式类型转换。WHERE phone=138（phone 是 VARCHAR）→ MySQL 把 phone 列转成数字来比 → 对列做函数 → 索引失效。写成 WHERE phone='138' 就没事。
Q5: RR 级别幻读彻底解决了吗？
快照读解决了，快照+当前读混用仍有。
事务 A 先 SELECT（快照读）→ B INSERT 提交 → A 再 SELECT FOR UPDATE（当前读）→ 会看到 B 插入的行。纯快照读不会幻读。
Q6: COUNT(*) 和 COUNT(1) 有区别吗？
没有。InnoDB 会选最小的索引来扫。COUNT(col) 不同——只统计 col 不为 NULL 的行。
Q7: 什么是索引下推（ICP）？
把 WHERE 条件中能用索引的部分，在引擎层就过滤掉，减少回表次数。比如索引 (name,age)，WHERE name LIKE '张%' AND age=20 → 引擎层直接过滤 age=20，不用每条都回表。
Q8: 秒杀系统怎么防超卖？
。行锁 + 条件更新，库存扣到 0 就自动不扣了。高并发再上 Redis 预扣库存 + 库存分片。

💬 苏格拉底式提问 —— 检验真理解
以下问题用来逼你追问本质。先自己想 30 秒，再点开验证。
索引为什么用 B+Tree 而不是 Hash 或 B-Tree？
Q1
Hash 的问题：
只支持等值查询，不支持范围查询（WHERE id > 100）、排序（ORDER BY）、前缀匹配（LIKE 'abc%'）。
B-Tree 的问题：
非叶子节点也存数据，导致每个节点能存的键更少，树更高，IO 次数更多。而且 B-Tree 的叶子节点没有链表连接，范围查询需要中序遍历，效率差。
B+Tree 的优势：
① 数据只存叶子节点 → 非叶子节点能存更多键 → 树更矮 → IO 更少；② 叶子节点用双向链表连接 → 范围查询直接遍历链表，O(log N + K)；③ 所有查询最终落到叶子层，查询时间稳定。
MVCC 真的能"完全避免锁"吗？
Q2
不能。
MVCC 通过 undo log 和 ReadView 实现快照读不加锁，但
写操作仍然需要锁
。
•
SELECT (快照读)：
不加锁，通过 ReadView 看到一致性快照。
•
SELECT ... FOR UPDATE (当前读)：
加行锁，读最新已提交数据。
•
UPDATE/DELETE：
始终是当前读，需要加锁。
本质：
MVCC 用"多版本"替代了"读锁"，但写-写冲突仍需锁解决。这就是为什么 RR 级别下仍然会出现死锁。
分库分表后，跨库 JOIN 怎么解决？每种方案有什么代价？
Q3
四种方案，各有代价：
①
应用层聚合：
分别查两个库，在代码里 JOIN。代价 = 网络开销 + 应用复杂度。
②
冗余字段：
把要 JOIN 的字段冗余到一张表。代价 = 数据不一致风险 + 更新双写。
③
全局表：
小表每个库都存一份。代价 = 更新需要广播到所有库。
④
异构索引（ES/Hive）：
把数据同步到搜索引擎或数仓做 JOIN。代价 = 数据延迟 + 维护两套系统。
核心思想：
分库分表本质是用"应用复杂度"换"数据库可扩展性"。没有银弹，选代价最小的。
为什么说"索引不是越多越好"？多建一个索引的真实代价是什么？
Q4
一个索引的代价包含三部分：
① 写入代价：
每次 INSERT/UPDATE/DELETE 都需要维护所有索引的 B+Tree。索引越多，写越慢。
② 空间代价：
每个索引都是一棵完整的 B+Tree，索引字段会被复制一份。大量索引可能比数据本身还大。
③ 优化器代价：
索引太多，优化器选择索引的时间变长，甚至可能选错索引（因为采样偏差）。
原则：
只为高频查询建索引，用慢查询日志和 EXPLAIN 验证索引是否被使用。未被使用的索引是纯负担。

🎙️ 口播总结
这篇文章帮你搞定 MySQL。
MySQL 就像一个餐厅后厨——连接器是迎宾员，解析器是服务员记菜单，优化器是厨师长决定怎么做最快，执行器动手做，InnoDB 是厨房设备。
索引是最核心的考点。B+Tree 为什么快？三层就能存千万数据，叶子节点有双向链表支持范围查询。聚簇索引叶子存完整数据，二级索引只存主键值，多一次回表操作。最左前缀原则就像快递地址——跳过省直接写市，邮局没法处理。索引失效最常见就四种：对列做函数、隐式类型转换、LIKE 前置百分号、负向查询。
事务和 MVCC 是第二个重点。ACID 靠 undo log 回滚、redo log 重做、MVCC 做隔离。四种隔离级别中，RR 是默认级别，通过 ReadView 快照实现可重复读。一条 SQL 在 RR 下看到的永远是事务开始时的快照。
日志系统记住两个核心：redo log 是引擎层的崩溃恢复日志，循环写；binlog 是 Server 层的主从复制日志，追加写。两阶段提交保证两者一致。
如果只能记住一个词，那就是"B+Tree"。
把 B+Tree 的结构、为什么选它、聚簇索引和二级索引的回表过程讲清楚，MySQL 这关就过了大半。
