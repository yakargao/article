📋 LangChain + LangGraph

一句话定位：LangChain 让你用搭积木的方式构建 LLM 应用。LangGraph 让你用画流程图的方式构建复杂的 Agent。

二者关系：LangChain = 瑞士军刀（单个工具很强大），LangGraph = 工厂流水线（把工具串起来，加入循环和分支）。

学习路线

阶段内容时间
入门LLM → Prompt → Chain → Tool30min
核心StateGraph → Node → Edge → Conditional Edge → Agent1h
进阶Human-in-Loop → Checkpointer → Multi-Agent → Streaming1h

🔗 LangChain：给 LLM 装上方向盘

🚗 类比：裸调 OpenAI API 就像坐在一辆没有方向盘的车上——你能让车跑，但控制不了方向。

LangChain 给这辆车装上：方向盘（Prompt模板）、导航（Chain流程）、工具箱（Tool调用）、记忆座椅（Memory）。

核心四件套

组件作用类比
LLM / ChatModel大模型接口统一封装发动机
PromptTemplate把变量注入提示词模板方向盘
Chain把多个步骤串起来 (A→B→C)流水线
Tool让 LLM 调用外部函数/API工具箱

最小可运行示例

# 1. 裸调 LLM
from langchain_openai import ChatOpenAI
llm = ChatOpenAI(model="gpt-4o")
response = llm.invoke("你好")           # 最简单用法

# 2. 加 Prompt 模板
from langchain_core.prompts import ChatPromptTemplate
prompt = ChatPromptTemplate.from_messages([
    ("system", "你是{role}专家"),
    ("user", "{question}")
])
chain = prompt | llm                     # LCEL 管道语法
result = chain.invoke({"role": "MySQL", "question": "什么是B+Tree"})

# 3. 加输出解析器
from langchain_core.output_parsers import StrOutputParser
chain = prompt | llm | StrOutputParser()  # 自动提取字符串

LCEL：LangChain 的管道语法

🔧 类比：Unix 管道的 AI 版。就像 cat file | grep error | wc -l，LCEL 让你用 | 把组件串起来。

chain = prompt | llm | output_parser    # 数据从左流到右
# 等价于：output_parser(llm(prompt(input)))

⛓️ Chain：AI 应用的流水线

🏭 类比：汽车装配流水线——工位1装发动机 → 工位2装轮胎 → 工位3喷漆。每个工位只做一件事，输出传给下一个。

Chain 就是这样的流水线：Prompt 模板 → LLM 调用 → 输出解析 → 存入数据库。

Chain 类型速查

类型场景关键词
LLMChain最简单的：提示词→LLM入门
SequentialChainA的输出是B的输入顺序执行
RouterChain根据输入内容选不同下游if-else
RetrievalQARAG：检索文档→LLM回答知识库问答
ConversationChain带记忆的多轮对话聊天机器人

RAG 实战（检索增强生成）

from langchain_community.vectorstores import Chroma
from langchain_openai import OpenAIEmbeddings

# 1. 文档切片 → 2. 向量化 → 3. 存向量库
vectorstore = Chroma.from_documents(docs, OpenAIEmbeddings())

# 4. 检索相关文档 → 5. 拼到 Prompt → 6. LLM 回答
retriever = vectorstore.as_retriever()

qa = RetrievalQA.from_chain_type(llm=llm, retriever=retriever)
qa.invoke("公司年假政策是什么？")  # 基于文档回答

🔧 Tool：给 LLM 装上"手"

🤖 类比：LLM 是一个超级聪明的大脑，但它没有手——不能查数据库、不能发邮件、不能调 API。

Tool = 给 LLM 装上机械臂。LLM 决定"该用什么工具、传什么参数"，工具执行动作，结果还给 LLM。

Tool Calling 流程：LLM 选择工具 → 执行 → 返回结果 → 生成回答

定义 Tool 的两种方式

# 方式1：@tool 装饰器（推荐）
from langchain_core.tools import tool

@tool
def get_weather(city: str) -> str:
    """查询指定城市的天气"""        # ← docstring 就是 Tool 描述！
    return f"{city}: 晴 25°C"

# 方式2：StructuredTool（更灵活）
from langchain_core.tools import StructuredTool
def search_db(query: str, limit: int = 10) -> list:
    return db.execute(query)[:limit]

search_tool = StructuredTool.from_function(
    func=search_db,
    name="search_database",
    description="在数据库中搜索，参数 query 是SQL，limit 是返回条数"
)

# 绑定工具到 LLM
llm_with_tools = llm.bind_tools([get_weather, search_tool])

🕸️ 为什么需要 LangGraph？Chain 不够吗？

🚇 类比：Chain 是"单行道"——A→B→C，只能直走。但真实 Agent 需要"地铁网络"——到了某站可以换乘、可以掉头、可以循环。

LangGraph = 用节点（Node）+ 边（Edge）画一个有向图。节点是处理逻辑，边决定去哪——支持循环和条件分支。

❌ Chain 的局限

  • 只能 A→B→C 顺序执行
  • 不能"如果XX就回到A"
  • 不能"做完B再决定去C还是D"
  • Agent 循环靠 while True 黑盒

✅ LangGraph 的能力

  • 图结构：节点+边，任意拓扑
  • 条件边：根据状态决定下一步
  • 循环：Agent 反复思考-执行
  • 持久化：任意节点可暂停/恢复
  • 流式：每个节点可实时输出

📊 StateGraph：像快递包裹一样传递状态

📦 类比：快递包裹（State）在物流中心之间流转。每个中心（Node）打开包裹→修改内容→重新打包→传给下一站（Edge）。

State 是一个字典/对象，所有节点共享并修改它。比如 {"messages": [...], "next": "tool"}。

StateGraph 流程：State 在节点间传递，条件边决定去向，支持循环

定义 State

from typing import TypedDict, Annotated
from langgraph.graph.message import add_messages

class AgentState(TypedDict):
    messages: Annotated[list, add_messages]  # add_messages = 追加而非覆盖
    next: str                                # 决定下一个节点

构建第一个图

from langgraph.graph import StateGraph, END

builder = StateGraph(AgentState)

# 1. 加节点
builder.add_node("agent", call_model)     # LLM 思考节点
builder.add_node("tools", call_tools)     # 工具执行节点

# 2. 加边
builder.set_entry_point("agent")         # 入口
builder.add_conditional_edges(             # 条件边：路由到哪
    "agent", router, {
        "tools": "tools",               # 需要工具 → 去 tools 节点
        "end": END                      # 不需要 → 结束
    }
)
builder.add_edge("tools", "agent")       # 工具执行完回到 agent 循环

graph = builder.compile()                 # 编译！

🔀 边与条件分支：Agent 的导航系统

🚦 类比：开车到路口——直行去公司、左转去超市、右转回家。条件边 = 路口的红绿灯 + 导航决策，根据当前位置（State）决定走哪条路。

三种边

类型代码何时用
普通边add_edge("A", "B")A 完了一定去 B
条件边add_conditional_edges("A", router, {"x":"B","y":"C"})A 结束后根据状态决定去 B 还是 C
入口边set_entry_point("A")图从哪个节点开始

Router 函数——条件边的"大脑"

def router(state: AgentState) -> str:
    """根据 State 中的 messages 决定下一步"""
    last_msg = state["messages"][-1]

    # LLM 返回了 tool_calls？→ 去执行工具
    if hasattr(last_msg, "tool_calls") and last_msg.tool_calls:
        return "tools"

    # 没有 tool_calls → 对话结束
    return "end"

Router 函数签名模板

(state: StateType) → str 返回的是 add_conditional_edges 映射字典里的 key。

映射字典：{"tools": "tools", "end": END, "human": "human_approval"}

🤖 构建 Agent：完整实战

🧠 类比：Agent = 大脑(LLM) + 手(Tools) + 记忆(State)。大脑思考→决定动手→看结果→再思考→直到满意。

这就是经典的 ReAct 模式：Reasoning（推理）+ Acting（行动）循环。

完整 Agent 代码（20行核心）

from langgraph.prebuilt import create_react_agent

# 最简方式：用预构建的 ReAct Agent
agent = create_react_agent(
    model=llm,
    tools=[get_weather, search_db, send_email]
)

# 运行
result = agent.invoke({
    "messages": [HumanMessage(content="查一下北京天气，如果下雨就发邮件提醒老王")]
})

# ---------- 手动构建版（理解原理） ----------
def call_model(state: AgentState):
    response = llm_with_tools.invoke(state["messages"])
    return {"messages": [response]}

def call_tools(state: AgentState):
    last_msg = state["messages"][-1]
    from langgraph.prebuilt import ToolNode
    return ToolNode(tools).invoke({"messages": [last_msg]})

graph = StateGraph(AgentState)
graph.add_node("agent", call_model)
graph.add_node("tools", call_tools)
graph.set_entry_point("agent")
graph.add_conditional_edges("agent", router, {"tools":"tools","end":END})
graph.add_edge("tools", "agent")
app = graph.compile()

👤 Human-in-the-Loop：关键操作要审批

✍️ 类比：公司报销流程——员工填单 → 主管审批 → 财务打款。Agent 也一样，发邮件、删数据、大额交易等操作，应该先"打断"让人确认。

Human-in-the-Loop：interrupt() 在关键节点前暂停
# 方式1：编译时指定中断点
app = graph.compile(interrupt_before=["tools"])

# 方式2：在节点内手动中断
from langgraph.types import interrupt
def sensitive_action(state):
    answer = interrupt("确认发邮件给老王？")  # 暂停，等待人输入
    if answer == "yes":
        send_email()

💾 Memory & Checkpointer：Agent 的存档功能

🎮 类比：玩 RPG 游戏——打 BOSS 前存档（Checkpoint），死了从这里重来，不用从开头打。Checkpointer = 每个节点做完后的自动存档。

一句话理解

加一个 checkpointer → 每次 .invoke() 后自动保存 State → 下次用同一个 thread_id 调用，从断点继续。

from langgraph.checkpoint.memory import MemorySaver

checkpointer = MemorySaver()                     # 开发环境用内存
# from langgraph.checkpoint.postgres import PostgresSaver  # 生产用 PG

app = graph.compile(checkpointer=checkpointer)

# 所有调用带上 config（thread_id 标识会话）
config = {"configurable": {"thread_id": "user-123"}}
app.invoke({"messages": [HumanMessage(content="你好")]}, config)
app.invoke({"messages": [HumanMessage(content="还记得我吗")]}, config)
# ↑ 同一个 thread_id → 自动加载之前的 State → "记得！"

支持的 Checkpointer 后端

后端场景

📡 Streaming：实时看到 Agent 在想什么

📺 类比：Chain 是"录播"——全部做完才给你看结果。Streaming 是"直播"——每做一个步骤就实时推送，用户能看到进度条。

四种 Stream Mode

Mode返回时机用途
"values"每个节点执行后，推送完整 State看每一步的完整状态
"updates"每个节点执行后，推送State 变更只看增量变化
"messages"LLM 每生成一个 token 就推送打字机效果（核心！）
"custom"节点内用 writer() 自定义推送推送进度/日志等
# 打字机效果：LLM 逐 token 输出
for chunk, metadata in app.stream(
    {"messages": [HumanMessage(content="写一首诗")]},
    config,
    stream_mode="messages"           # ← 关键参数
):
    print(chunk.content, end="", flush=True)  # 逐字输出

常用组合

# 同时用两种 mode：打字机效果 + 节点切换通知
for event in app.stream_events(input, config, version="v2"):
    kind = event["event"]
    if kind == "on_chat_model_stream":
        print(event["data"]["chunk"].content, end="")  # token
    elif kind == "on_tool_start":
        print(f"\n🔧 调用工具：{event['name']}")

🧬 Deep Agents：让 Agent 会规划、会反思、会自我纠错

🧠 类比：普通 Agent 像「打工人」——接到任务就做，做错了重来。Deep Agent 像「老工程师」——先想再做、做完检查、做错了复盘改进，下次不再犯同样的错。

核心区别：Deep Agent 在"行动"之外多了规划（Plan）、反思（Reflect）、纠错（Self-Correct）三个能力。

Deep Agent 四大模式

模式核心思路适用场景
Plan-and-Execute先制定完整计划 → 再逐步执行 → 执行中可调整复杂多步任务（写代码、做调研）
Reflexion执行后自我评估 → 总结教训 → 写进记忆 → 下次改进需要迭代优化的任务
Self-Correction输出后自检 → 发现错误 → 自动修正 → 重新输出代码生成、数学推理
Tree-of-Thought同时探索多条路径 → 评估每条 → 选择最优 → 回溯复杂决策、策略规划

🌳 架构全景图

Deep Agent 核心循环：Plan → Execute → Reflect → Self-Correct

🔷 模式一：Plan-and-Execute（规划执行）

🏗️ 类比：盖楼先画图纸——规划 Agent 拆解任务为 N 个步骤，执行 Agent 按步骤干活。干到一半发现不对劲？回到规划调整再继续。

from langgraph.prebuilt import create_react_agent
from langgraph.graph import StateGraph, END
from typing import TypedDict, Annotated

# State：包含计划列表和当前步骤
class PlanState(TypedDict):
    task: str
    plan: list[str]              # [\"步骤1\", \"步骤2\", ...]
    current_step: int            # 当前执行到第几步
    results: Annotated[list, lambda x, y: x + y]
    final_answer: str

def planner(state: PlanState):
    \"\"\"LLM 生成执行计划\"\"\"
    prompt = f\"将任务拆成 3-5 个步骤：{state['task']}\"
    response = llm.invoke(prompt)
    # 解析 LLM 输出的步骤列表
    steps = [s.strip() for s in response.content.split(\"\\n\") if s.strip()]
    return {\"plan\": steps, \"current_step\": 0}

def executor(state: PlanState):
    \"\"\"执行当前步骤\"\"\"
    step = state[\"plan\"][state[\"current_step\"]]
    result = agent.invoke({\"messages\": [step]})
    return {
        \"results\": [result],
        \"current_step\": state[\"current_step\"] + 1
    }

def should_continue(state: PlanState) -> str:
    if state[\"current_step\"] >= len(state[\"plan\"]):
        return \"summarize\"      # 全部完成，汇总
    return \"executor\"           # 继续下一步

graph = StateGraph(PlanState)
graph.add_node(\"planner\", planner)
graph.add_node(\"executor\", executor)
graph.add_node(\"summarize\", summarize)
graph.set_entry_point(\"planner\")
graph.add_edge(\"planner\", \"executor\")
graph.add_conditional_edges(\"executor\", should_continue, {
    \"executor\": \"executor\", \"summarize\": \"summarize\"
})
graph.add_edge(\"summarize\", END)
app = graph.compile()

🔷 模式二：Reflexion（反思改进）

📝 类比：学生做错题 → 看答案分析原因 → 把错题记到错题本 → 下次遇到同类题不再错。Reflexion = 给 Agent 加一个"错题本"。

class ReflexionState(TypedDict):
    task: str
    attempt: int                    # 第几次尝试
    output: str                     # 本次输出
    reflections: Annotated[list, lambda x, y: x + y]  # 教训累积

def actor(state):
    \"\"\"执行任务（带历史教训）\"\"\"
    tips = \"\\n教训：\" + \"\\n\".join(state[\"reflections\"]) \
            if state[\"reflections\"] else \"\"
    response = llm.invoke(f\"{state['task']}{tips}\")
    return {\"output\": response.content, \"attempt\": state[\"attempt\"] + 1}

def evaluator(state):
    \"\"\"评估输出质量\"\"\"
    score_prompt = f\"评估质量（0-10）：{state['output']}\"
    score = int(llm.invoke(score_prompt).content)
    return {\"score\": score}

def reflect(state):
    \"\"\"如果不满意，生成改进建议\"\"\"
    if state[\"score\"] >= 8:
        return {\"reflections\": [\"✅ 已完成\"]}  # 达标
    lesson = llm.invoke(f\"为什么质量只有{state['score']}分？给一条改进建议\")
    return {\"reflections\": [lesson.content]}

def route_after_reflect(state):
    if state[\"score\"] >= 8 or state[\"attempt\"] >= 3:
        return END
    return \"actor\"  # 带着教训重试

🧬 Reflexion 的三要素

  1. Actor（执行者）：接收任务 + 历史教训，生成输出
  2. Evaluator（评估者）：给输出打分（正确性、完整性、质量）
  3. Reflector（反思者）：如果不及格，分析失败原因，生成"教训"文本

教训以纯文本追加到 State，下次 Actor 执行时拼到 Prompt 里 → Agent 越用越聪明

🔷 模式三：Self-Correction（自我纠错）

🔍 类比：写完代码跑一下 linter——发现 bug → 自动 fix → 再跑。Self-Correction 内建了一个"质量检查器"，每次输出后自动审一遍，有问题当场改。

def generate(state):
    \"\"\"生成初始输出\"\"\"
    response = llm.invoke(state[\"task\"])
    return {\"output\": response.content}

def critique(state):
    \"\"\"自检输出中的问题\"\"\"
    check = llm.invoke(
        f\"检查以下内容是否有错误或改进空间：\\n{state['output']}\"
    )
    return {\"critique\": check.content}

def revise(state):
    \"\"\"根据自检结果修正\"\"\"
    revised = llm.invoke(
        f\"原始输出：{state['output']}\\n问题：{state['critique']}\\n请修正\"
    )
    return {\"output\": revised.content, \"revision_count\": state[\"revision_count\"] + 1}

# 流程：Generate → Critique → Revise → Critique → ... → Final
graph.add_edge(\"generate\", \"critique\")
graph.add_conditional_edges(\"critique\", needs_revision, {
    \"revise\": \"revise\",     # 有问题 → 去修正
    \"done\": END            # 完美 → 结束
})
graph.add_edge(\"revise\", \"critique\")   # 修正后再自检

🔷 模式四：Tree-of-Thought（思维树搜索）

🌳 类比：下棋时在脑子里推演——走 A 三步后是什么局面？走 B 呢？选最优的走。ToT 让 Agent 同时探索多条推理路径，评估每条的质量，选最好的继续。

# 核心思想（伪代码）
def tree_of_thought(problem, breadth=3, depth=4):
    thoughts = [{\"text\": problem, \"score\": 10}]

    for step in range(depth):
        # 1. 为每个当前思路生成 breadth 个下一步思路
        candidates = []
        for t in thoughts:
            next_steps = llm.invoke(
                f\"当前思路：{t['text']}\\n给出{breadth}个下一步思考方向\"
            )
            candidates.extend(parse_steps(next_steps))

        # 2. 评估每个候选思路的质量
        for c in candidates:
            c[\"score\"] = int(llm.invoke(
                f\"评分(0-10)：{c['text']}\"
            ).content)

        # 3. 只保留 top-breadth 个最优思路
        candidates.sort(key=lambda x: x[\"score\"], reverse=True)
        thoughts = candidates[:breadth]

        # 4. 如果某条思路得分够高，提前结束
        if thoughts[0][\"score\"] >= 9:
            return thoughts[0]

    return thoughts[0]  # 返回最优解

场景推荐模式原因

🎯 一句话总结：Deep Agents = Agent + Plan（规划）+ Reflect（反思）+ Self-Correct（纠错）。核心是把单次"思考-行动"循环升级为"规划→执行→评估→改进"的多层循环，让 Agent 从"会做"变成"会做好、会改进"。

LangGraph 天然支持这些模式——StateGraph 的循环和条件边是构建 Deep Agent 的完美基础设施。

❓ 核心题 & 最佳实践

Q1: LangChain 和 LangGraph 的关系？什么时候用哪个？
LangChain 是 LLM 应用的 SDK（提示词模板、链、工具封装）。LangGraph 是 Agent 编排框架（图结构的状态机）。简单线性流程用 LangChain；需要循环、分支、多人审批的复杂 Agent 用 LangGraph。
Q2: StateGraph 的核心概念？
State（状态字典）→ Node（处理函数）→ Edge（连接）→ Conditional Edge（条件路由）→ Compile（编译运行）。State 是全局可变的，所有 Node 读写它。Reducer 控制 State 字段的合并方式（覆盖 vs 追加）。
Q3: 怎么实现 Agent 的工具调用循环？
条件边 + Router 函数。Agent Node 调用 LLM → Router 检查是否有 tool_calls → 有就去 Tools Node → Tools Node 执行完回到 Agent Node → 循环直到 Router 返回 END。
Q4: Human-in-the-Loop 怎么实现？
两种方式：① compile(interrupt_before=["节点名"]) 在指定节点前自动暂停；② 节点内调用 interrupt("确认？") 暂停并等人工输入。暂停后 graph 状态持久化到 Checkpointer。
Q5: 多 Agent 怎么通信？
Supervisor 模式：一个主管 Agent 用条件边分派任务。Agent 之间靠 State 共享信息。Swarm 模式：用 handoff 工具直接移交控制权给另一个 Agent。推荐用 langgraph_supervisor 库。
Q6: 生产环境 Checkpointer 怎么选？
PostgresSaver。单机开发用 MemorySaver 或 SqliteSaver。生产必须用持久化存储（PostgreSQL），支持多实例共享、断点恢复、状态查询。
Q7: 怎么处理长对话 Token 爆炸？
① 在 Node 中手动裁剪 state["messages"][-20:]；② 用 TrimMessages 或 FilterMessages；③ 摘要压缩：定期调用 LLM 把历史消息总结成一段。

🦜 生产环境最佳实践清单

  1. State 用 Pydantic 不要用 TypedDict：类型安全 + 验证
  2. checkpointer 必须持久化：别用 MemorySaver 上生产
  3. 加超时和重试：LLM 调用可能超时，用 with_retry
  4. 工具执行要容错：try/except 包住每个 Tool，返回错误消息而非抛异常
  5. 限制 Agent 循环次数：compile(interrupt_after=20) 防止死循环烧钱
  6. 记录所有 State 变更：方便排查"Agent 为什么做了这个决策"
  7. Streaming 给前端：用户体验提升巨大，打字机效果是标配

💬 苏格拉底式提问 —— 检验真理解

以下问题用来逼你追问本质。先自己想 30 秒，再点开验证。

Agent 为什么用"图"而不是"链"来编排？Q1
链（Chain）的局限：Chain 是线性的 A→B→C，无法处理条件分支和循环。Agent 的核心是"根据观察决定下一步"，本质上是有环有分支的有向图。
图的优势：① 条件边 → 根据 LLM 输出路由；② 循环 → "观察-思考-行动"循环；③ 并行 → 同时调用多个工具。
一句话：Chain 是"流水线"，Graph 是"决策树+循环"。
Human-in-the-loop 为什么是必选项？Q2
三个硬理由：
① 安全边界：Agent 可能执行危险操作，关键操作前暂停等人工确认是最低成本的保险。
② 质量兜底：LLM 有幻觉率（5-15%），人类审核关键决策可大幅降低错误传播。
③ 渐进信任：先在关键节点加 human check → 运行稳定后逐步去掉 → 最终全自动。这不是"功能不完善"，而是"工程化的信任构建策略"。
Tool Calling 和手工解析 JSON 的本质区别？Q3
手工 JSON：让 LLM 输出 JSON → 解析 → 调用。JSON 不一定合法（漏括号、多逗号），需要大量容错和重试。
Tool Calling：模型被微调过，输出特殊的 function_call token，保证格式结构化、可预测。
本质：Tool Calling = 原生协议，手工 JSON = prompt hack。前者是合同，后者是祈祷。

🎙️ 口播总结

这篇文章把 LangChain 和 LangGraph 从入门到精通讲透了。

LangChain 是给 LLM 装上方向盘——Prompt 模板做输入、Chain 串流程、Tool 让 LLM 调外部函数。核心是 LCEL 管道语法，用竖线把组件串起来，数据从左流到右。

LangGraph 解决 Chain 的致命缺陷：Chain 只能直走，不能循环和分支。LangGraph 用图结构——Node 是处理节点，Edge 是连接，Conditional Edge 根据 State 决定下一步。State 就像快递包裹在所有节点之间流转。最经典的 Agent 模式就是 LLM 节点和 Tool 节点之间的无限循环：思考→调用工具→看结果→再思考，直到不需要工具为止。

进阶内容覆盖了 Human-in-the-Loop（关键操作暂停等人工审批）、Checkpointer（游戏存档、随时断点续传）、多 Agent 协作（Supervisor 主管模式）、Streaming 四种模式。

如果只能记住一个概念，那就是"StateGraph"。理解了 State→Node→ConditionalEdge→Compile 这个流程，就知道怎么用 LangGraph 构建任意复杂的 Agent 工作流。
