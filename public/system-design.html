🏛️ 系统设计方法论

系统设计方法论不是考你背答案，是考你"怎么想"。考官想看的是：你能不能把模糊的需求变成清晰的架构，能不能在约束下做合理的权衡。

本文分两部分：方法论教你怎么解题，五个经典案例带你走完完整流程。

🎯 四步解题框架

🏗️ 类比：系统设计方法论 = 盖房子——①问清楚要几室几厅（需求澄清）②算算需要多少砖（容量估算）③画户型图（高层设计）④选材料定细节（深入组件）。

黄金 45 分钟分配

时间阶段做什么禁忌
0-5min需求澄清问功能/非功能需求，明确边界别上来就画图
5-10min容量估算QPS、存储、带宽大概算一下别精确到小数点
10-25min高层设计画系统架构图，定组件和数据流别陷入一个组件细节
25-40min深入组件挑 2-3 个核心组件深挖别每个都蜻蜓点水
40-45min总结权衡复盘瓶颈、讨论扩展方案别说"这个很简单"

📋 需求澄清：不问清楚就设计 = 耍流氓

🏠 类比：客户说"我要盖个房子"。你得问：几个人住？几层？预算多少？要车库吗？要求抗震几级？不问你直接盖了个别墅，客户说"我只要个平房"——白干了。

必问三类问题

类型问题模板为什么重要
功能需求"用户能做什么？核心功能是什么？"决定做什么
非功能需求"QPS 多大？延迟要求？一致性要求？"决定怎么做
边界条件"多少用户？读写比例？数据保留多久？"决定做多大

万能提问清单（实战时直接套用）

  1. 用户量：DAU 多少？峰值 QPS？
  2. 读写比：读多写少还是写多读少？
  3. 数据量：每天产生多少数据？需要存储多久？
  4. 一致性：需要强一致还是最终一致？
  5. 延迟：P99 延迟要求多少毫秒？
  6. 可用性：几个 9？能接受多长时间的不可用？

📐 容量估算：用"信封背面"算清楚

🧮 类比：去超市买东西，你不需要算到分——知道大概 300 块就够了。系统设计估算同理：不必精确，数量级对就行。

常用数字（背下来）

操作耗时记住这个
L1 缓存访问~1 ns—
内存访问~100 ns比 L1 慢 100 倍

估算公式

// 每日活跃用户 1000 万，每人每天发 5 条，读写比 10:1
写 QPS = 10M × 5 / 86400 ≈ 580
读 QPS = 580 × 10 = 5800
峰值 QPS = 5800 × 2 = 11600

// 存储：每条消息 1KB，保存 5 年
日增量 = 10M × 5 × 1KB = 50 GB
5 年 = 50 × 365 × 5 ≈ 91 TB

关键原则：数量级对就行，考官不抠数字。

⚖️ 权衡取舍：考官最想听的部分

⚖️ 类比：买车——动力强就费油，省油就动力弱，又强又省油就贵。系统设计处处是这种不可能三角。

五个经典权衡

权衡选 A选 B怎么选
SQL vs NoSQL强一致、支持 JOIN水平扩展容易、Schema 灵活需要事务/SQL → SQL；海量+扩展 → NoSQL

🔗 案例1：设计一个短链接系统（TinyURL）

📎 类比：你把一长串网址缩成 t.cn/abc123。就像给文件起短名——原名太长记不住，短名好记好分享。

Step 1：需求澄清

  • 功能：输入长 URL → 返回短 URL；访问短 URL → 重定向到长 URL
  • QPS：写 100/秒，读 10,000/秒（读写比 100:1）
  • 存储：每天 10 亿条短链接，保存 10 年
  • 短码长度：7 位（62^7 ≈ 3.5 万亿，够用）

Step 2：核心设计

ID 生成方案对比

方案优点缺点选它？
Hash(MD5/SHA256)简单碰撞、长度不可控❌
UUID → 62 进制无碰撞22 字符太长❌
雪花 ID → 62 进制短、有序、分布式依赖机器时钟✅ 推荐
自增 ID + 62 进制最短单点瓶颈、可被遍历⚠️

Step 3：架构图

短链接系统架构：API 网关 + 无状态服务 + 雪花 ID 生成 + MySQL 分片 + Redis 缓存

Step 4：深入要点

  • 302 还是 301？临时重定向用 302（可以统计点击），永久用 301（浏览器缓存）
  • 短码预生成：ID 生成器后台预生成一批短码放内存，服务直接取，无需每次调 ID 服务
  • 布隆过滤器：读请求先查布隆，不存在直接 404，避免穿透到 DB
  • 自定义短码：用户想 t.cn/my-brand，单独一张表存，优先匹配

🚦 案例2：设计一个限流器（Rate Limiter）

🚗 类比：高速公路收费站——不管后面排多长队，闸机每 5 秒放一辆。限流器就是保护下游服务的收费站。

四种算法对比

算法原理允许突发？内存消耗场景
固定窗口每分钟固定 N 次边界突刺⭐ 极小简单场景
滑动窗口日志记每条请求时间戳精确⭐⭐⭐ 大精确限流
滑动窗口计数器前窗口×权重+当前窗口较平滑⭐ 小推荐
令牌桶固定速率放令牌✅ 支持⭐ 极小最常用
漏桶请求排队，固定速率流出❌ 削峰⭐ 极小严格平滑

分布式限流架构

// Redis + Lua 实现滑动窗口计数器（原子执行）
local key = KEYS[1]          -- rate:user:123:minute
local limit = tonumber(ARGV[1])  -- 100
local window = tonumber(ARGV[2])  -- 60 秒
local now = tonumber(ARGV[3])

redis.call('ZREMRANGEBYSCORE', key, 0, now - window)
local count = redis.call('ZCARD', key)
if count >= limit then return 0 end
redis.call('ZADD', key, now, now .. ':' .. math.random())
redis.call('EXPIRE', key, window)
return 1  -- 放行

多层限流架构

① 网关层：全局 QPS 限制（Nginx/Kong）→ ② 服务层：单服务 QPS（令牌桶中间件）→ ③ 用户层：每用户每分钟 N 次（Redis 滑动窗口）→ ④ 业务层：发短信/支付等敏感操作（独立计数器）

💬 案例3：设计一个聊天系统（WhatsApp）

📱 类比：聊天系统 = 快递 + 信箱。你发消息（寄快递），服务器帮你投递到对方信箱。对方在线直接投递（WebSocket 推送），不在线存到信箱等他来取（离线消息）。

核心抉择

问题选择原因
通信协议WebSocket双向实时、长连接、低开销
存储MySQL + Redis + 对象存储消息存 MySQL 分片，图片/视频存 S3
一对一消息服务端中转不直连，安全、离线消息支持
群聊消息队列广播一条消息 → MQ 分发给所有群成员
已读回执异步更新不阻塞消息发送，后台批量更新 last_read

架构要点

  1. WebSocket 长连接：客户端连到 WebSocket 服务器，服务端维护 userId → connection 映射。Go 用 gorilla/websocket
  2. 消息投递：发送方 → API → 消息服务 → 存 DB → 查接收方是否在线 → 在线推 WebSocket / 离线存 Redis 等拉取
  3. 群聊扩散：一个 500 人群发一条消息 → 消息服务把消息写入 MQ → MQ 广播给 500 个消费者的 WebSocket 连接
  4. 在线状态：WebSocket 心跳（每 10s），断开后标离线，通知好友

📰 案例4：设计一个信息流系统（Twitter Feed）

📬 类比：你关注了 100 个人。每个人发推文就像往你的信箱投一封信。你打开信箱（刷新 Feed），看到所有关注的人的信件按时间排序。

推模式 vs 拉模式 vs 推拉结合

模式原理读延迟写压力存储
纯推发推时写入所有粉丝的 Feed⭐ 低大 V 爆炸每用户存一份 Feed
纯拉刷 Feed 时去拉所有关注者的推文⭐⭐⭐ 高⭐ 低只存一份
推拉结合 ✅普通用户推，大 V 拉⭐ 低⭐⭐ 可控混合

推拉结合的架构

推（普通用户，粉丝 < 10 万）

用户发推 → 写入自己的推文表 → 粉丝 Feed 列表中追加推文 ID → Redis List/ZSet 存储 → 粉丝打开 App 直接读自己的 Feed 列表。

拉（大 V，粉丝 > 10 万）

大 V 发推 → 存入大 V 专属缓存 → 粉丝刷 Feed → 合并"推来的 Feed"+"拉来的大 V Feed"→ 排序 → 展示。

信息流推拉结合：普通用户推（预先写Feed） + 大V拉（刷新时拉取），各取所长

Feed 存储选型

每个用户的 Feed 只需存最近 N 条（如 1000 条），用 Redis Sorted Set——推文 ID 为 member，时间戳为 score。ZREVRANGE 按时间倒序取。

🗄️ 案例5：设计一个分布式 KV 存储（DynamoDB-like）

📕 类比：字典——每个词（key）对应一个解释（value）。分布式 KV 存储 = 把一本超厚的字典拆成多本，分别放在不同书架上，但查词时你只需要知道去哪个书架。

核心设计决策

问题选择原因
分片策略一致性哈希 + 虚拟节点加节点只迁移局部数据
数据复制N=3（每个 key 存 3 副本）平衡可靠性和成本
一致性最终一致（ hinted handoff + read repair）高可用 > 强一致
冲突解决向量时钟（Vector Clock）判断更新顺序
故障检测Gossip 协议去中心化、容错
存储引擎LSM-Tree写优化 ＞ 读优化

LSM-Tree（写优化的秘密）

📝 类比：写日记——不是每次写都翻到对应日期那页（随机写），而是在最后一页一直往下写（顺序写）。LSM-Tree 就是数据的"一直往下写"模式——先写内存 MemTable，满了刷到磁盘 SSTable，后台合并（Compaction）去重。

// LSM-Tree 写流程
写入 → MemTable (内存) → 满了 → Immutable MemTable → 刷盘 SSTable
读取 → 查 MemTable → 查 Immutable → 查 SSTable（布隆过滤器加速）
后台 → Compaction 合并多个 SSTable → 去重 + 删除过期数据

为什么选 LSM-Tree 而不是 B+Tree？

B+Tree 写是随机 IO（要找到对应页再修改），LSM-Tree 是顺序 IO（追加写）。对于写多读少的 KV 存储，顺序写比随机写快 100 倍。

💬 苏格拉底式提问 —— 检验真理解

以下问题用来逼你追问本质。先自己想 30 秒，再点开验证。

短链接用 302 还是 301？这个选择会影响什么？Q1
选 302（临时重定向），不能用 301（永久重定向）。
• 301 的问题：浏览器会缓存 301——第一次访问后，浏览器直接跳转目标 URL，不再请求短链接服务器。后果：你再也统计不到这个链接的点击量。更糟的是，如果你改了目标 URL，老用户仍然被缓存导向旧地址。
• 302 的行为：每次都会请求短链接服务器 → 可以统计点击 → 可以动态修改目标 URL → 可以做 A/B 测试。
技术之外：302 还有一个好处——可以在跳转前做风控（检查 IP、频率限制）、用户画像（记录 User-Agent、地理位置）等。301 完全跳过了服务端。
信息流为什么是"推拉结合"而不是全推或全拉？Q2
全推和全拉各有致命缺陷，结合才能互补。
全推的问题：大 V 有 1 亿粉丝，发一条动态 → 写入 1 亿个收件箱 → 写放大灾难。而且大部分粉丝根本不活跃，白白浪费存储和 IO。
全拉的问题：你关注了 1000 人，打开 App → 查询 1000 个人的最新动态 → 合并排序 → 非常慢。
推拉结合：
• 普通用户 → 推（关注者少，写入成本低）。
• 大 V → 拉（关注者太多，推送成本太高）。
• 活跃用户 → 推（保证体验）。
• 非活跃用户 → 拉（他们不常打开，推了也浪费）。
本质：这是一个"计算换存储"还是"存储换计算"的权衡。推 = 写时计算 + 存储放大，拉 = 读时计算 + 延迟放大。
限流器应该放在哪一层？网关层还是应用层？有什么区别？Q3
应该多层都放，各司其职。
网关层限流：
• 作用：保护整个系统不被单个用户/IP 打垮。
• 粒度：粗粒度——按 IP/用户 ID/API Key。
• 优点：请求在到达业务层之前就被拦截，节省下游资源。
• 缺点：无法感知业务语义（比如"这个用户虽然是高频请求，但都是轻量操作"）。
应用层限流：
• 作用：保护具体的服务资源（DB 连接池、线程池）。
• 粒度：细粒度——按 API 接口 + 用户等级 + 时间段。
• 优点：可以感知业务——VIP 用户配额更高，付费接口单独限流。
• 缺点：请求已进入业务层，消耗了部分资源。
最佳实践：网关 → 粗粒度全局限流 + 应用层 → 细粒度业务限流 + 存储层 → 连接池限流。三道防线层层过滤。

🎙️ 口播总结

这篇文章帮你拿下系统设计方法论。

系统设计方法论的核心不是背答案，是展示你的思维过程。记住四步框架：需求澄清问清楚做什么、容量估算算清楚做多大、高层设计画架构图、深入组件挑 2 到 3 个核心模块深挖。

五个经典案例覆盖了实战中最高频的题型。短链接系统的关键是雪花 ID 生成短码，读写比 100 比 1 所以强依赖缓存。限流器首选令牌桶算法，用 Redis 加 Lua 做分布式滑动窗口。聊天系统用 WebSocket 长连接做实时双向通信，群聊用消息队列做扩散。信息流系统的核心是推拉结合——普通用户推、大 V 拉，Redis Sorted Set 存 Feed 按时间排序。分布式 KV 存储用一致性哈希分片、LSM-Tree 做写优化、Gossip 协议做故障检测、向量时钟解决冲突。

实战中最打动考官的不是你"知道某个方案"，而是你能说出"为什么选这个方案而不是那个"。每个选择背后都要有理由——数据量、读写比、一致性要求、延迟要求决定了你的架构决策。

如果只能记住一个框架，那就是"需求→估算→设计→深挖"四步走。按这个节奏来，45 分钟刚好，考官也听得舒服。
