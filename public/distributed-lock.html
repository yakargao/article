🔐 分布式锁

一句话：分布式锁 = 在分布式系统中，保证同一时刻只有一个进程能执行某段代码。

核心挑战：锁要可靠（不能两个人同时拿到）+ 要自动释放（拿到锁的人挂了不能死锁）+ 要高性能（加锁不能成为瓶颈）。

🔐 什么是分布式锁？

🚻 类比：商场公共厕所——门锁就是"分布式锁"。一个人进去锁门（获取锁），用完开门（释放锁）。外面排队的人看到"有人"就等着。如果进去的人晕倒了（进程崩溃），门锁不会自动打开（死锁）。

分布式锁要解决的就是"晕倒自动开门"（超时释放）的问题。

本地锁 vs 分布式锁

类型作用范围实现失效场景
本地锁单进程内sync.Mutex多实例部署时各自有各自的锁，互不感知
分布式锁跨进程、跨机器Redis / ZK / etcd—

✅ 分布式锁必须满足的五个条件

  1. 互斥性 —— 任意时刻只有一个客户端持有锁
  2. 无死锁 —— 持有锁的客户端崩溃了，锁最终能自动释放（TTL）
  3. 容错性 —— 锁服务本身部分节点挂了，锁仍然可用
  4. 加锁和解锁必须是同一个客户端 —— 不能解别人的锁
  5. 性能 —— 加锁/解锁延迟要低，不能成为瓶颈

🔴 Redis 分布式锁：最常用的方案

🎫 类比：Redis 锁 = 抢红包——SET lock:stock "owner-A" NX EX 30。先到先得，30 秒后不管有没有用完都自动释放（防止某人不主动释放）。

基于 SETNX 的正确姿势

// ❌ 错误：SETNX + EXPIRE 不是原子的——中间崩溃锁永远不释放
// SETNX lock:key "value"  （拿到锁）
// EXPIRE lock:key 30     （设过期——但这一步可能没执行到！）

// ✅ 正确：SET NX EX 是原子操作（Redis 2.6.12+）
func TryLock(ctx context.Context, key, value string, ttl time.Duration) (bool, error) {
    // SET key value NX EX seconds —— 一条命令，原子完成
    ok, err := rdb.SetNX(ctx, key, value, ttl).Result()
    return ok && err == nil, err
}

// ✅ 解锁必须用 Lua 脚本：先检查 value 是自己，才删
const unlockScript = `
    if redis.call("GET", KEYS[1]) == ARGV[1] then
        return redis.call("DEL", KEYS[1])
    else
        return 0
    end
`
func Unlock(ctx context.Context, key, value string) error {
    return rdb.Eval(ctx, unlockScript, []string{key}, value).Err()
}

value 为什么要用唯一 ID？

假设 A 拿到锁，执行超时锁自动过期 → B 拿到锁 → A 执行完去解锁 → A 把 B 的锁删了！value 用 UUID 标识身份，解锁时判断"这把锁是不是我拿的"。

Watch Dog（看门狗）—— 续租机制

🐕 类比：你租了个车位，30 分钟到期。你每隔 10 分钟去续一次（看门狗自动续期）。这样即使你停车停了 2 小时，也不会被拖走——因为锁一直在续。

参数含义典型值
锁 TTL锁的过期时间30s
续期间隔看门狗每隔多久续一次TTL / 3（10s）
续期命令EXPIRE key 30（仅当 value 匹配）Lua 脚本保证原子

🏰 Redlock 算法：单机 Redis 不够安全

🗳️ 类比：单机 Redis 挂了 → 锁全丢了。Redlock = 在 5 台独立 Redis 上分别加锁，拿到 ≥3 台（多数派）才算成功。即使 2 台挂了，锁还在。

Redlock：在 N 个独立 Redis 上分别加锁，拿到多数派（N/2+1）才算成功

Redlock 的争议

Martin Kleppmann（《设计数据密集型应用》作者）指出：Redlock 假设时钟不会跳变，但实践中 GC 停顿、时钟漂移会导致锁提前/推迟过期。如果你需要绝对安全，用 ZooKeeper/etcd；如果性能优先且能容忍极小概率冲突，用 Redis。

🦓 ZooKeeper 分布式锁：最安全的方案

🏦 类比：银行排队叫号——每个客户取一个号码（临时顺序节点），叫到最小号的客户办业务（持有锁）。办完叫号机跳到下一个（锁释放），下一个人自动获得锁。如果有客户中途离开（会话超时），他的号码作废自动跳过。

ZK 锁的核心原理

  1. 所有客户端在同一路径下创建临时顺序节点（EPHEMERAL_SEQUENTIAL）
  2. 客户端获取所有子节点，序号最小的获得锁
  3. 其他客户端对前一个节点注册 Watcher（监听删除事件）
  4. 锁释放（主动删除）或客户端崩溃（Session 超时自动删除）→ 下一个客户端收到通知 → 重新检查是否最小

ZK 的三个优势

特性说明
无死锁临时节点 + Session 超时 → 客户端挂了锁自动释放，不依赖 TTL
公平锁顺序节点 → 先到先得，不会饥饿
强一致ZAB 协议 → CP 系统 → 不会出现两个客户端同时以为拿到锁

ZK 锁的代价

性能比 Redis 低（毫秒级 vs 微秒级），依赖较重的 ZK 集群（3-5 节点）。如果系统已经用了 ZK（如 Kafka），直接复用；否则专门为锁引入 ZK 有点重。

🔵 etcd 分布式锁：Go 生态首选

📋 类比：etcd 锁 = ZK 的现代化版本。租约（Lease）= 签了一份合同，合同到期自动失效。事务（Txn）= 签合同前先检查条件是否满足。

// etcd 分布式锁（Go 官方库）
import clientv3 "go.etcd.io/etcd/client/v3"
import "go.etcd.io/etcd/client/v3/concurrency"

cli, _ := clientv3.New(clientv3.Config{Endpoints: []string{"localhost:2379"}})

// 创建 Session（自动续租的 Lease）
session, _ := concurrency.NewSession(cli, concurrency.WithTTL(30))
defer session.Close()

// 创建分布式锁
mutex := concurrency.NewMutex(session, "/lock/my-resource")

// 阻塞获取锁
if err := mutex.Lock(ctx); err != nil { return err }
defer mutex.Unlock(ctx)
// 执行业务...

etcd 的关键特性

特性说明
Lease（租约）自动续期，客户端崩溃后 Lease 到期自动释放锁
Watch监听 key 变化，锁释放时自动通知等待者
Txn原子 CAS——"如果 key 不存在，就创建它"

🗄️ 数据库分布式锁：最简单的方案

用数据库的唯一约束或行锁实现，适合已有数据库、不想引入新组件的场景。

两种实现方式

方式SQL释放缺点
唯一索引INSERT INTO locks (key) VALUES ('order_1')DELETE WHERE key='order_1'无 TTL——必须手动释放，崩了就死锁
行锁(FOR UPDATE)SELECT * FROM locks WHERE key='x' FOR UPDATECOMMIT依赖数据库连接，连接断了锁才释放、性能差
乐观锁UPDATE locks SET version=2 WHERE id=1 AND version=1无需释放冲突多时重试成本高，适合低冲突

数据库锁的升级版：带 TTL

INSERT INTO locks (key, holder, expires_at) VALUES ('x','A',NOW()+30s)，后台定时任务清理 WHERE expires_at < NOW()。但精度受清理任务间隔影响。

💣 分布式锁的常见陷阱

陷阱1：锁过期了业务还没执行完
A 拿到锁（TTL=10s）→ 执行到 12s（GC 停顿）→ 锁已过期被 B 拿走 → A 和 B 同时执行。解法：Watch Dog 自动续期 + Fencing Token。
陷阱2：Fencing Token——即使锁过期了也不怕
每次拿到锁时服务端返回一个单调递增的 Token（如 ZooKeeper 的 zxid/cversion）。写数据时带上 Token，存储层拒绝旧 Token 的写入。即使锁被抢了，旧持有者也无法搞破坏。
Fencing Token：即使锁过期，旧持有者也无法搞破坏——存储层拒绝低 Token 写入
陷阱3：Redis 主从切换丢锁
A 在主节点拿到锁 → 主节点挂了但还没同步到从 → 从节点升主 → B 在新主又拿到同一把锁 → A 和 B 同时持有。解法：用 Redlock（多主）或用 ZK/etcd（CP 系统）。
陷阱4：时钟漂移
机器 A 时钟快了 5 秒 → A 的锁提前过期 → B 拿到锁 → A 还在执行。Redlock 依赖时钟同步，但 NTP 只能保证毫秒级精度。解法：加时钟漂移 buffer（如 TTL 扣 1-2 秒）。

🔧 Go 生产级分布式锁实现

type RedisLock struct {
    client  *redis.Client
    key     string
    value   string           // 唯一标识（UUID）
    ttl     time.Duration
    stopCh  chan struct{}
}

func NewRedisLock(client *redis.Client, key string, ttl time.Duration) *RedisLock {
    return &RedisLock{
        client: client, key: key, ttl: ttl,
        value: uuid.New().String(), stopCh: make(chan struct{}),
    }
}

func (l *RedisLock) Lock(ctx context.Context) error {
    ok, err := l.client.SetNX(ctx, l.key, l.value, l.ttl).Result()
    if err != nil || !ok { return fmt.Errorf("lock failed") }
    go l.watchdog()  // 启动看门狗自动续期
    return nil
}

func (l *RedisLock) watchdog() {
    ticker := time.NewTicker(l.ttl / 3)
    defer ticker.Stop()
    for {
        select {
        case <-ticker.C:
            l.client.Eval(ctx, renewScript, []string{l.key}, l.value, int(l.ttl.Seconds()))
        case <-l.stopCh:
            return
        }
    }
}

func (l *RedisLock) Unlock(ctx context.Context) error {
    close(l.stopCh)  // 停止看门狗
    return l.client.Eval(ctx, unlockScript, []string{l.key}, l.value).Err()
}

❓ 高频考题

Q1: 分布式锁用 Redis 有什么问题？怎么解决？
三大问题：① 主从切换丢锁 → Redlock 多主。② 锁过期业务没完 → Watch Dog 续期 + Fencing Token。③ 时钟漂移 → ZK/etcd 的 CP 方案或加时钟 buffer。如果只是简单场景（防止重复提交），单机 Redis SETNX 就够了。
Q2: Redlock 和 ZooKeeper 锁选哪个？
性能优先 → Redlock：微秒级，适合秒杀、限流。
安全优先 → ZK/etcd：绝对互斥，适合金融、库存扣减。
已有 ZK/etcd → 直接复用，不要再引入新组件。
Q3: 不用 Redis 不用 ZK，数据库能实现分布式锁吗？
能，但不推荐生产长期用。① 唯一索引 + expire 字段（需定时清理过期锁）② FOR UPDATE 行锁（依赖数据库连接，连接断开锁释放）。适合快速开发、并发量低的场景。致命弱点：没有 Watch 机制，只能轮询。
Q4: 分布式锁和本地锁能同时用吗？
可以，而且推荐。先抢本地锁（减少 Redis 压力）→ 抢到本地锁再抢分布式锁。这就是 Redisson 的做法：本地锁 = 同一 JVM 内竞争，分布式锁 = 跨 JVM 竞争。Go 中类似：先 sync.Mutex → 再 Redis SETNX。
Q5: etcd 的 Lease 和 Redis 的 TTL 有什么本质区别？
Redis TTL：客户端自己续期，不续就过期。如果客户端忘了续或崩溃，锁就丢了。
etcd Lease：服务端管理，客户端定期发心跳。客户端崩溃 → 心跳停 → Lease 过期 → 锁自动释放。服务端保证 Lease 到期一定释放，不需要客户端主动 unlock。

💬 苏格拉底式提问 —— 检验真理解

以下问题用来逼你追问本质。先自己想 30 秒，再点开验证。

Redis SETNX 获取锁真的是原子的吗？Q1
单命令是原子的：SET NX PX 一个命令同时"加锁+设过期"。
但释放锁不是：GET 锁值 → 判断是否自己的锁 → DEL 不是原子的。必须用 Lua 脚本：if redis.call("get",KEYS[1]) == ARGV[1] then return redis.call("del",KEYS[1]) else return 0 end
Redlock 解决了什么？Kleppmann 为什么说它不安全？Q2
Redlock：解决单 Redis 故障时锁丢失。N/2+1 个节点成功才算加锁。
Kleppmann 的致命一击：A 拿到锁 → GC 停顿 30s → 锁过期 → B 拿到锁 → A 的 GC 结束仍以为持有锁。分布式系统中没有可靠的时钟，多节点无法解决这个问题。
正确解法：Fencing Token。加锁时附带递增 Token，存储层拒绝旧 Token 的写入。
为什么锁是"最后的手段"？Q3
锁的本质是"串行化"，违背分布式的"并行"初衷。
无锁替代：① 幂等设计 → 同一请求多次执行结果相同；② 乐观锁 → UPDATE WHERE version=oldVersion；③ 原子操作 → Redis INCR、DB ON DUPLICATE KEY；④ 队列串行化 → MQ 单线程消费。
核心金句："先问能不能用幂等/乐观锁/CAS 解决，实在不行才上分布式锁。锁是最后一招。"

🎙️ 口播总结

这篇文章覆盖了分布式锁的四种实现方案和所有常见陷阱。

分布式锁要满足五个条件：互斥性、无死锁、容错性、加解锁同客户端、高性能。Redis 是最常用的方案——一条 SETNX 命令原子加锁，Lua 脚本判断身份后解锁，Watch Dog 看门狗自动续期防止锁过期。单机 Redis 有主从切换丢锁的风险，Redlock 用多主多数派投票来解决。

ZooKeeper 的锁是最安全的——临时顺序节点实现公平锁，客户端崩溃 Session 超时自动释放。etcd 是 Go 生态首选，Lease 租约自动续期，Watch 监听释放通知。数据库锁最简单但不推荐生产长期用。

四大陷阱务必记住：锁过期业务没执行完要用 Watch Dog 加 Fencing Token 防御、主从切换丢锁要用 Redlock 或 CP 系统、时钟漂移要加缓冲区、A 解 B 的锁要用唯一 value 标识身份。

选型口诀：性能优先用 Redis，安全第一用 etcd，已有 ZooKeeper 直接复用。
