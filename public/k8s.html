📋 总览：Kubernetes 是什么

Kubernetes（K8s）是 Google 开源的容器编排平台，是目前云原生时代的事实标准。它负责自动化容器的部署、伸缩、运维——简单说，你告诉 K8s"我要跑 3 个 Nginx 容器，CPU 使用率超过 70% 就自动加一个"，剩下的它帮你搞定。

想象你开了一家连锁餐厅：刚开始只有一家店（单体应用），你亲自管后厨、收银、清洁。

生意火了，开了 50 家分店（微服务），每家店都配后厨、收银、仓库——这时候你需要一个总部的管理系统：哪家店缺厨师就调人过去（自动调度），哪家店排队太长就加人手（弹性伸缩），哪家店机器坏了就换备用的（自愈）。

Kubernetes 就是这个"餐饮连锁管理系统"，容器就是你的厨师和服务员——它们按标准流程（Docker 镜像）工作，K8s 保证每个岗位都有人干活的节奏。

🤔 为什么需要 Kubernetes

在没有 K8s 的世界里，运维工程师的日常是：SSH 到服务器 → 拉镜像 → docker run → 祈祷不要挂。当你有 3 个服务还能忍，30 个服务就疯了，300 个服务？直接提离职。

🐌 没有 K8s

  • 手动登录每台服务器部署
  • 服务挂了必须等人去重启
  • 流量暴增只能手动加机器
  • 升级版本：先停服 → 升级 → 重启
  • 不同环境的配置靠复制粘贴
  • 端口冲突经常发生

🚀 有 K8s

  • kubectl apply -f deploy.yaml
  • K8s 自动重启/重新调度
  • HPA 自动扩缩（Horizontal Pod Autoscaler）
  • 滚动更新：零停机逐步替换
  • ConfigMap/Secret 集中管理配置
  • 每个 Pod 独立 IP + Service 隔离

没有 K8s 的运维就像开一家没有管理系统的连锁餐厅——每家店的大厨自己决定放多少盐、什么时候开业、坏了找谁修。

有了 K8s 就相当于有了总部的标准化运营系统：统一菜品配方（镜像）、统一排班调度（调度器）、统一配送（Service）、统一巡检（kubelet 健康检查）。哪家店生意好就多派厨师（弹性伸缩），哪家店厨房着火了自动从后门逃生并呼叫新厨师（自愈）。

🎯 一句话记住 K8s 的定位

Kubernetes 不是"又一种容器技术"——它是管理容器的基础设施操作系统。如果 Docker 是"打包和运行应用的箱子"，K8s 就是"管理成千上万个箱子的智能仓库系统"。

🏗️ 核心架构

Kubernetes 采用Master-Worker 架构（也叫 Control Plane + Data Plane）。一个集群通常有 1 个或多个 Master 节点（高可用部署）和多个 Worker 节点。

核心组件详解

组件一句话类比
kube-apiserver所有操作唯一入口🏢 公司前台，所有访客必须经过它
etcd分布式 key-value 存储🔐 保险柜，存着公司所有档案
kube-scheduler给 Pod 找合适的节点🧑‍💼 HR，把新人分配到最合适的部门

kube-proxy网络代理 + 负载均衡📡 快递分拣员，包裹正确送达

📦 Pod —— K8s 最小的部署单元

Pod 是 Kubernetes 中最小的可调度单元。一个 Pod 可以包含一个或多个容器（通常是一个 Pod 一个容器），它们共享网络栈和存储卷。

Pod 的英文原意是"豌豆荚"——Pod 就是豌豆荚（Pod），里面的容器就是一颗颗豌豆（container）。

豌豆荚（Pod）把几颗豌豆（容器）包在一起：它们共享同一个壳（网络命名空间——同一个 IP、端口空间）、同一份土壤（存储卷——共享文件系统）。

如果你需要两个容器"紧密协作"（例如 Nginx + sidecar 日志采集器），就放在同一个 Pod 里。它们用 localhost 就能互相通信，像哥俩住在同一个房间。

图 3：Pod 结构示意——容器共享网络和存储

Pod 的生命周期

阶段描述状态码
PendingPod 已被接受但还没启动（拉镜像中）⏳
RunningPod 已绑定到节点，所有容器已启动✅
Succeeded所有容器正常退出（Job 类任务）✅
Failed至少一个容器非零退出码❌
CrashLoopBackOff容器反复崩溃，K8s 在退避重启🔄

💡 关键理解

Pod 是临时性的——它可能随时被销毁和重建（IP 会变）。所以永远不要直接访问 Pod 的 IP！要用 Service 来暴露 Pod。

⚙️ 工作负载：Deployment → Service → Ingress

K8s 中最常见的"三件套"工作流：Deployment 管理 Pod → Service 暴露内部访问 → Ingress 暴露外部访问。

图 4：用户 → Ingress → Service → Deployment(Pods) 的请求流转

三种 Service 类型

Type访问方式场景
ClusterIP集群内部 IP 访问内部微服务互调
NodePort节点 IP + 端口访问开发测试环境
LoadBalancer云厂商 LB + 公网 IP生产环境对外暴露

🎯 核心工作流

  1. 写 YAML：定义 Deployment（要啥镜像、几个副本）、Service（如何暴露）、Ingress（域名和路径规则）
  2. kubectl apply：告诉 K8s"按这个配置干活"
  3. K8s 自动执行：创建 Pod → 调度到节点 → 启动容器 → 配置网络 → 对外暴露
  4. 持续维护：Pod 挂了？重新创建。副本不够？自动扩容。版本更新？滚动替换。

🎯 调度策略：把 Pod 放到合适的"床位"

Scheduler 给 Pod 选节点的过程就像"酒店前台给客人分配房间"——既要满足客人的硬性要求，也要考虑酒店各楼层的均衡负载。

图 5：调度流程——先筛选（硬约束），后打分（软偏好）

调度策略对比

策略类型描述示例
nodeSelector硬约束节点必须有指定标签disktype: ssd
nodeAffinity硬/软约束高级节点选择required: gpu=true
podAffinity软约束跟某类 Pod 放一起降低网络延迟
taints/tolerations排斥/容忍节点排斥不带通行证的 Pod专用 GPU 节点

想象酒店前台（Scheduler）分配房间：

nodeSelector = "客人要求必须在 3 楼以上"（筛选）
nodeAffinity = "最好能住海景房，但如果没有也行"（软偏好）
taint/toleration = "VIP 楼层在门口写着'非 VIP 请勿入'，只有 VIP 客人能刷卡进入"（排斥机制）
podAffinity = "这两位客人希望住隔壁房间"（亲和性）

🔧 配置管理：ConfigMap & Secret

在 K8s 中，配置和敏感信息应该从镜像中解耦出来。这就是 ConfigMap（非敏感配置）和 Secret（敏感信息，如密码/密钥）的作用。

📋 ConfigMap

  • 存配置文件、环境变量
  • 非敏感文本
  • 以文件挂载或 env 注入
  • Base64 编码（纯文本）
  • 例：nginx.conf、application.yml

🔒 Secret

  • 存密码、API Key、证书
  • 敏感信息
  • 以文件或 env 注入
  • Base64 编码（建议加密）
  • 例：DB 密码、TLS 证书

💡 最佳实践

千万别把密码写在镜像里！ 用 ConfigMap/Secret 的好处：改配置不用重新构建镜像，不同环境（dev/staging/prod）可以挂载不同的配置。Secret 默认只存内存（tmpfs），不会写到磁盘。

# 创建 ConfigMap
kubectl create configmap app-config --from-literal=APP_ENV=production

# 创建 Secret
kubectl create secret generic db-secret --from-literal=DB_PASSWORD=S3cur3!

# 在 Pod 中挂载
env:
  - name: APP_ENV
    valueFrom:
      configMapKeyRef:
        name: app-config
        key: APP_ENV

💾 存储：PV / PVC / StorageClass

Pod 是临时性的，但数据需要持久化。K8s 通过 PV（PersistentVolume）、PVC（PersistentVolumeClaim）、StorageClass 三层抽象来解决存储问题。

想象你去租办公室：

PV（PersistentVolume） = 房东的房源（服务器上的存储空间），有 500GB，在北京朝阳区（zone: beijing-a）。
PVC（PersistentVolumeClaim） = 你的租房需求："我要 200GB 以上、北京区域的办公室"（存储申请）。
StorageClass = "经纪人"——自动从云厂商那租新房子（动态创建 PV），比如"高性能 SSD"或"大容量 HDD"类型。

K8s 的匹配逻辑：看到 PVC 申请 → 找符合条件的 PV → 找不到就用 StorageClass 动态创建 → 绑定 → Pod 使用。

图 6：存储体系——Pod 通过 PVC 申请存储，由 PV 或 StorageClass 供应

存储场景对比

场景方案说明
临时日志emptyDirPod 删除即消失
持久化数据库PV + PVC + StatefulSetPod 重建后数据还在
多 Pod 共享读写ReadWriteMany（如 NFS）文件存储场景
高 IO 数据库Local SSDs + StorageClass性能敏感场景

⛵ Helm —— K8s 的"包管理器"

一个微服务可能有 10+ 个 YAML 文件（Deployment、Service、ConfigMap、Ingress……）。Helm 就是 K8s 的 apt-get/homebrew——把一堆 YAML 打包成可复用的 Chart，一行命令部署整个应用。

图 7：Helm Chart 结构及模板引擎工作原理

🎯 为什么要用 Helm？

  • 复用：官方 Artifact Hub 上 1000+ 现成 charts（Redis、MySQL、Nginx……）
  • 参数化：同一 chart + 不同 values = 不同环境配置
  • 版本管理：helm list / helm history / helm rollback 轻松管理版本
  • 依赖管理：一个 chart 可以依赖另一个 chart（如应用 chart 依赖 MySQL chart）

❓ 实战问答（6 题）

点击问题展开答案，先自己想想再看解析 👇

Q1：Kubernetes 的架构是怎样的？Master 和 Worker 节点各包含哪些组件？

核心思路：

K8s 采用 Master-Worker（控制面-工作节点）架构。

Master 节点：kube-apiserver（入口）、etcd（存储）、kube-scheduler（调度）、controller-manager（运行控制器）。

Worker 节点：kubelet（管理 Pod）、kube-proxy（网络代理）、容器运行时（Docker/containerd）。

加分点：

提到"所有组件只通过 API Server 通信"和"etcd 是唯一有状态的组件"。高可用部署需要多 Master + etcd 集群。

Q2：Pod 是什么？为什么 K8s 不直接调度容器？

核心思路：

Pod 是 K8s 的最小调度单元，包含一个或多个容器，共享网络命名空间和存储卷。Pod 是"逻辑主机"的概念。

不直接调度容器的原因：

  1. 资源共享：需要"紧耦合"的容器（如 Nginx + sidecar）共享 localhost 通信
  2. 原子调度：一组容器要么全在一个节点上，要么都不调度
  3. 生命周期绑定：共享网络、存储、健康检查和重启策略

一句话：Pod 是豌豆荚，容器是豌豆——几颗豌豆共享一个荚。

Q3：Deployment 和 StatefulSet 的区别是什么？

核心思路：

特性DeploymentStatefulSet
身份无状态，Pod 名字随机有状态，Pod 名字固定（-0, -1, -2）
存储所有 Pod 共享卷或临时卷每个 Pod 绑定独立 PVC
启停顺序并行启停有序启停（从-0到-n）
网络随机 IP稳定的网络标识（DNS）
场景Web / API 服务数据库 / 消息队列 / 缓存

StatefulSet 保证"每个 Pod 有固定的身份和存储"——状态敏感应用需要这种保证。

Q4：Service 有哪几种类型？ClusterIP、NodePort、LoadBalancer 分别适用于什么场景？

核心思路：

类型访问范围场景
ClusterIP集群内部微服务间内部调用（默认）
NodePort节点 IP + 端口（外网可访问）开发测试，或自建 LB 场景
LoadBalancer云厂商 LB + 公网 IP生产环境对外暴露

额外知识：Headless Service（clusterIP: None）用于有状态应用，直接返回所有 Pod IP 列表而不是负载均衡。

Q5：K8s 的调度器是如何为 Pod 选择节点的？有哪些调度策略？

核心思路：

调度分两阶段：

  1. Predicates（筛选）：过滤掉不满足硬性条件的节点（资源不足、不匹配 nodeSelector、不容忍 taint）
  2. Priorities（打分）：对剩余节点打分（资源利用率均衡、节点亲和性、Pod 亲和性）

关键策略：

  • nodeSelector / nodeAffinity：节点标签匹配
  • podAffinity / podAntiAffinity：Pod 间亲和/反亲和
  • Taints & Tolerations：节点排斥 / Pod 容忍

最终：评分最高的节点胜出，Pod 绑定到该节点。

Q6：K8s 中如何进行滚动更新和回滚？

核心思路：

滚动更新：修改 Deployment 的镜像版本 → kubectl set image 或更新 YAML → K8s 逐步创建新 Pod（ReplicaSet），销毁旧 Pod。

滚动更新策略参数：

  • maxSurge：更新期间最多超出的 Pod 数（默认 25%）
  • maxUnavailable：更新期间最多不可用的 Pod 数（默认 25%）

回滚：kubectl rollout undo deployment/nginx → K8s 把 ReplicaSet 回退到上一个版本。

注意：每个 Deployment 变更都会生成新的 ReplicaSet，所以 K8s 可以"记住"历史版本用于回滚。

Q7（附加）：ConfigMap 和 Secret 的区别是什么？如何安全使用 Secret？

核心思路：

特性ConfigMapSecret
存储内容非敏感配置敏感数据（密码、密钥）
编码明文Base64（建议额外加密）
内存存储可写磁盘默认 tmpfs（仅内存）
大小限制1MB1MB

安全使用 Secret 的最佳实践：

  • 启用 Encryption at Rest（etcd 加密）
  • 使用 External Secrets Operator 从 AWS Secrets Manager / Vault 同步
  • 开启 RBAC 限制 Secret 的访问权限
  • 不要将 Secret 提交到 Git（.gitignore + 外部管理）

Q8（附加）：Ingress 和 Service 的区别是什么？

核心思路：

Service 是第 4 层（传输层）的负载均衡，基于 IP+端口转发流量。只能在集群内部或通过 NodePort/LB 暴露。

Ingress 是第 7 层（应用层）的路由，基于域名、路径、Header 等规则转发 HTTP/HTTPS 流量。需要 Ingress Controller（如 Nginx Ingress、Traefik）作为实际执行者。

Service 是"快递分拣到楼栋"，Ingress 是"快递分拣到具体房间号"。

📌 核心要点

读者问 K8s 时最看重的不是背 API，而是理解设计哲学：

  • 声明式 vs 命令式：告诉 K8s"要什么"而不是"怎么做"
  • 控制器模式：Reconciliation Loop（实际状态 → 期望状态的死循环检查）
  • 分层抽象：Pod → Deployment → Service → Ingress，每一层解决一个问题
  • etcd 的分布式共识：Raft 协议保证数据一致性

🎙️ 口播总结

好，我们来快速捋一遍 Kubernetes 的实战知识体系。

第一，K8s 是"连锁餐厅的管理系统"——当你有几十上百个容器要管理时，手动操作是不现实的。K8s 提供自动部署、弹性伸缩、自愈、服务发现四大核心能力，核心思想是声明式 API：你告诉它"要什么状态"，它负责实现。

第二，Master-Worker 架构是 K8s 的骨架。Master 节点上的 API Server 是唯一的入口，etcd 存所有状态，Scheduler 负责调度，Controller Manager 确保系统状态符合期望。Worker 节点上的 kubelet 是"管家"，管理 Pod 的生命周期。

第三，Pod 是最小单元，相当于"豌豆荚"，里面一个或多个容器共享网络和存储。但 Pod 是临时性的，所以需要用 Deployment 来管理副本，用 Service 来暴露稳定的网络入口，用 Ingress 来做域名路由。

第四，各种高级特性：调度策略（NodeSelector、Affinity、Taint/Toleration）控制 Pod 放哪里；ConfigMap/Secret 管配置；PV/PVC/StorageClass 管持久化存储；Helm 管应用打包。

K8s 内容很多，但抓住"声明式 + 控制器 + 分层抽象"这三个设计哲学，你就能理解 80% 的 K8s 行为。实战时展示对这些理念的理解，比背一堆 YAML 字段强得多。
