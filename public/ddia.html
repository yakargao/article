Ch1 可靠性 · 可扩展性 · 可维护性

🏠 类比：盖房子三个要求——不能塌（可靠）、人多也能住（可扩展）、水管坏了容易修（可维护）。DDIA 开篇就用这三个词定义了"好系统"的标准。

🔒 可靠性 Reliability

即使出现硬件故障、软件错误、人为失误，系统仍能正常工作。

  • 硬件故障：冗余（RAID、双电源）
  • 软件错误：隔离、熔断、监控
  • 人为失误：易用API、沙箱、回滚

📈 可扩展性 Scalability

负载增长时，系统仍有合理的性能。

  • 描述负载：QPS、读写比、活跃用户
  • 描述性能：p50/p95/p99 延迟
  • 垂直扩展 vs 水平扩展

🔧 可维护性 Maintainability

维度含义
可运维性运维团队轻松：监控、部署、配置
简单性新人也看得懂：消除意外复杂度
可演化性需求变了容易改：敏捷 + TDD + 重构

🎯 核心要点

描述系统设计时从这三个维度切入：可靠性怎么保证（冗余/容错）、可扩展性怎么设计（分片/无状态）、可维护性怎么考虑（监控/CI/CD）。

Ch2 数据模型与查询语言

🗺️ 类比：关系模型是 Excel 表格（行列整齐），文档模型是 JSON 文件（嵌套自由），图模型是地铁线路图（关系为王）。选模型就是选"怎么描述世界"。

三大数据模型对比

模型代表适合场景弱点
关系模型MySQL / PostgreSQL多对多、JOIN 查询模式变更困难
文档模型MongoDB / CouchDB一对多、自包含数据多对多弱、JOIN 需应用层做
图模型Neo4j / Dgraph多对多关系密集学习曲线陡

关系模型 vs 文档模型的经典争论

文档模型优势

  • 模式灵活（schemaless）
  • 局部性更好（一次读整个文档）
  • 和 OOP 数据结构对应

关系模型优势

  • JOIN 支持（多对多查询）
  • 强 schema 约束保证数据质量
  • 几十年的查询优化器积累

查询语言：声明式 vs 命令式

-- 声明式（SQL）：说"要什么"，不管"怎么做"
SELECT * FROM users WHERE age > 18 ORDER BY name;

-- 命令式（代码）：说"怎么做"
result = []
for user in users:
    if user.age > 18:
        result.append(user)
result.sort(key=lambda u: u.name)

声明式优势：优化器可以自由选择索引、并行化、重排顺序，不会被代码实现"锁死"。

Ch3 存储与检索

📚 类比：哈希索引是"每本书放固定位置"（O(1) 查，但顺序扫全库），LSM-Tree 是"先记小本本，定期整理到大书架"（写快读慢），B-Tree 是"图书馆分类号"（读写平衡）。

存储引擎两大流派

LSM-TreeB-Tree
代表LevelDB / RocksDB / CassandraMySQL InnoDB / PostgreSQL
写性能✅ 极快（顺序写）⚠️ 随机写（页分裂）
读性能⚠️ 需查多个文件✅ O(log N) 稳定
空间放大可配置（压缩）有碎片
核心操作MemTable → SSTable → Compaction页分裂 / WAL / 缓冲池

LSM-Tree 三级结构

写入路径：
写请求 → MemTable (内存有序树，如红黑树)
     → 满了后冻结 → 刷到磁盘成 SSTable (不可变)
     → 后台 Compaction 合并多个 SSTable (去重+删 tombstone)

读取路径：
先查 MemTable → 再查 Bloom Filter → 再查各级 SSTable

B-Tree 核心机制

B-Tree 写入：
找到叶子页 → 写入 → 页满了？→ 页分裂（一分为二，父节点加指针）

崩溃恢复：
WAL (Write-Ahead Log)：先写日志，再改 B-Tree 页。崩溃后重放 WAL。

关键参数：
分支因子 ≈ 500（一个 4KB 页存 500 个指针）
→ 4 层 B-Tree 可存 500⁴ ≈ 62.5 亿条记录
LSM-Tree（左）写优化：MemTable → SSTable → Compaction | B-Tree（右）读写平衡：树形结构 + WAL + 页分裂

OLTP vs OLAP

OLTP（在线事务处理）

面向用户，少量读写

  • 行存（按行组织）
  • B-Tree / LSM-Tree
  • MySQL / PostgreSQL

OLAP（在线分析处理）

面向分析，大量扫表

  • 列存（按列组织）
  • 压缩 + 向量化
  • ClickHouse / Doris

列存为什么快？分析查询通常只涉及几列——列存只需读这几列的数据，比行存少读 90%+ 的磁盘 IO。再加上压缩（同列数据相似性高）、向量化执行，查询速度可以比行存快 100 倍。

Ch4 编码与演化

🌐 类比：两个人通信——一个说中文，一个说英文，中间需要翻译。序列化格式就是数据在网络上的"语言"。好的格式能处理"语言演变"（加字段、改类型）而不破坏通信。

编码格式演进

格式可读性Schema兼容性适用
JSON / XML人类可读无弱（无 schema 约束）REST API
Protobuf / Thrift二进制有（.proto）强（向前+向后）RPC / 微服务
Avro二进制有（动态）强（writer+reader schema）Hadoop / Kafka

向前兼容 vs 向后兼容

向后兼容 Backward

新代码读旧数据

新版本能处理旧版本写的数据。方法是：新字段必须有默认值。

向前兼容 Forward

旧代码读新数据

旧版本能忽略新版本加的字段。Protobuf 天然支持（未知字段跳过）。

Avro 的独特设计

# Avro 不存 schema 在数据里，而是分开传递
# Writer Schema: 写入时用的 schema
# Reader Schema: 读取时用的 schema
# Avro 自动做"格式转换"（字段名匹配、类型转换、默认值填充）

优点：schema 可以独立演化，数据本身不冗余

Ch5 复制 —— 让多台机器拥有相同数据

📋 类比：公司公告栏——领导（Leader）写了通知，大家（Follower）抄到自己本子上。如果有人想改通知，只能通过领导。这就是主从复制。

三种复制策略

策略原理优点缺点代表
主从复制一个 Leader 写，多个 Follower 读简单、读写分离单点写入瓶颈MySQL / MongoDB
多主复制多个 Leader 都可写，互相同步就近写入、容灾写冲突处理复杂CouchDB / GoldenGate
无主复制所有人可写，Quorum 决定高可用、无单点一致性弱、冲突多Cassandra / Dynamo

主从复制的三种同步方式

1. 同步复制：Leader 等所有 Follower 确认后才返回
   优点：强一致  缺点：一个 Follower 挂了整个系统阻塞

2. 异步复制：Leader 写完就返回，Follower 异步追
   优点：高性能  缺点：Leader 挂了可能丢数据

3. 半同步（推荐）：一个 Follower 同步，其余异步
   折中方案：至少有一个备份是同步的（保证持久化 + 可用性）

处理复制滞后：Read-After-Write 一致性

问题：用户写了一条数据，立刻去读——由于异步复制，从库可能还没收到这条数据，用户看到"旧数据"（感觉像数据丢了）。

解法：读"用户自己修改的数据"时强制走主库；其他数据可以读从库。这叫 Read-After-Write 一致性。

Ch6 分区 —— 数据太大，一台机器放不下

📦 类比：图书馆按字母分架——A-G 在一楼，H-N 在二楼，O-Z 在三楼。分区就是把数据按规则散到多台机器，每台只管自己那部分。

分区策略

策略原理优点致命缺陷
范围分区按 key 范围切（A-C, D-F...）范围查询快热点：时间戳分区，今天的数据全写一个节点
哈希分区hash(key) % N均匀分布范围查询需扫所有分区
一致性哈希哈希环 + 虚拟节点增删节点只影响相邻实现复杂

分区 + 复制 = 标配

每个分区通常在 多个节点 上存副本。例如 Kafka 的每个 Partition 有 3 个 Replica（1 Leader + 2 Follower）。

二级索引：本地 vs 全局

本地索引（文档分区）

每个分区独立建索引

写快，但读需查所有分区（scatter/gather）

全局索引（词条分区）

索引本身也分区存储

读快（只查一个分区），但写需跨分区更新索引

Ch7 事务 —— 要么全做，要么全不做

💸 类比：银行转账——从 A 扣 100，给 B 加 100。如果扣完 A 后系统崩溃了，B 没收到钱，钱去哪了？事务保证这两个操作绑在一起，要么全成功，要么全回滚。

ACID 详解

字母含义通俗理解实现
Atomicity原子性一个事务要么全做，要么全不做Undo Log 回滚
Consistency一致性事务前后数据满足所有约束外键、CHECK、触发器
Isolation隔离性并发事务互不干扰MVCC / 锁
Durability持久性事务提交后不丢失WAL + fsync

⚠️ 重点：事务隔离级别

这是高频必考点。从弱到强：

级别脏读不可重复读幻读性能
读未提交❌ 可能❌ 可能❌ 可能最快
读已提交✅ 防止❌ 可能❌ 可能快
可重复读✅ 防止✅ 防止❌ 可能中
可串行化✅ 防止✅ 防止✅ 防止最慢
脏读示例（Read Uncommitted 下可能发生）：
事务A：UPDATE account SET balance = 200 WHERE id = 1  -- 还没提交
事务B：SELECT balance FROM account WHERE id = 1       → 200  -- 读到未提交数据！
事务A：ROLLBACK                                        -- A 回滚了
事务B 基于 200 做了决策...  -- 基于"脏数据"的决策，系统状态错乱

不可重复读示例（Read Committed 下可能发生）：
事务B：SELECT balance FROM account WHERE id = 1       → 100
事务A：UPDATE account SET balance = 200 WHERE id = 1; COMMIT;
事务B：SELECT balance FROM account WHERE id = 1       → 200  -- 同一事务两次读到不同值！

幻读示例（Repeatable Read 下可能发生）：
事务B：SELECT COUNT(*) FROM account WHERE balance > 100  → 5
事务A：INSERT INTO account VALUES (6, 'new', 150); COMMIT;
事务B：SELECT COUNT(*) FROM account WHERE balance > 100  → 6  -- "多了一行"

MVCC：多版本并发控制

📸 类比：MVCC 就像 Git——每个事务看到的是数据库的"某个快照"，即使其他事务在修改，你看到的也不会变。每行数据都有created_by 和 deleted_by 两个版本号。

核心机制：读不阻塞写，写不阻塞读。每次 UPDATE 不覆盖原数据，而是创建新版本，旧版本保留给正在读的事务。

MVCC 版本链：每行多版本共存，Read View 决定事务看到哪个版本

🔗 MySQL 实战对照

DDIA 讲的是事务的通用原理，MySQL 的 InnoDB 是这些原理的工业级实现。对照着看事半功倍：

DDIA 概念MySQL InnoDB 实现详见
MVCC 多版本并发控制每行隐藏字段 DB_TRX_ID（事务ID）+ DB_ROLL_PTR（回滚指针）→ 通过 Undo Log 构建版本链📘 MySQL
Read View（快照读）RR 级别下事务开始时创建 Read View，RC 级别下每条语句重新创建📘 MySQL
可重复读（Repeatable Read）InnoDB 默认级别，通过 MVCC + Next-Key Lock（Gap Lock）防止幻读📘 MySQL
WAL 预写日志InnoDB 的 redo log（崩溃恢复）+ undo log（事务回滚+MVCC）📘 MySQL
隔离级别的并发问题脏读/不可重复读/幻读的 MySQL 实战演示📘 MySQL

Ch8 分布式系统的麻烦

🌩️ 类比：单机是独栋别墅（什么都可控），分布式是公寓楼——邻居可能半夜装修（网络延迟）、电梯可能坏（节点故障）、快递可能送错门（消息乱序）。DDIA 这章把分布式系统所有坑都列了一遍。

八大分布式谬误

#谬误现实
1网络是可靠的丢包、延迟、分区（网络分区）
2延迟为零跨机房 RTT 100-300ms
3带宽无限拥塞、限流
4网络是安全的中间人、DDoS
5拓扑不变加节点、减节点
6只有一个管理员多团队、多配置源
7传输成本为零序列化、压缩有开销
8网络是同质的不同链路延迟差异大

时钟问题

分布式系统中各节点的时钟不一定同步——NTP 可能有几个 ms 的偏差。所以不能用墙上时钟（Wall Clock）判断事件顺序，必须用逻辑时钟（如 Lamport 时间戳）。

真相由多数派定义

一句话：在分布式系统中，一个节点不能相信自己的判断——它可能已经"脑裂"了。真相由 Quorum（多数派）投票决定。

Ch9 一致性与共识

🗳️ 类比：一致性是"大家看到的东西一样吗"，共识是"大家投票选出一个决定了吗"。线性一致性是最强的保证——让分布式系统"假装"只有一台机器。

一致性模型谱系（从强到弱）

模型保证典型系统
线性一致性所有操作看起来像原子操作一样瞬间完成，全局有序etcd / ZooKeeper
顺序一致性操作按某个全局顺序执行，但不保证实时理论模型
因果一致性有因果关系的操作按序，无关的可以乱序无主数据库
最终一致性只要没有新写，最终所有副本一致DNS / Cassandra
一致性模型谱系：强一致（左）→ 最终一致（右），CAP 的 C 并非二选一

共识协议对比

协议特点性能应用
Paxos理论优雅，实现极难2 RTTChubby
Raft可理解性优先1 RTTetcd / TiKV
ZAB为 ZooKeeper 定制1 RTTZooKeeper

Ch10 批处理 —— 离线处理海量数据

🏭 类比：批处理 = 工厂夜班——白天收集订单，晚上统一处理一整天的数据。不在乎单条多快，在乎吞吐量（单位时间处理多少）。

MapReduce 三部曲

Map:    将输入拆成 (key, value) 对 → 分到不同 Reducer
Shuffle: 按 key 排序 + 分组（最耗时的阶段）
Reduce:  对每个 key 的 values 做聚合

示例：统计每个单词出现次数（Word Count）
def map(doc):
    for word in doc.split():
        emit(word, 1)         # 输出 (word, 1)

def reduce(word, counts):
    emit(word, sum(counts))   # 求和

MapReduce → Spark 的进化

MapReduce 的痛点：每步都写磁盘，中间结果巨大。Spark 的核心创新：RDD 放内存，DAG 调度，避免不必要的磁盘 IO。这就是为什么 Spark 比 MapReduce 快 10-100 倍。

MapReduce Join 算法

算法原理适用
Reduce-side JoinMap 阶段打标签，Shuffle 按 join key 分组，Reduce 阶段做 join通用、两表都大
Map-side Join把小表加载到内存，Map 阶段直接 join，不走 Shuffle一大一小（broadcast join）

Ch11 流处理 —— 实时处理，来了就做

📡 类比：批处理是"每天晚上算昨天的销售额"，流处理是"每卖出一单立刻更新仪表盘"。一个看历史，一个看现在。

批处理 vs 流处理

批处理 Batch

  • 数据：有界（固定大小文件）
  • 延迟：分钟到小时级
  • 代表：Spark / Hive

流处理 Stream

  • 数据：无界（持续产生）
  • 延迟：毫秒到秒级
  • 代表：Flink / Kafka Streams

消息系统的两种模型

AMQP/JMS 模型日志模型（Kafka）
消费确认消费者 ack，broker 删除消费者记 offset，消息保留
重复消费难（消息已删）易（重置 offset 重读）
多消费者负载均衡（一条消息一人消费）发布订阅（多组各自消费）

时间窗口

四种窗口类型：
1. Tumbling Window（滚动窗口）：不重叠，固定大小
   例：每分钟统计一次 [10:00, 10:01), [10:01, 10:02)

2. Hopping Window（滑动窗口）：固定大小，固定步长
   例：每 30 秒统计过去 1 分钟

3. Sliding Window（滑动窗口）：仅按时间滑动
   例："过去 5 分钟内的数据"（不停更新）

4. Session Window（会话窗口）：按活动间隔切分
   例：用户一顿操作后 30 分钟没动静 = 一个 session

流表二象性

DDIA 最深刻的思想之一：表和流是同一枚硬币的两面。

流 → 表：把流（INSERT/UPDATE/DELETE 事件）累计起来就是表（当前最新状态）。

表 → 流：把表的变更日志（CDC，Change Data Capture）发出去就是流。

启示：数据库、消息队列、流处理器本质上在做同一件事——维护数据的物化视图。Kafka 可以当数据库用，数据库的 binlog 可以当消息队列用。

流表二象性：流累积成表，表的变更日志成流——两者是同一数据的不同物化形式

Ch12 数据系统的未来

DDIA 最后一章不是讲技术，而是讲 哲学——构建数据系统时应该坚持什么原则。

核心洞察

原则含义
派生数据 > 分布式事务与其用 2PC 跨系统协调，不如用 CDC 把数据流向各个系统（松散耦合）
不可变日志是金Kafka 式的 immutable log 是系统间数据交换的最佳抽象
端到端原则不要把正确性全压在底层（如 TCP 重传不够，应用层也要校验）

❓ DDIA 高频考题

Q1: LSM-Tree 和 B-Tree 的区别？各适合什么场景？
LSM-Tree：写优化（顺序写），读放大（需查多个 SSTable），适合写多读少（日志、时序数据）。B-Tree：读写平衡（O(log N)），写放大（页分裂），适合读多写少或读写均衡（OLTP）。代表：LSM-Tree → RocksDB/Cassandra，B-Tree → InnoDB/PostgreSQL。
Q2: 事务隔离级别有哪些？MySQL 默认是什么？
读未提交 → 读已提交 → 可重复读 → 可串行化。MySQL InnoDB 默认是可重复读（Repeatable Read），通过 MVCC + Next-Key Lock（Gap Lock）防幻读。PostgreSQL 默认是读已提交。
Q3: MVCC 的实现原理？
每行数据有隐藏字段：trx_id（创建该版本的事务ID）和 roll_pointer（指向 undo log 旧版本）。SELECT 根据事务开始的 Read View 判断哪些版本可见。UPDATE 创建新版本而不是覆盖 → 写不阻塞读。
Q4: 什么是线性一致性？和最终一致性的区别？
线性一致性：让分布式系统看起来像单机——写完后所有后续读都能看到新值，操作按全局实时顺序。etcd 提供。最终一致性：只要没有新写，最终所有副本一致——但中间可能读到旧值。Cassandra/DNS 提供。
Q5: 主从复制的延迟问题怎么解决？
① Read-After-Write：用户自己刚写的数据走主库读；② 单调读：同一用户始终读同一个从库副本；③ 前缀一致性读：保证因果顺序。终极方案是用共识协议（Raft）替代主从。
Q6: 什么场景适合用列存？
OLAP（分析查询），查询只涉及少数列、大量行。列存压缩率高（同列数据相似）、向量化执行快。典型：ClickHouse、Doris。不要用列存做 OLTP——单行写入代价极高。
Q7: 说一下流表二象性？
表和流是等价的：流（变更事件）累积起来是表；表的 changelog（CDC）发出去是流。这意味着数据库和消息队列没有本质区别——只是数据的两种物化形式。这催生了 Kappa 架构（用流处理统一批和流）。

💬 苏格拉底式提问

为什么说"分布式事务（2PC）是反模式"？Q1
2PC 的致命缺陷：① 阻塞协议——协调者挂了，参与者锁住资源死等（可能 forever）；② 性能差——两轮 RPC + 所有参与者的 fsync；③ 耦合紧——参与者必须在线。DDIA 推崇用 CDC + 补偿事务（Saga）替代，通过不可变日志松散耦合各系统。
B-Tree 用了几十年了，为什么 LSM-Tree 能在现代崛起？Q2
硬件变了。B-Tree 诞生于磁盘时代（随机读写差距不大）。现代 SSD 的随机写仍然有写放大问题（最小写入单位是 page），而LSM-Tree 的顺序写完美匹配 SSD 的物理特性。另外大数据时代的数据量暴增，B-Tree 的碎片化在大规模下成了灾难。
如果让你从零设计一个数据库，你会选什么存储引擎？Q3
取决于场景。OLTP 不用想，直接 B-Tree（InnoDB）——它是 40 年锤炼过的方案。时序/日志/写密集场景，LSM-Tree（RocksDB）。列存场景 ClickHouse。关键是不要自己从零写——WAL + 页管理 + 崩溃恢复的坑深不见底。DDIA 全书都在告诉我们：理解原理，但在工程上站在巨人肩膀上。
