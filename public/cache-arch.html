📋 总览：多级缓存架构是什么

缓存是计算机科学中唯一真正的银弹——几乎所有性能问题，都能通过加一层缓存来解决。多级缓存架构就是将缓存按速度、容量、成本分层，从 CPU 寄存器到浏览器缓存，每一级解决不同粒度的问题。

在企业级后端系统中，典型的多级缓存包括：本地缓存（Caffeine/Guava）→ 分布式缓存（Redis Cluster）→ CDN 缓存，层层递进，形成完整的缓存防御体系。

想象你开了一家大型图书馆：读者借书流程就是缓存的多级查询。

第一级（L1 — 本地缓存）：你口袋里的小笔记本，记着最常借的几本书的位置——翻口袋就知道在哪，1 毫秒。

第二级（L2 — 分布式缓存）：图书馆大厅的电子查书终端——查书号、看剩余数量，5 毫秒。

第三级（L3 — DB/源站）：你走到对应的书架翻书——虽然能找到，但要走几步路，50 毫秒甚至更久。

缓存的本质就是：用空间换时间，把高频访问的数据放在离"读者"最近的地方。

图 1：缓存金字塔——从 L1 到 DB 的五层体系

⚡ 为什么需要缓存

没有缓存的世界：每个用户请求都去查数据库 → 数据库连接数打满 → SQL 查询排队 → 响应时间飙升 → 数据库 OOM → 服务雪崩 → 老板找你喝茶。有了缓存：热数据在内存中 → 99% 的请求不碰数据库 → 响应时间从 200ms → 2ms。

🐌 没有缓存

用户 → Nginx → 应用 → 数据库

  • 每个请求都穿透到 DB
  • 热点数据反复查询
  • 100 QPS 数据库就吃力
  • 响应时间 > 200ms
  • 数据库是单点瓶颈
  • 流量翻倍=数据库翻倍成本

🚀 有缓存

用户 → CDN → Nginx → 本地缓存 → Redis → 数据库

  • 99% 请求在缓存层命中
  • 热点数据存内存
  • 10,000+ QPS 轻松应对
  • 响应时间 ~5ms
  • 缓存扛住流量洪峰
  • DB QPS 降低 100 倍

没有缓存 = 没有收银台的超市——每个顾客都要去仓库找商品。

有缓存 = 超市门口摆了一排高频商品货架：可乐、矿泉水、方便面——90% 的顾客直接拿走，只有特殊商品才去仓库（数据库）。

缓存就是\"高频商品前置货架\"——把最常被买的东西放在离顾客最近的地方。

🗼 多级缓存架构

一个成熟的生产级缓存架构通常包含三层：本地缓存（进程内）→ 分布式缓存（跨进程共享）→ CDN 缓存（边缘加速）。每一级承担不同的职责并互为兜底。

分布式缓存：Redis Cluster 架构

Redis Cluster 通过分片（Sharding）将数据分布到多个节点，每个节点负责一部分哈希槽。客户端直连节点，无中心化代理，支持水平扩展。

🎯 各层缓存的定位

  • 本地缓存：极速（~1ms）、不跨进程、容量小（GB 级以内）、适合热点数据兜底
  • Redis Cluster：共享数据、跨进程、容量大（TB 级）、适合公共热点、支持持久化
  • CDN：静态资源加速、地域级缓存、降低源站压力、T+0 级缓存

📐 缓存模式（Cache Patterns）

缓存与数据库的交互有几种经典模式，每种模式解决不同场景下的读写一致性和复杂度问题。

图 3：三种经典缓存模式——Cache-Aside / Read-Through / Write-Through

三种模式就像三种购物方式：

Cache-Aside = 自己去超市（缓存）看看有没有牛奶，没有就去仓库（DB）拿，然后放一箱在超市货架上。

Read-Through = 让便利店店员帮你去取——你说"我要牛奶"，店员先看货架，没有就去仓库拿回来放上。

Write-Through = 你往货架上放牛奶的同时，店员同步在仓库也放一份——两边都有记录。

Cache-Aside 的读写流程（最常用）

// 读流程
Object get(String key) {
    Object val = cache.get(key);     // 1. 先查缓存
    if (val == null) {               // 2. 缓存未命中
        val = db.query(key);         // 3. 查数据库
        cache.put(key, val, ttl);    // 4. 回填缓存（设置过期时间）
    }
    return val;
}

// 写流程
void update(String key, Object data) {
    db.update(key, data);            // 1. 先更新数据库
    cache.del(key);                  // 2. 再删除缓存（不是更新！）
}

💡 为什么是删缓存而不是更新缓存？

因为写操作的并发场景：两个线程同时写同一个 key，先写后写更新缓存可能导致数据不一致。删除缓存让下次读时自然回填，避免复杂的并发问题。这就是经典的 Cache-Aside 写删策略。

🔄 缓存一致性（Cache 更新策略）

缓存一致性是缓存系统最难的问题——如何保证缓存里的数据和数据库里的数据是一致的？在分布式系统中，强一致性的代价很高，通常我们追求最终一致性。

图 4：四种缓存一致性策略对比

延迟双删（最常用的生产方案）

void updateWithDelayDelete(String key, Object data) {
    // 第一步：先删缓存（让旧数据失效）
    cache.del(key);

    // 第二步：更新数据库
    db.update(key, data);

    // 第三步：延迟一段时间（如 500ms），再删一次缓存
    // 目的是清除并发读线程在第一步之后写入缓存的旧数据
    Thread.sleep(500);
    cache.del(key);
}

缓存一致性就像图书管理员更新书架：

TTL 策略 = 在书架上放个便签"这本书的信息每晚 10 点更新"——中间有人看到旧信息也认了。

更新删缓存 = 书内容变了，直接把书架上的书扔了——下次有人要看，管理员去仓库拿新书。

Binlog 方案 = 图书馆总馆有个"变更记录本"（Binlog），后台程序监控这个本子，发现哪本书变了就去更新对应书架。

延迟双删 = 扔旧书 → 换仓库 → 再等几秒再扔一次——防止中间有人捡了旧书放回去。

🛡️ 缓存雪崩三兄弟：穿透 / 击穿 / 雪崩

这是缓存系统最经典的三个灾难场景，实战必问。理解它们并掌握对应的解决方案，是缓存架构师的基本功。

图 5：缓存三大灾难——穿透 / 击穿 / 雪崩，及其完整解决方案体系

缓存三大灾难，用图书馆来理解：

缓存穿透 = 有人借一本不存在的书（ISBN 号都不对）→ 查电脑（缓存）没有 → 去仓库（DB）翻也没有。恶意者反复借不存在的书，仓库管理员累瘫。解决方案：告诉管理员\"查过一次没有的书，记在小本本上（缓存空对象），下次就不用去仓库了\" 或者布隆过滤器：像安检门口的名单——不在名单上的人直接不让进。

缓存击穿 = 最火爆的《哈利·波特》秘籍过期下架了（缓存失效），100 个人同时来借→100 个人同时去仓库找。解决方案：在秘籍下架时只让一个人去仓库取，其他人排队等着（互斥锁）。

缓存雪崩 = 图书馆断电了（Redis 挂了），或者所有热门书同一天到期。所有人直接涌向仓库，仓库门被挤爆。解决方案：热门书不要设置同一天到期（TTL 随机化），在仓库门口设限流闸机（降级），给图书馆配送发电机（Redis 持久化 + 主从）。

代码示例：缓存击穿互斥锁方案

// 缓存击穿防护：互斥锁 + 双重检查
Object getWithLock(String key) {
    Object val = cache.get(key);
    if (val != null) return val;       // 缓存命中，直接返回

    String lockKey = "lock:" + key;
    if (redis.tryLock(lockKey, 3s)) {  // 尝试获取锁
        try {
            // 双重检查：锁等待期间可能其他线程已回填
            val = cache.get(key);
            if (val != null) return val;

            val = db.query(key);        // 查数据库
            cache.put(key, val, 60s);   // 回填缓存
            return val;
        } finally {
            redis.unlock(lockKey);      // 释放锁
        }
    } else {
        // 没拿到锁的线程等待一会儿再重试
        Thread.sleep(50);
        return getWithLock(key);        // 递归重试
    }
}

📋 实战最佳实践

以下最佳实践来自一线互联网公司的生产经验，涵盖缓存使用、监控、运维等方方面面。

图 6：缓存实战的 8 条黄金法则

生产级缓存配置模板

// Caffeine 本地缓存配置示例
Cache localCache = Caffeine.newBuilder()
    .maximumSize(10_000)              // 最大 1 万条
    .expireAfterWrite(60, TimeUnit.SECONDS)  // 写入后 60 秒过期
    .refreshAfterWrite(30, TimeUnit.SECONDS) // 30 秒后异步刷新
    .recordStats()                    // 开启命中率统计
    .build();

// Redis 分布式缓存配置建议
redis:
  cluster: true                       # 启用集群模式
  max-redirects: 3                    # 最大重定向次数
  timeout: 2000ms                     # 连接超时
  pool:
    max-active: 200                   # 最大连接数
    max-wait: -1                      # 获取连接最大等待
  ttl-default: 60s                    # 默认过期时间
  key-prefix: "myapp:"                # 统一 key 前缀

🎯 核心原则总结

  • 缓存是加速而不是存储：所有数据必须以 DB 为准，缓存可以丢
  • 永远假设缓存会失效：设计时考虑缓存穿透、击穿、雪崩三个场景
  • 多级缓存互备：本地缓存兜底 Redis，Redis 兜底 DB
  • 缓存不是银弹：不是所有数据都适合缓存——频繁更新的数据不适合

Q1：讲一下多级缓存架构？每一层的作用和选型是什么？

核心思路：

多级缓存架构从近到远分为三层：本地缓存 → 分布式缓存 → CDN 缓存。

① 本地缓存（L1）：Caffeine/Guava，进程内缓存，速度最快（~1ms），容量有限（GB 级），适合缓存热点数据。选型推荐 Caffeine（W-TinyLFU 淘汰算法，性能是 Guava 的 2-3 倍）。

② 分布式缓存（L2）：Redis Cluster/Memcached，跨进程共享数据，容量大（TB 级），适合公共热点。选型推荐 Redis Cluster（支持持久化、丰富数据结构、高可用）。

③ CDN 缓存（L3）：静态资源加速，边缘节点缓存。适合图片、CSS/JS、HTML 页面等。常用 Cloudflare、阿里云 CDN、AWS CloudFront。

加分点：

提到各层互备逻辑——Redis 挂了本地缓存还能扛一会儿，本地缓存通过 RefreshAfterWrite 异步刷新避免阻塞。

Q2：缓存穿透、缓存击穿、缓存雪崩的区别是什么？分别怎么解决？

核心思路：

缓存穿透：key 在缓存和 DB 都不存在，请求直接打到 DB。解决方案：布隆过滤器（预判 key 是否存在）或缓存空对象（key-null，短 TTL）。

缓存击穿：热点 key 过期瞬间，大量并发请求打到 DB。解决方案：互斥锁（只让一个线程查 DB 回填）或逻辑过期 + 异步刷新。

缓存雪崩：大量 key 同时过期或 Redis 宕机，导致请求全部打向 DB。解决方案：TTL 加随机偏移 + 多级缓存 + 限流降级 + Redis 主从集群。

一句话区分：穿透是"不存在"，击穿是"一个热点过期"，雪崩是"一大片同时失效"。

Q3：解释一下 Cache-Aside 模式？为什么更新缓存时是"删缓存"而不是"更新缓存"？

核心思路：

Cache-Aside（旁路缓存）是最常用的缓存模式。读：先查缓存 → 未命中查 DB → 回填缓存。写：先更新 DB → 再删除缓存。

为什么删而不是更新？因为并发写场景下，更新缓存会导致数据不一致：

举例：线程 A 和 B 同时写 key=1。A 写入 DB（value=2），B 写入 DB（value=3）。如果 A 先更新缓存为 2，B 后更新缓存为 3——结果缓存是 3（正确）。但如果执行顺序是：A 写 DB(2) → B 写 DB(3) → B 更新缓存(3) → A 更新缓存(2)，缓存变成了旧的 2。删缓存策略避免了这个问题——让下次读时自然拉取最新数据。

加分点：

提到"延迟双删"——先删缓存 → 写 DB → 延迟再删一次，解决并发读请求在第一步之后写入旧缓存的问题。

Q4：Redis 有哪些数据结构？分别适合什么缓存场景？

核心思路：

结构场景示例
String缓存对象、计数器、分布式锁缓存用户信息、点赞数
Hash对象的部分字段读写用户资料（单独改头像）
List消息队列、最新列表微博时间线、消息通知
Set去重、社交关系共同好友、标签系统
Sorted Set排行榜、延时队列热门文章、定时任务
Bitmap布尔状态统计签到记录、布隆过滤器
HyperLogLog基数统计（近似）UV 统计（误差 0.81%）

选择原则：String 最常用（JSON 序列化），Hash 适合部分字段更新，Sorted Set 适合排行榜等排序场景。

Q5：Redis 的过期删除策略是什么？内存淘汰机制有哪些？

核心思路：

过期删除策略：

  • 惰性删除：访问 key 时检查是否过期，过期则删除（懒加载）
  • 定期删除：每 100ms 随机扫描部分 key，删除过期 key（平衡 CPU 和内存）

两者配合——惰性删除保证不浪费 CPU，定期删除防止过期 key 一直占内存。

内存淘汰机制（maxmemory-policy）：

策略说明场景
noeviction内存满直接报错（默认）不推荐生产
allkeys-lru淘汰最近最少使用的 key最常用
allkeys-lfu淘汰使用频率最低的 key高频场景推荐
volatile-lru从设置过期时间的 key 中淘汰混合存储场景
volatile-ttl淘汰快过期的 key少用

生产推荐：allkeys-lru 或 allkeys-lfu（Redis 4.0+）。

Q6：什么是布隆过滤器？怎么用在缓存穿透防护上？

核心思路：

布隆过滤器（Bloom Filter）是一种空间效率极高的概率性数据结构。它用一个 bitmap 和多个哈希函数来判断一个元素是否在集合中。

特点：

  • 说"不存在"就一定不存在（不漏报）
  • 说"存在"可能误判（有误报率 ~1%）
  • 空间极小——1 亿个 key 只需要约 120MB

防穿透方案：系统启动时将所有可能存在的主键（如用户 ID、商品 ID）哈希到布隆过滤器。查询流程变成：

请求 → 布隆过滤器判断 key 是否存在 → 不存在直接返回（不查缓存和 DB）→ 存在才查缓存 → 未命中查 DB。

加分点：

提到 Redis 的 Bitmap 可以实现简单布隆过滤器，或者用 Guava BloomFilter（本地）和 Redisson RBloomFilter（分布式）。

Q7：如何保证缓存和数据库的双写一致性？

核心思路：

缓存和数据库双写一致性是分布式系统经典难题，没有完美的强一致方案（除非用 2PC，但性能太差）。生产上追求最终一致性。

常用方案（从简单到可靠）：

方案做法一致性复杂度
TTL 过期缓存设 TTL，到期自动拉新弱（TTL 内不一致）⭐
Cache-Aside 删缓存写 DB 后删缓存，下次读时回填最终一致⭐⭐
延迟双删删缓存 → 写 DB → 延迟再删一次最终一致（窗口极小）⭐⭐⭐
Binlog 监听Canal 监听 MySQL binlog 异步更新缓存最终一致（毫秒级）⭐⭐⭐⭐

加分点：

提到 分布式锁 + 版本号方案：缓存中存版本号，写请求先获取锁 → 递增版本号 → 写入 DB → 更新缓存。读请求对比版本号，不一致则从 DB 拉最新数据。

Q8：你在实际项目中是如何设计和优化缓存策略的？

核心思路（以电商系统为例）：

场景：商品详情页，日均 1000 万 UV，高峰期 1 万 QPS。

缓存设计：

  1. CDN 缓存：商品图片、CSS/JS 等静态资源，CDN 边缘节点缓存 7 天
  2. Nginx 本地缓存：商品详情页 HTML 片段，lua 共享字典缓存 10s
  3. Caffeine 本地缓存：热点商品数据（TOP 1000 商品），expireAfterWrite=60s，refreshAfterWrite=30s
  4. Redis Cluster 分布式缓存：所有商品数据，Key 设计为 "product:{id}:detail"，TTL=60s+随机偏移

防雪崩措施：

  • 布隆过滤器拦截非法商品 ID 请求
  • 互斥锁防缓存击穿（Redis SETNX）
  • TTL 加随机值 [0, 30s] 防雪崩
  • Sentinel 限流，超过阈值直接返回缓存中的旧数据

效果：CDN 命中率 60% + 本地缓存 30% → 只有 10% 请求到 Redis → < 1% 请求到 DB。接口 P99 延迟从 200ms 降到 8ms，DB QPS 从 5000 降到 < 50。

加分点：

提到缓存监控告警和容量规划——命中率监控、内存水位告警、定期压测。并强调"架构没有银弹，要根据业务场景和数据特征做权衡"。

📌 核心要点

读者问缓存时最关注的是深度思考而非背方案：

  • 懂原理：知道为什么删缓存而不是更新，知道布隆过滤器的数学原理
  • 懂权衡：一致性 vs 可用性 vs 性能 的取舍（CAP 理论在缓存中的体现）
  • 懂场景：不同模式适合不同业务特点（读多写少 vs 写多读少）
  • 懂兜底：缓存挂了怎么办？多级缓存、限流降级、本地缓存兜底

🎙️ 口播总结

好，我们来快速捋一遍多级缓存架构设计的知识体系。

第一，缓存是性能优化的第一选择——几乎任何系统遇到性能问题，第一个想到的解决方案就是"加一层缓存"。缓存的本质是用空间换时间，把读多写少的热点数据放在离计算最近的地方。

第二，多级缓存架构是一个金字塔——从上到下依次是 CDN 缓存、本地缓存（Caffeine/Guava）、分布式缓存（Redis Cluster）、数据库。越往上速度越快容量越小成本越高，越往下一致性越强。每一层都是上一层失效时的兜底方案。

第三，缓存模式方面，最常用的是 Cache-Aside（旁路缓存）——读时先查缓存，未命中再查 DB 并回填；写时先更新 DB 再删缓存。记住核心原則：删缓存而不是更新缓存，以避免并发写导致的数据不一致。

第四，缓存一致性无法做到完美的强一致，我们追求的是最终一致性。生产上常用延迟双删和 Binlog 监听（Canal）方案。

第五，防雪崩三板斧：缓存穿透用布隆过滤器，缓存击穿用互斥锁，缓存雪崩用 TTL 随机化 + 多级缓存 + 限流降级。记住一句话——永远不要让请求直接穿透到数据库。

缓存的实战点很多，但抓住"速度、容量、一致性"的三角权衡和"多级兜底"这两个核心思想，你就能应对 80% 的缓存问题了。
