🤖 多 Agent 协作模式

从单体智能到群体智能的架构演进 · 深度技术指南

1. 为什么需要多 Agent

想象一位顶级厨师独自操办满汉全席：他要同时掌勺 108 道菜、雕刻萝卜花、熬高汤、摆盘……即使他有三头六臂，出餐速度和质量也必然崩溃。多 Agent 协作就像给这位大厨配上一支专业团队——配菜师、面点师、冷盘师傅各司其职，主厨只需调度和品控。

单一 Agent 在实际生产中会面临三重瓶颈：

  • 上下文窗口溢出：单个 LLM 承载过多角色和知识，token 消耗爆炸，注意力分散
  • 单点脆弱性：一个环节出错导致全流程失败，没有冗余和回退机制
  • 专业化不足：让同一个模型既写代码又做设计又做测试，每个任务都达不到专家水准

2. Orchestrator-Workers 模式

Orchestrator（编排器）是中心调度节点，负责将复杂任务分解为子任务，分发给专用的 Worker Agent，最后收集结果并汇合输出。这是最常用的多 Agent 模式，适合有明确流程结构的任务。

核心流程

  • 任务分解：Orchestrator 分析用户意图，将其拆解为可独立执行的子任务
  • 智能分发：根据 Worker 的能力标签进行匹配分派
  • 并行执行：Workers 各自独立运行，支持阻塞/非阻塞模式
  • 汇合与排序：收集所有结果，按依赖关系组装成最终应答

Go 实现：Orchestrator

package agent

import (
    "context"
    "fmt"
    "sync"
)

// Task 代表一个可分发的工作单元
type Task struct {
    ID          string
    Description string
    RequiredCap string // 所需能力
    Payload     interface{}
}

// Result 封装 Worker 的执行结果
type Result struct {
    TaskID string
    Output string
    Err    error
}

// Worker 接口：每个 worker 必须实现 Execute
type Worker interface {
    Execute(ctx context.Context, task Task) Result
    Capability() string
}

// Orchestrator 核心调度器
type Orchestrator struct {
    workers map[string][]Worker
}

func NewOrchestrator(workers []Worker) *Orchestrator {
    reg := make(map[string][]Worker)
    for _, w := range workers {
        cap := w.Capability()
        reg[cap] = append(reg[cap], w)
    }
    return &Orchestrator{workers: reg}
}

// Run 分解任务 -> 分发 -> 收集 -> 汇合
func (o *Orchestrator) Run(ctx context.Context, goal string) (string, error) {
    // 1. 分解目标为子任务
    tasks := o.decompose(goal)

    // 2. 并发分发
    var wg sync.WaitGroup
    results := make(chan Result, len(tasks))

    for _, task := range tasks {
        wg.Add(1)
        go func(t Task) {
            defer wg.Done()
            worker := o.assign(t)
            results <- worker.Execute(ctx, t)
        }(task)
    }

    go func() {
        wg.Wait()
        close(results)
    }()

    // 3. 收集并汇合
    return o.synthesize(results)
}

func (o *Orchestrator) decompose(goal string) []Task {
    // 使用 LLM 将 goal 分解为子任务列表
    // 此处省略具体调用，返回示例
    return []Task{
        {ID: "t1", Description: "需求分析", RequiredCap: "analysis"},
        {ID: "t2", Description: "代码实现", RequiredCap: "coding"},
        {ID: "t3", Description: "单元测试", RequiredCap: "testing"},
    }
}

func (o *Orchestrator) assign(t Task) Worker {
    return o.workers[t.RequiredCap][0]
}

func (o *Orchestrator) synthesize(results <-chan Result) (string, error) {
    var sb strings.Builder
    for r := range results {
        if r.Err != nil {
            return "", fmt.Errorf("task %s failed: %w", r.TaskID, r.Err)
        }
        sb.WriteString(r.Output)
        sb.WriteString("\n")
    }
    return sb.String(), nil
}

3. Supervisor-Subordinate 模式

在 Supervisor（监督者）模式中，一个上级 Agent 负责监控和评估下级 Agent 的工作质量，当下级产出不达标时触发重做。这解决了 Agent 输出不可靠的核心痛点。

关键机制： 监督者不直接执行任务，而是对 Worker 的输出进行质量检查。发现缺陷后，将任务连同反馈一起退回给 Worker 重做，直到质量达标或达到最大重试次数。

4. Swarm 模式

Swarm（群体）模式取消中心节点，所有 Agent 通过消息传递相互通信、协商和协作。每个 Agent 都是对等的（peer-to-peer），依靠局部信息做出决策，整体涌现出智能行为。

就像蜂群中的蜜蜂——没有"总指挥"，每只蜜蜂根据周围同伴的信息素信号自主行动。整个蜂群却能高效完成采蜜、筑巢、防御等复杂任务。Swarm 模式的 Agent 群同样通过消息总线交换信息，自组织完成任务。

特点

  • 去中心化：没有单点故障，弹性极高
  • 可扩展：加入新 Agent 无需修改其他节点
  • 鲁棒性：部分 Agent 失效不影响整体
  • 挑战：调试困难、消息风暴、最终一致性

5. Debate 辩论模式

Debate（辩论）模式让多个 Agent 扮演不同角色，针对同一问题进行多轮辩论和反驳，最后通过投票或 LLM-as-Judge 达成共识。这种模式能显著提升答案的深度、广度和准确性。

Go 实现：Debate 循环

package debate

import (
    "context"
    "fmt"
)

// Agent 观点
type Statement struct {
    Role   string
    Content string
    Round   int
}

// Debate 辩论引擎
type Debate struct {
    Topic    string
    Roles    []string
    History  []Statement
    MaxRound int
}

func NewDebate(topic string, roles []string) *Debate {
    return &Debate{
        Topic:    topic,
        Roles:    roles,
        MaxRound: 3,
    }
}

// Run 执行多轮辩论
func (d *Debate) Run(ctx context.Context, llm func(string) string) []Statement {
    for round := 1; round <= d.MaxRound; round++ {
        for _, role := range d.Roles {
            prompt := d.buildPrompt(role, round)
            response := llm(prompt)

            d.History = append(d.History, Statement{
                Role:    role,
                Content: response,
                Round:   round,
            })

            // 打印辩论过程
            fmt.Printf("[Round %d] %s: %s\n", round, role, response[:min(80, len(response))])
        }
    }
    return d.History
}

func (d *Debate) buildPrompt(role string, round int) string {
    prompt := fmt.Sprintf("议题：%s\n你的身份：%s\n当前是第 %d 轮辩论。\n", d.Topic, role, round)

    if round > 1 {
        prompt += "以下是之前的辩论记录，请根据这些观点进行反驳或补充：\n"
        for _, s := range d.History {
            prompt += fmt.Sprintf("  [%s]: %s\n", s.Role, s.Content)
        }
        prompt += "\n请针对以上观点提出新的论据或反驳。\n"
    }
    return prompt
}

6. 通信协议

多 Agent 之间的通信方式是架构设计的核心决策之一。三种主流方案各有优劣：

特性 直接调用 (RPC/HTTP) 消息队列 (MQ) 共享内存 (Redis/DB)
延迟 低 (ms 级) 中等 (10-100ms) 低 (μs 级)
解耦程度 紧耦合 高 中等
可靠性 需自建重试 持久化 + ACK 依赖持久化方案
扩展性 线性增长连接 水平扩展 水平扩展
调试 简单直接 需追踪消息 需监控状态
适用场景 小规模、确定性流程 高吞吐、异步解耦 状态共享、实时协作

7. 任务分配策略

如何将任务分配给最合适的 Agent 是多 Agent 系统效率的关键。以下是四种主流策略：

选择策略的核心考量： Worker 的异构程度越高，分配策略越需要智能化。同质 Worker 集群用 Round Robin 即可，异构集群则需要 Capability 或 Auction 策略。
策略 原理 优点 缺点 适用场景
Round Robin 轮询依次分配 实现简单、负载平均 忽略能力差异 同质 Worker
Capability 根据能力标签匹配 专业对口、质量高 需维护能力注册表 异构高技能集群
Load-aware 根据当前负载分配 避免过载、响应快 需实时监控状态 负载波动大
Auction Worker 竞标任务 动态最优匹配 计算开销大、复杂 复杂决策场景 
// LoadAwareBalancer 负载感知分配器
type LoadAwareBalancer struct {
    workers    []Worker
    loadCount  []int
    mu         sync.Mutex
}

func (lb *LoadAwareBalancer) Assign(task Task) Worker {
    lb.mu.Lock()
    defer lb.mu.Unlock()

    minIdx := 0
    for i := 1; i < len(lb.workers); i++ {
        if lb.loadCount[i] < lb.loadCount[minIdx] {
            minIdx = i
        }
    }
    lb.loadCount[minIdx]++
    return lb.workers[minIdx]
}

func (lb *LoadAwareBalancer) Complete(workerIdx int) {
    lb.mu.Lock()
    lb.loadCount[workerIdx]--
    lb.mu.Unlock()
}

8. LLM-as-Judge

在多 Agent 系统中，Agent 之间难免产生冲突和分歧。LLM-as-Judge 模式让一个独立的评判 Agent 根据预定义的评分标准对各方输出进行裁决，解决冲突并给出最终方案。

评判标准

  • 事实准确性：声明是否有事实依据，是否可验证
  • 逻辑一致性：论点内部是否自洽，推理是否严谨
  • 完整性：是否覆盖了问题的所有关键方面
  • 实用性：方案是否切实可行，是否有具体步骤
  • 安全性：是否存在偏见、有害内容或安全风险

Go 实现：Judge 仲裁器

import (
    "context"
    "fmt"
)

// Criterion 评判维度
type Criterion struct {
    Name   string
    Weight float64 // 权重 0-1
}

// Score 打分结果
type Score struct {
    AgentID    string
    Scores     map[string]float64 // 维度 -> 分数
    Total      float64
    Reasoning  string // 评判理由
}

// Judge 仲裁器
type Judge struct {
    Criteria []Criterion
    LLM      func(prompt string) string
}

func NewJudge(llm func(string) string) *Judge {
    return &Judge{
        Criteria: []Criterion{
            {Name: "accuracy", Weight: 0.30},
            {Name: "logic", Weight: 0.25},
            {Name: "completeness", Weight: 0.20},
            {Name: "practicality", Weight: 0.15},
            {Name: "safety", Weight: 0.10},
        },
        LLM: llm,
    }
}

// Arbitrate 对多个 Agent 的输出进行仲裁
func (j *Judge) Arbitrate(ctx context.Context, submissions map[string]string) (*Score, error) {
    var bestScore *Score

    for agentID, output := range submissions {
        prompt := j.buildEvalPrompt(agentID, output)
        result := j.LLM(prompt)

        score, err := j.parseScore(result)
        if err != nil {
            return nil, fmt.Errorf("parse score for %s: %w", agentID, err)
        }

        if bestScore == nil || score.Total > bestScore.Total {
            bestScore = score
        }
    }

    return bestScore, nil
}

9. 实战：装修系统多 Agent

以一个智能装修助手为例，展示多 Agent 协作的完整流程。用户输入"我要装修一套 120㎡ 的三居室，预算 30 万，现代简约风格"，系统自动调度多个专业化 Agent 协作完成。

每个 Agent 的输出都会经过 Supervisor 质检，质检不合格的退回重做。最终 Orchestrator 汇合所有子方案，形成完整的装修提案交付用户。

10. QA + 口播总结

多 Agent 和单 Agent 调 Prompt 有什么区别？

单 Agent 调 Prompt 是在"教一个人做所有事"——你得把全部规则塞进一个上下文，容易冲突且 token 爆炸。多 Agent 是"组建一支专业团队"——每个 Agent 只专注自己的领域，Prompt 简洁、职责清晰、可独立优化。从工程实践看，多 Agent debug 更容易定位问题，因为你知道是"预算 Agent 算错了"而不是"整个模型表现不稳定"。

Orchestrator 本身会不会成为瓶颈？

会。Orchestrator 是中心节点，如果设计不当确实会成为单点瓶颈。解决方案：1) 将 Orchestrator 设计为无状态，可水平扩展；2) 引入异步事件驱动架构，Orchestrator 只管分发和策略，不阻塞等待 Worker 结果；3) 对于超大规模场景，采用分层 Orchestrator——根 Orchestrator 只调度子 Orchestrator，子 Orchestrator 再调度具体 Workers。

Debate 模式真的能提高答案质量吗？

研究（如 Du et al. 2023）和实践都表明：多轮辩论能显著提升答案的准确性和深度。原因是：1) 不同角色迫使模型从多角度思考；2) 反驳机制迫使模型检查自己的逻辑漏洞；3) 多轮迭代让论证越来越严谨。但在简单问题上 Debate 是过度设计，建议仅用于复杂、高风险的决策场景。

Swarm 模式如何避免消息风暴？

Swarm 的消息风暴是真实挑战。常用策略：1) TTL 机制——每条消息设置有效时长，过期自动丢弃；2) 路由过滤——Agent 只订阅自己关心的消息类型；3) 去重逻辑——同一 Agent 在短时间内收到多条相同消息时只处理一次；4) 退避算法——负载高时增大消息发送间隔。生产环境建议用消息队列 + Swarm 的混合架构。

什么时候用 Capability 分配，什么时候用 Auction？

Capability 分配适合：Worker 能力标签明确且稳定，任务类型可预判（如"翻译任务→翻译 Agent"）。优点是快、确定性强。Auction 分配适合：Worker 能力标签模糊或动态变化，任务需要综合评估（如"写一篇技术博客→同时考量写作能力、技术深度、风格匹配"）。Auction 更灵活但带来额外延迟和计算成本。经验法则：能预判时用 Capability，需综合评判时用 Auction。
