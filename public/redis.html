🔴 Redis

一句话：Redis = 内存数据结构服务器。快是因为数据在内存 + 单线程无锁 + 高效数据结构。

考察：数据结构底层、持久化对比、高可用三件套、缓存三大问题、分布式锁。

🔬 底层实现：数据结构会"变形"

🔄 类比：变形金刚——同一种 Redis 类型，数据量小时用紧凑结构（省内存），数据量大时自动切换为高效结构（快）。这个过程叫编码转换。

Skiplist（跳表）—— ZSet 的秘密武器

🚇 类比：地铁有快线和慢线——慢线每站都停（链表遍历），快线只停大站（跳表）。你要去某站，先坐快线到大站附近，再换慢线精准定位。

跳表 = 多层索引的链表。每层跳过一半节点，查询 O(log N)，插入 O(log N)，且实现比红黑树简单。Redis 选跳表不选红黑树的根本原因：跳表支持范围查询更自然。

📸 RDB 快照：定期拍照

📷 类比：每 5 分钟给数据拍一张照片（dump.rdb）。系统崩了？恢复到最近那张照片。优点：恢复快（直接加载文件）。缺点：两次拍照之间的数据丢了。

触发方式

方式命令/配置阻塞？
手动SAVE（阻塞）/ BGSAVE（fork 子进程）BGSAVE 不阻塞
自动save 900 1（900s 内 ≥1 次修改）自动 BGSAVE

BGSAVE 的 Copy-on-Write

fork 子进程时父子共享内存（只读）。父进程修改数据时触发 COW——复制要改的内存页给父进程，子进程读到的始终是 fork 瞬间的快照。

⚠️ 风险：写操作多 → COW 复制大量内存页 → 内存翻倍 → OOM。

📝 AOF 日志：行车记录仪

🚗 类比：行车记录仪持续录像（每条写命令都记录）。车出事故（系统崩溃）？回放录像还原现场。优点：最多丢 1 秒数据。缺点：文件比 RDB 大，恢复慢。

三种刷盘策略

策略配置丢数据性能
每次刷盘appendfsync always不丢最慢
每秒刷盘（推荐）appendfsync everysec最多 1s平衡
从不主动刷appendfsync no不可控最快

AOF 重写（Rewrite）—— 减肥

AOF 文件一直追加会越来越大。重写 = 把多条命令合并。例如 INCR counter 100次 → SET counter 100。BGREWRITEAOF fork 子进程后台做。

RDB vs AOF vs 混合：恢复速度和安全性的三维权衡

混合持久化：RDB + AOF 合体

方案恢复速度数据安全文件大小推荐场景
仅 RDB⭐⭐⭐ 快⭐⭐ 丢分钟级⭐⭐⭐ 紧凑缓存、可接受丢失
仅 AOF⭐ 慢⭐⭐⭐ 丢 1s⭐ 大严格不丢数据
混合 4.0+⭐⭐⭐ 快⭐⭐⭐ 丢 1s⭐⭐ 中等生产推荐 ✅

混合持久化 = AOF 重写时，前半部分写 RDB 格式（快），后半部分写 AOF 增量。恢复时先加载 RDB，再重放 AOF，兼顾速度和安全性。

📡 主从复制：读写分离 + 数据冗余

📰 类比：报社——主编写好文章（Master 写），复印机把文章分发到各个报刊亭（Slave 读）。读者去最近的报刊亭买报纸（读分离），不用跑报社。

复制流程三阶段

  1. 全量同步：从节点连上 → 主节点 BGSAVE 生成 RDB → 发给从节点 → 从节点加载 + 追增量
  2. 增量同步：主节点把写命令写入 replication buffer → 发给从节点
  3. 心跳：从节点每秒 PING 主节点，主节点 10s 一次 PING 从节点

复制风暴

多个从节点同时重启 → 同时请求全量同步 → 主节点多次 BGSAVE → IO/CPU 打满。解法：级联复制（主→从→从），不要让所有从直连主。

🛡️ Sentinel 哨兵：自动故障转移

🏥 类比：ICU 监护仪——持续监控患者心跳（ping Master），心跳停了（Master 挂了）？立即启动抢救（Sentinel 投票选新主），通知护士站（通知客户端新地址）。

Sentinel 架构：哨兵集群监控 → 主观下线 → 客观下线 → 投票选新主

故障转移全流程

  1. SDOWN（主观下线）：一个 Sentinel 发现 Master 不可达
  2. ODOWN（客观下线）：≥quorum 个 Sentinel 都认为 Master 挂了 → 确认
  3. 选举 Leader：Sentinel 之间 Raft 投票选出一个 Leader 执行故障转移
  4. 选新主：Leader 从 Slave 中选最优的（复制偏移量最大、优先级最高）
  5. 通知客户端：Leader 通知其他 Sentinel 和新主地址，客户端通过 Pub/Sub 收到通知

客户端怎么发现新主？

客户端先连 Sentinel，Sentinel 返回当前 Master 地址。切换时 Sentinel 通过 Pub/Sub 推送新地址。Go 客户端（go-redis）自动处理。

🧩 Cluster 集群：16,384 个抽屉

🗄️ 类比：中药房——16,384 个小抽屉（slot），每种药（key）按方子放到固定抽屉。抓药（查询）时直接去对应抽屉，不用翻遍所有柜子。加新柜子？把部分抽屉搬到新柜子上。

Cluster 核心机制

机制说明
Slot 分片16384 个 slot，slot = CRC16(key) % 16384
Hash Tagkey{user1}:name → 只对 {} 内的部分计算 slot → 相关 key 落在同一 slot
MOVED 重定向key 不在当前节点 → 返回 MOVED + 正确节点地址 → 客户端自动跳转
ASK 重定向slot 正在迁移中 → ASK 临时重定向（不影响后续请求）
Gossip 协议节点之间通过 Gossip 交换集群状态（谁负责哪些 slot）
主从每个 slot 的主节点可以有多个从节点 → 主挂了从自动顶上

Cluster vs Sentinel 选型

维度SentinelCluster

🎯 缓存三大经典问题

🔨 缓存穿透

查一个根本不存在的数据 → 每次都穿透缓存打到 DB

解法：布隆过滤器 + 缓存空值（短 TTL）

⚡ 缓存击穿

热点 key 过期 → 瞬间大量请求打到 DB

解法：互斥锁 / 永不过期+异步刷新

❄️ 缓存雪崩

大量 key 同时过期或 Redis 宕机 → DB 瞬间压垮

解法：过期时间加随机值 + 多级缓存 + 熔断限流 + Redis 高可用

🗑️ 淘汰策略：内存满了怎么办？

策略行为适用
noeviction不淘汰，拒绝写入，返回错误不允许丢数据
allkeys-lru所有 key 中淘汰最近最少用的缓存场景（最常用）
volatile-lru有过期时间的 key 中淘汰 LRU缓存 + 持久数据混合
allkeys-lfu所有 key 中淘汰最不频繁用的避免偶发性热点被误淘汰
volatile-ttl有过期时间的 key 中淘汰 TTL 最短的优先淘汰快过期的
allkeys-random所有 key 中随机淘汰简单场景

LRU vs LFU

📚 类比：LRU = 图书馆淘汰最久没被借的书（最近不用的）。LFU = 图书馆淘汰借阅次数最少的书（总是不用的）。LRU 只看时间，LFU 看重频率——热点 key 即使最近没访问也不淘汰。

Redis 的 LRU 是近似 LRU：随机采样 N 个 key，淘汰其中最久未用的（N 可通过 maxmemory-samples 配置）。不是严格 LRU 但够用且省内存。

💾 内存优化：用最少的空间存最多的数据

内存构成

Redis 内存 = 数据内存 + 进程内存 + 缓冲内存（客户端缓冲、复制积压、AOF 缓冲）。

数据内存 = used_memory = 键大小 + 值大小 + 指针开销。键越短越好（"u:1001" 比 "user:1001:profile" 省 10 字节 × 百万 key = 10MB）。

内存优化手段

手段效果
短 key 名百万 key 省几十 MB
Hash 代替 String 存对象ziplist 编码极度省内存
设置 maxmemory + 淘汰策略防止 OOM
hash-max-ziplist-entries 调大更多 Hash 用 ziplist 编码

⚡ Pipeline / 事务 / Lua

Pipeline：一次发一堆命令

📦 类比：快递集包——10 个包裹分 10 次寄 = 10 次往返（RTT）。Pipeline = 10 个包裹打成一个包一次寄出 = 1 次 RTT。注意：不保证原子性！

// Go Pipeline
pipe := rdb.Pipeline()
for i := 0; i < 100; i++ { pipe.Set(ctx, fmt.Sprintf("k%d",i), i, 0) }
_, err := pipe.Exec(ctx)  // 一次 RTT 发 100 条命令

事务：MULTI/EXEC

特性说明
原子性⚠️ 不是严格原子——语法错误全部回滚，运行时错误不回滚
隔离性✅ 事务内命令串行执行，不被其他客户端打断
WATCH乐观锁——watch 的 key 被修改则事务取消

Lua 脚本：真正的原子操作

多条命令要原子执行 → 写 Lua 脚本。整个脚本作为一个原子操作执行，不会被其他命令插入。替代事务的首选。

-- 限流脚本（原子执行）
local current = redis.call('INCR', KEYS[1])
if current == 1 then redis.call('EXPIRE', KEYS[1], ARGV[1]) end
if current > tonumber(ARGV[2]) then return 0 end  -- 超过限制
return 1  -- 放行

Pub/Sub 发布订阅

PUBLISH channel msg → 所有 SUBSCRIBE channel 的客户端收到。注意：消息不持久化，离线后消息丢失。需要可靠消息用 Stream。

❓ 高频考题

Q1: Redis 为什么快？单线程为什么快？
① 内存操作（核心）② 单线程无锁（无上下文切换、无竞争）③ IO 多路复用（epoll）④ 高效数据结构（SDS/跳表/压缩列表）。6.0+ 引入多线程 IO（只处理网络读写，执行命令仍是单线程）。
Q2: Redis 过期 key 怎么删除的？
惰性删除 + 定期删除。惰性：访问 key 时检查，过期就删（省 CPU）。定期：每秒 10 次随机抽 20 个 key 检查，过期比例 >25% 继续抽（主动清理）。两者互补——极端情况可能有大量过期 key 占用内存但未被访问和抽到。
Q3: 缓存和数据库双写一致性怎么保证？
Cache Aside 模式：先更新 DB → 再删除缓存（不是更新缓存！）。为什么要删而不是更新？更新缓存可能导致并发写覆盖。延迟双删兜底：删一次 → 等 500ms → 再删一次。追求强一致性 → 分布式事务/监听 binlog（Canal）。
Q4: Redis Cluster 的 MOVED 和 ASK 有什么区别？
MOVED：slot 永久迁移完成 → 客户端更新本地 slot 映射，后续请求直接走新节点。
ASK：slot 正在迁移中（部分 key 还在旧节点）→ 临时重定向，仅本次请求走新节点，客户端不更新本地映射。
Q5: 怎么用 Redis 实现一个可靠的延迟队列？
ZSet：ZADD delay_queue <执行时间戳> <任务数据>。定时任务 ZRANGEBYSCORE 0 NOW LIMIT 0 100 取出到期任务 → 删除（ZREM）→ 执行业务。删除和执行业务不在同一事务内 → 可能重复执行 → 业务方必须幂等。
Q6: 大 key 和热 key 怎么处理？
大 key：单个 key 的 value 过大（如 hash 百万字段）→ redis-cli --bigkeys 扫描 → 拆分（按日期/用户分片）。
热 key：某个 key QPS 极高 → 多副本（key 复制到多个节点，随机读一个）+ 本地缓存。提前发现：客户端统计 QPS 或 Redis 的 --hotkeys。

💬 苏格拉底式提问 —— 检验真理解

以下问题用来逼你追问本质。先自己想 30 秒，再点开验证。

用 String 存 JSON 有什么问题？Q1
三个问题：
① 更新浪费：改一个字段 → 序列化整个 JSON → 传输整个对象。Hash 可以只更新一个 field。
② 读取浪费：只要一个字段 → 读取整个 JSON → 反序列化。Hash 可以 HGET 一个 field。
③ 内存浪费：字段名在多 key 中重复。Hash 底层用 ziplist/listpack 压缩。
决策：频繁整体读写 → String。频繁更新单个字段 → Hash。
RDB 和 AOF 能同时开吗？用哪个恢复？Q2
可以同时开，强烈建议。
• 恢复优先级：AOF > RDB。AOF 数据更完整（最多丢 1 秒）。
• RDB：快照 → 文件小、恢复快 → 灾备。
• AOF：日志 → 丢数据少 → 主节点。
• 混合持久化（4.0+）：AOF 前半 RDB 快照 + 后半增量命令。
建议：主节点开 AOF（everysec），从节点可只开 RDB。
穿透、击穿、雪崩的本质区别？Q3
三种不同问题，解法不同：
• 穿透：数据根本不存在（DB 里也没有）。📌 布隆过滤器 + 缓存空值。
• 击穿：热点 key 过期，大量请求同时查 DB。📌 互斥锁 + 逻辑过期。
• 雪崩：大量 key 同时过期。📌 过期时间加随机扰动 + 多级缓存 + 限流。
核心区别：穿透 = 数据不存在，击穿 = 一个热点，雪崩 = 批量过期。解题思路完全不同，不可混用。

🎙️ 口播总结

这篇文章是 Redis 的完整攻略。

Redis 为什么快？四个字：内存操作。加三个辅助：单线程无锁、IO 多路复用、高效数据结构。6.0 后加了多线程 IO，但执行命令还是单线程。

九大数据结构各有用途：String 做缓存和计数器、Hash 存对象、List 做消息队列、Set 做共同好友、ZSet 做排行榜、Bitmaps 做签到（365 天只需 46 字节）、HyperLogLog 做 UV 统计（12KB 统计上亿）、GEO 做附近的人、Stream 做可靠消息队列。底层编码会变形——数据少时用紧凑的 ziplist 省内存，数据多时切换为 hashtable 或跳表。

持久化有三种：RDB 定期拍照恢复快但会丢数据、AOF 行车记录仪丢 1 秒但文件大、混合持久化 4.0 版本合二为一。高可用三件套：主从做读写分离、Sentinel 哨兵做自动故障转移、Cluster 集群用 16384 个 slot 做水平扩展。

缓存三大经典问题：穿透查不存在的数据用布隆过滤器挡、击穿热点过期用 singleflight 兜底、雪崩大量同时过期给 TTL 加随机值。淘汰策略首选 allkeys-lru。

如果只能记住一个数字，那就是 16384。16384 个 slot 决定了 Redis Cluster 的分片逻辑、决定了多 key 操作的限制、决定了扩容迁移的粒度。
