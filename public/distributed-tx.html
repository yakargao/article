📋 总览：分布式事务是什么

分布式事务是指跨越多个独立数据库（或服务）的事务操作，保证它们要么全部成功，要么全部失败。在微服务架构中，一个业务操作往往涉及多个服务各自的数据源——传统的本地事务（ACID）无法跨服务生效。

想象你给朋友跨行转账 1000 元：你的钱在 A 银行（账户扣 1000），朋友的钱在 B 银行（账户加 1000）。

如果 A 银行扣了钱，但 B 银行系统挂了没加上——那这 1000 元去哪了？分布式事务就是保证这种"跨行操作"要么两边都完成，要么都不发生，绝不出现"钱扣了没到账"的情况。

就像寄一张双联支票——你和收件人各拿一联，要么两张同时生效，要么都作废。

图 1：跨行转账——分布式事务的典型场景，核心是保证多个参与者的数据一致性

🎯 分布式事务的两种类型

刚性事务：强一致性（ACID），典型代表 2PC/XA。适合对一致性要求极高、性能要求不高的场景。

柔性事务：最终一致性（BASE），典型代表 TCC、Saga、事务消息。适合高性能、高可用的互联网场景。

核心选择：强一致性 vs 最终一致性，没有银弹。

🤔 为什么需要分布式事务

在单体应用时代，所有操作都在同一个数据库里——一条 SQL 搞定 UPDATE account SET balance=balance-100 WHERE id=1 和 UPDATE account SET balance=balance+100 WHERE id=2，数据库的 ACID 事务保证一切。

但到了微服务时代：用户服务、订单服务、支付服务、库存服务各用各的数据库——一个下单操作要扣库存、扣余额、创建订单、发优惠券，跨 4 个数据库，传统事务无力回天。

🟢 单体应用

  • 一个 MySQL 数据库
  • BEGIN TRANSACTION ... COMMIT
  • ACID 完美支持
  • 本地事务就够了
  • 性能高，一致性好

🔴 微服务架构

  • 每个服务独立数据库
  • 没有全局事务管理器
  • 网络可能超时、宕机
  • 部分成功 = 数据不一致
  • 需要分布式事务方案

困境：网络不可靠 + 状态分布式

分布式事务之所以"难"，根源在于两个核心挑战：

  • 网络不可靠：请求可能丢失、超时、重试、乱序。你无法区分"对方处理了但回包丢了"和"对方根本没收到"。
  • 状态分布式：没有一台机器能看到全局状态。协调者只知道"我发了什么"，不知道"参与者实际执行了什么"。

这两个挑战导致了一个理论上的"不可能"——我们接下来看 CAP 和 FLP 定理。

🎯 CAP 理论与 FLP 不可能定理

CAP 定理是分布式系统的"不可能三角"：一致性（Consistency）、可用性（Availability）、分区容错性（Partition Tolerance）三者最多同时满足两个。

分布式事务的选择本质是在 CAP 中做权衡：

  • CP 系统（放弃可用性）：2PC/XA — 保证强一致性，但协调者故障时整个系统不可用
  • AP 系统（放弃强一致性）：TCC / Saga / 事务消息 — 保证最终一致性，系统始终可用
图 2：CAP 三角——分布式事务方案的选型本质是一致性与可用性的权衡

FLP 不可能定理

比 CAP 更底层的是 FLP 不可能定理（Fischer, Lynch, Paterson, 1985）：在异步通信系统中，即使只有一个进程可能故障，也不存在一个确定性的共识算法能保证所有非故障进程在有限时间内达成一致。

翻译成人话：在分布式系统中，你永远无法确定"对方真的挂了"还是"只是回得慢"。 这意味着所有分布式事务方案都是"有损的"——要么牺牲可用性（2PC 阻塞等待），要么牺牲一致性（最终一致），要么引入超时和重试（但超时的设定本身是猜测）。

💡 FLP 对分布式事务的启示

  • 2PC 的阻塞问题源于 FLP：协调者无法区分"参与者挂了"和"参与者慢"，只能等
  • 3PC 引入超时也没根治：超时只是一个"猜"的阈值，猜错了就数据不一致
  • TCC/Saga 的补偿本质是"事后兜底"：承认无法实时达成一致，改为最终一致
  • Paxos/Raft 能做共识，但只解决了状态复制，没解决业务原子性

🔁 2PC / XA —— 两阶段提交

两阶段提交（Two-Phase Commit）是最经典的分布式事务协议，由协调者（Coordinator）和多个参与者（Participant）组成。XA 是 X/Open 组织定义的分布式事务规范，2PC 是其核心实现。

想象你组织一场公司团建聚餐：

第一阶段（Prepare）：你（协调者）给所有人发消息："明天晚上 7 点去 XXX 餐厅，能来的请回复'确认'。"
第二阶段（Commit/Abort）：大家回复后，你统计——如果所有人回复"确认"，你通知"好，确认去！"（Commit）；如果有人回复"去不了"或没回复，你通知"取消，计划作废"（Abort）。

关键问题：如果有人回复确认后突然失联（挂了），你也不知道他明天到不到——这就是 2PC 的"阻塞"问题。

2PC 详细状态机

2PC 不是一个简单的"问一圈再决定"，它的内部状态机如下：

协调者

INIT → WAITING → COMMITTED 或 ABORTED

参与者

INIT → PREPARED → COMMITTED 或 ABORTED

日志关键点

参与者写入 undo/redo 日志后才回复 YES

脑裂场景

协调者在 COMMIT 阶段挂了 → 参与者不确定提交还是回滚

图 3：2PC 两阶段提交——Prepare 阶段询问，Commit/Abort 阶段执行

2PC 的四大致命问题

问题描述发生场景
同步阻塞Prepare 阶段参与者锁定资源，等待协调者决策高并发下资源被长时间占用，系统吞吐骤降
单点故障协调者挂了，参与者不知道是 Commit 还是 Abort协调者进程 OOM、机器宕机
数据不一致协调者发了 Commit，部分参与者因网络没收到网络分区，参与者只收到了 Prepare 没收到 Commit
脑裂协调者与一半参与者网络断开，两边各自决策交换机故障导致集群分裂

3PC：为什么没能解决问题？

三阶段提交（3PC）在 2PC 的基础上引入超时机制和PreCommit 阶段，试图解决 2PC 的阻塞问题：

  1. CanCommit：协调者问"能不能提交？"——参与者只需回复 yes/no，不需要写日志或锁资源
  2. PreCommit：协调者通知"准备提交"——参与者执行事务写 undo/redo 日志，回复 ACK
  3. DoCommit：协调者收到所有 ACK 后通知"正式提交"——参与者提交事务

关键改进：参与者在 PreCommit 后如果超时没收到 DoCommit，自行提交（而不是像 2PC 那样一直等）。

⚠️ 3PC 仍然没有完全解决问题

问题 1：PreCommit 后，如果网络分区导致部分参与者收到了 DoCommit、部分没收到——没收到的参与者超时自行提交，但这时候协调者其实想 Abort（因为在 PreCommit 阶段已经有参与者超时了）。这就产生了数据不一致。

问题 2：3PC 引入了更多的网络交互（3 轮 vs 2 轮），性能更差。

问题 3：FLP 定理说"异步系统无法达成确定性共识"——3PC 只是在概率上降低了风险，但无法根除。

3PC 就像升级了"团建计划"：先问大家"想不想去？"（不锁资源）、再通知"准备去"、最后说"出发"。但如果你说"准备去"后失联，别人只能自己猜——猜错了就出问题。

Go 代码：2PC 协调者伪代码

package twopc

import (
	"context"
	"fmt"
	"sync"
	"time"
)

// Participant 定义事务参与者接口
type Participant interface {
	Prepare(ctx context.Context, txID string) error       // 阶段一：准备
	Commit(ctx context.Context, txID string) error        // 阶段二：提交
	Rollback(ctx context.Context, txID string) error      // 阶段二：回滚
}

// Coordinator 2PC 协调者
type Coordinator struct {
	mu           sync.Mutex
	participants []Participant
	timeout      time.Duration
}

func NewCoordinator(participants []Participant, timeout time.Duration) *Coordinator {
	return &Coordinator{participants: participants, timeout: timeout}
}

// Execute 执行两阶段提交
func (c *Coordinator) Execute(ctx context.Context, txID string) error {
	// === 阶段一：Prepare ===
	c.mu.Lock()
	prepared := make([]int, 0, len(c.participants))
	for i, p := range c.participants {
		pCtx, cancel := context.WithTimeout(ctx, c.timeout)
		if err := p.Prepare(pCtx, txID); err != nil {
			cancel()
			c.mu.Unlock()
			// 任一 Prepare 失败 → 执行回滚
			c.rollbackAll(ctx, txID, prepared)
			return fmt.Errorf("prepare phase failed at participant %d: %w", i, err)
		}
		cancel()
		prepared = append(prepared, i)
	}
	c.mu.Unlock()

	// === 阶段二：Commit ===
	for i, p := range c.participants {
		if err := p.Commit(ctx, txID); err != nil {
			// Commit 失败→数据已不一致，记录严重错误
			// 实际生产需人工介入或重试
			return fmt.Errorf("commit phase failed at participant %d: %w", i, err)
		}
	}
	return nil
}

func (c *Coordinator) rollbackAll(ctx context.Context, txID string, prepared []int) {
	for _, i := range prepared {
		_ = c.participants[i].Rollback(ctx, txID)
	}
}

💡 实际应用

2PC/XA 在传统金融、银行核心系统中仍有应用（如 Tuxedo、WebLogic JTA）。但在互联网高并发场景中，几乎没人用 2PC——性能和可用性无法接受。取而代之的是 TCC、Saga 等柔性事务方案。

2PC 就像让所有人等你一个回复——在快节奏的互联网世界等不起。

✂️ TCC 模式（Try-Confirm-Cancel）

TCC 是分布式事务中应用广泛的补偿型方案。它将一个分布式事务分成三个阶段：Try（预留资源）→ Confirm（确认执行）→ Cancel（取消回滚）。

想象你在美团上预订酒店：

Try 阶段：你提交订单——系统预占了一间房，但还没扣款。就像酒店前台说"好的，我先帮你锁一间房，保留 15 分钟"。
Confirm 阶段：15 分钟内你确认入住——系统扣款，正式生成订单。酒店前台说"已确认入住，房间归你了"。
Cancel 阶段：你取消订单——系统释放预占的房间，不扣钱。酒店前台说"好的，房间释放了"。

关键是：Try 阶段是"占坑"而不是"干活"——把最危险的资源竞争变成资源预留。

图 4：TCC 三阶段流程——Try 预占 → Confirm 确认 → Cancel 回滚

空回滚（Hanging Rollback）与悬挂（Suspension）

TCC 最容易被面试官追问的两个边界问题：

空回滚

场景：网络超时导致 Try 请求丢失，协调者超时后直接调用了 Cancel。

问题：Cancel 执行时，Try 并没有被执行过——如果 Cancel 直接尝试"解冻余额"，而余额根本没被冻结过，就会导致数据错误。

解决：在事务日志中记录 Try 的执行状态。Cancel 先查日志：如果没找到 Try 记录，标记为"空回滚"，直接返回成功。

悬挂

场景：Cancel 已经执行完（可能是空回滚），但 Try 请求因为网络延迟姗姗来迟。

问题：Try 不知道 Cancel 已经执行过了，正常执行预留操作——但这次预留永远不会有 Confirm 来消费它，变成了"孤儿资源"。

解决：Try 方法先查事务日志，如果已存在 Cancel 记录，直接返回成功（不执行预留操作）。

Go 代码：TCC 接口定义与空回滚处理

package tcc

import (
	"context"
	"fmt"
	"sync"
)

// TccService 每个参与者需要实现的 TCC 接口
type TccService interface {
	Try(ctx context.Context, txID string, bizPayload interface{}) error
	Confirm(ctx context.Context, txID string) error
	Cancel(ctx context.Context, txID string) error
}

// TxLog 事务日志——用于处理空回滚和悬挂
type TxLog struct {
	mu      sync.Mutex
	records map[string]int // txID → 0=tryed, 1=canceled, 2=confirmed
}

func NewTxLog() *TxLog {
	return &TxLog{records: make(map[string]int)}
}

func (l *TxLog) Set(txID string, state int) {
	l.mu.Lock()
	defer l.mu.Unlock()
	l.records[txID] = state
}

func (l *TxLog) Get(txID string) (int, bool) {
	l.mu.Lock()
	defer l.mu.Unlock()
	s, ok := l.records[txID]
	return s, ok
}

// AccountService 账户服务 TCC 实现——带空回滚/悬挂处理
type AccountService struct {
	TxLog *TxLog
}

func (a *AccountService) Try(ctx context.Context, txID string, _ interface{}) error {
	state, exists := a.TxLog.Get(txID)
	if exists && state == 1 { // Cancel 已执行→悬挂检测
		fmt.Printf("[%s] 悬挂检测：Cancel已执行，跳过Try\n", txID)
		return nil
	}
	// 正常执行：冻结余额
	fmt.Printf("[%s] Try：冻结余额 100元\n", txID)
	a.TxLog.Set(txID, 0)
	return nil
}

func (a *AccountService) Confirm(ctx context.Context, txID string) error {
	fmt.Printf("[%s] Confirm：扣除冻结余额\n", txID)
	a.TxLog.Set(txID, 2)
	return nil
}

func (a *AccountService) Cancel(ctx context.Context, txID string) error {
	state, exists := a.TxLog.Get(txID)
	if !exists { // Try 没执行→空回滚
		fmt.Printf("[%s] 空回滚：无对应Try，直接返回\n", txID)
		a.TxLog.Set(txID, 1)
		return nil
	}
	if state != 0 { // 已经被处理过了→幂等
		return nil
	}
	fmt.Printf("[%s] Cancel：解冻余额\n", txID)
	a.TxLog.Set(txID, 1)
	return nil
}

// TccCoordinator 协调者
type TccCoordinator struct {
	services []TccService
}

func (c *TccCoordinator) Execute(ctx context.Context, txID string, payload interface{}) error {
	// Try 阶段
	for _, s := range c.services {
		if err := s.Try(ctx, txID, payload); err != nil {
			// 任一 Try 失败→全部 Cancel
			c.cancelAll(ctx, txID)
			return fmt.Errorf("try failed: %w", err)
		}
	}
	// Confirm 阶段
	for _, s := range c.services {
		if err := s.Confirm(ctx, txID); err != nil {
			// Confirm 失败→重试，最终人工介入
			return fmt.Errorf("confirm failed: %w", err)
		}
	}
	return nil
}

func (c *TccCoordinator) cancelAll(ctx context.Context, txID string) {
	for _, s := range c.services {
		_ = s.Cancel(ctx, txID)
	}
}

💡 TCC 的关键点

幂等性：Confirm 和 Cancel 可能被多次调用，必须保证幂等（多用事务日志或去重表）。
空回滚：Cancel 可能在 Try 还没执行时就触发了（比如网络超时），Cancel 必须能正确处理"无操作可回滚"的情况。
悬挂：Cancel 执行完后 Try 才到达，Try 要能检测到"已被取消"并拒绝执行。

🎭 Saga 模式（编排型/协同型）

Saga 是一种长事务（Long-Lived Transaction）解决方案，将一个分布式事务拆分为一系列本地事务，每个本地事务都有对应的补偿事务用于回滚。Saga 有两种实现方式：编排型（Choreography）和协同型（Orchestration）。

想象你订了一份"旅行套餐"：机票 + 酒店 + 租车。

你用编排型 Saga（事件驱动）：
订机票成功 → 发布"机票已订"事件 → 酒店服务监听到事件自动订房 → 酒店订成功发布"酒店已订"事件 → 租车服务监听到事件自动租车。

但如果租车失败了呢？
租车服务发布"租车失败"事件 → 酒店服务监听到后取消预订（补偿） → 机票服务监听到后取消机票（补偿）。

就像多米诺骨牌——往前推是正常流程，往后倒就是补偿回滚。

图 5：Saga——正向事务依次执行，失败后补偿事务逆向回滚

Saga 的隔离性问题

Saga 最大的缺点是缺乏隔离性。不像 ACID 事务的"隔离"（I）——其他事务看不到中间状态，Saga 中每个本地事务提交后，其修改对其他事务立即可见。这可能导致：

  • 丢失更新：事务 A 扣了库存（提交），事务 B 也扣了同一库存（提交），但事务 A 后来补偿回滚了——库存加了回来，相当于事务 B 的扣减被"丢失"。
  • 脏读：事务 A 创建了订单（提交），事务 B 读了这条订单并发了货，但事务 A 后来补偿回滚了订单——B 发了"不存在"的货。

💡 解决方案：Countermeasure（对策）

语义锁：在数据上加一个"脏"标记，其他事务看到后能识别这是"未最终确认"的数据。

交换式更新：使用可逆的操作而不是不可逆的操作（如"订单状态=已取消"而不是"删除订单行"）。

重新读验证：在补偿前重新读取数据，如果发现数据已被其他事务修改，走人工审核。

Go 代码：编排型 Saga 协调者

package saga

import (
	"context"
	"fmt"
)

// SagaStep 定义 Saga 中的一个步骤
type SagaStep struct {
	Name       string
	Action     func(ctx context.Context) error // 正向操作
	Compensate func(ctx context.Context) error // 补偿操作
}

// SagaCoordinator 协同型 Saga 协调者
type SagaCoordinator struct {
	steps []SagaStep
}

func NewSagaCoordinator(steps []SagaStep) *SagaCoordinator {
	return &SagaCoordinator{steps: steps}
}

// Execute 执行 Saga——依次执行 Action，任一失败就执行补偿
func (s *SagaCoordinator) Execute(ctx context.Context) error {
	completed := make([]int, 0, len(s.steps))

	for i, step := range s.steps {
		fmt.Printf("[Saga] 执行步骤 %d: %s\n", i, step.Name)
		if err := step.Action(ctx); err != nil {
			fmt.Printf("[Saga] 步骤 %d (%s) 失败: %v，开始补偿\n", i, step.Name, err)
			// 逆向执行补偿
			s.compensate(ctx, completed)
			return fmt.Errorf("step %s failed: %w", step.Name, err)
		}
		completed = append(completed, i)
	}
	return nil
}

func (s *SagaCoordinator) compensate(ctx context.Context, completed []int) {
	// 逆向遍历已完成的步骤
	for i := len(completed) - 1; i >= 0; i-- {
		step := s.steps[completed[i]]
		fmt.Printf("[Saga] 补偿步骤 %d: %s\n", completed[i], step.Name)
		if err := step.Compensate(ctx); err != nil {
			// 补偿也可能失败→记录日志，最终人工介入
			fmt.Printf("[Saga] ⚠️ 补偿步骤 %s 也失败了: %v\n", step.Name, err)
		}
	}
}

// 使用示例
func Example() {
	saga := NewSagaCoordinator([]SagaStep{
		{
			Name:       "创建订单",
			Action:     func(ctx context.Context) error { return createOrder(ctx) },
			Compensate: func(ctx context.Context) error { return cancelOrder(ctx) },
		},
		{
			Name:       "扣减库存",
			Action:     func(ctx context.Context) error { return deductStock(ctx) },
			Compensate: func(ctx context.Context) error { return restoreStock(ctx) },
		},
		{
			Name:       "扣减余额",
			Action:     func(ctx context.Context) error { return deductBalance(ctx) },
			Compensate: func(ctx context.Context) error { return restoreBalance(ctx) },
		},
	})

	if err := saga.Execute(context.Background()); err != nil {
		fmt.Printf("Saga 最终失败: %v\n", err)
	} else {
		fmt.Println("Saga 执行成功！")
	}
}

💡 Saga 的注意事项

没有隔离性：Saga 不保证隔离性，所以其他事务可能看到中间状态。解决方案有语义锁和交换式更新。
补偿必须是幂等的：补偿事务可能被执行多次，必须保证最终结果一致。
Saga 是"向前恢复"：出错了不退回到初始状态，而是通过一系列补偿事务"对冲"到一致状态。

📨 事务消息（RocketMQ 半消息 vs Kafka 事务）

事务消息将本地事务和消息发送绑定为一个原子操作——要么本地事务和消息都成功，要么都失败。RocketMQ 的事务消息是最经典的实现，通过半消息（Half Message）机制实现。

想象你去邮局寄一封"挂号信"（需要回执的那种）：

你把信寄出，邮局给你一个回执——但这个时候信其实还没真正寄出，只是邮局系统里"暂存"着（半消息状态）。
等你确认内容没问题了，打电话给邮局说"可以发了"——邮局才真正把信投递出去。
如果你说"内容有误，取消"——邮局就把这封信销毁（回滚）。
关键点是：邮局保证"你说发我才发，你说取消我就销毁"——这就是事务消息的"两阶段确认"。

图 6：RocketMQ 事务消息——半消息 + 本地事务 + 反查机制

RocketMQ 事务消息 vs Kafka 事务 Exactly-Once

特性RocketMQ 事务消息Kafka 事务（Exactly-Once）
核心机制半消息 + 反查事务 Coordinator + 幂等 Producer
本地事务绑定通过 TransactionListener 回调通过 Kafka 事务 API 绑定生产与消费

Go 代码：事务消息（模拟 RocketMQ 半消息）

package txmsg

import (
	"context"
	"fmt"
	"sync"
	"time"
)

// HalfMessage 半消息——暂存在 Broker 中不可消费
type HalfMessage struct {
	Topic   string
	Payload interface{}
	Status  string // "pending" | "committed" | "rolledback"
}

// Broker 模拟消息队列
type Broker struct {
	mu       sync.Mutex
	messages map[string]*HalfMessage
}

func NewBroker() *Broker {
	return &Broker{messages: make(map[string]*HalfMessage)}
}

func (b *Broker) SendHalf(txID, topic string, payload interface{}) {
	b.mu.Lock()
	defer b.mu.Unlock()
	b.messages[txID] = &HalfMessage{
		Topic:   topic,
		Payload: payload,
		Status:  "pending",
	}
	fmt.Printf("[半消息] 已暂存 tx=%s topic=%s (消费者不可见)\n", txID, topic)
}

func (b *Broker) Commit(txID string) {
	b.mu.Lock()
	defer b.mu.Unlock()
	if m, ok := b.messages[txID]; ok && m.Status == "pending" {
		m.Status = "committed"
		fmt.Printf("[事务消息] tx=%s 已提交，消息投递到消费者\n", txID)
	}
}

func (b *Broker) Rollback(txID string) {
	b.mu.Lock()
	defer b.mu.Unlock()
	if m, ok := b.messages[txID]; ok && m.Status == "pending" {
		m.Status = "rolledback"
		fmt.Printf("[事务消息] tx=%s 已回滚，消息已删除\n", txID)
	}
}

// TransactionProducer 事务消息生产者
type TransactionProducer struct {
	Broker    *Broker
	LocalTxFn func(ctx context.Context) error // 本地事务
}

func (p *TransactionProducer) SendTransaction(ctx context.Context, txID, topic string, payload interface{}) error {
	// 1. 发送半消息
	p.Broker.SendHalf(txID, topic, payload)

	// 2. 执行本地事务
	if err := p.LocalTxFn(ctx); err != nil {
		// 本地事务失败→回滚消息
		p.Broker.Rollback(txID)
		return fmt.Errorf("local tx failed, message rolled back: %w", err)
	}

	// 3. 本地事务成功→提交消息
	p.Broker.Commit(txID)
	return nil
}

// 使用示例
func Example() {
	broker := NewBroker()
	producer := &TransactionProducer{
		Broker: broker,
		LocalTxFn: func(ctx context.Context) error {
			// 模拟本地事务：创建订单
			fmt.Println("[本地事务] 创建订单成功")
			return nil
		},
	}
	ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
	defer cancel()

	_ = producer.SendTransaction(ctx, "tx-001", "order-topic", map[string]string{"orderId": "123"})
}

💡 事务消息的核心优势

解耦：生产者和消费者通过消息队列解耦，生产者只保证"消息一定发出去"，消费者自行消费。
高性能：相比 2PC 的同步阻塞，事务消息只需一次网络调用 + 本地事务，性能损失极小。
可靠：半消息 + 反查机制解决了"生产者本地事务成功但没来得及提交消息"的问题。
适合场景：下单送积分、注册送优惠券——不需要强一致但必须最终一致。

package main

import (
	"github.com/lithammer/dtm/dtmcli"
	"github.com/lithammer/dtm/dtmutil"
)

// DTM TCC 示例——只需要定义 Try/Confirm/Cancel 三个接口
func main() {
	server := dtmutil.NewServer()

	// 定义 TCC 服务
	server.GET("/api/tcc/trans_out_try", dtmutil.WrapHandler(func(c *dtmcli.Context) {
		// Try：冻结转出金额
		c.JSON(200, dtmcli.ResultSuccess)
	}))
	server.GET("/api/tcc/trans_out_confirm", dtmutil.WrapHandler(func(c *dtmcli.Context) {
		// Confirm：扣减余额
		c.JSON(200, dtmcli.ResultSuccess)
	}))
	server.GET("/api/tcc/trans_out_cancel", dtmutil.WrapHandler(func(c *dtmcli.Context) {
		// Cancel：解冻余额
		c.JSON(200, dtmcli.ResultSuccess)
	}))

	// TCC 全局事务调用
	dtmcli.TccGlobalTransaction(dtmutil.DefaultServer, func(tcc *dtmcli.Tcc) error {
		err := tcc.CallBranch(nil, "http://localhost:8080/api/tcc/trans_out_try",
			"http://localhost:8080/api/tcc/trans_out_confirm",
			"http://localhost:8080/api/tcc/trans_out_cancel")
		return err
	})
}

❓ 实战问答（8 题）

Q1：面试官问你"2PC 有什么问题"，你怎么答才显得有深度？

核心思路：

不要只说"同步阻塞"四个字——展示你对 2PC 状态机的理解：

  1. 同步阻塞：Prepare 阶段参与者写了 undo/redo 日志、锁了资源，但不知道协调者下一步是 Commit 还是 Abort，只能阻塞等待
  2. 单点故障：协调者在 Commit 阶段挂了——参与者处于"不确定状态"，日志已写但不知道提交还是回滚
  3. 脑裂：网络分区导致协调者向一半参与者发了 Commit、另一半没收到——两边永久不一致

加分点：

主动引出 FLP 不可能定理——"2PC 的问题本质上是异步系统无法达成确定性共识的体现。3PC 引入超时也没根除，因为超时阈值本身是猜的。" 这会让面试官眼前一亮。

Q2：2PC 和 TCC 的核心区别是什么？什么场景选 TCC 而不是 2PC？

核心区别：

维度2PCTCC
资源锁定数据库锁（物理锁，同步阻塞）业务预留（逻辑锁，不阻塞）
一致性强一致性（ACID）最终一致性（BASE）
性能低（锁等待时间长）极高（无锁，预留即可）
业务侵入低（XA 接口由数据库实现）高（需实现 Try/Confirm/Cancel）
适用范围参与方都支持 XA 协议任何有网络接口的服务

选 TCC 的场景：

  • 高并发、低延迟要求（互联网核心交易链路）
  • 参与者不支持 XA（如 Redis、MongoDB）
  • 需要精确的资源控制（比如"预占库存 15 分钟"）

Q3：TCC 的空回滚和悬挂问题到底怎么解决？给我看代码思路。

核心思路：

用事务日志表记录每个 txID 的状态（Try / Cancel / Confirm），所有接口先查日志再执行：

空回滚处理（Cancel 中）：

func Cancel(ctx, txID) {
    state := txLog.Get(txID)
    if state == nil {
        // Try 根本没执行过 → 空回滚
        txLog.Set(txID, "canceled")
        return nil // 直接成功，不要报错
    }
    if state != "tryed" {
        return nil // 幂等：已经被处理过了
    }
    // 正常执行解冻
    txLog.Set(txID, "canceled")
}

悬挂处理（Try 中）：

func Try(ctx, txID, ...) {
    state := txLog.Get(txID)
    if state == "canceled" {
        // Cancel 已经执行过了 → 悬挂
        return nil // 跳过预留操作
    }
    // 正常执行预留
    txLog.Set(txID, "tryed")
}

核心口诀：空回滚：还没 Try 就别报错；悬挂：Cancel 过就别执行。

Q4：Saga 的隔离性问题具体是什么？怎么补偿？

隔离性缺失的表现：

Saga 中每个本地事务提交后，修改立即可见——没有"隔离"的保护：

  • 丢失更新：事务 A 和事务 B 同时扣库存。A 扣了 10 → 提交，B 扣了 10 → 提交。然后 A 补偿回滚，库存加了 10 回来——但 B 的 10 是合法扣减，被 A 的补偿"覆盖"了。
  • 脏读：事务 A 创建订单 → 提交，B 读取该订单并发了货。然后 A 补偿取消订单——B 发了"不存在"的货。

怎么补偿（解决）：

  • 语义锁：在每个行上加"脏"标记（如 order.dirty = true），其他事务看到后延迟处理
  • 交换式更新：用"增加一个补偿操作"代替"直接回滚"（加库存而不是恢复库存快照）
  • 后验检查：补偿前检查数据是否被其他事务修改过

Q5：RocketMQ 事务消息和 Kafka 事务（Exactly-Once）有什么区别？

核心区别：

维度RocketMQ 事务消息Kafka 事务
设计目标绑定本地事务和消息发送的原子性流处理的 Exactly-Once 语义
核心机制半消息（先存再确认） + 反查事务 Coordinator + 幂等 Producer
反查有（Broker 回调 Producer）无（超时自动 Abort）
性能1 次额外网络交互2 次额外网络交互（Init + Commit）
典型场景下单→发券（跨服务）Kafka Streams 中的 EOS 语义

一句话：RocketMQ 事务消息适合"业务代码发消息"的场景，Kafka 事务适合"流处理框架内部"的场景。 如果你的业务是"用户下单后发一条消息通知积分服务"，用 RocketMQ 事务消息；如果你在 Kafka Streams 中做"读取→处理→写入"，用 Kafka 事务。

Q6：DTM 的原理是什么？和 Seata 比有什么优劣？

DTM 原理：

DTM（https://github.com/dtm-labs/dtm）是一个 Go 原生实现的分布式事务框架。核心架构：

  • DTM Server（单独部署的协调者服务）：维护全局事务状态，调度子事务
  • 业务服务：只需暴露 Try/Confirm/Cancel（TCC）或 Action/Compensate（Saga）HTTP/gRPC 接口
  • DTM 通过子事务屏障（SubTransaction Barrier）实现空回滚/悬挂/幂等的自动处理

DTM vs Seata：

维度DTMSeata
语言Go（纯 Go 实现，无 Java 依赖）Java（Seata-Go 是移植版）
协议HTTP + gRPC自研 TCP 协议
AT 模式不支持支持（零侵入）
子事务屏障原生支持（防空回滚/悬挂）需手动处理
学习成本低（接口简单）中（AT 模式复杂）

如果你是全 Go 技术栈，选 DTM；如果需要和 Java Spring Cloud 互通，选 Seata。

Q7：Seata AT 模式怎么实现"零侵入"分布式事务的？

核心原理：

Seata AT 模式通过代理数据源 + 自动解析 SQL + undo log 实现零侵入：

  1. 代理数据源：Seata 的 DataSourceProxy 拦截所有 SQL 执行
  2. 前镜像：执行 SQL 前，自动查一次当前数据（如 SELECT balance FROM account WHERE id=1）
  3. 执行业务 SQL：正常执行 UPDATE account SET balance=balance-100 WHERE id=1
  4. 后镜像：执行 SQL 后，再查一次当前数据
  5. 生成 undo log：前镜像和后镜像的差异就是 undo log（用来恢复的"反向 SQL"）
  6. 行锁：在 undo_log 表中写入行锁，防止其他全局事务修改同一行
  7. 事务结束：成功则删除 undo log + 行锁；失败则用前镜像恢复数据

开发者只需要加一个 @GlobalTransactional 注解（Java 中）或 tm.WithGlobalTx()（Seata-Go 中），不需要写任何补偿逻辑。

AT 模式就像请了个自动记账员——你只管做业务，它偷偷拍"前后快照"，出错了自动恢复。

Q8：我有一个 Go 微服务（订单 + 库存 + 支付），怎么选分布式事务方案？给我具体建议。

具体场景分析：

假设你的业务是"用户下单 → 扣库存 → 扣余额 → 创建订单→ 发积分"。

方案选型（按推荐顺序）：

优先级方案理由
🥇 首选DTM TCC纯 Go，性能极高，Try 阶段预留库存/冻结余额，Confirm 阶段正式扣减，Cancel 阶段释放。DTM 自带子事务屏障处理空回滚/悬挂。
🥈 备选DTM Saga如果"发积分"是可选步骤（失败了不影响主流程），用 Saga。积分扣失败只补偿积分，不影响订单和库存。

选型口诀：高性能用 TCC，长事务用 Saga，异步解耦用消息，强一致才用 2PC。

🎙️ 口播总结

好，我们来快速捋一遍分布式事务的实战知识体系。

第一，分布式事务为什么难？核心原因有两个：网络不可靠和状态分布式。CAP 定理告诉我们，一致性、可用性、分区容错性三者只能选两个。FLP 不可能定理更进一步——在异步系统中，你永远无法确定"对方挂了"还是"只是回得慢"，所以所有分布式事务方案都是"有损的"。

第二，四大核心技术方案。2PC 两阶段提交——协调者问一圈再决定，最大问题是同步阻塞和单点故障。理解它用团建聚餐来想：先问大家能不能来，再定去不去。3PC 引入超时也没根除问题——超时阈值本身是猜的。TCC 用 Try-Confirm-Cancel 三阶段——用酒店预订来理解：先占坑，再确认入住，或取消释放。特别注意空回滚（没 Try 就 Cancel）和悬挂（Cancel 了 Try 才到）。Saga 用正向事务+补偿事务——用旅行套餐来想：先订机票再订酒店，租车失败就反向取消一切。Saga 最大的坑是没有隔离性——小心丢失更新和脏读。事务消息用 RocketMQ 半消息——用挂号信来想：先写好感暂存，确认无误再真正投递。

第三，Go 生态怎么做？推荐 DTM（纯 Go 实现，支持 TCC/Saga/2PC/事务消息，HTTP+gRPC）和 Seata-Go（和 Java Seata 互通）。不要从零自己造轮子——空回滚、悬挂、幂等、并发控制这些边界问题极其复杂，用成熟的框架比自己实现靠谱得多。

最后记住选型口诀：高性能用 TCC，长事务用 Saga，异步解耦用消息，强一致才用 2PC。实战时除了讲原理，一定要主动说出每种方案的缺点和边界问题——这才能体现你真的理解分布式事务的本质。
