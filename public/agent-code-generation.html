💻 代码生成 Agent — SWE-Agent/Cursor/Devin 原理 & Go 实现

从「补全助手」到「自主程序员」—— 代码生成 Agent 的技术全景与实践

1. 为什么代码生成需要 Agent

想象一家汽车修理厂。传统 AI 编程助手像是一个只会拧螺丝的实习生——你告诉他"拧这个螺栓"，他照做，但不知道为什么要拧、下一步做什么。而 Agent 像一位高级工匠：他看到刹车片磨损、检查制动液、对比维修手册、决策更换方案、执行修复、最后路试验证。

只会写 Hello World 的程序员和能独立构建微服务架构的程序员之间的差距，就是 工具调用 → 目标驱动 Agent 的差距。

传统辅助编程 vs Agent 编程：

❌ 传统 AI 辅助
  • 单次补全，无上下文记忆
  • 用户手动定位问题
  • 不会运行代码验证
  • 无错误修复闭环
  • "问一句答一句"
✅ Agent 编程
  • 多步推理，维护对话图
  • 自动探索仓库结构
  • 运行测试验证修改
  • 解析错误 → 自愈循环
  • 目标驱动，自主规划

2. 代码仓库理解

Agent 要像资深开发者一样理解仓库，需要三层能力：AST 解析 → 依赖分析 → 调用链追踪。

2.1 AST 解析 + 调用图构建 (Go 实现)

以下 Go 代码读取 Go 源文件，提取函数声明和调用关系，构建调用图：

package codegraph

import (
    "go/ast"
    "go/parser"
    "go/token"
    "os"
    "fmt"
)

// FuncNode 代表一个函数节点
type FuncNode struct {
    Name     string
    File     string
    Line     int
    Calls    []string // 调用的函数名列表
    CalledBy []string // 被谁调用
}

type CallGraph struct {
    Nodes map[string]*FuncNode
}

func BuildCallGraph(filePath string) (*CallGraph, error) {
    fset := token.NewFileSet()
    f, err := parser.ParseFile(fset, filePath, nil, parser.ParseComments)
    if err != nil {
        return nil, err
    }

    cg := &CallGraph{Nodes: make(map[string]*FuncNode)}

    // 第一遍：收集所有函数声明
    for _, decl := range f.Decls {
        funcDecl, ok := decl.(*ast.FuncDecl)
        if !ok { continue }
        name := funcDecl.Name.Name
        pos := fset.Position(funcDecl.Pos())
        cg.Nodes[name] = &FuncNode{
            Name:     name,
            File:     filePath,
            Line:     pos.Line,
            Calls:    []string{},
            CalledBy: []string{},
        }
    }

    // 第二遍：提取调用关系
    for _, decl := range f.Decls {
        funcDecl, ok := decl.(*ast.FuncDecl)
        if !ok { continue }
        caller := funcDecl.Name.Name
        ast.Inspect(funcDecl.Body, func(n ast.Node) bool {
            callExpr, ok := n.(*ast.CallExpr)
            if !ok { return true }
            ident, ok := callExpr.Fun.(*ast.Ident)
            if !ok { return true }
            callee := ident.Name
            // 跳过自身
            if callee == caller { return true }
            if node, exists := cg.Nodes[callee]; exists {
                cg.Nodes[caller].Calls = append(cg.Nodes[caller].Calls, callee)
                node.CalledBy = append(node.CalledBy, caller)
            }
            return true
        })
    }
    return cg, nil
}

func main() {
    cg, err := BuildCallGraph("main.go")
    if err != nil {
        fmt.Println(err)
        return
    }
    for name, node := range cg.Nodes {
        fmt.Printf("%s (%s:%d) calls: %v\n", name, node.File, node.Line, node.Calls)
    }
}

2.2 依赖分析与调用链追踪

理解代码不仅仅是解析语法。Agent 还需：

  • 模块依赖图：分析 go.mod / package.json / Cargo.toml 等
  • 跨文件引用：追踪类型定义和使用的关系
  • 数据流分析：变量从创建到使用的完整路径
  • 脏数据标记：当文件 A 修改时，标记所有依赖 A 的文件为"需检查"

3. 代码编辑三步曲 — 搜索→定位→修改→验证

代码编辑是 Agent 的核心能力。我们将其拆解为四个阶段：

3.1 搜索相关代码片段

使用语义搜索 + 关键词 + 正则组合，快速定位相关代码。

3.2 定位修改点

根据搜索到的信息，结合 AST 定位需要修改的精确位置（行号、AST 节点路径）。

3.3 生成补丁 + 运行测试

package codeedit

import (
    "bytes"
    "fmt"
    "os/exec"
    "regexp"
    "strings"
)

// SearchResult 搜索结果
type SearchResult struct {
    File    string
    Line    int
    Content string
}

type EditEngine struct {
    workspace string
}

func NewEditEngine(workspace string) *EditEngine {
    return &EditEngine{workspace: workspace}
}

// Search 支持文本搜索和正则搜索
func (e *EditEngine) Search(pattern string, isRegex bool) ([]SearchResult, error) {
    cmd := exec.Command("rg", "-n", pattern, e.workspace)
    if !isRegex {
        cmd = exec.Command("rg", "-F", "-n", pattern, e.workspace)
    }
    out, err := cmd.Output()
    if err != nil {
        return nil, fmt.Errorf("search failed: %w", err)
    }
    var results []SearchResult
    for _, line := range strings.Split(string(out), "\n") {
        if line == "" { continue }
        parts := strings.SplitN(line, ":", 3)
        if len(parts) < 3 { continue }
        lineNo := 0
        fmt.Sscanf(parts[1], "%d", &lineNo)
        results = append(results, SearchResult{
            File:    parts[0],
            Line:    lineNo,
            Content: parts[2],
        })
    }
    return results, nil
}

// Replace 执行文件内替换 — 生成类补丁操作
func (e *EditEngine) Replace(file, oldText, newText string) error {
    cmd := exec.Command("sed", "-i",
        fmt.Sprintf("s/%s/%s/g", oldText, newText),
        file)
    return cmd.Run()
}

// RunTests 运行测试并返回结果
func (e *EditEngine) RunTests(pkg string) (bool, string, error) {
    cmd := exec.Command("go", "test", "-v", "-run", pkg, e.workspace)
    var out bytes.Buffer
    cmd.Stdout = &out
    cmd.Stderr = &out
    err := cmd.Run()
    output := out.String()
    if err != nil {
        return false, output, nil // 测试失败但不是 fatal
    }
    // 检查是否全部通过
    passed := strings.Contains(output, "PASS") || !strings.Contains(output, "FAIL")
    return passed, output, nil
}

4. 错误定位与自动修复

编译错误和测试失败是软件开发中最耗时的环节。Agent 可以自动化这一过程：

  1. 解析错误信息：从编译器/测试框架输出中提取文件、行号、错误类型
  2. 定位代码：根据行号和错误类型找到问题代码
  3. 生成修复：LLM 根据错误上下文生成修复方案
  4. 重跑测试：应用修复后重新运行验证
🔧 实际案例：SWE-Agent 在 SWE-bench 中通过该循环解决了超过 12% 的 GitHub Issue，完全自主地定位、修改并验证修复。

4.1 编译错误解析示例

package errorfix

import (
    "regexp"
    "strconv"
)

// CompileError 编译错误
type CompileError struct {
    File    string
    Line    int
    Column  int
    Message string
}

var goErrorRe = regexp.MustCompile(
    `^([^:]+):(\d+):(\d+):\s*(.+)$`,
)

func ParseGoError(output string) []CompileError {
    var errors []CompileError
    for _, match := range goErrorRe.FindAllStringSubmatch(output, -1) {
        line, _ := strconv.Atoi(match[2])
        col, _ := strconv.Atoi(match[3])
        errors = append(errors, CompileError{
            File:    match[1],
            Line:    line,
            Column:  col,
            Message: match[4],
        })
    }
    return errors
}

5. SDK 和 API 适配

代码生成 Agent 另一个强大能力是自动学习第三方 API 并生成调用代码。核心技术是 RAG over 文档：

  1. 文档索引：爬取官方文档，分块并向量化
  2. 检索增强：用户需求 → 检索相关文档片段
  3. 代码生成：LLM 根据文档片段 + 用户需求生成 API 调用代码
  4. 验证：可选运行测试检查 API 调用是否正确
📚 示例场景：用户想使用 Stripe API 创建订阅。Agent 检索 Stripe 文档中关于 checkout.sessions.create 的章节，理解参数要求，然后生成完整的 Go 调用代码，包括错误处理和重试逻辑。

7. MCP 在代码场景的应用

MCP (Model Context Protocol) 为 Agent 提供了标准化的工具接口。在代码场景中，三个 MCP 服务器至关重要：

  • FileSystem MCP：读写文件、目录遍历、文件搜索
  • Git MCP：提交、分支、diff、log
  • Terminal MCP：执行命令、编译、运行测试

7.1 代码编辑 MCP Server (Go)

package main

import (
    "context"
    "encoding/json"
    "fmt"
    "os"
    "os/exec"
)

// MCP 工具定义
type Tool struct {
    Name        string          `json:"name"`
    Description string          `json:"description"`
    InputSchema map[string]interface{} `json:"inputSchema"`
}

type MCPServer struct {
    tools []Tool
}

func NewCodeEditorMCP() *MCPServer {
    return &MCPServer{
        tools: []Tool{
            {
                Name:        "read_file",
                Description: "读取文件内容",
                InputSchema: map[string]interface{}{
                    "type": "object",
                    "properties": map[string]interface{}{
                        "path": map[string]string{"type": "string"},
                    },
                },
            },
            {
                Name:        "write_file",
                Description: "写入文件（覆盖）",
                InputSchema: map[string]interface{}{
                    "type": "object",
                    "properties": map[string]interface{}{
                        "path":    map[string]string{"type": "string"},
                        "content": map[string]string{"type": "string"},
                    },
                },
            },
            {
                Name:        "git_diff",
                Description: "查看当前变更",
            },
            {
                Name:        "run_test",
                Description: "运行测试",
                InputSchema: map[string]interface{}{
                    "type": "object",
                    "properties": map[string]interface{}{
                        "pkg": map[string]string{"type": "string"},
                    },
                },
            },
        },
    }
}

// HandleToolCall 处理工具调用请求
func (s *MCPServer) HandleToolCall(name string, args json.RawMessage) (string, error) {
    switch name {
    case "read_file":
        var params struct { Path string `json:"path"` }
        json.Unmarshal(args, &params)
        data, err := os.ReadFile(params.Path)
        if err != nil { return "", err }
        return string(data), nil

    case "write_file":
        var params struct {
            Path    string `json:"path"`
            Content string `json:"content"`
        }
        json.Unmarshal(args, &params)
        err := os.WriteFile(params.Path, []byte(params.Content), 0644)
        if err != nil { return "", err }
        return fmt.Sprintf("written %d bytes to %s", len(params.Content), params.Path), nil

    case "git_diff":
        out, _ := exec.Command("git", "diff").Output()
        return string(out), nil

    case "run_test":
        var params struct { Pkg string `json:"pkg"` }
        json.Unmarshal(args, &params)
        cmd := exec.Command("go", "test", "-v", "-run", params.Pkg)
        out, _ := cmd.CombinedOutput()
        return string(out), nil
    }
    return "", fmt.Errorf("unknown tool: %s", name)
}

8. 安全考量

代码生成 Agent 涉及 代码执行，安全风险不容忽视：

🛡️ 代码执行沙箱

Agent 生成的代码不应直接在宿主机执行。推荐使用 Docker 容器、gVisor、Firecracker 微VM 等隔离技术。每次任务启动独立的沙箱环境，任务结束销毁。

🔑 权限控制

遵循最小权限原则：Git MCP 只给仓库写权限，FileSystem MCP 限制可操作的目录范围，Terminal MCP 禁止网络访问除非显式授权。

📦 供应链安全

Agent 可能自动安装依赖。需要：依赖来源校验（checksum）、已知漏洞扫描、许可合规检查、安装前审批流程。

9. QA & 总结

常见问题

代码生成 Agent 和 Copilot 有什么区别？

Copilot 是补全工具，Agent 是编程伙伴。 Copilot 根据上下文预测下一段代码，但不会主动理解仓库结构、不会运行测试、不会修复错误。而 Agent 像一位 junior developer 实习生：给它一个 Issue，它能自主定位代码、修改、运行测试、提交 PR。

打个比方：Copilot 是智能输入法，Agent 是写作助理。

SWE-Agent 的 Agent-SW 交互协议是什么？

Agent-SW (Agent-Computer Interface) 是 SWE-Agent 团队提出的一种人机交互协议，定义了 Agent 与软件环境交互的标准化操作集，包括：

  • 查看：view_file、search_dir、grep
  • 编辑：edit_file（基于行号的替换）
  • 执行：run_command（沙箱内）

相比 MCP，Agent-SW 更关注代码场景的专有操作，而 MCP 是通用协议。

如何评估代码生成 Agent 的能力？

目前最权威的评测基准是 SWE-bench（以及 SWE-bench Multilingual）：

  • 从真实 GitHub 仓库中提取 Issue + 关联 PR
  • Agent 需要理解 Issue、定位代码、生成修改、通过测试
  • 评分方式：修改后的代码是否通过所有单元测试

其他评测：HumanEval（函数补全）、RepoBench（仓库级任务）、CodeEditorBench（编辑类任务）。

Go 语言适合实现代码生成 Agent 吗？

非常适合。 原因：

  • 标准库 AST 支持：go/ast、go/parser、go/token 让 Go 代码分析开箱即用
  • 编译速度快：Agent 需要频繁编译验证，Go 的秒级编译是巨大优势
  • 并发模型：goroutine 天然适合并行搜索、多文件分析
  • 静态链接：单二进制部署，便于沙箱分发

不过，如果 Agent 需要分析多语言（Python/JS/TS），Go 需要借助 Tree-sitter 等通用解析器。

代码生成 Agent 的 token 成本大概多少？

以 SWE-Agent 使用 GPT-4o 为例：

  • 平均每次任务：~200K 输入 token + ~10K 输出 token
  • 成本：约 $0.3-$0.5/任务（GPT-4o 价格）
  • 使用 Claude 3.5 Sonnet：约 $0.2-$0.4/任务
  • 复杂任务（跨多个文件、多次迭代）：可达 $1-$2

相比人工开发者数小时的工作，这个成本非常有竞争力。Cursor 的订阅制 ($20/月) 则适合日常高频使用。

MCP 协议在代码场景中最大的挑战是什么？

主要有三个挑战：

  1. 原子性：多步编辑操作需要事务支持——要么全部成功要么全部回滚。目前 MCP 没有内建事务语义
  2. 上下文管理：Agent 可能同时打开多个文件、执行多个命令，MCP 目前缺乏请求追踪能力
  3. 安全边界：Terminal MCP 如果暴露给不信任的模型，可能造成严重安全风险。需要严格的授权和审计

社区正在推动 MCP 规范增加批处理和安全相关扩展。

代码生成 Agent 会取代程序员吗？

不会取代，但会重塑工作方式。

就像 IDE 没有取代程序员、版本控制没有取代程序员一样，Agent 是生产力的又一次跃升。它会：

  • 接管重复性编码（CRUD、样板代码、测试编写）
  • 加速 Bug 修复和调试
  • 帮助新成员快速理解代码仓库
  • 让程序员聚焦于架构设计、系统思考、用户理解

引用一位 SWE-Agent 用户的话："Agent 不是来抢工作的，是来消除无聊的。"

📺 口播总结

大家好，今天我们一起深入探讨了代码生成 Agent 的原理与实践。

我们从为什么需要 Agent 开始——传统 AI 编程助手就像只会写 Hello World 的实习生，而 Agent 是能独立解决问题的高级工匠。两者的差距在于：工具调用 → 目标驱动。

然后我们看了 SWE-Agent 是如何通过 Agent-SW 协议 理解仓库、定位代码、生成修复的；Cursor 是如何在 IDE 中提供丝滑的交互体验的；Devin 是如何在云端全自主完成复杂任务的。

我们用 Go 代码实现了 AST 解析 + 调用图构建、代码搜索与编辑引擎、以及 MCP 代码编辑 Server——这些是构建你自己的代码 Agent 的基础组件。

MCP 协议 正在成为 Agent 工具调用的标准——FileSystem、Git、Terminal 三个 MCP Server 就能覆盖大部分代码场景。而 安全 始终是底线：沙箱隔离、最小权限、供应链检查，缺一不可。
