📋 总览：NoSQL 是什么

NoSQL（Not Only SQL）是非关系型数据库的统称。它打破了关系型数据库（MySQL/PostgreSQL）的"表-行-列"模式，专为高并发、海量数据、灵活 Schema 的场景而生。NoSQL 不是要取代 SQL，而是——用合适的工具解决合适的问题。

想象一个超级市场：传统 SQL 数据库像超市的仓储系统——所有商品按品类、货架号、层数精确登记（严格 Schema），找东西很准但每次都要查登记本（Join），人多就排队。

NoSQL 像地摊集市——每个摊位（不同数据库）想怎么摆就怎么摆（灵活 Schema），你直接去对应摊位拿东西（按 Key 查），速度快得多。但代价是——如果你想"找出所有红色的商品"，可能得跑遍全场（缺乏复杂查询能力）。

所以 NoSQL 和 SQL 是互补关系：SQL 管精度，NoSQL 管速度；SQL 管关系，NoSQL 管扩展。

图1：NoSQL 四大类型概览，本文重点讲解文档型(MongoDB)、搜索引擎(ES)、列族型(ClickHouse)

🤔 为什么需要 NoSQL

传统关系型数据库在处理以下场景时力不从心：

  • 高并发写入：每秒数万条日志写入，MySQL 单库扛不住
  • 灵活 Schema：日志字段不固定，改表结构需要停机
  • 海量数据：PB 级数据，分库分表维护成本高
  • 全文检索：MySQL LIKE '%keyword%' 不走索引，性能差
  • 大数据分析：行式存储对聚合查询不友好

想象一个图书馆（SQL）：每本书都要按 ISBN、分类、作者精确登记。找书非常精准。但是——

突然来了 100 万本没有 ISBN 的手稿（非结构数据），你没法按老规矩登记了。这时候你需要不同的"知识管理方式"：

• MongoDB = 大箱子，每份手稿随便往里塞，帖个标签就完事（文档存储）

• ElasticSearch = 把每份手稿的每个字都抄到索引卡上，方便搜索（全文检索）

• ClickHouse = 把所有手稿按"颜色"分类统计（列式分析）

所以 NoSQL 不是来替代\"图书馆\"的——是来管理那些\"图书馆管不了的东西\"。

🍃 MongoDB — 文档型数据库之王

1. 文档模型（Document Model）

MongoDB 的数据结构是 BSON（Binary JSON），每条记录是一个 Document，多个 Document 组成 Collection（相当于 SQL 的"表"）。最大的特点是：同一个 Collection 里的文档字段可以不一样。

📦 SQL（关系模型）

-- 用户表
CREATE TABLE users (
  id INT PRIMARY KEY,
  name VARCHAR(50),
  email VARCHAR(100),
  address_id INT
);
-- 订单表（需要 JOIN）
CREATE TABLE orders (...);
-- 地址表（需要 JOIN）
CREATE TABLE addresses (...);

🍃 MongoDB（文档模型）

// 一个文档搞定一切！
{
  _id: ObjectId("..."),
  name: "张三",
  email: "zhang@example.com",
  orders: [
    { item: "手机", price: 2999 },
    { item: "壳", price: 29 }
  ],
  address: {
    city: "北京",
    street: "长安街"
  }
}

SQL 像\"图书馆登记系统\"——书（订单）和人（用户）分两张表登记，想看谁借了什么书要做 JOIN（关联查询）。

MongoDB 像\"每个人的档案袋\"——直接把这个人借的所有书、地址、联系方式都塞进一个档案袋里。想看一个人的全部信息，打开一个档案袋就行，不用翻不同柜子（JOIN）。

这种 反范式（Denormalization） 的设计，适合读多写少、数据关系简单的场景。但如果你需要复杂的多对多关系（社交网络），文档模型就不如关系型或图数据库。

2. 复制集（Replica Set）

复制集是 MongoDB 的高可用方案。每个复制集通常包含 3 个节点：

图2：MongoDB 复制集 — 主节点接收写请求，从节点异步复制 Oplog，主节点宕机后自动选举新主

核心机制

  • Oplog（操作日志）：Primary 的所有写操作记录在 Oplog 中，Secondary 不断拉取并重放
  • 自动选举：Primary 宕机后，Secondary 通过 Raft 协议变种 选举新 Primary
  • 多数派写入：w: "majority" 保证写操作被多数节点确认后才返回，防止数据丢失
  • 读写分离：读操作可配置为 readPreference: secondaryPreferred 分摊主节点压力

3. 分片集群（Sharding）

当单机无法承载数据量和吞吐量时，MongoDB 通过 分片（Sharding） 实现水平扩展。

图3：MongoDB 分片集群 — mongos 路由请求，Config Server 存元数据，每个分片是一个独立复制集

分片键（Shard Key）选择

分片键的选择极其重要，选错了会导致数据倾斜或性能瓶颈：

  • ✅ 好 高基数：如 _id、用户 ID，值分布均匀
  • ✅ 好 单调递增：如时间戳（需配合 Hashed Sharding）
  • ❌ 差 低基数：如性别（只有男女），数据倾斜严重
  • ❌ 差 纯单调递增：如自增 ID，所有新数据写入同一个分片

分片策略：Range Sharding（范围分片，适合范围查询）vs Hashed Sharding（哈希分片，适合均匀分布）。

🔍 ElasticSearch — 搜索引擎的标杆

1. 倒排索引（Inverted Index）

ElasticSearch 的核心数据结构是倒排索引。传统 SQL 的索引是"文档→关键词"，而倒排索引是"关键词→文档"。

想象你在图书馆找书：

正向索引（SQL）就像——你去书架上一本一本地翻书皮（全表扫描），或者先查目录看哪些页有你要的词（B+树索引）。

倒排索引（ES）就像——图书馆门口有个大公告板，上面写着：

  • "分布式" → 出现在第1、3、7、12本书
  • "数据库" → 出现在第2、3、5、8、11本书
  • "一致性" → 出现在第3、6、9本书

你要找同时包含"分布式"+"数据库"的书——直接在公告板上取交集！第3本书命中。根本不需要翻书！

图4：正向索引 vs 倒排索引对比 — 倒排索引以"词"为索引，搜索时直接查词典即可

2. 分词（Analysis）与聚合（Aggregation）

分词器（Analyzer）

ES 的分词器由三部分组成：Character Filter（字符过滤）→ Tokenizer（分词器）→ Token Filter（词元过滤）。

// 中文分词示例
POST _analyze
{
  "analyzer": "ik_smart",  // IK 中文分词器
  "text": "我是中国人"
}
// 结果: ["我", "是", "中国人"]

// 英文分词
POST _analyze
{
  "analyzer": "standard",
  "text": "I love ElasticSearch"
}
// 结果: ["i", "love", "elasticsearch"]  → 小写化

聚合（Aggregations）

ES 的聚合分析非常强大，可以替代部分 SQL 的 GROUP BY 功能：

// 按价格区间统计订单数
GET orders/_search
{
  "size": 0,
  "aggs": {
    "price_ranges": {
      "range": {
        "field": "price",
        "ranges": [
          { "to": 100 },
          { "from": 100, "to": 500 },
          { "from": 500 }
        ]
      }
    }
  }
}

分词 = 把一句话切成有意义的\"积木块\"。

"我是中国人" → IK 分词器知道\"中国人\"是一个整体词，而不是切成\"中\"\"国\"\"人\"（那样就失去语义了）。

聚合 = 把搜索结果按条件\"分堆\"。就像你在水果摊问\"帮我统计一下每种水果的均价\"——摊主把所有苹果放一堆算均价，所有香蕉放一堆算均价。这就是 ES 的 Terms Aggregation。

3. 集群架构

图5：ES 集群架构 — 每个 Index 分为多个 Shard，Shard 有 Primary/Replica 副本，分布在多个节点上

节点角色

  • Master Node：管理集群状态、分片分配。建议 3 个专用 Master 节点（奇数）
  • Data Node：存储数据、执行搜索和聚合。核心节点，需要大内存+快速磁盘
  • Ingest Node：文档写入前的预处理管道（数据清洗、字段提取）
  • Coordinating Node：仅做请求转发和结果归并，不存数据

写入流程

  • Client 向任意节点发送写请求 → 协调节点计算文档属于哪个 Shard → 转发到 Primary Shard
  • Primary Shard 写入 Lucene 的 Memory Buffer（内存缓冲区）
  • 定时 Refresh（默认 1s）→ 生成 Segment（段），此时文档才可被搜索
  • 定时 Flush → 将 Segment 持久化到磁盘并写入 Commit Point + Translog

ES 写入就像餐厅出菜流程：

• Memory Buffer = 厨师的案板（还没下锅，别人吃不到）

• Refresh = 把菜倒进展示柜（虽然还没完全凝固，但客人可以看到了）

• Flush = 做好的菜摆到正式冷柜（不怕停电丢了）

所以 ES 是近实时搜索（默认 1 秒后可见），不是实时！如果写完后立刻搜索不到，很正常。

🟢 ClickHouse — 列式存储的扛把子

1. 列式存储（Columnar Storage）

ClickHouse 的核心是列式存储——数据按列而不是按行组织。这和 MySQL/PostgreSQL 的行式存储有本质区别：

图6：行式存储 vs 列式存储 — 列存只读取需要的列，I/O 减少 + 高压缩比

行式存储就像\"每个人的档案袋\"——想看工资总和？你得把所有人的档案袋拆开，只看工资那一行，其他信息（姓名、地址、爱好）都白白读了。

列式存储就像\"把所有工资写在同一张纸上\"——想看工资总和？直接拿这张纸加一下就行，根本不用碰其他信息。

而且同一列的数据类型相同（全是数字），压缩率极高——比如\"性别\"列全是\"男\"\"女\"，可以压缩到原始大小的 1/10。

2. 向量化执行（Vectorized Execution）

ClickHouse 的执行引擎不是逐行处理，而是批量（Block）处理——一次处理一批数据（默认 4096 行），利用 CPU 的 SIMD（单指令多数据） 指令集并行计算。

-- 逐行处理（传统数据库）
for each row: result += row.salary

-- 向量化处理（ClickHouse）
-- 一次加载 4096 个 salary 值到 CPU 寄存器
-- 一条 SIMD 指令完成 4096 次加法
-- 效率提升数十倍

3. 查询优化技巧

  • 分区（Partition）：按日期分区，查询时直接裁剪不相关分区。如 PARTITION BY toYYYYMM(date)
  • 排序键（ORDER BY）：数据按指定列物理排序，范围查询极快。如 ORDER BY (date, city)
  • 物化视图（Materialized View）：预聚合加速，写入时实时计算。如每分钟统计一次 PV/UV
  • 稀疏索引：每 8192 行记录一个 Mark，避免全表扫描
  • 采样查询：SAMPLE 0.1 只查 10% 数据，快速获取近似结果

🛠️ ClickHouse vs MySQL 聚合速度对比

场景：对 10 亿行日志按日期统计 PV

  • MySQL：扫描 10 亿行，读整个行 → 耗时约 30-60 秒
  • ClickHouse：只读 3 列（日期、URL），列存压缩后仅扫描 1/10 数据，向量化执行 → 耗时 0.3-1 秒

这个量级的差距，正是列式存储 + 向量化执行带来的红利。

⚖️ 选型对比：MongoDB vs ES vs ClickHouse vs Redis

四种 NoSQL 数据库各有侧重，选型的关键是匹配业务场景：

图7：NoSQL 四巨头全维度对比 — 红色标⭐为优势领域

🔍 选型决策树

  • 需要全文搜索 + 日志分析？ → ES
  • 需要灵活 Schema + 高速读写 + 复制集高可用？ → MongoDB
  • 需要海量数据聚合分析（OLAP）？ → ClickHouse
  • 需要微秒级缓存 + 计数器 + 分布式锁？ → Redis
  • 既需要搜索又需要存储？ → MongoDB（存）+ ES（搜），使用 MongoDB Connector for ES 同步
  • 实时分析 + 大屏展示？ → Kafka → ClickHouse 是经典组合

选数据库就像选择交通工具：

🍃 MongoDB = SUV（家庭用车，啥都能干一点，但不是最强的）

🔍 ES = 搜索救援队（专精找东西，其他方面一般）

🟢 ClickHouse = 货运卡车（专拉数据分析这趟活，超强）

🔑 Redis = 跑车（速度就是一切，但装不了多少东西）

没有最好的数据库，只有最合适的。一个成熟的业务通常会同时使用 2-3 种 NoSQL + SQL 组合。

❓ 实战问答

以下 8 道题覆盖 NoSQL 实战的常见考点：

Q1：MongoDB 复制集如何实现故障自动切换？

核心思路：

MongoDB 复制集通过 Raft 协议变种 实现自动选主：

  1. 所有 Secondary 节点每 2 秒 向 Primary 发送心跳
  2. 如果 10 秒 内未收到心跳响应，Secondary 认为 Primary 已宕机
  3. 触发选举：剩余节点通过 多数派投票（需要超过半数节点同意）选出新 Primary
  4. 新 Primary 开始接受写入，从节点切换 Oplog 同步源
  5. 旧 Primary 恢复后，自动降级为 Secondary 并重新同步

关键参数：electionTimeoutMillis（默认 10s）可调整切换灵敏度。

Q2：MongoDB 分片键如何选择？分片策略有哪些？

核心思路：

分片键是分片集群的最关键决策，直接影响性能和扩展能力。

好分片键的特征：高基数（值分布广泛）、写入均匀、查询带分片键。

分片策略：

  • Range Sharding：按值范围分片（如 0-100 → Shard A, 100-200 → Shard B）。适合范围查询，但可能导致数据倾斜
  • Hashed Sharding：对分片键 Hash 后取模。写入均匀分布，但不支持范围查询
  • Zone Sharding：按地域或标签分片（如中国用户存国内节点，美国用户存美西节点）

注意：分片键一旦选定，不能修改！所以需要在业务设计阶段就规划好。

Q3：ElasticSearch 写入后为什么不能立即搜索到？

核心思路：

ES 是近实时（NRT, Near Real-Time）搜索，写入到可搜索之间有 Refresh Interval（默认 1秒）的延迟。

原因在于 Lucene 的写入流程：

  1. 写请求 → Memory Buffer（内存缓冲区，此时不可搜索）
  2. 定时 Refresh（默认 1s）→ 生成内存中的 Segment，此时文档被"刷入"索引，立即可搜
  3. 定时 Flush → 将 Segment 持久化到磁盘

如果需要写入即搜索，可以通过 ?refresh=true 强制 Refresh，但会显著降低写入吞吐量。

Q4：ES 的倒排索引是什么？为什么搜索比 MySQL LIKE 快？

核心思路：

倒排索引是以"词"为核心组织文档的索引结构：

  • 将每个文档的文本内容分词成一个个 Term（词条）
  • 建立 Term → 文档列表（Posting List）的映射
  • 搜索时直接查词典找到对应的文档列表，不需要遍历所有文档

而 MySQL 的 LIKE '%keyword%' 无法使用 B+树索引，必须全表扫描，走磁盘 I/O 逐行匹配。

所以 ES 搜索的速度与文档总量无关，只取决于词典的查找效率（通常是 O(1) 或 O(log n)）。

Q5：ClickHouse 的列式存储为什么比行式存储快那么多？

核心思路：三个层面

① I/O 大幅减少：聚合查询只读取需要的列，行式存储需要读取整行（所有列）。假设 20 列的表、只查 2 列，I/O 减少 90%。

② 高压缩比：同一列数据类型相同，压缩率极高（通常 5-10 倍）。如性别列只有"男/女"，用字典编码可压缩到极小的体积。

③ 向量化执行：ClickHouse 一次处理 4096 行数据，利用 CPU 的 SIMD 指令集并行计算。相比传统的逐行处理，指令级并行效率高数倍。

实际效果：对 10 亿行数据的聚合查询，CK 通常比 MySQL 快 50-100 倍。

Q6：MongoDB 和 ElasticSearch 如何选择？什么场景两者一起用？

核心思路：

维度MongoDBElasticSearch
数据存储✅ 强（文档可靠性高）❌ 弱（不支持事务，有丢数据风险）
全文搜索❌ 弱（正则匹配，不索引分词）✅ 强（倒排索引 + 分词器）
聚合分析⚠️ 中等（Agg Pipeline）✅ 强（各种 Bucket/Metric）
数据一致性✅ 强⚠️ 近实时

一起用：MongoDB 作为主存储（Source of Truth），ES 作为搜索/分析引擎。通过 MongoDB Connector for ES 或 CDC 工具实时同步。

典型架构：写入走 MongoDB → Oplog Tailer → ES → 查询走 ES。

Q7：ClickHouse 的分区键和排序键有什么区别？如何优化查询？

核心思路：

分区键（PARTITION BY）：影响数据在磁盘上的目录结构，查询时通过分区裁剪跳过不相关的分区目录。如 PARTITION BY toYYYYMM(date) 按年月分区，查询单月数据只扫描一个目录。

排序键（ORDER BY）：影响数据在每个分区内部的物理存储顺序，ClickHouse 利用稀疏索引（每 8192 行一个 Mark）快速定位数据范围。

优化技巧：分区键粒度不要太小（避免太多小文件），排序键要包含高频过滤字段，且列的排列顺序非常重要——最常过滤的列放最前面。

例如 ORDER BY (date, city, user_id) — 如果经常按 date 范围查询，date 放在第一位能使稀疏索引发挥最大作用。

Q8：你们项目里实际用什么数据库组合？为什么要这样搭？

参考答案（示例）：

在我们生产环境中，采用 PostgreSQL + MongoDB + ES + Redis 的组合：

  • PostgreSQL：核心业务数据（订单、用户、财务）——需要事务、强一致性、复杂关联查询
  • MongoDB：用户资料、内容管理（CMS）——字段灵活、不需要 JOIN、读写频繁
  • ElasticSearch：商品搜索、日志分析——全文检索是核心需求
  • Redis：缓存（热点数据）、计数器（PV/UV）、分布式锁（Redlock）

关键设计原则：每种数据库只做它最擅长的事。PGSQL 管"钱"和"关系"，Mongo 管"灵活内容"，ES 管"搜"，Redis 管"快"。不混用、不滥用。

数据同步方面：通过 CDC（Debezium + Kafka）实现跨数据库的数据流转，保证最终一致性。

  • 组合思维：知道 NoSQL 之间以及 NoSQL+SQL 如何搭配使用

🎙️ 口播总结

好，我们来快速捋一遍 NoSQL 数据库的实战知识体系。

第一，NoSQL 不是"没有 SQL"，而是"Not Only SQL"——文档型、键值型、列族型、图型、搜索引擎，每种解决不同的问题。核心特征：灵活 Schema、水平扩展、高吞吐。

第二，MongoDB（文档型）的核心是 JSON/BSON 文档模型，支持二级索引和聚合管道。高可用靠复制集（Raft 选举），水平扩展靠分片集群（Range/Hashed 分片）。分片键的选择是最关键的决策——选错了数据会倾斜。

第三，ElasticSearch（搜索引擎）的核心是倒排索引——以"词"为索引，搜索速度与数据量无关。写入是近实时的（默认 1 秒后可见），因为有 Refresh 机制的延迟。中文搜索需要搭配 IK 分词器。集群架构分 Master、Data、Ingest、Coordinating 四种角色。

第四，ClickHouse（列式存储）是 OLAP 场景的天花板——列式存储只读需要的列，加上向量化执行引擎（SIMD），聚合查询比 MySQL 快 50-100 倍。核心优化：分区键裁剪、排序键稀疏索引、物化视图预聚合。

第五，选型没有银弹——业务往往需要 2-3 种数据库组合。关键是每种数据库做它最擅长的事：MongoDB 管灵活内容，ES 管搜索，ClickHouse 管分析，Redis 管缓存。实战时展示你的选型思路和实战经验，比背语法加分得多。
