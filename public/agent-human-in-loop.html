👤 Human-in-the-Loop 设计模式

确认点 / 异常上报 / 审批流 & Go 实现 — 让 AI 辅助人，而不是替代人

1. 为什么需要 Human-in-the-Loop

费曼类比

全自动驾驶 看起来很酷，但遇到从没见过的路况就会出错。而 辅助驾驶 把决策权交给人类驾驶员，AI 负责感知、建议和执行辅助操作——人做决策，车来执行。这才是在真实世界中安全落地的方案。

Human-in-the-Loop（HITL）就是 AI 世界的辅助驾驶：Agent 负责跑腿、计算、建议，人类负责判断、确认、批准。机器做不了的价值判断、伦理决策、高风险审批，交给人类；重复性、信息密集型的工作，交给 Agent。

在装修助手这样的实际系统中，HITL 解决三个核心问题：

  • 安全 — Agent 推荐的建材、报价、施工方案，涉及真金白银和安全责任，不能全自动执行
  • 信任 — 用户看到每一步都要自己确认，才能逐步建立对 AI 的信赖
  • 边界 — Agent 遇到不确定性（模糊的用户需求、歧义指令）时，优雅地求助而非瞎猜

Agent执行 纯人工 自动化程度 →              人类参与程度 → ⚖️ 自动化光谱 — HITL 的位置 HITL 甜区

2. 确认点模式（Confirmation Gate）

核心理念： Agent 提出建议，但必须等待用户显式确认后才能执行操作。这是 HITL 最基础、最常用的模式。

典型场景：装修助手推荐产品

  • Agent 根据用户需求筛选出 3 款瓷砖，附上价格、评测和推荐理由
  • 用户浏览后点击"确认添加"
  • 只有确认后，Agent 才把产品写入装修清单
  • 用户也可以修改数量、备注或直接拒绝

Go 实现：确认点中间件

// confirmation_gate.go — 确认点中间件模式

package hitl

import (
    "context"
    "fmt"
    "time"
)

// Action 代表一个待确认的操作
type Action struct {
    ID          string
    Type       string      // product_add, price_update, schedule_change
    Payload     any
    SuggestedBy string      // agent name
    CreatedAt   time.Time
    Timeout     time.Duration
}

// ConfirmationGate 确认点门卫
type ConfirmationGate struct {
    pending  map[string]*Action
    callback func(ctx context.Context, act *Action, approved bool) error
}

func NewConfirmationGate(cb func(ctx context.Context, a *Action, approved bool) error) *ConfirmationGate {
    return &ConfirmationGate{
        pending:  make(map[string]*Action),
        callback: cb,
    }
}

// Propose 提出一个待确认的动作，返回 action ID
func (g *ConfirmationGate) Propose(ctx context.Context, a *Action) string {
    a.ID = fmt.Sprintf("act_%d", time.Now().UnixNano())
    a.CreatedAt = time.Now()
    g.pending[a.ID] = a
    return a.ID
}

// Confirm 用户确认通过，执行回调
func (g *ConfirmationGate) Confirm(ctx context.Context, actionID string) error {
    a, ok := g.pending[actionID]
    if !ok {
        return fmt.Errorf("action %s not found or expired", actionID)
    }
    delete(g.pending, actionID)
    return g.callback(ctx, a, true)
}

// Reject 用户拒绝，执行回滚逻辑
func (g *ConfirmationGate) Reject(ctx context.Context, actionID string) error {
    a, ok := g.pending[actionID]
    if !ok {
        return fmt.Errorf("action %s not found", actionID)
    }
    delete(g.pending, actionID)
    return g.callback(ctx, a, false)
}

// TimeoutCheck 扫描超时未确认的操作并自动拒绝
func (g *ConfirmationGate) TimeoutCheck(ctx context.Context) {
    now := time.Now()
    for id, a range g.pending {
        if now.Sub(a.CreatedAt) > a.Timeout {
            delete(g.pending, id)
            _ = g.callback(ctx, a, false) // 超时视为拒绝
        }
    }
}

3. 异常上报模式（Exception Escalation）

核心理念： Agent 遇到自己无法判断的场景时，主动暂停执行，打包上下文上报给人类处理。这是一种"自知之明"的设计。

不确定性检测 + 自动上报

Agent 在执行过程中持续计算置信度：

  • 置信度 > 90% → 全自动执行
  • 70% ~ 90% → 建议 + 确认点（上一节）
  • < 70% → 触发异常上报，等待人工裁决

上下文打包

上报时附带完整的上下文快照，让人能快速决策：

  • 原始用户输入 / 对话历史
  • Agent 已尝试的方案及其推理过程
  • 多个候选答案及各自置信度
  • 相关数据源引用

Go 代码：异常上报器

// exception_escalation.go — 异常上报

package hitl

import (
    "context"
    "encoding/json"
    "fmt"
    "time"
)

// ConfidenceLevel 置信度级别
type ConfidenceLevel int

const (
    ConfidenceHigh   ConfidenceLevel = 90
    ConfidenceMedium ConfidenceLevel = 70
)

// EscalationTicket 异常上报工单
type EscalationTicket struct {
    ID          string          `json:"id"`
    UserQuery   string          `json:"user_query"`
    AgentReason string          `json:"agent_reason"`
    Candidates  []string        `json:"candidates"`
    Confidence  float64         `json:"confidence"`
    Context     map[string]any `json:"context"`
    Status      string          `json:"status"` // pending / resolved / rejected
    CreatedAt   time.Time       `json:"created_at"`
    ResolvedAt  *time.Time      `json:"resolved_at,omitempty"`
}

// EscalationHandler 异常上报处理器
type EscalationHandler struct {
    tickets map[string]*EscalationTicket
    notifier func(ticket *EscalationTicket)
    resolver func(ticket *EscalationTicket, decision string) (any, error)
}

func NewEscalationHandler(
    notify func(*EscalationTicket),
    resolve func(*EscalationTicket, string) (any, error),
) *EscalationHandler {
    return &EscalationHandler{
        tickets:  make(map[string]*EscalationTicket),
        notifier: notify,
        resolver: resolve,
    }
}

// Escalate 触发异常上报
func (h *EscalationHandler) Escalate(ctx context.Context, query string,
    reason string, candidates []string, confidence float64,
    contextData map[string]any) string {

    ticket := &EscalationTicket{
        ID:          fmt.Sprintf("esc_%d", time.Now().UnixNano()),
        UserQuery:   query,
        AgentReason: reason,
        Candidates:  candidates,
        Confidence:  confidence,
        Context:     contextData,
        Status:      "pending",
        CreatedAt:   time.Now(),
    }
    h.tickets[ticket.ID] = ticket
    if h.notifier != nil {
        h.notifier(ticket)
    }
    return ticket.ID
}

// Resolve 人工裁决结果注入
func (h *EscalationHandler) Resolve(ticketID string, decision string) (any, error) {
    ticket, ok := h.tickets[ticketID]
    if !ok {
        return nil, fmt.Errorf("ticket %s not found", ticketID)
    }
    now := time.Now()
    ticket.ResolvedAt = &now
    ticket.Status = "resolved"
    return h.resolver(ticket, decision)
}

// Decision: "use_first" / "use_second" / "reject" / custom

4. 审批流集成（Approval Workflow）

核心理念： 多级审批，类似工单系统。Agent 提建议 → 主管审批 → 执行。适用于需要多人或多角色确认的场景。

典型流程

  1. Agent 生成一份装修报价单（含材料、人工、工期）
  2. 系统自动提交给项目经理初审
  3. 项目经理确认后，流转到财务审批
  4. 全部审批通过后，Agent 执行下单操作
  5. 任一环节驳回，流程终止并通知 Agent 调整方案

Go 实现：审批流引擎

// approval_workflow.go — 多级审批流引擎

package hitl

import (
    "context"
    "fmt"
    "time"
)

// ApprovalStatus 审批状态
type ApprovalStatus string

const (
    StatusPending   ApprovalStatus = "pending"
    StatusApproved  ApprovalStatus = "approved"
    StatusRejected  ApprovalStatus = "rejected"
    StatusCancelled ApprovalStatus = "cancelled"
)

// ApprovalStep 审批步骤
type ApprovalStep struct {
    ID        string
    Name      string          // "初审", "财务审批", "终审"
    Approver  string          // userID 或 roleID
    Status    ApprovalStatus
    Comment   string
    ActedAt   *time.Time
    Timeout   time.Duration
}

// ApprovalWorkflow 审批流实例
type ApprovalWorkflow struct {
    ID         string
    Title      string
    Payload    any             // 待审批的数据（报价单、合同等）
    Steps      []*ApprovalStep
    CurrentIdx int
    Status     ApprovalStatus
    CreatedAt  time.Time
    OnComplete func(ctx context.Context, w *ApprovalWorkflow)
}

// ApprovalEngine 审批流引擎
type ApprovalEngine struct {
    workflows map[string]*ApprovalWorkflow
}

func NewApprovalEngine() *ApprovalEngine {
    return &ApprovalEngine{
        workflows: make(map[string]*ApprovalWorkflow),
    }
}

// StartWorkflow 启动一个新审批流
func (e *ApprovalEngine) StartWorkflow(title string, payload any,
    steps []*ApprovalStep, onComplete func(ctx context.Context, w *ApprovalWorkflow),
) string {
    w := &ApprovalWorkflow{
        ID:         fmt.Sprintf("wf_%d", time.Now().UnixNano()),
        Title:      title,
        Payload:    payload,
        Steps:      steps,
        CurrentIdx: 0,
        Status:     StatusPending,
        CreatedAt:  time.Now(),
        OnComplete: onComplete,
    }
    e.workflows[w.ID] = w
    return w.ID
}

// Approve 当前步骤通过，流转到下一步或完成
func (e *ApprovalEngine) Approve(ctx context.Context,
    workflowID string, comment string) error {

    w, ok := e.workflows[workflowID]
    if !ok {
        return fmt.Errorf("workflow %s not found", workflowID)
    }
    if w.Status != StatusPending {
        return fmt.Errorf("workflow already %s", w.Status)
    }
    step := w.Steps[w.CurrentIdx]
    now := time.Now()
    step.Status = StatusApproved
    step.Comment = comment
    step.ActedAt = &now

    if w.CurrentIdx + 1 >= len(w.Steps) {
        // 所有步骤完成
        w.Status = StatusApproved
        if w.OnComplete != nil {
            w.OnComplete(ctx, w)
        }
    } else {
        w.CurrentIdx++
    }
    return nil
}

// Reject 驳回，终止整个流程
func (e *ApprovalEngine) Reject(workflowID string, comment string) error {
    w, ok := e.workflows[workflowID]
    if !ok {
        return fmt.Errorf("workflow %s not found", workflowID)
    }
    w.Status = StatusRejected
    w.Steps[w.CurrentIdx].Status = StatusRejected
    w.Steps[w.CurrentIdx].Comment = comment
    now := time.Now()
    w.Steps[w.CurrentIdx].ActedAt = &now
    return nil
}

// GetStatus 查询审批流状态
func (e *ApprovalEngine) GetStatus(workflowID string) (ApprovalStatus, string) {
    w, ok := e.workflows[workflowID]
    if !ok {
        return "", "not found"
    }
    currentStep := "completed"
    if w.Status == StatusPending {
        currentStep = w.Steps[w.CurrentIdx].Name
    }
    return w.Status, currentStep
}

5. 人机协作编辑

核心理念： Agent 生成草稿/初稿，人工进行编辑和审核，然后发布。这是"AI 辅助创作"的核心模式。

代码审核模式

Agent 生成 Pull Request（代码变更 + 描述），人工 Review 后合并：

  • Agent 分析代码问题 → 生成修复 PR
  • 开发者收到 PR 通知 → Review 变更 → Approve / Request Changes
  • Approve 后自动合并并部署

内容生产模式

Agent 写初稿，人工修改后发布：

  • Agent 生成装修方案描述（含图片、材料清单、施工步骤）
  • 设计师/编辑在可视化编辑器中修改
  • 确认无误后一键发布到业主端

🛠️ 代码审核模式

✅ Agent 生成 PR + 测试

✅ 开发者 Review 差异

✅ 分级：LGTM / Comment / Request Changes

⚠️ 需要开发者具备代码判断力

📝 内容生产模式

✅ Agent 生成初稿 + 配图建议

✅ 富文本编辑器实时修改

✅ 版本对比 + 一键发布

⚠️ 需要人工润色风格和语气

6. 模式选择指南

不同的场景和风险级别，应该选择不同的 HITL 模式：

绿色=全自动可行 ｜ 黄色=确认点/异常上报 ｜ 红色=需要审批流

场景 推荐模式 理由

7. 用户体验设计

HITL 做不好会让用户觉得烦——每次弹窗确认就像 Windows Vista 的 UAC。好的设计应该让 HITL 变得自然、不打断、可批量。

核心原则：减少用户的认知负担

每次让人做决定，都在消耗用户的注意力。设计目标是在安全与效率之间找到最佳平衡点。

三大设计策略

更多设计细节

  • 上下文展示 — 需要在确认界面显示足够的上下文，用户不需要跳转到其他页面才能判断
  • 一键驳回+原因采集 — 拒绝时要轻松，但也轻量采集原因用于 Agent 改进
  • 可撤销操作 — 如果用户误确认，在一定时间窗口内允许撤销
  • 渐进式授权 — 初期频繁确认，随着信任建立逐渐减少干预频率
  • 通知渠道 — 站内通知 + 邮件 + 即时通讯（飞书/钉钉），确保不遗漏

8. 实战：装修助手的 HITL 设计

装修助手系统全面采用了上述四种 HITL 模式，形成了完整的人机协作体系：

装修助手各模块使用的 HITL 模式

模块 HITL 模式 说明
商品推荐 确认点 Agent 推荐产品 → 用户确认 → 加入清单
需求理解 异常上报 用户描述模糊时，上报给人工客服澄清
报价生成 审批流 Agent 生成报价 → 主管审批 → 发送客户

价格监控 全自动 定时更新建材价格，无需人工干预

9. QA & 口播总结

Q1: HITL 会不会拖慢系统响应速度？

会，但这是设计上的取舍。关键策略是：异步流程 + 预判。Agent 可以在等待用户确认的同时，继续执行其他不需要确认的任务。另外，聚合通知和批量确认能减少用户的决策次数。对于低风险操作，可以设置较短的超时自动通过，避免流程卡死。

Q2: 什么时候应该使用审批流而不是确认点？

判断标准是 责任边界。如果操作仅影响当前用户（如个人购物车），确认点就够了。如果操作影响多个角色或涉及资金安全（如报价单、促销活动），就需要审批流。简单说：确认点 = 自己决策，审批流 = 多人决策。

Q3: 异常上报中，Agent 怎么判断自己"拿不准"？

多种信号综合判定：

  • NLP 置信度 — 模型输出的 logits 概率分布
  • 规则兜底 — 匹配到模糊关键词（"随便"、"看着办"）
  • 用户行为 — 用户反复修改同一条输入、连续拒绝建议
  • 业务规则 — 金额超过阈值、涉及特殊品类

这些信号加权计算出一个综合置信度，低于阈值就触发上报。

Q4: 协作编辑模式下，Agent 生成的初稿质量如何保证？

从三个层面保证：

  1. 结构化模板 — Agent 按照预定义的 Schema 生成内容，确保格式规范
  2. 质量打分 — 输出后先经过一个轻量质量检测模型，不合格的自动重生成
  3. 编辑痕迹 — 记录人工修改的内容，形成反馈数据持续微调 Agent
Q5: HITL 在装修助手中遇到的最大挑战是什么？

最大的挑战是 用户疲劳。初期我们设置了过多的确认点，用户反馈"每次买根螺丝都要确认"。后来我们做了三件事改进：

  • 引入 信任等级 — 老用户、低风险操作减少确认频率
  • 聚合 + 批量 — 把"添加 5 件商品"合并成一个确认操作
  • 智能默认 — 根据用户历史行为预填确认选项
Q6: 这些 HITL 模式可以组合使用吗？

完全可以，而且推荐灵活组合。例如：

  • 确认点 + 超时策略 — 用户 5 分钟不操作自动驳回
  • 异常上报 + 审批流 — 异常先上报给一线支持，解决不了升级到主管审批
  • 协作编辑 + 确认点 — Agent 生成草稿 → 人工编辑 → 确认发布

不同的组合适用于不同的风险等级和用户场景。

Q7: Go 的并发特性在 HITL 中有什么优势？

Go 的 goroutine 和 channel 让 HITL 的异步处理变得非常优雅：

  • 每个确认点可以跑在一个独立的 goroutine 中，用 channel 监听确认/拒绝信号
  • 审批流的每一步都可以异步推进，不阻塞主流程
  • 超时检测可以用 select + time.After 轻松实现
  • 上下文打包通过 context.Context 传播，天然支持链路追踪

好，我们来快速回顾一下 Human-in-the-Loop 设计模式的核心要点。

第一，HITL 的哲学。 它不是要把人变成机器的监督员，而是让机器在人类擅长的领域主动求助。就像费曼说的：让 AI 做辅助驾驶，而不是自动驾驶。全自动在封闭环境可行，但在开放业务中，必须有人类在回路中把控价值判断和风险的边界。

第二，四大模式。 确认点模式是最常用的，Agent 提议、用户拍板，适合个人决策场景。异常上报模式让 Agent 有"自知之明"，拿不准时优雅求助。审批流集成用于多角色协作场景，像工单系统一样逐级流转。协作编辑模式让 Agent 当助手而不是替代者，生成草稿、人工修改、最终发布。

第三，选型判断。 风险矩阵告诉我们：低风险低影响的操作全自动，中等风险用确认点或异常上报，高风险高影响的一定要走审批流。没有银弹，要根据业务场景灵活选择。

第四，用户体验。 再好的安全设计，如果让用户觉得烦，就失败了。聚合通知、批量确认、默认超时策略、渐进式信任——这些细节决定了一个 HITL 系统是被人喜爱还是被人厌恶。

最后想说：HITL 不是 AI 能力不足的妥协，而是构建可信 AI 系统的工程智慧。它让人类保持在决策环中，既利用了 AI 的效率，又保留了人类的判断力。在装修助手这样的实际系统中，这四种模式的组合使用，让系统既能高效运行，又能安全可靠地为用户创造价值。
