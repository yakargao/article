🧭 领域驱动设计 DDD

一句话：DDD 不是写代码的技巧，是把业务建模成代码的方法论。核心 = 通用语言 + 限界上下文 + 聚合。

和传统开发的本质区别：传统开发是"数据库表驱动"（先设计表再写代码），DDD 是"业务模型驱动"（先理解业务再建模，数据库只是实现细节）。

🧭 什么是 DDD？让代码说业务的语言

🗺️ 类比：盖房子有两种方式——① 直接搬砖，看哪里需要砌哪里（CRUD 式开发）。② 先画蓝图（领域建模），标注每个房间的用途和关系，然后按图施工（DDD）。

DDD 的核心价值：当业务变复杂时，你的代码结构仍然清晰反映业务结构。改需求时不用"翻译"（业务语言→技术语言），直接改。

DDD 的两层设计

层做什么产出关键词
战略设计划分业务边界、理顺上下文关系限界上下文 + 上下文映射"这个功能属于哪个模块？"
战术设计在每个上下文内，怎么建模实体、值对象、聚合、资源库"这个对象是什么角色？"

DDD 不是银弹——什么时候该用？

适合：业务逻辑复杂（流程、规则、状态机多）、长期维护的核心系统（电商、金融、ERP）

不适合：简单 CRUD（后台管理）、纯技术性系统（网关、日志采集）、一次性项目

🗣️ 通用语言：让产品、开发、测试说同一种话

🏥 类比：医生之间说"ST 段抬高型心肌梗死"，而不是"心脏那个血管堵了"。DDD 的通用语言 = 团队定义的一套精确术语，代码中的类名、方法名直接使用这套术语。

一个例子看懂

没有通用语言（传统）有通用语言（DDD）
产品说"用户下单后要锁定库存"
代码里：UpdateOrderStatus(1, "paid")
数据库：UPDATE orders SET status=2 WHERE id=1		 产品说"用户提交订单后，订单进入已确认状态"
代码里：order.Confirm()
数据库中触发：OrderConfirmed 事件

DDD 的代码直接映射业务术语——不需要脑内翻译。

通用语言的落地原则

  1. 同一个词在不同上下文可以不同含义。"客户"在售前上下文=潜在买家，在售后上下文=已购买用户
  2. 类名、方法名、变量名全用业务术语。不用 setStatus(1)，用 confirmOrder()
  3. 代码即文档。读 DDD 代码应该像读业务说明书

📦 限界上下文：DDD 最重要的概念

🏢 类比：一个大公司——财务部说的"资产"（现金、应收账款）和 IT 部说的"资产"（服务器、电脑）不是同一个东西。每个部门有自己的术语体系和职责边界。DDD 把每个部门叫一个限界上下文 (Bounded Context)。

三个限界上下文 + 上下文映射：订单、商品、支付各自独立，通过 ID 和事件通信

限界上下文的物理形态

一个限界上下文通常对应一个微服务或一个模块（Go package）。但不是绝对的——一个复杂的上下文可以拆成多个微服务，一个简单的上下文也可以和其他合并。

关键原则：每个上下文有自己的数据库，不共享表。上下文之间通过 API 或领域事件通信。

🗺️ 上下文映射：上下文之间怎么协作？

关系类型含义实现方式例子
共享内核两个上下文共享一部分模型公共 library / package用户 ID、地址值对象
客户-供应商上游提供，下游消费，上游主导API 调用订单服务调用商品服务
防腐层 ACL下游加一层翻译，保护自己不受上游变化影响Adapter 模式对接第三方支付 SDK
开放主机 OHS上游提供标准化 API 供多下游消费REST / gRPC / 事件用户中心提供统一的用户 API
各行其道两个上下文没有直接关系不集成报表系统和订单系统无关

防腐层（ACL）——最重要的集成模式

🔌 类比：出国旅游需要电源转换插头——你的设备（领域模型）不变，只是翻译一下外部接口。防腐层 = 转换插头，让外部系统的脏模型不污染你的领域。

// 防腐层：把外部支付 SDK 的模型翻译成领域模型
type PaymentACL struct { client *alipay.Client }

func (acl *PaymentACL) ProcessPayment(order *domain.Order) (*domain.Payment, error) {
    // 把领域对象翻译成支付宝的请求格式
    aliReq := &alipay.PayRequest{
        OutTradeNo:  order.ID.String(),
        TotalAmount: order.TotalAmount().String(),
        Subject:     order.Description(),
    }
    aliResp, err := acl.client.Pay(aliReq)
    if err != nil { return nil, err }
    // 把支付宝的响应翻译成领域对象
    return domain.NewPayment(order.ID, aliResp.TradeNo, aliResp.Amount), nil
}
限界上下文映射：5 种集成关系可视化——同一个系统的不同上下文如何协作

🏷️ 子域分类：分清主次，把钱花在刀刃上

类型是什么核心竞争力？投入策略例子（电商）
核心域让你区别于竞争对手的业务✅ 是重兵投入、自研、精雕推荐算法、定价引擎
支撑域支撑核心业务但不独特❌ 不是自研或定制化采购订单管理后台
通用域所有人都需要的通用能力❌ 不是用现成的、别自己造用户认证、文件存储

🆔 实体 Entity：有身份证号的对象

👤 类比：你这个人 = 实体。改了名字、换了发型、搬了家，你还是你——因为你有唯一的身份证号。实体通过标识 ID 来区分，而不是属性。

// 实体：有唯一标识、可变、有生命周期
type Order struct {
    ID          OrderID        // 唯一标识（UUID / 雪花ID）
    CustomerID  CustomerID
    Status      OrderStatus    // 可变状态
    Items       []OrderItem
    TotalAmount Money
    CreatedAt   time.Time
    UpdatedAt   time.Time
}

func (o *Order) Confirm() error {   // 行为 > 属性
    if o.Status != Pending {
        return ErrInvalidTransition
    }
    o.Status = Confirmed
    o.UpdatedAt = time.Now()
    return nil
}

💵 值对象 Value Object：只关心面值的 100 元纸币

💵 类比：两张 50 元纸币——你不在乎哪张是哪张，只要面值一样就完全等价。值对象没有 ID，通过属性值判断相等性，且不可变。

值对象 vs 实体，一张表看懂

实体 Entity

  • 有唯一 ID
  • 可变（状态会变）
  • 通过 ID 判等
  • 有生命周期
  • 例：User, Order

值对象 Value Object

  • 无 ID
  • 不可变（改了就是新对象）
  • 通过属性值判等
  • 无生命周期
  • 例：Money, Address, PhoneNumber
// 值对象：不可变、无 ID、通过值判等
type Money struct {
    Amount   int64
    Currency string
}

// 不可变：Add 返回新对象，不修改原对象
func (m Money) Add(other Money) Money {
    if m.Currency != other.Currency { panic("currency mismatch") }
    return Money{Amount: m.Amount + other.Amount, Currency: m.Currency}
}

type Address struct {  // 地址天然是值对象
    Province, City, District, Street string
}
// 两个地址 Province+City+District+Street 相同 → 就是同一个地址
Entity（有身份） vs Value Object（只关心值）——同一概念在不同上下文可以不同

什么时候用值对象而不是实体？

问自己三个字："它变吗？它有身份吗？"。金额不变、地址在你搬家前也不变、电话号码会变但它本身没有"身份"——这些都是值对象。

🏰 聚合 Aggregate：DDD 的核心武器

🛂 类比：入境海关——你不能随便进入一个国家，必须通过海关检查（聚合根）。海关检查你的护照，决定你能不能入境。聚合 = 一组紧密相关的对象，只能通过"海关"（聚合根）访问内部。

聚合的四条铁律

  1. 只有一个聚合根：外部只能通过聚合根修改内部对象
  2. 通过 ID 引用其他聚合：不持有其他聚合的引用，只存 ID
  3. 最终一致性：聚合之间的状态通过领域事件最终一致（不要跨聚合事务）
  4. 小聚合：一个聚合尽量小，只包含必须一起保证一致性的对象
// Order 聚合：Order 是聚合根，OrderItem 是内部实体
type Order struct {          // ← 聚合根 (Aggregate Root)
    ID          OrderID
    CustomerID  CustomerID      // 引用其他聚合 → 只存 ID！
    Status      OrderStatus
    items       []OrderItem     // 内部实体，不对外暴露
    totalAmount Money
}

type OrderItem struct {       // 聚合内部实体
    ProductID   ProductID       // 引用商品聚合 → 只存 ID
    Quantity    int
    UnitPrice   Money
}

// ✅ 正确：通过聚合根操作，保证业务规则
func (o *Order) AddItem(productID ProductID, qty int, price Money) error {
    if o.Status != Pending { return ErrOrderLocked }
    if qty <= 0 { return ErrInvalidQuantity }
    o.items = append(o.items, OrderItem{...})
    o.recalculateTotal()
    return nil
}
// ❌ 错误：外部直接修改 items——绕过了业务规则
// order.Items = append(order.Items, ...)

聚合大小设计——DDD 的最大难点

太大：性能差、并发冲突多。太小：一致性保证不了。

经验值：一个聚合通常包含根实体 + 1~3 个内部实体。如果犹豫，往小了拆，用领域事件保证最终一致性。

📁 资源库 Repository：聚合的"档案室管理员"

📂 类比：公司的档案室——你要取一份合同（聚合），跟管理员说要合同编号（聚合 ID），管理员去档案柜（数据库）找出来给你。你不需要知道档案柜是什么牌子的。

// Repository 接口（定义在领域层）
type OrderRepository interface {
    Save(ctx context.Context, order *Order) error
    FindByID(ctx context.Context, id OrderID) (*Order, error)
    FindByCustomerID(ctx context.Context, customerID CustomerID) ([]*Order, error)
}

// Repository 实现（在基础设施层）
type orderRepo struct { db *sql.DB }
func (r *orderRepo) Save(ctx context.Context, order *Order) error {
    return r.db.ExecContext(ctx,
        "INSERT INTO orders (id, status, ...) VALUES (?,?,...) ON DUPLICATE KEY UPDATE ...",
        order.ID, order.Status,
    )
}
func (r *orderRepo) FindByID(ctx context.Context, id OrderID) (*Order, error) {
    row := r.db.QueryRowContext(ctx, "SELECT ... FROM orders WHERE id=?", id)
    return scanOrder(row)  // 从数据库行重建聚合
}

Repository 的关键原则

  • 一个聚合一个 Repository——不为内部实体单独建 Repository
  • 接口定义在领域层，实现在基础设施层——依赖倒置
  • Save 而不是 Update——Repository 负责整存整取，不暴露字段级更新

⚙️ 领域服务：不属于任何一个实体的业务逻辑

⚖️ 类比：转账——既不单独属于 A 账户，也不单独属于 B 账户，是"银行转账系统"这个服务的职责。领域服务 = 跨聚合的业务逻辑。

// 领域服务：协调两个聚合之间的操作
type TransferService struct {
    accountRepo AccountRepository
}

func (s *TransferService) Transfer(
    ctx context.Context,
    fromID, toID AccountID,
    amount Money,
) error {
    from, _ := s.accountRepo.FindByID(ctx, fromID)
    to, _   := s.accountRepo.FindByID(ctx, toID)
    if err := from.Debit(amount); err != nil { return err }
    to.Credit(amount)
    return s.accountRepo.SaveBoth(ctx, from, to)
  // 注意：跨聚合事务最好用最终一致性（发事件），而非 XA 事务
}

📨 领域事件：发生了重要的事，告诉相关方

📢 类比：公司 OA 系统——张三提交了请假申请 → 系统自动通知部门经理审批 + HR 备案 + 工资系统扣钱。领域事件 = 业务领域内发生了重要事情，广播给所有关注方。

// 领域事件：不可变的、描述已发生的事实
type OrderConfirmed struct {
    OrderID    OrderID    `json:"order_id"`
    CustomerID CustomerID `json:"customer_id"`
    TotalAmount Money     `json:"total_amount"`
    ConfirmedAt time.Time `json:"confirmed_at"`
}

// 聚合内部产生事件
func (o *Order) Confirm() error {
    if o.Status != Pending { return ErrInvalidTransition }
    o.Status = Confirmed
    o.AddDomainEvent(OrderConfirmed{       // 记录事件
        OrderID:    o.ID,
        TotalAmount: o.totalAmount,
        ConfirmedAt: time.Now(),
    })
    return nil
}

// Repository Save 时发布事件
func (r *orderRepo) Save(ctx context.Context, order *Order) error {
    tx, _ := r.db.BeginTx(ctx)
    // 1. 保存聚合
    tx.Exec("UPDATE orders SET status=? WHERE id=?", order.Status, order.ID)
    // 2. 在同一个事务中写入事件表 (Outbox Pattern)
    for _, evt := range order.DomainEvents() {
        tx.Exec("INSERT INTO outbox (type, payload) VALUES (?,?)", ...)
    }
    return tx.Commit()
}

❓ 高频考题

Q1: 用 DDD 设计一个电商系统的订单模块
限界上下文：订单上下文（独立于商品、支付、用户）。
聚合根：Order（包含 OrderItem）。只能通过 Order 修改内部项。
值对象：Money（金额）、Address（收货地址）、OrderID。
领域事件：OrderPlaced → 扣库存（商品上下文）+ 发支付请求（支付上下文）。
Repository：OrderRepository 只负责存取 Order 聚合。
Q2: 实体和值对象怎么区分？一个对象今天当实体明天能当值对象吗？
看业务场景。电话号码——在用户上下文是值对象（换号就换号），在运营商 CRM 系统是实体（每个号码有归属、套餐、账单）。同一概念在不同上下文可以是不同类型。
Q3: 聚合和微服务的关系？
一个微服务 ≈ 一个限界上下文（不是绝对的）。一个限界上下文内可以有多个聚合。微服务之间通过 API/事件通信，聚合之间通过 ID 引用 + 领域事件保持最终一致。不要把多个聚合塞进一个微服务事务——用 Saga 或 Outbox 模式。
Q4: DDD 和 Clean Architecture / 六边形架构的关系？
DDD 是"建什么"（业务建模），Clean Architecture 是"怎么建"（代码组织）。两者完美互补：DDD 的领域层 = Clean Architecture 的 Entity + UseCase 层。DDD 不规定目录结构，Clean Architecture 提供了分层的实践方案。
Q5: 为什么聚合要设计得尽量小？
① 并发冲突：大聚合 = 多人同时改不同字段也会冲突（锁整个聚合）。
② 性能：每次加载聚合要加载所有内部对象 → 内存和查询开销。
③ 事务边界：DDD 建议一个事务只改一个聚合，大聚合意味着大事务。
Q6: DDD 落地最大的困难是什么？
① 领域专家难找——需要懂业务的人持续参与。② 团队共识难——通用语言需要所有人遵守。③ 过度设计——简单 CRUD 硬上 DDD 是浪费。建议：先在一个核心模块试点，成功了再推广。

💬 苏格拉底式提问 —— 检验真理解

以下问题用来逼你追问本质。先自己想 30 秒，再点开验证。

限界上下文按什么划分？Q1
按"语言边界"。同一个词在不同上下文有不同含义——"订单"在交易是待支付，在物流是待发货。
判断标准：① 同一术语有不同含义 → 拆；② 两个功能独立演化 → 拆；③ 两个团队对同一概念无法达成一致 → 拆。
常见错误：按数据库表拆（技术驱动），按组织结构拆（政治驱动）。边界应该是业务语言的边界。
聚合根之间为什么只用 ID 引用？Q2
三个原因：
① 事务边界：一个事务只能修改一个聚合。持有对象引用容易跨聚合修改。
② 加载成本：加载 Order 时如果还要加载整个 Product——性能灾难。
③ 独立演化：聚合 A 和 B 应能独立修改和部署。
本质：ID 引用 = 弱关联。"不持有对象"不是偷懒，是保护聚合的独立性和事务边界。
领域事件和 MQ Event 的本质区别？Q3
MQ Event：技术事件。"用户下单了，通知物流系统"。关注"数据传递"。
领域事件：业务事件。"订单已提交"是业务事实。关注"业务含义的不可变性"。
差异：① 命名：技术 = "OrderCreated"，领域 = "OrderSubmitted"；② 用途：技术用于通知，领域用于驱动流程；③ 领域事件一旦产生不可修改。
一句话：领域事件是"发生了什么事"，MQ Event 是"请做这个操作"。

🎙️ 口播总结

这篇文章帮你理解领域驱动设计的全部核心概念。

DDD 不是写代码的技巧，是把业务建模成代码的方法论。核心三板斧：通用语言、限界上下文、聚合。通用语言让产品、开发、测试说同一种话，代码直接映射业务术语。限界上下文就像公司部门——财务部说的"资产"和 IT 部说的"资产"不同，每个上下文有自己的术语和数据。

战术设计层有五个核心概念：实体是有身份证号的对象，值对象是没有 ID 只关心值的对象，聚合是一组紧密相关对象的堡垒只能通过聚合根访问，资源库是聚合的档案室管理员负责存取，领域服务是不属于任何一个实体的跨聚合业务逻辑。

聚合的四条铁律是 DDD 最难掌握的地方：只有一个聚合根、通过 ID 引用其他聚合、最终一致性、尽量小。防腐层就像出国转换插头，保护你的领域模型不被外部系统的脏模型污染。

领域事件让上下文之间解耦通信。Go 中用四层目录结构落地 DDD：domain 领域层零外部依赖、application 应用层做用例编排、infrastructure 基础设施层实现技术细节、interfaces 接口层对外暴露。

如果只能记住一个概念，那就是"聚合"。理解聚合根、聚合的四条铁律、聚合大小设计，就掌握了 DDD 的精髓。
