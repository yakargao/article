🔄 高并发系统设计

从 Go 并发原语到实战场景 · 限流/削峰/熔断/缓存/热点/秒杀 · 12 张配图

🏃 类比：高并发系统就像一座地铁站。高峰时 10 万人涌入——限流（闸机控制进站速度）= 不让站台挤爆。削峰填谷（排队缓冲区）= 大家排着队慢慢进。熔断降级（某条线故障时跳台换乘）= 一个区域崩了不影响全站。多级缓存（电子屏显示车次信息）= 不用每次去调度室问。热点应对（热门线路加开班次）= 发现哪条线人多就多派车。

本文不只讲 Go 并发原语——那是工具箱里的螺丝刀。我们还要讲整个工具箱怎么用、什么时候用哪把、遇到突发流量时怎么应变。

🧵 并发编程基础

并发 vs 并行

并发 (Concurrency)

  • 逻辑上同时处理
  • 时间片轮转
  • 单核也能实现
  • 关注任务调度

并行 (Parallelism)

  • 物理上同时执行
  • 需要多核
  • 真正的"同时"
  • 关注计算加速

🎯 四大并发模式

Worker Pool

func WorkerPool(tasks []Task, workers int) {
    ch := make(chan Task, len(tasks))
    var wg sync.WaitGroup
    for i := 0; i < workers; i++ {
        wg.Add(1)
        go func() { defer wg.Done(); for t := range ch { t.Process() } }()
    }
    for _, t := range tasks { ch <- t }
    close(ch); wg.Wait()
}

Pipeline / Fan-Out-Fan-In / Pub-Sub

Pipeline： Stage1→Stage2→Stage3，每阶段用 channel 连接，天然并行。

Fan-Out Fan-In： 一个输入分发到多个 Worker 并行处理，结果汇聚到一个 channel。

Pub-Sub： 一对多广播，用 channel + select 实现，解耦生产者和消费者。

🚦 限流策略 — 不让系统被流量冲垮

限流是高并发系统的第一道防线。目标是：系统能处理多少，就放进来多少。多出来的要么排队等，要么直接拒绝。

🚪 类比：酒吧门口排队——里面满 200 人了，保安（限流器）在外面拦着，出来一个放进去一个。限流不是拒绝用户，而是保护系统不为用户。

四种限流算法

算法原理优缺点适用场景
计数器（固定窗口）每秒最多 N 请求，过了整秒重置简单但窗口边界有突发粗粒度限流
滑动窗口按时间片切割，滑动统计比固定窗口精准，内存略高API 限流
令牌桶（Token Bucket）匀速往桶里加令牌，请求消耗令牌允许突发（桶里有攒的令牌）最常用
漏桶（Leaky Bucket）请求进桶，匀速流出严格平滑，不能突发削峰填谷

Go 令牌桶实现

type TokenBucket struct {
    rate  float64       // 每秒放入 rate 个令牌
    burst int          // 桶容量（最多攒多少个）
    tokens float64     // 当前令牌数
    last   time.Time      // 上次取令牌时间
    mu     sync.Mutex
}

func NewTokenBucket(rate float64, burst int) *TokenBucket {
    return &TokenBucket{rate: rate, burst: burst, last: time.Now()}
}

func (tb *TokenBucket) Allow() bool {
    tb.mu.Lock()
    defer tb.mu.Unlock()

    now := time.Now()
    elapsed := now.Sub(tb.last).Seconds()
    tb.tokens = math.Min(float64(tb.burst), tb.tokens+elapsed*tb.rate)
    tb.last = now

    if tb.tokens >= 1 {
        tb.tokens--
        return true
    }
    return false
}

分布式限流

单机限流不够——你有 10 台机器，用户用同一个 IP 刷，每台都放 100/s，实际上 1000/s 打到了 DB。

Redis + Lua 方案

用 Redis 的 INCR + EXPIRE 做滑动窗口，Lua 脚本保证原子性。性能好，但 Redis 本身可能成为瓶颈。

适用：全局限流（用户级、IP 级）

Sentinel 方案

每个节点本地限流 + 定期同步配额。没有中心化瓶颈，适合大规模集群。

适用：服务级限流（每个服务实例独立限流）

⛰️ 削峰填谷 — 让流量变得更平滑

限流是"不让太多人进来"，削峰是"把高峰期的流量摊平到低谷期处理"。核心思想：用异步化换时间。

💧 类比：城市排水系统——暴雨来临时（流量高峰），雨水先进入蓄水池（消息队列），等雨停了再慢慢排到污水处理厂（后端处理）。如果没有蓄水池，排水管会直接被冲爆。

图 2 · 削峰填谷：消息队列是蓄水池，把瞬时洪峰变成平缓的流水

三种削峰方式

方式实现延迟适用
MQ 异步化请求入队即返回，Worker 消费秒级下单、发通知、日志
请求合并攒一批请求再处理（batch）毫秒级批量写、批量推送
本地缓冲应用层缓冲队列毫秒级高频埋点、上报

⚡ 实战：一个 HTTP 请求的削峰路径

请求到达 → Nginx 限流（令牌桶）→ 应用层排队（Channel Buffer）→ Worker Pool 处理 → MQ 持久化 → 异步消费。每一层都在"削"——Nginx 削尖峰，应用层削抖动，MQ 削长期波动。三层削峰确保后端永远在安全水位。

🛡️ 熔断降级 — 防雪崩的最后防线

🔌 类比：家里的空气开关（熔断器）——用电量超过线路负荷时，不是让电线烧起来，而是主动跳闸断电。熔断 = 牺牲局部保全局。

Circuit Breaker 三态

图 3 · Circuit Breaker 三态：CLOSED → OPEN → HALF-OPEN → CLOSED

Go 熔断器实现

type CircuitBreaker struct {
    mu            sync.Mutex
    state         int        // 0=CLOSED 1=OPEN 2=HALF-OPEN
    failures      int
    threshold     int        // 触发 OPEN 的失败次数
    timeout       time.Duration // OPEN→HALF-OPEN 等待时间
    lastFailTime  time.Time
}

func (cb *CircuitBreaker) Call(fn func() error) error {
    cb.mu.Lock()
    if cb.state == 1 { // OPEN
        if time.Since(cb.lastFailTime) < cb.timeout {
            cb.mu.Unlock()
            return fmt.Errorf("circuit breaker is OPEN")
        }
        cb.state = 2 // HALF-OPEN
    }
    cb.mu.Unlock()

    err := fn()
    cb.mu.Lock()
    defer cb.mu.Unlock()

    if err != nil {
        cb.failures++
        cb.lastFailTime = time.Now()
        if cb.failures >= cb.threshold {
            cb.state = 1 // OPEN
        }
    } else {
        cb.failures = 0
        if cb.state == 2 {
            cb.state = 0 // CLOSED
        }
    }
    return err
}

降级策略

功能降级

推荐页挂了 → 展示静态默认推荐。订单详情页挂了 → 只展示基础信息，不展示推荐商品。核心原则：主流程必须有降级方案。

数据降级

DB 挂了 → 读缓存。缓存也挂了 → 返回 5 秒前的快照（stale data）。连快照都没了 → 返回友好的"稍后重试"页面。逐层降级，绝不抛 500。

🗼 多级缓存 — 逐层挡，层层过滤

🏰 类比：城堡的防御体系——外围的护城河挡住大部分人（CDN），过了护城河还有城墙（本地缓存），破了城墙还有内城（Redis），内城攻破了还有王宫（DB）。每一层都拦截一部分请求，最后一层（DB）永远不被打穿。

图 4 · 多级缓存架构：CDN → 本地缓存 → Redis → DB，99.9% 的请求在进入 DB 前被拦截

缓存更新的经典问题

问题现象解法
缓存穿透请求的数据缓存和 DB 都没有布隆过滤器 + 空值缓存
缓存击穿热 key 过期，瞬间打满 DB互斥锁重建 + 永不过期

🔥 热点应对 — 当某个 key 被打爆时

🎫 类比：某个明星演唱会门票开售（热点 key），几百万人同时抢。如果所有人跑去同一个卖票窗口（单 Redis key），窗口会被挤破。热点 = 流量全部打到同一个资源上。

热点问题三要素

层次热点类型典型场景
Redis热 Key一个商品 key 每秒几百万读
HTTP热点 URL秒杀页面、公告页面
DB热点行同一个商品 ID 的库存行

热点应对三板斧

① 热 key 发现

在 Redis 侧监控每秒访问量。超过阈值（如 100k/s）自动上报到热点中心。

方式：Redis 的 hotkey 命令、本地统计 + 定时上报、代理层拦截

② 本地缓存分摊

热 key 缓存到每个节点的本地缓存中。即使 Redis 被打爆，本地缓存还能挡一阵。

方式：freecache、bigcache、go-cache

③ 热 key 散列

把热 key product_123 拆成 product_123_0 到 product_123_9 共 10 个副本。请求随机访问一个副本。一个 Redis 节点扛 100k qps → 10 个节点各扛 10k qps。

代价：一致性变弱（不同副本可能不同步），读多写少的场景才适用。

📦 池化技术 — 不要重复造轮子，要反复用

🚗 类比：打车 vs 买车——每次出门买一辆新车（每次 new 对象），还是租共享汽车（对象池复用）？池化 = 把高频创建/销毁的对象缓存起来重复用。

Go 中的四种池化方式

池类型复用对象Go 实现效果
连接池DB 连接 / Redis 连接 / HTTP 连接database/sql、redis.Pool、http.Transport减少 TCP 握手，QPS 提升 10x
对象池临时对象（protobuf、byte buffer）sync.Pool减少 GC，延迟降低 50%
goroutine 池goroutine（避免创建/销毁开销）ants、gammazero/workerpool控制并发数，防止 goroutine 泄漏
内存池固定大小内存块bytebufferpool、自制 slab pool减少内存分配和 GC 压力

sync.Pool 最佳实践

var bufPool = sync.Pool{
    New: func() interface{} {
        return bytes.NewBuffer(make([]byte, 0, 1024))
    },
}

func HandleRequest(r Request) {
    buf := bufPool.Get().(*bytes.Buffer)
    buf.Reset()
    defer bufPool.Put(buf)

    json.NewEncoder(buf).Encode(r)
    // 用 buf 做后面的处理 ...
    // 不用每次都 new bytes.Buffer，减少 GC 压力
}

💡 池化实战经验

  • 连接池大小不是越大越好。太大反而因为线程竞争而降低性能。标准做法：压测找最优值。
  • sync.Pool 不保证对象还在。GC 会清理 Pool 里的对象。只适合"有就用，没有就新创建"的场景。
  • goroutine 池 vs 按需创建：短任务（<1ms）用 goroutine 池，长任务（>100ms）按需创建即可。

⚡ 秒杀系统设计 — 高并发实战案例

秒杀是高并发系统设计的终极考题。它同时考验了限流、削峰、熔断、缓存、热点应对——所有策略全上还不够，还要额外加几层。

🎪 类比：超市搞促销，前 100 名鸡蛋 1 毛钱（秒杀）。门口涌进 10 万人——保安（限流）只看健康码就慢，所以要提前发入场券（预约抢资格）。进去之后不是直接去货架抢鸡蛋，而是先抽号（异步扣库存），抽到了再去领。秒杀 = 分层削峰 + 异步化 + 最终一致。

图 5 · 秒杀系统多层防御：从浏览器到 DB 共 6 层过滤，每层拦截 90% 以上的请求

秒杀核心流程（Go 实现）

// 1. 预热：Redis 提前设置库存
client.Set(ctx, "stock:1001", 100, 0)

// 2. Lua 脚本：原子扣库存+判断
luaScript := `
local stock = redis.call("GET", KEYS[1])
if not stock or tonumber(stock) <= 0 then
    return 0  -- 卖光了
end
redis.call("DECR", KEYS[1])
return 1     -- 扣减成功
`

// 3. 应用层限流+扣库存
func Seckill(ctx context.Context, userId, skuId int) error {
    // 第一步：本地限流（令牌桶）
    if !limiter.Allow() {
        return fmt.Errorf("请求太频繁")
    }

    // 第二步：布隆过滤器判断用户是否重复抢
    if bloom.Test([]byte(fmt.Sprintf("%d:%d", userId, skuId))) {
        return fmt.Errorf("已抢过")
    }

    // 第三步：Redis 原子扣库存
    script := redis.NewScript(luaScript)
    ok, _ := script.Run(ctx, client, []string{"stock:1001"}).Int()
    if ok == 0 {
        return fmt.Errorf("库存不足")
    }

    // 第四步：异步落库（MQ 削峰）
    mq.Publish("order_create", OrderMsg{UserId: userId, SkuId: skuId})
    return nil
}

秒杀设计的核心原则

  • 不要信任客户端 — 所有限流在后端做，前端只是辅助
  • 尽量让流量提前被拦截 — 浏览器→Nginx→应用→Redis→MQ→DB，越早拦截越好
  • Redis 不是万能的 — Redis 热 key 会打爆 Redis 集群，需要用本地缓存分摊
  • 最终一致性就够了 — 秒杀不需要强一致。用户扣款成功但订单落库延迟几秒，完全 OK

💬 实战问答

🤔 Q：限流和削峰有什么区别？能只用一种吗？
限流是拒绝超额请求，削峰是缓冲超额请求。限流说"不行，你被拒绝了"，削峰说"行，但你要等一下"。两者不是替代关系——限流保护系统不被冲垮，削峰提升系统利用率。正确的做法是两处都用：先用消息队列削峰，队列满了再用限流拒绝新请求。
🤔 Q：服务挂了怎么快速发现热点 key？有没有自动化的方案？
有两种主流方案：① 代理层拦截（如 Codis、Twemproxy）统计每个 key 的访问频次，超过阈值自动上报。② 客户端采样（如 Redis 的 hotkey 命令，或 Go 客户端的 local top-k 统计）。更自动化的是基于滑动窗口的实时热点发现：每 1s 统计一次 top keys，超过 N 次/秒就自动标记为热点，触发本地缓存策略。
🤔 Q：熔断后 HALF-OPEN 阶段放多少请求合适？
通常放少量请求（如 1-5 个）——只需要验证后端是否恢复。全部成功就关闭熔断器，有一个失败就继续 OPEN。HALF-OPEN 的请求可以用一个单独的 goroutine 定期探测，而不是在正常请求路径上随机放行。这样用户不会偶然体验到"时好时坏"的服务。
🤔 Q：本地缓存 + Redis 两级缓存，一致性怎么保证？
两级缓存的一致性通常做近似保证（最终一致性），不追求强一致。常用策略：① 本地短 TTL（如 10s），过期后重新从 Redis 拉取。② 定时刷新（每 5s 从 Redis 同步一次热点 key）。③ 消息通知（更新发生时 MQ 广播通知各节点清除本地缓存）。实际践中，对缓存的期望是"几乎一致"就够了——如果要求强一致，就不用缓存。
追问：如果同时面临 100 倍流量洪峰，限流/削峰/熔断全部用上还是扛不住，还有什么办法？ #极限思考

实战中还剩最后三张底牌：

  • 弹性扩容（Auto Scaling） — 流量上来时 K8s HPA 自动加 Pod。配合负载均衡预热，从 10 台扩到 100 台。
  • 静态化 + CDN 扛 99% — 把页面提前生成静态 HTML 推到 CDN。用户看到的就是静态页面，根本不打到后端。秒杀页面提前渲染，"立即抢购"按钮的时间到了才变成可点击，但页面早在 CDN 上了。
  • 请求合并 + 批量写入 — 1 万个扣库存请求合并成 1 个批量操作。Redis pipeline + Lua 脚本，从 1 万次网络 IO 变成 1 次。

如果这三张底牌都用了还是扛不住……那就是资源不够。这时候不是技术问题，是预算问题了 🙂

追问：本地缓存怎么保证不成为新的瓶颈？GC 和内存上限怎么处理？ #Go 性能

本地缓存用不好确实会出问题——比不用缓存还糟糕。几个关键点：

  • 限制缓存大小（Hard Limit） — 用 freecache 或 bigcache 这种预分配内存的库，不要用 go-cache（无限增长）。给本地缓存分配 512MB 就是 512MB，超出自动淘汰最旧的。
  • 对象复用 + 减少 GC 扫描 — bigcache 把数据序列化成字节数组存，GC 不用扫描。freecache 用 ring buffer 预分配，GC 友好。
  • 分 shard 降低锁竞争 — freecache 默认 256 个 shard，每个 shard 有自己的锁。热点 key 被散列到不同 shard，多核并发时不会互相等锁。
  • 监控缓存命中率 — 命中率低于 50% 说明缓存策略有问题，需要调整 TTL 或扩容。

推荐直接使用 freecache（最佳 Go 本地缓存库）：零 GC、预分配内存、分 shard、支持过期。20 万 QPS 毫无压力。

🎯 口播总结

高并发系统设计 = 并发编程原语（怎么安全地操作共享数据）+ 系统级策略（怎么让流量在合理范围内）+ 实战组合拳（具体场景怎么应对）。
