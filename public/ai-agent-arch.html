🤖 AI Agent 架构原理

2026 · 四大范式 & 状态机设计 & Go 实现 · 共 8 张配图

🍳 类比：Agent 就像一个厨师。最简单的厨师是"固定菜单"（单一 Prompt 调用），但一个真正能干的厨师会 — 看菜名查菜谱（感知→推理），发现缺食材就去买（工具调用），发现灶台不够就调整顺序（规划），做完尝一尝再调味道（反思→修正）。Agent 架构就是教厨师怎么做饭的"思维操作系统"。

本文会拆解四种主流 Agent 范式：ReAct（思考-行动-观察循环）、Plan-Execute（计划-执行分离）、Multi-Agent（多角色协作）、Reflection（自我反思与修正），并深入它们背后的状态机设计和认知架构。

🔄 ReAct 范式 — 思考·行动·观察

ReAct（Reasoning + Acting）是 2023 年由 Yao 等人提出的最经典 Agent 范式。它将推理链和工具调用交织在一起，让模型在"想"和"做"之间切换。

图 1 · ReAct 范式循环：思考→行动→观察→再思考，直到得到最终答案

ReAct 的核心规则

  • 每步必观察：每次工具调用后，把结果塞回上下文让模型"看到"
  • 思考驱动行动：模型先想"我需要做什么"，再决定调什么工具
  • 最大迭代限制：防止无限循环（典型值 10-25 步）
  • Early Stopping：当模型输出"最终答案"标志时提前结束

Prompt 结构

ReAct 的核心 Prompt 模板大概长这样：

// System Prompt 中注入的 ReAct 格式
Available Tools:
- get_weather(city: string) // 查询天气
- search_web(query: string) // 搜索互联网

Format:
Thought: 你当前的想法和下一步计划
Action: 工具名称(参数列表)
Observation: 工具返回的结果
...（重复 Thought→Action→Observation）...
Thought: 我现在有足够的信息了
Final Answer: 对用户的最终回复

ReAct 的三大缺陷

问题 1 缺乏全局规划

ReAct 是"走一步看一步"的贪心策略。遇到需要 5 步以上的复杂任务时，容易迷失在局部细节中，忘记最终目标。

问题 2 Token 爆炸

每一步的 Thought + Action + Observation 都累积到上下文中。15 步循环后，光 Agent 的"思考历史"就能吃掉几万 Token。

问题 3 没有回退机制

一旦选错了工具或理解错了上下文，ReAct 没有"撤销"能力。错误会沿着推理链传播，后面再怎么想也是基于错误前提。

📋 Plan-Execute 范式 — 先规划再执行

Plan-Execute 把"想"和"做"彻底分开。Agent 先制定一个多步计划，然后按计划逐个执行，遇到异常再重新规划。这就像你装修房子先画施工图，再按图施工，而不是边想边砸墙。

图 2 · Plan-Execute 架构：规划器生成多步计划，执行器逐步执行，失败触发重新规划

两种实现模式

维度单模型内循环规划器 + 执行器分离
架构同一个 LLM 既当 Planner 又当 Executor两个独立 LLM 调用（或同模型不同 System Prompt）
步骤状态存储在上下文窗口中存储在外部数据结构（JSON/DAG）中
容错整个计划重做只回滚到失败步骤
复杂度低，适合简单任务高，适合复杂工作流
代表框架LangChain Plan-and-ExecuteBabyAGI, AutoGPT, Voyager

🍳 类比：装修房子，ReAct 是"边想边砸墙"——砸错了再糊上。Plan-Execute 是先出设计图（规划），项目经理按图施工（执行），发现管道位置不对就更新图纸（重新规划），而不是继续在那砸。后者成本低 10 倍。

Plan-Execute 的陷阱

  • 计划过时：如果执行过程中外部环境变化（比如数据源返回了意料之外的结果），原始计划可能已经无效
  • 过度规划：有些任务根本不需要 5 步计划，但 Planner 仍然生成复杂的 DAG，浪费 Token 和时间
  • 计划粒度难控：太粗 → 一步之内还是需要 ReAct 思维；太细 → Token 开销反超 ReAct

👥 Multi-Agent 范式 — 多角色协作

Multi-Agent 不是一个 Agent 单打独斗，而是多个 Agent 各司其职，像一支篮球队有控卫、分卫、中锋，每个人负责不同的战术角色。

图 3 · Multi-Agent 架构：Orchestrator 将大任务拆解，分发给 Specialist Agent，汇总结果

三种协作模式

👑 编排式 (Orchestrator)

一个中心调度器负责任务分解、分发和汇合。Worker 之间不直接通信。

适用：写作助手（搜索→大纲→撰写→配图）
优势：流程可控，调试容易

🤝 协商式 (Debate)

多个 Agent 围绕一个问题交换意见，直到达成共识。每个 Agent 有自己的角色偏好。

适用：代码审查、设计评审
优势：减少偏见，质量更高

🏗️ 层级式 (Hierarchical)

Orchestrator 可以再招募 Sub-Orchestrator，形成树的层级结构。每个 Sub-Orchestrator 管理自己的 Worker 团队。这在大型复杂系统中特别有用（比如一个装修管理系统：总调度→水电组、木工组、油漆组→每个组再细分任务）。

🧠 Reflection 范式 — 自我反思与修正

Reflection 是最接近"人类学习方式"的范式。Agent 先完成一个任务，然后自己"回头看"哪里做得不好，再修正。这就像你写完一篇文章先自己读一遍，圈出错别字和逻辑漏洞再改。

图 4 · Reflection 架构：Actor 生成 → Critic 审查 → 反馈→修正循环

Reflection 的实践技巧

技巧做法效果
角色隔离Actor 用"自信的专家"Prompt，Critic 用"苛刻的审稿人"Prompt减少自我偏差，Critic 更容易发现 Actor 的问题
结构化反馈Critic 按格式输出：问题类型 | 位置 | 严重程度 | 修改建议Actor 不需要"猜测"哪里错了，直接执行修改
轮次上限最多 3 轮反思，否则边际效益递减防止无限循环（Reflection 最危险的陷阱）
渐进回顾先检查语法/格式 → 再检查逻辑 → 最后检查风格避免低层次问题干扰高层次思考

⚙️ Agent 状态机设计

无论哪种范式，Agent 本质上都是一个有限状态机（FSM）。理解状态机设计是构建可靠 Agent 的基石。

核心状态定义

图 6 · Agent 核心状态机：IDLE → THINKING → ACTING → OBSERVING → THINKING/IDLE

状态管理的三个关键设计

1. 幂等性（Idempotency）

工具调用可能在网络超时后重试，同一个 Action 不能被执行两次。实现方式：每个 Action 生成一个唯一 action_id，状态机记录已执行的 ID，遇到重复则直接返回缓存的结果。

2. 超时处理（Timeout）

每个状态必须有超时保护：

  • THINKING 超时 → 回到 IDLE，回复"思考超时"
  • ACTING 超时 → 标记工具调用失败，进入 OBSERVING 记录错误
  • 总步数超限 → 回到 IDLE，回复"任务太复杂无法完成"

3. 上下文压缩（Context Compression）

超过 N 步后，自动压缩历史：保留系统 Prompt + 用户原始问题 + 最近的 M 步记录，中间的步骤用"省略了 K 步中间过程"替代。这是 Multi-Agent 和 Reflection 范式中最容易忽略但最关键的性能优化。

💻 Go 实现 — 从零构建 Agent 状态机

下面用 Go 实现一个可扩展的 Agent 状态机框架。支持任意范式的核心循环。

7.1 状态机核心

package agent

import (
    "context"
    "fmt"
    "time"
)

// State 定义 Agent 状态
type State int

const (
    StateIDLE      State = 0
    StateThinking  State = 1
    StateActing    State = 2
    StateObserving State = 3
    StateDone      State = 4
    StateError     State = 5
)

func (s State) String() string {
    return []string{"IDLE", "THINKING", "ACTING", "OBSERVING", "DONE", "ERROR"}[s]
}

// Action 记录一次工具调用
type Action struct {
    ID        string        // 唯一 ID，用于幂等
    Name      string        // 工具名
    Arguments map[string]any // 参数
    Result    string        // 结果
    Error     error         // 执行错误
    Timestamp time.Time
}

// AgentStatus 包含运行中状态的快照
type AgentStatus struct {
    State      State
    Step       int
    MaxSteps   int
    StartTime  time.Time
    LastAction *Action
    History    []*Action  // 执行历史
}

7.2 Agent 引擎 — 支持任意范式

// Agent 引擎：负责状态转换
type Agent struct {
    status    AgentStatus
    llm       LLMProvider     // 接口化 LLM
    tools     map[string]Tool   // 注册工具
    memory    MessageStore    // 消息存储
    compressor ContextCompressor // 上下文压缩
    stopFunc  func(string) bool // 提前停止判断
}

// Run 运行 Agent 主循环（ReAct 模式）
func (a *Agent) Run(ctx context.Context, userInput string) (string, error) {
    a.status.State = StateThinking
    a.status.StartTime = time.Now()

    for a.status.Step < a.status.MaxSteps {
        select {
        case <-ctx.Done():
            return "", ctx.Err()
        default:
        }

        // 1. THINKING — 调用 LLM 决定下一步
        thought, actionSpec := a.think(ctx)
        if a.hasFinalAnswer(thought) {
            a.status.State = StateDone
            return a.extractAnswer(thought), nil
        }

        // 2. ACTING — 执行工具
        a.status.State = StateActing
        action := a.executeAction(ctx, actionSpec)
        a.status.LastAction = action
        a.status.History = append(a.status.History, action)

        // 3. OBSERVING — 把结果塞回上下文
        a.status.State = StateObserving
        a.memory.AddObservation(action)

        // 4. 上下文压缩（每 5 步执行一次）
        if a.status.Step > 0 && a.status.Step%5 == 0 {
            a.memory = a.compressor.Compress(a.memory)
        }

        a.status.Step++

        // 提前停止检查
        if a.stopFunc != nil && a.stopFunc(a.memory.Latest()) {
            a.status.State = StateDone
            return a.memory.Latest(), nil
        }

        a.status.State = StateThinking
    }

    a.status.State = StateError
    return "", fmt.Errorf("max steps (%d) exceeded without final answer", a.status.MaxSteps)
}

// think 调用 LLM，返回思考内容和 Action 规范
func (a *Agent) think(ctx context.Context) (string, *ActionSpec) {
    msgs := a.memory.GetMessages()
    resp, err := a.llm.Complete(ctx, msgs)
    if err != nil {
        return "", nil
    }
    // 解析输出，提取 Action（如果有）
    return parseThoughtAction(resp)
}

7.3 接口设计 — 替换式架构

// LLMProvider 抽象 — 可替换任意模型
type LLMProvider interface {
    Complete(ctx context.Context, msgs []Message) (string, error)
}

// Tool 接口 — 所有工具都必须实现
type Tool interface {
    Name() string
    Description() string
    Execute(ctx context.Context, args map[string]any) (string, error)
}

// ContextCompressor 接口 — 控制 Token 消耗
type ContextCompressor interface {
    Compress(ms MessageStore) MessageStore
}

// 用法示例
func main() {
    agent := &Agent{
        status: AgentStatus{MaxSteps: 15},
        llm:    NewOpenAIProvider("gpt-4"),
        tools: map[string]Tool{
            "search": NewSearchTool(),
            "calc":   NewCalculator(),
        },
        compressor: NewSlidingWindowCompressor(10), // 保留最近 10 条
    }

    result, err := agent.Run(context.Background(), "上海今天天气怎么样？需要带伞吗？")
    if err != nil {
        fmt.Printf("Agent error: %v\n", err)
        return
    }
    fmt.Println(result)
}

📌 代码设计要点

  • LLMProvider 接口 — 可以无缝切换 GPT-4 / Claude / MiniMax 或本地模型
  • Tool 接口 — 每个工具独立测试，通过 Execute 方法隔离副作用
  • ContextCompressor — 滑动窗口压缩，保留最近的 N 条消息和原始的 System Prompt
  • 幂等性 — Action ID 机制保证同一动作不会被执行两次（网络重试安全）

💬 实战问答

Q1: ReAct 和 Plan-Execute 怎么选？分界线在哪里？
经验法则：如果任务能在 5 步以内完成（比如查天气、简单问答），用 ReAct。如果需要 5 步以上且有明确的子任务依赖关系（比如"先搜索三个来源，再对比分析，最后生成报告"），用 Plan-Execute。分界线是"你是否能在写下任务时，就预见到步骤顺序"。
Q2: Multi-Agent 一定比单 Agent 好吗？
不一定。Multi-Agent 的 Token 消耗是单 Agent 的 3-5 倍，延迟也更高。只在以下情况使用：(1) 任务需要截然不同的专业知识（比如代码 + 设计），(2) 需要多角色交叉验证（写代码 + 安全审查），(3) 单个 Agent 的上下文窗口装不下。一个常见的错误：任务明明是单步 Tool Calling，却搞了 3 个 Agent 来回传消息。
Q3: Reflection 的 Critic 怎么设计才能有效？
关键三点：(1) 结构化输出 — Critic 必须按固定格式输出反馈（如 JSON），而不是自由文本。(2) 评分标准明确 — 在 Critic 的 Prompt 里写明检查清单：语法、逻辑、完整性、风格。(3) 多次反思边际递减 — 实测第 3 轮以后改进不到 5%，所以默认最多 3 轮就够了。
Q4: Agent 上下文窗口满了怎么办？
三种策略按优先级排列：
① 滑动窗口：保留 System Prompt + 用户问题 + 最近 N 轮交互，中间历史用"省略了 K 步"替代。
② 摘要压缩：用 LLM 把历史浓缩成一段摘要，替换原始对话。③ 外部记忆：把关键状态存到数据库，Agent 只在需要时载入。第一种实现成本最低，推荐优先尝试。
Q5: Agent 调用工具超时了怎么处理？
三步恢复：(1) 尝试重试 1 次（可能有网络抖动），(2) 如果还是超时，记录错误并告诉 LLM"工具调用失败，原因：超时"，(3) 让 LLM 决定是否换一个工具或换一个策略。不要让 Agent 在同一个工具上无限重试。安全的做法是：每个工具调用设独立超时（默认 10s），超过后触发 fallback。

🎤 本文一句话总结

Agent 架构的核心是状态机：四种范式（ReAct / Plan-Execute / Multi-Agent / Reflection）是四种不同的"思考→行动→观察"调度策略，可以自由组合。生产级 Agent 需要关心：幂等性（同一个 Action 不执行两次）、超时控制（每个状态有超时保护）、上下文压缩（防止 Token 爆炸）。Go 实现的接口化框架让你一小时内在任何范式之间切换。
