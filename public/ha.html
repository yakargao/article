⚡ 高可用服务设计 · 从入门到精通

一句话：高可用 = 系统在部分组件挂掉时，仍然能对外正常服务。不是"不挂"，是"挂了也不怕"。

核心思想贯穿全文：冗余 + 隔离 + 快速恢复 = 高可用

📐 什么是高可用？

✈️ 类比：飞机有双发动机——一个坏了另一个还能飞。双电源、双网络、双机房，高可用的本质就是"双份保险"。

可用性公式

Availability = MTTF / (MTTF + MTTR)

术语含义你要怎么优化
MTTF (Mean Time To Failure)平均多久坏一次提高代码质量、冗余部署 → 延长 MTTF
MTTR (Mean Time To Recover)平均多久能修好自动故障转移、快速回滚 → 缩短 MTTR

核心策略：MTTR 比 MTTF 更重要。系统一定会坏，关键是多久恢复。

📊 SLA 与"几个 9"

可用性年宕机时间月宕机什么级别
99% (两个9)3.65 天7.2 小时内部工具
99.9% (三个9)8.76 小时43.8 分钟一般 SaaS
99.99% (四个9)52.6 分钟4.4 分钟支付/金融
99.999% (五个9)5.26 分钟26 秒电信级

SLA vs SLO vs SLI

SLI（指标）：实际测量的数字，如"错误率 0.1%"
SLO（目标）：内部定的目标，如"错误率 < 0.5%"
SLA（协议）：对外承诺的合同，如"超过 0.5% 赔钱"

SLO 应该比 SLA 更严格，留出缓冲空间。

🔄 冗余设计：多备几份永远没错

🔋 类比：主机 + UPS + 柴油发电机。市电断了 UPS 顶上，UPS 快耗尽了发电机启动。层层冗余，绝不单点。

三种冗余模式

模式描述适用场景
Active-Standby主节点干活，备节点待命。主挂 → 备接管数据库、消息队列
Active-Active所有节点同时干活，LB 分发流量无状态服务（API）
N+1需要 N 台，多备 1 台。成本最低的冗余预算有限的场景

单点故障（SPOF）—— 高可用的死敌

任何只有一个的东西都是 SPOF：单台服务器、单个数据库、单个机房、单人掌握密码。

消除 SPOF = 高可用的第一步。问自己："这个挂了怎么办？"→ 加备份。

⚖️ 负载均衡：银行叫号系统

🏦 类比：银行大厅——你取号后，叫号系统（负载均衡器）把你分配到空闲柜台（后端服务）。某个柜台临时关闭？系统自动跳过它（健康检查）。没人排队时空转？柜台数量弹性伸缩。

四层/七层负载均衡架构 + 健康检查自动摘除故障节点

负载均衡算法速查

算法原理适用
轮询 (Round Robin)依次分配，循环往复服务器配置相同
加权轮询按权重比例分配服务器配置不同
最少连接发给当前连接数最少的长连接场景
IP Hash / 一致性哈希同一 IP 固定到同一节点需要会话保持 / 缓存亲和
最短响应时间发给响应最快的服务性能不均

🔁 故障转移：坏了自动换，用户无感知

⚡ 类比：双路供电——A 路断电，ATS（自动切换开关）在 20ms 内切到 B 路。你的灯闪了一下？不，你根本没感觉。这就是故障转移的目标。

关键设计点

  1. 故障检测要快：心跳 + 超时。常用：TCP 连接探测、HTTP /health 端点、gRPC Health Check
  2. 切换要快：DNS 切换可能分钟级（TTL 缓存），VIP 漂移秒级，客户端重试毫秒级
  3. 避免脑裂：两个节点都以为自己是主 → 用法定人数（Quorum）+ 租约（Lease）防止
  4. 优雅降级：全都挂了也比雪崩好 → 返回缓存/降级数据/限流拒绝，不要硬撑

Go 中的健康检查示例

// 一个简单的健康检查端点
func HealthHandler(w http.ResponseWriter, r *http.Request) {
    // 检查依赖：DB、Redis、MQ...
    ctx, cancel := context.WithTimeout(r.Context(), 2*time.Second)
    defer cancel()

    if err := db.PingContext(ctx); err != nil {
        w.WriteHeader(http.StatusServiceUnavailable)
        json.NewEncoder(w).Encode(map[string]string{"status": "unhealthy", "db": "down"})
        return
    }
    w.WriteHeader(http.StatusOK)
    json.NewEncoder(w).Encode(map[string]string{"status": "healthy"})
}

🚪 API 网关：大楼的前台

🏢 类比：写字楼前台——访客统一登记（认证）、包裹统一收发（路由）、可疑人员拦截（安全）、电梯分流（限流）。各公司（微服务）不用各自设前台。

能力说明
统一入口所有流量经过网关，单点管控
认证鉴权JWT 验证、OAuth2 在网关层完成
限流熔断保护后端，拒绝超额请求
路由转发按路径/Header 分发到不同服务
灰度发布按用户 ID/百分比分流到新版本
日志/监控统一记录请求耗时、状态码

Go 生态常用网关：Kong、APISIX、Traefik、自研（基于 net/http + 中间件模式）。

💾 数据层高可用：最麻烦的一层

📚 类比：图书馆的珍本——不能搞丢，不能写错，多人借阅还不能冲突。数据层高可用 = 既要多副本（不丢），又要一致性（不错），还要性能（不慢）。

🔺 CAP 定理：三者只能得其二的铁律

🎯 类比：项目管理三角——快、好、省，只能选两个。CAP = 一致性、可用性、分区容错，只能同时满足两个。

CAP 定理：P 必须选，然后在 C 和 A 之间权衡

关键理解

P（分区容错）必须选。网络分区在分布式系统中必然发生，你不可能假设网络永远正常。

剩下C 和 A 二选一：网络断了时，你是拒绝写入（选 C）还是接受不一致（选 A）？

CP（ZooKeeper/etcd）：分区时拒绝服务，保证数据一致
AP（Cassandra/Eureka）：分区时继续服务，但数据可能不一致

🗳️ 共识算法：会议室里怎么达成一致

🤝 类比：三人合伙开公司，重大决策需要两人同意。这就是 Raft —— 领导者提案 → 多数派同意 → 执行。如果领导者失联，重新选举。

Raft 核心三件事

子问题怎么解决
Leader 选举随机超时 → 先超时的发起投票 → 拿到多数票成为 Leader
日志复制Leader 收到写请求 → 追加到日志 → 发给 Followers → 多数确认 → 提交
安全性只有拥有最新日志的节点才能当选 Leader；已提交的日志不会被覆盖

为什么是奇数节点？

3 节点容忍 1 台挂，5 节点容忍 2 台挂。偶数节点浪费——4 节点也只容忍 1 台，和 3 节点一样（多数 = N/2+1 → 4 节点多数是 3，和 3 节点多数 2 能力相同）。

🔌 熔断器：电路保险丝

⚡ 类比：你家总闸的保险丝——某路电流过大 → 保险丝熔断 → 保护全屋电路不烧。熔断器 = 下游服务挂了 → 快速返回错误 → 不把上游拖垮。

熔断器三态转换：CLOSED → OPEN → HALF-OPEN → CLOSED

熔断器配置核心参数

参数含义典型值
失败阈值连续失败多少次熔断5~10
冷却时间OPEN 多久后进入 HALF-OPEN30s~60s
半开试探次数HALF-OPEN 放多少请求试探1~3
滑动窗口统计失败率的时间窗口10~30s

🚦 限流：高速入口的闸机

🚗 类比：高速公路入口——车流量太大？闸机控制每 5 秒放一辆。不管后面排多长，进了高速的车不会堵。限流 = 保护下游不被冲垮。

四种限流算法

算法原理特点Go 思路
固定窗口每分钟 N 次，到了就拒绝简单但有边界突刺问题map + atomic
滑动窗口统计过去 N 秒内的请求数平滑，无突刺Redis ZSET / sorted set
漏桶请求先进桶，固定速率流出强行平滑，突发被削峰channel + ticker
令牌桶固定速率放令牌，有令牌才能过允许一定突发（攒的令牌一次用完）golang.org/x/time/rate

Go 标准库令牌桶示例

import "golang.org/x/time/rate"

// 每秒放 100 个令牌，最多攒 200 个（允许突发 200）
limiter := rate.NewLimiter(100, 200)

func handler(w http.ResponseWriter, r *http.Request) {
    if !limiter.Allow() {
        w.WriteHeader(429)  // Too Many Requests
        return
    }
    // 处理请求...
}

🔄 重试 & 超时：不放弃，但也不能傻等

📞 类比：打电话——对方没接？等 3 秒再打一次（重试）。但连打 10 次就骚扰了（重试上限）。打了 5 分钟还没人接？挂了吧（超时）。

重试三要素

  1. 退避策略：不要立刻重试，等 100ms → 200ms → 400ms...（指数退避 + 随机抖动）
  2. 重试上限：最多 3 次，无限重试 = 自残
  3. 幂等性：扣款操作不能重试两次！确保重试安全——用 幂等 key 去重

超时链条

每一层都要设超时，且上游超时必须大于下游超时之和：

客户端 5s ＞ API 网关 4s ＞ 服务 A 3s ＞ DB 查询 2s 

// Go context 超时链
ctx, cancel := context.WithTimeout(r.Context(), 3*time.Second)
defer cancel()

// 调用下游时传入 ctx，超时自动取消
resp, err := httpClient.Do(req.WithContext(ctx))

🧱 仓壁隔离：一个舱进水，不沉整艘船

🚢 类比：泰坦尼克号为什么沉？相邻水密舱之间的隔板不够高，水从一个舱漫到下一个。现代轮船的仓壁隔离——一个舱进水，其他舱完好，船不沉。

三种隔离方式

方式怎么隔离例子
线程池隔离每个服务用独立线程池/goroutine pool服务A的请求占满线程也不影响B
信号量隔离限制某服务的最大并发数make(chan struct{}, maxConcurrency)
连接池隔离不同下游用不同连接池DB 连接池 vs Redis 连接池分开
// Go 中简单的信号量隔离：用 buffered channel
type Service struct {
    sem chan struct{}  // 信号量
}

func (s *Service) Handle(ctx context.Context) error {
    select {
    case s.sem <- struct{}{}:  // 获取信号量
        defer func() { <-s.sem }()
        // 处理请求...
    case <-ctx.Done():
        return ctx.Err()           // 达到上限，拒绝
    }
}

🌍 多活架构：一个城市地震了，另一个城市还能用

🏙️ 类比：跨国公司有北京和新加坡两个数据中心。北京机房光纤被挖断 → 流量自动切到新加坡。用户在 app 上完全无感。这就是多活。

同城双活/异地多活：每个 Region 独立运行，数据异步同步

多活的挑战

挑战解决思路
数据同步延迟用户粘性路由（同一用户固定到同一 Region）
写冲突避免多主写入；用 CRDT（无冲突数据结构）
全局 ID 唯一雪花算法（含 Region ID 位段）
流量调度DNS 智能解析 / GSLB / Anycast
成本先从冷备开始，逐步到双活

🔍 可观测性：看不见的问题最难修

🏥 类比：病人进医院——体温计（Metrics）告诉你发烧了，病历（Logging）记录之前得过什么病，CT 扫描（Tracing）看到底是哪个器官出问题。三样结合才能确诊。

三大支柱

支柱回答什么问题工具
Metrics 指标"系统正常吗？" QPS/延迟/错误率Prometheus + Grafana
Logging 日志"发生了什么？" 请求的详细记录ELK / Loki
Tracing 链路"哪个环节慢了？" 跨服务的请求追踪Jaeger / Zipkin / OpenTelemetry

RED 方法论（服务监控四指标）

Rate（每秒请求量）· Errors（错误率）· Duration（延迟 P50/P95/P99）

一个服务只要监控这三个指标 + 资源（CPU/内存/GC），就能发现 90% 的问题。

❓ 高频考题

Q1: 设计一个高可用的秒杀系统
入口层：CDN + WAF → API 网关（限流 + 鉴权）→ 负载均衡
应用层：无状态多副本 + 预热缓存 + 熔断兜底
数据层：Redis 预扣库存 + MySQL 异步扣减 + 主从读写分离
关键：防超卖（Redis DECR + 条件判断）、防流量冲击（多层限流）、数据一致性（最终一致）
Q2: 熔断、降级、限流有什么区别？
限流：主动控制流量，超过阈值就拒绝（"只能进 100 个人"）
熔断：下游挂了，自动切断，避免级联故障（"这扇门封了，走别处"）
降级：系统压力大时，关闭非核心功能保核心（"先暂停推荐，保证下单"）
Q3: 怎么保证分布式系统数据一致性？
没有银弹，按场景选：
强一致：Raft/Paxos（etcd）、两阶段提交（XA）——牺牲性能
最终一致：消息队列 + 本地事务表 + 定时补偿——高性能，有延迟窗口
读修复 + 反熵：Cassandra 的 Hinted Handoff + Read Repair
Q4: Go 服务怎么做优雅关闭？
signal.Notify 监听 SIGTERM → 停止接收新请求 → 等待现有请求处理完 → 关闭 DB/Redis 连接 → 退出。 关键：LB 先摘除流量（等一个 health check 周期），再发 SIGTERM。
Q5: Redis 挂了怎么办？
分场景：
缓存场景：熔断 + 降级（返回空数据/默认值），DB 扛一会
计数/库存场景：Sentinel 自动切换 + 持久化（RDB/AOF），恢复后数据回补
分布式锁：RedLock 多节点 + 锁续期防止锁丢失
Q6: 你的系统能抗多少 QPS？怎么知道的？
压测 → 找到瓶颈 → 优化 → 再压测。工具：wrk/vegeta/k6。流程：单接口 → 混合场景 → 全链路。关注：P99 延迟陡增的拐点就是瓶颈 QPS。

💬 苏格拉底式提问 —— 检验真理解

以下问题用来逼你追问本质。先自己想 30 秒，再点开验证。

多活真的在任何场景都比主备好吗？Q1
不是。多活的核心代价是"数据一致性问题"。
• 主备：简单，只有主机房写。代价 = 备机房资源闲置 + 切换有窗口丢失。
• 多活：资源利用率高，切换零感知。代价 = 必须解决写冲突（CRDT、LWW、全局锁），架构复杂度指数级上升。
决策公式：最终一致性 → 多活；强一致（余额、库存）→ 主备。多活不是升级，是 trade-off。
熔断器打开后，什么时候尝试关闭？Q2
Half-Open 状态：熔断器定时放少量请求试探下游是否恢复。
• 试探成功：关闭熔断器。
• 试探失败：重新打开。
关键参数：① 试探间隔 ② 试探比例 ③ 连续成功次数。
经验值：间隔 30s，试探 1-3 个请求，连续成功 3 次再恢复。
CAP 定理是否被过度简化了？Q3
是的。CAP 被严重滥用了。
• 真实含义：发生网络分区（P）时，才必须在 C 和 A 之间选。没分区时二者可兼得。
• 常见误解："我们选 CP 架构"——分区不发生时不需要做选择。
• PACELC：有分区→选 A 或 C？无分区→选 L 或 C？
实战：别纠结"我是 CP/AP"，设计分区时如何降级、恢复后如何修复。

🎙️ 口播总结

这篇文章讲透了高可用系统设计的核心方法论。

高可用的本质就六个字：冗余、隔离、快速恢复。可用性公式：MTTF/(MTTF+MTTR)，MTTR 比 MTTF 更重要——系统一定会坏，关键是多久能恢复。

负载均衡就像银行叫号系统，把流量均匀分发到后端。L4 只看 IP 端口，L7 能理解 HTTP 内容做路由。健康检查自动摘除故障节点，用户无感知。API 网关是所有流量的统一入口，做认证、限流、路由、灰度。

数据层是最麻烦的。数据库从单库到主从到半同步到 MGR 到分库分表，每一步都在平衡一致性和性能。Redis 高可用有 Sentinel 哨兵模式和 Cluster 集群模式。CAP 定理是理论基础——P 必须选，然后在 C 和 A 之间取舍。

弹性设计有四个核心武器：熔断器（下游挂了快速失败）、限流（令牌桶控制流量）、重试（指数退避加随机抖动）、仓壁隔离（一个舱进水不沉船）。

如果只能记住一句话，那就是"冗余+隔离+快速恢复=高可用"。消除所有单点故障、层层设防、自动化故障转移，这就是高可用系统的全部秘诀。
