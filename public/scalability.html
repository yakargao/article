📐 高可扩展性系统设计

一句话：可扩展 = 用户从 1 万涨到 1 亿，系统不需要重写，加机器就能扛住。

核心公式：无状态 + 分片 + 异步 = 无限扩展。和高性能的区别：高性能关心"单机跑多快"，可扩展性关心"加机器能不能线性提速"。

📐 什么是可扩展性？

🏋️ 类比：垂直扩展 = 一个人吃蛋白粉练成施瓦辛格——但人类有极限，不能无限变强。水平扩展 = 雇 100 个普通人一起搬砖——没有单点瓶颈，来多少活就加多少人。

垂直 vs 水平扩展 + 阿姆达尔定律：串行部分是扩展性的死穴

扩展性的四个维度

维度问自己关键指标
流量扩展QPS 翻 10 倍，架构要改吗？加机器后 QPS 是否线性增长
数据扩展数据量从 GB→TB→PB，查询还快吗？分片后查询延迟不劣化
功能扩展加新功能需要改动几个服务？新功能上线的影响面
团队扩展10 个团队并行开发，会不会冲突？服务边界是否清晰（康威定律）

🧩 无状态设计：水平扩展的前提

🍔 类比：快餐店——你点餐时，收银员不认识你，不记得你上次点了什么。任何收银员都能服务你。如果一个收银台排队太长，开一个新的就行——因为所有收银员都一样（无状态）。

如果有状态（如"老王只认小李点餐"）→ 小李请假你就没法点餐 → 不可扩展。

🍕 数据库分片：一张表存不下就切成 N 张

📚 类比：图书馆藏书太多——A-D 作者放一楼，E-M 放二楼，N-Z 放三楼。你要找《三体》（作者刘慈欣，L开头）→ 直接去二楼，不用翻整个图书馆。

Hash 分片：user_id % 4 路由到 4 个 Shard

策略做法优点缺点

难题解法

🔘 一致性哈希：加节点只影响邻居

🪑 类比：圆桌吃饭——8 个人围坐，每人负责照顾自己右手边的菜（数据）。加一个人（加节点）只需要重新分配他右手边那道菜，其他菜不动。而普通 Hash 分片（id%N 改 N）等于所有人站起来重新分菜。

一致性哈希环：key 顺时针找最近的节点，加/删节点只影响局部

🧮 虚拟节点：用数学解决数据倾斜

🎯 类比：4 个人围坐圆桌分菜——有人面前菜多、有人面前菜少（数据倾斜）。现在给每个人发 150 张"替身卡"随机撒在桌上——每个替身代表原主人拿菜。由于替身均匀散布，每个人最终拿到的菜量趋近相等。这就是虚拟节点的魔力。

问题：为什么普通一致性哈希会倾斜？

一致性哈希环上，4 个物理节点把环切成 4 段。但 4 个点不是均匀分布的——它们是通过 hash(node_id) 算出来的，落到环上的位置完全随机。

结果：4 个点在环上形成的 4 段弧长差异巨大——最长的可能是最短的几十倍。数据落到哪段就归哪个节点管，导致有的节点扛 50% 流量，有的只扛 5%。

左：无虚拟节点 → 节点聚集，数据倾斜 | 右：虚拟节点 → 均匀散布，负载均衡

数学原理 #1：为什么节点多了就更均匀？—— 大数定律

一致性哈希的分配本质上是：在 [0, 2³²) 的环上随机撒 K 个点（虚拟节点），数据 key 也随机落在环上。每个虚拟节点管辖的弧长是一个随机变量。

K 个独立同分布的弧长 → 由大数定律，K 越大，各虚拟节点分管的数据量越趋近期望值。方差 ∝ 1/K。

设：环上共有 K 个虚拟节点，数据总量为 D
每个虚拟节点期望分管：D / K

无虚拟节点时：K = N（物理节点数），通常 N ∈ {3, 5, 7}
→ 一个节点可能分到 2-3 倍期望值的数据（方差极大）

有虚拟节点时：K = N × V，V 是每个物理节点的虚拟节点数
→ 由中心极限定理，每个物理节点负载 ~ N(D/N, σ²/V)
→ V 越大，方差越小，负载越均匀

数学原理 #2：负载标准差与虚拟节点数的关系

🧮 形式化推导

设 N 个物理节点，每个节点 V 个虚拟节点，总共 K = N×V 个虚拟节点均匀分布在环上。

一个物理节点 i 的负载 = 它的 V 个虚拟节点分到的数据之和。由于每个虚拟节点的数据量独立同分布（均期望 = D/K）：

Var(负载i) = V × Var(单个虚拟节点) = V × (D/K)² = D² / (N²·V)

标准差与均值的比值（变异系数 CV）：

CV = σ / μ = √(D²/(N²·V)) / (D/N) = 1 / √V

核心结论：负载不均匀程度 ∝ 1/√V，只跟虚拟节点数有关，跟物理节点数 N 无关！

实践选择：V 取多少？

虚拟节点数 VCV (负载变异系数)最轻/最重节点比适用
1（无虚拟节点）100%可能差 10-50 倍—
1031.6%~2 倍小规模、可容忍不均
100-1508-10%< 1.3 倍 ← 业界推荐生产环境标配
10003.2%~1.06 倍极致均匀，但路由表变大

⚖️ 权衡：不是 V 越大越好

V 太小V 太大
❌ 数据倾斜严重❌ 内存开销大（路由表 = N×V 条目）
❌ 加节点时大量数据迁移❌ 节点变更时需更新大量虚拟节点映射
✅ 业界标准：100-150 个虚拟节点/物理节点（Cassandra 默认 256，Dynamo 用 100-150）

加/删节点时的数学

加一个物理节点（V 个虚拟节点插入环上）→ 平均每个现有节点只损失它数据的 1/(N+1)，而不是像 id % N 那样几乎全量重分配。

普通 Hash：
N=4 → N=5，需迁移 80% 数据 (4/5 的 key 的 hash 值变了)

一致性哈希 + 虚拟节点：
N=4, V=150 → 600 个虚拟节点
N=5, V=150 → 750 个虚拟节点
→ 新节点 V 个虚拟节点插入环 → 只从相邻虚拟节点"抢"数据
→ 平均每个旧节点只需迁移 1/(N+1) = 20% 的数据
→ 实际迁移量 ≈ 1/(N+1)，与 V 无关！

🔧 微服务拆分：不是越细越好

🏢 类比：创业公司 5 个人坐在一个房间里，沟通靠喊（单体）。公司 500 人必须分部门（微服务）——财务部、技术部、市场部各管各的，通过邮件和会议（API）协作。

拆分的黄金法则

  1. 按业务域拆分（DDD 限界上下文）：用户服务、订单服务、支付服务
  2. 高内聚、低耦合：一个服务改代码不改另一个。改订单促销逻辑只改订单服务
  3. 数据独立：每个服务有自己的数据库，不共享。通过 API 访问别人数据
  4. 先不拆，等痛了再拆：过早拆分 = 分布式系统的复杂度超过收益

拆分信号——什么时候该拆了？

信号说明
不同模块的部署频率差异大改 A 模块要连带部署 B、C
不同模块的流量差异大A 需要扩容但 B 跟着扩浪费资源
团队冲突两个团队改同一个代码库经常冲突
技术栈差异A 适合 Go 但 B 必须用 Python（ML）

📖 CQRS：读和写用两套模型

🏦 类比：银行柜台——存款走 1 号窗（写操作：校验、记账、持久化），查余额走大厅的 ATM（读操作：直接显示）。读写分离，各自优化。

CQRS 架构

端数据源优化方向
Command（写）主库（MySQL/PostgreSQL）强一致性、事务、高可靠
Query（读）只读副本（ES/Redis/从库）高性能、可任意 JOIN、可面向页面优化

写操作更新主库 → 通过 MQ/CDC 同步到读模型 → 查询从读模型出。读模型可以按前端页面的需要反范式化，极致优化查询性能。

CQRS 适合什么时候？

  • 读写比 > 10:1（读远多于写）
  • 查询逻辑复杂、涉及多表 JOIN（面向页面建模）
  • 读写性能要求差异大（写要强一致，读要毫秒级）

📜 事件溯源：会计账簿模式

📒 类比：传统的记账是直接改余额（UPDATE balance=100）。事件溯源 = 记录每笔交易流水：+50、-30、+80。余额 = 所有交易之和。好处：永远不会丢数据，可以回溯到任意时间点。

🌐 分布式缓存：Redis Cluster 怎么扩展到 TB 级

🗄️ 类比：一个文件柜（单机 Redis）最多 64GB → 不够用？换成 16 个文件柜的档案室（Redis Cluster），每个柜子存一部分，总共 1TB。

Redis Cluster 架构要点

  • 16384 个 slot —— 所有 key 通过 CRC16(key) % 16384 映射到 slot
  • 每个节点负责一部分 slot —— 加节点时 slot 自动迁移
  • 每个 slot 有主从 —— 主节点挂了从节点自动顶上
  • 客户端直连 —— 不需要代理，客户端知道 slot 分布

客户端怎么知道 key 在哪个节点？

// Go Redis Cluster 客户端自动处理路由
rdb := redis.NewClusterClient(&redis.ClusterOptions{
    Addrs: []string{"node1:6379", "node2:6379", "node3:6379"},
})
// MOVED 重定向：key 不在这个节点 → 客户端自动跳转到正确节点
val := rdb.Get(ctx, "mykey").Val()

❓ 高频考题

Q1: 设计一个短链接系统（如 t.cn），怎么扩展到 100 亿条？
ID 生成：雪花算法/号段模式 生成 62 进制短码。
存储分片：按短码 Hash 分片 MySQL（100 亿 / 每片 1 亿 = 100 片）。
缓存：热门链接缓存到 Redis（8 小时过期）。
扩展：无状态 API + 一致性哈希 Redis Cluster + DB 动态扩容。
Q2: 分库分表后怎么扩容？数据怎么迁移？
推荐：双写 + 历史迁移 + 灰度切流。
① 新数据同时写旧分片和新分片（双写）→ ② 后台任务把历史数据从旧分片迁到新分片 → ③ 校验数据一致后切读流量到新分片 → ④ 下掉旧分片双写。
Q3: 微服务和单体到底怎么选？
团队 < 5 人 → 单体，别折腾。团队 > 20 人 + 部署冲突频发 + 模块流量差异大 → 拆分。核心判断标准：拆分的收益（独立部署、独立扩容）> 分布式成本（网络延迟、数据一致性、运维复杂度）。
Q4: 一致性哈希和普通哈希分片的区别？什么时候用哪个？
普通 Hash：id % N，N 变了数据全迁移 → 适合分片数固定（如提前规划好 64 个分片）。
一致性哈希：加/减节点只迁移 1/N 数据 → 适合节点数频繁变化（如 Redis Cluster、CDN 边缘节点）。
生产实践：很多系统用预分片——一次性建 1024 个逻辑分片，物理扩容时移动逻辑分片。
Q5: CQRS 和 Event Sourcing 必须一起用吗？
不必须，但经常一起用。CQRS 解决读写分离，Event Sourcing 是写端的实现方式之一。你可以只用 CQRS（写 MySQL 主库 + 读 ES 副本），不用 Event Sourcing。Event Sourcing 适合审计、回溯、事件驱动架构，但有复杂度成本。
Q6: 怎么看待"过早优化"和"过度设计"？
可扩展性设计是"留接口"而非"提前实现"。比如：设计时用抽象层隔离存储（type UserRepo interface），但不提前写 Redis 实现。选分片 key 时考虑未来扩容，但不提前建 100 个分片。核心：架构上支持扩展，代码上不过度实现。

💬 苏格拉底式提问 —— 检验真理解

以下问题用来逼你追问本质。先自己想 30 秒，再点开验证。

无状态就一定能水平扩展吗？Q1
只对了一半。瓶颈可能在：
① 有状态的下游：应用从 10→100 实例，DB 连接数暴涨。
② 负载均衡器：LB 本身可能成为瓶颈。
③ 共享资源：Redis、MQ——"无状态背后的有状态"是真正的天花板。
真谛：无状态只是"应用层无状态"，基础层一定有状态。把有状态的集中管理，让无状态的任意复制。
CQRS 的代价是什么？Q2
隐藏代价：
① 最终一致性：写完命令模型，查询模型可能还没更新。
② 代码量翻倍：命令和查询各一套代码。
③ 运维复杂度：两个模型独立部署。
不该用的场景：简单 CRUD、团队小、强一致性要求。
黄金法则：CQRS 解决"读写模型根本不同"的问题，不是"代码拆两半"。
一致性哈希解决了什么又引入了什么？Q3
解决：普通哈希 hash(key) % N 在节点变化时数据全乱。
一致性哈希：节点和数据映射到同一个环，增删节点只影响相邻节点。
新问题：① 数据倾斜 → 虚拟节点；② 节点挂了 → 数据全落到下一个 → 可能雪崩；③ 维护复杂度。
本质：用"少量不一致"换"可扩展性"。

这篇文章讲的是系统怎么从 1 万用户撑到 1 亿用户。

可扩展性的核心公式就五个字：无状态加分片。无状态就像快餐店——任何一个收银员都能服务你，因为不记你是谁。加机器就能加容量。有状态就像理发店固定师傅——师傅不在你就没法理发。

阿姆达尔定律揭示了扩展的天花板：串行部分决定你加多少机器都没用。所以 DDD 的聚合设计要坚持"小聚合"原则，减小锁范围和事务边界。

数据层扩展有三种策略：Hash 分片均匀但扩容难、范围分片扩容容易但热点集中、目录分片灵活但路由表本身要保证高可用。一致性哈希是精髓——加节点只影响邻居，不像普通哈希要全量迁移。Redis Cluster 把 16384 个 slot 分到不同节点，客户端直接路由。

CQRS 读写分离让读和写用不同的模型甚至不同的数据库。事件溯源像会计账簿——只记录每笔流水，余额是算出来的，永远可以回溯历史。

终极方案是单元化架构——每个机房是一个自给自足的单元，用户路由到最近单元，无限水平扩展。

如果只能记住一句话，那就是"无状态+分片+异步=无限扩展"。
