🗳️ 分布式一致性协议

一句话：一致性协议 = 让多个节点在不可靠的网络上对同一个值达成一致。

核心矛盾：网络会延迟/分区、节点会崩溃、消息会乱序——但系统还是要"看起来像一个整体"。

❓ 为什么需要共识协议？

👨‍👩‍👦 类比：三兄弟分家产，大哥说房子归我，二哥说不行归我，小弟也说归我——意见不统一。共识协议 = 一套规则，让三兄弟最终都同意"房子归大哥"（即使二哥小弟一开始不同意）。

共识协议的三个要求

要求含义反例
终止性最终一定会做出决定（不死循环）一直投票没结果
一致性所有节点做出的决定相同A 认为主是 N1，B 认为主是 N2
有效性决定的值必须是某个节点提出的无中生有一个值

🚫 FLP 不可能定理：异步网络不存在完美共识

📞 类比：你打电话给朋友商量晚上去哪吃——电话中途可能断线（节点崩溃），断多久也不确定（异步网络）。FLP 说：在纯异步环境中，只要有一个节点可能故障，就无法保证在有限时间内达成共识。

FLP 定理的实践意义：不是共识不可能，而是共识协议必须在安全性和活性之间权衡。Paxos/Raft 通过引入超时（部分同步假设）绕过了 FLP 的绝对限制。

🗳️ 多数派 (Quorum)：共识的数学基础

N 个节点要达成一致，至少需要 ⌊N/2⌋+1（超过半数）同意。这就是 Quorum。

📜 Paxos：共识协议的鼻祖

📜 类比：古希腊议会制——任何人都可以提交法案（Proposer），但必须获得多数长老（Acceptor）的批准才能成为法律。整个流程分两步：第一步"预约提案号"，第二步"正式提案"。两阶段的设计目的只有一个：防止多个提案同时通过导致冲突。

三个角色

角色职责数量
Proposer（提案者）发起提案，推动共识达成可以多个（竞争）
Acceptor（接受者）对提案投票（接受/拒绝），存储已接受的值N 个（通常 3/5/7）
Learner（学习者）学习最终达成的共识值，通知客户端多个

🎯 核心设计目标

Paxos 保证：一旦某个值被多数 Acceptor 接受（Chosen），后续任何提案只能提出这个值。这是 Paxos 最精妙的地方——不管有多少 Proposer 同时提案，最终只有一个值能通过。

提案编号机制

每个提案有唯一编号 n，编号越大优先级越高。Acceptor 只接受编号 ≥ 自己见过的最大编号的提案——这是协议安全性的根基。

# Acceptor 的状态（每个节点维护）
min_proposal = 0          # 见到的最小提案号，拒绝 < 此号的 Prepare
accepted_proposal = 0     # 上次接受的提案编号
accepted_value = None     # 上次接受的值

🔷 两阶段协议 —— 逐步骤拆解

Phase 1：Prepare（预约阶段）——"我要提案，谁支持？"

步骤谁 → 谁消息含义
1.1Proposer → 所有 AcceptorPrepare(n)"我准备用编号 n 提案，给我反馈"
1.2Acceptor → ProposerPromise(n, v_accepted) 或 忽略"我承诺不接受 <n 的提案，另外我之前接受了值 v"（若没接受过则为 null）

💡 Acceptor 收到 Prepare(n) 后的逻辑

def on_prepare(n):
    if n <= min_proposal:
        return IGNORE        # 拒绝：你编号太小/过时了
    min_proposal = n          # 更新"最小接受号"，承诺不再接受 <n
    return Promise(n, accepted_proposal, accepted_value)
    # 返回当前最大编号 + 上次接受的值（如果有）

关键点：Phase 1 不传值，只"占坑"。Proposer 需要收到 Quorum（多数） 个 Promise 才能进入 Phase 2。

Phase 2：Accept（接受阶段）——"现在正式提案"

步骤谁 → 谁消息含义
2.1Proposer → 所有 AcceptorAccept(n, value)"我正式提案：编号 n，值为 value"
2.2Acceptor → ProposerAccepted(n, value) 或 忽略"我接受这个值" 或 "我已经承诺过更大的编号了"

💡 Acceptor 收到 Accept(n, v) 后的逻辑

def on_accept(n, v):
    if n < min_proposal:
        return IGNORE        # 拒绝：我已经承诺过更大的编号
    accepted_proposal = n
    accepted_value = v
    return Accepted(n, v)

最关键的决策逻辑：Proposer 选什么值来提案？

# Proposer 收集到多数 Promise 后
def choose_value(promises):
    # 从所有 Promise 中找出 accepted_proposal 最大的那个值
    highest = max(promises, key=lambda p: p.accepted_proposal)
    if highest.accepted_value is not None:
        return highest.accepted_value  # ✅ 有旧值，必须沿用！
    else:
        return my_own_value            # 可以提自己的值

⚠️ 这是 Paxos 最难理解也最精妙的地方：如果 Proposer 在收集的 Promise 中发现某个 Acceptor 已经接受过值了，那它不能提自己的值——必须沿用那个已经被部分接受的值。这就是为什么无论多少 Proposer 同时跑，最终只有一个值能通过。

🎬 完整示例：正常流程

场景：3 个 Acceptor（A1, A2, A3），P1 想提案值 "X"。

完整流程：Phase 1 Prepare → Phase 2 Accept → Learner 学习

🎬 冲突示例：两个 Proposer 同时提案

场景：P1（n=5，想提 "X"）和 P2（n=8，想提 "Y"）几乎同时发起。

冲突场景：P1 先部分成功 → P2 继承 P1 的值 → 最终一致

🔐 Paxos 的安全性证明（口头版）

为什么 Paxos 安全？归结为一句话：如果一个值 v 已经被多数 Acceptor 接受，那么后续任何通过 Phase 1 的 Proposer，其收集的 Promise 中一定至少有一个 Acceptor 带有 v，从而强制沿用 v。

数学原因：任意两个 Quorum（多数集）必有交集。第一个值的 Acceptor 多数集 ∩ 第二个 Proposer 拿 Promise 的多数集 ≠ ∅，所以必然"撞上"旧值。

Multi-Paxos：从单次共识到连续日志

🏭 类比：Basic Paxos 是"每次选一个值都要重新投票"——效率太低。Multi-Paxos 是"先选出一个常任主席（Leader），以后所有提案由主席优先处理，不走 Phase 1"，效率直接翻倍。

Basic Paxos:  每个值 → Phase1 + Phase2  (两轮RPC)
Multi-Paxos:  先选Leader → 之后每个值只需 Phase2  (一轮RPC)

# Multi-Paxos 的核心优化
# 1. 选出一个 Leader（通过 Basic Paxos 选）
# 2. Leader 在连续提案中跳过 Prepare，直接用 Accept
# 3. 如果 Leader 崩溃 → 重新选 Leader（又是一轮 Basic Paxos）

# Raft 本质上就是 Multi-Paxos + 三大拆分：
#  Leader Election  ·  Log Replication  ·  Safety

🚢 Raft：可理解性优先的共识协议

🏫 类比：班级选班长（Leader Election）→ 班长在黑板上写通知（Log Replication）→ 下面同学抄到笔记本上（Commit）。Raft 把共识拆成三个独立的子问题，比 Paxos 好理解十倍。

Raft：Leader 选举 + Log 复制 + 安全性保证，三大子问题独立解决

Raft 的三大子问题

子问题核心机制关键细节
Leader Election随机超时 + Term + RequestVote RPC150-300ms 随机超时防止分裂投票
Log ReplicationAppendEntries RPC + committedIndexLeader 强制 Follower 日志与自己一致
Safety选举限制 + 提交限制只有拥有所有已提交日志的节点才能当选

Raft 为什么比 Paxos 成功？

可理解性。Paxos 的论文极其难懂，Lamport 本人后来不得不写一篇"Paxos Made Simple"来解释。Raft 的设计目标就是"让研究生能看懂"，把共识拆成独立的 Leader 选举、日志复制、安全性三个模块。

🦓 ZAB：ZooKeeper 的共识引擎

ZAB = ZooKeeper Atomic Broadcast。和 Raft 本质相似但有细微区别：

特性RaftZAB
Leader 选举完成数据可能有空洞需补新 Leader 必须拥有所有已提交事务
日志提交多数派确认即可提交所有 Follower 确认后才能提交（更严格）
读请求可走 Follower（可能 stale）默认走 Leader（保证线性一致）
实现—ZK 专有，和 ZK 强绑定

💬 Gossip 协议：一传十，十传百

🗣️ 类比：村里传话——张三告诉李四"村口有免费西瓜"，李四又告诉王五、赵六。几分钟后全村都知道了。没有中心节点，不需要所有人同时在线。

Gossip 的特点

特性说明
最终一致性不是强一致——存在传播延迟窗口
去中心化每个节点对等，无 Leader
容错性极强节点随便挂，剩下的继续传
指数传播O(log N) 轮覆盖所有节点

谁在用 Gossip？

Redis Cluster 的节点发现、Cassandra 的集群状态同步、Consul 的成员管理、Bitcoin 的区块传播。

🧩 CRDT：不用共识也能达成一致

➕ 类比：加法满足交换律——A 加 5、B 加 3，无论谁先算，总和都是 8。CRDT = 设计一种满足交换律、结合律、幂等律的数据结构，让并发修改自动收敛到相同结果，不需要协商。

类型操作合并方式例子
GCounter只能加取 max点赞计数
PNCounter可加可减两个 GCounter 分别合并库存计数
GSet只能添加取并集去重集合
ORSet可添加可删除带墓碑的并集购物车

CRDT 适合不需要强一致、但对可用性要求极高的场景（如协同编辑、分布式计数器）。

❓ 高频考题

Q1: 为什么需要分布式共识？举一个具体场景
选主：3 个节点的主从集群，主节点挂了，谁当新主？如果两个从节点同时认为自己是主 → 脑裂 → 数据写坏。共识协议保证同一时刻只有一个节点被选为 Leader。etcd/ZK 就是这么做分布式锁和配置管理的。
Q2: Raft 的 Term（任期）是什么？为什么重要？
Term 是一个全局单调递增的整数（逻辑时钟）。每次选举 Term++。作用：① 区分不同任期——旧 Term 的 Leader 发的心跳被拒绝 ② 防止脑裂——一个 Term 最多一个 Leader ③ 日志一致性——Log Entry 带 Term 号，冲突时 Term 大的优先。
Q3: Paxos 和 Raft 的本质区别？
Paxos：非 Leader 驱动，任何人都可以提案，两阶段每次都要走。Multi-Paxos 加了 Leader 优化。
Raft：强 Leader 驱动，日常只需一轮 AppendEntries。把选举、复制、安全拆成三个独立模块。
核心区别：设计哲学——Paxos 追求理论的优雅，Raft 追求工程的可理解性。
Q4: 为什么 Raft 要求奇数节点？
偶数浪费。3 节点容 1 个故障（Quorum=2），4 节点也只容 1 个故障（Quorum=3）。可用性相同，多一台机器的成本。4 节点的唯一好处是读性能高一点，但写性能反而下降（需要等 3 个确认而不是 2 个）。
Q5: Gossip 协议怎么保证消息一定传到所有节点？
不保证"一定"，保证"极高概率"。每个节点每轮随机选 K 个节点传播（通常 K=3）。传播轮数 ≈ log(N)。比如 10000 个节点，约 10 轮覆盖全部。如果某条消息丢了？没关系——定期全量同步（反熵）兜底。
Q6: 用 Paxos/Raft 实现一个可靠的分布式锁，和 Redis 的锁有什么区别？
CP 系统锁（Paxos/Raft）：绝对互斥——即使网络分区，也不会有两个客户端同时拿到锁。代价：性能低（毫秒级），依赖多数派。
AP 系统锁（Redis）：性能高（微秒级），但在网络分区或主从切换时可能丢锁。选哪个取决于业务：扣库存必须 CP，防重复提交用 AP 就行。

💬 苏格拉底式提问 —— 检验真理解

以下问题用来逼你追问本质。先自己想 30 秒，再点开验证。

Raft 为什么需要强 Leader？Q1
可以没有 Leader（如 Paxos），但 Raft 选择 Leader 是为了"可理解性"。
① 简化决策：所有写通过 Leader → 日志顺序由 Leader 确定。
② 减少通信：1→N 推送，不是 N²。
③ 可读性：Follower 可直接响应读请求。
代价：Leader 宕机需选举 → 选举期间不可写。但"简单易懂"比"理论上更优"更重要。
Paxos 过时了吗？Q2
没过时，但多数场景用 Raft。
选 Raft：大多数工程场景（etcd、TiKV、Consul）。
Paxos 仍有价值：• 无 Leader Paxos（EPaxos）→ 对延迟极敏感；• Google Chubby/Spanner 用 Paxos 变体；• 学术研究。
一句话：新项目无脑 Raft。只有"Leader 是瓶颈"成为具体约束时，才考虑 Paxos 变体。
CRDT 能替代共识协议吗？Q3
不同问题。
共识协议：对"操作顺序"达成一致。代价 = Leader 协调 + 分区牺牲可用性。
CRDT：数据结构保证合并不冲突，不需协调。代价 = 只能用 CRDT 数据结构。
CRDT 适用：协作编辑、最终一致性计数器、多活的非关键数据。
Raft 适用：强一致、操作顺序关键、元数据存储。
可共存：Raft 管元数据，CRDT 管可冲突的用户数据。

🎙️ 口播总结

这篇文章梳理了分布式一致性协议的全家桶。

共识协议解决的核心问题是：在不可靠的网络上，让多个节点对同一个值达成一致。FLP 定理告诉我们纯异步网络不可能完美共识，但 Paxos 和 Raft 通过引入超时绕过了这个限制。Quorum 多数派是共识的数学基础——N 个节点需要超过半数同意才能做决定。

Paxos 是共识协议的鼻祖，两阶段提交：先 Prepare 申请提案号，再 Accept 正式提案。但论文极其难懂，所以有了 Raft。Raft 的设计哲学是可理解性优先——把共识拆成三个独立子问题：Leader 选举用随机超时避免分裂投票、Log 复制靠强 Leader 强制同步、安全性保证日志最新的节点才能当选。etcd、TiKV、CockroachDB 都是 Raft 的工程实现。

Gossip 是另一种思路——不强求一致，追求最终一致。每个节点每轮随机告诉几个邻居，指数级传播，O(log N) 轮覆盖全集群。Redis Cluster 用它做节点发现。

选型口诀：需要强一致用 Raft（etcd），能接受最终一致用 Gossip（Redis Cluster），极致可用性用 CRDT（无冲突数据结构）。
