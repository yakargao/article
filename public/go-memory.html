🧠 Go 内存与 GC

一句话：Go 的内存系统分两半——分配（TCMalloc 三级缓存 + 逃逸分析决定堆/栈）+ 回收（三色标记并发 GC + 混合写屏障）。这两半的精心设计让 Go 在"不用手动管理内存"和"低延迟"之间找到了绝佳平衡点。

模块核心内容关键词
分配器mcache→mcentral→mheap→OS67 size classes, span, tiny allocator
逃逸分析编译期决定堆 vs 栈escape to heap, -gcflags="-m"
GC三色标记+并发清扫+写屏障white/gray/black, Dijkstra+Yuasa hybrid
调优GOGC, GOMEMLIMIT, traceGC pacing, assist, pprof

🏬 TCMalloc 风格的三级分配器

🏬 类比：Go 的分配器像三层仓库——每个工位 P 有自己的小抽屉（mcache，无锁），抽屉空了去楼层仓库（mcentral，有锁）补货，楼层空了去总仓库（mheap）调拨，总仓库空了向 OS 要地（mmap）。

Go 内存分配器全景：mcache(P级无锁)→mcentral(全局有锁)→mheap→OS mmap

mspan 结构

type mspan struct {
    startAddr  uintptr      // span 起始地址
    npages     uintptr      // 占用 page 数（1 page = 8KB）
    spanclass  spanClass     // size class (0-67)
    allocBits  *gcBits       // 位图标记哪些槽位已分配
    allocCount uint16       // 已分配对象数
    // 对象大小 = class_to_size[spanclass]
    // 一个 span 可容纳 E = npages×8KB / objectSize 个对象
}

🔍 逃逸分析：谁上堆，谁留栈

🚪 类比：逃逸分析是编译期的"门卫"——局部变量如果"逃出"了函数（返回指针、被闭包捕获、太大），就分配在堆上（GC 来回收）；如果只在函数里用，就留在栈上（函数返回自动回收，极快）。

决定逃逸的四个条件

场景示例结果
返回局部变量的指针return &x逃逸到堆
Interface 装箱fmt.Println(x)可能逃逸
闭包捕获变量go func() { use(x) }()逃逸到堆
分配较大对象slice cap > 64KB 或变量 > 10MB直接堆分配
# 查看逃逸分析
go build -gcflags="-m -m" main.go

# 输出示例：
# ./main.go:5:2: x escapes to heap       ← x 逃逸了
# ./main.go:8:3: y does not escape       ← y 留在栈上
# ./main.go:10:6: moved to heap: largeBuf ← 大对象直接上堆

💡 Go 逃逸分析的意义

Go 能开百万协程的关键——每个 goroutine 初始栈只 2KB，按需增长。变量不逃逸就留在栈上，goroutine 结束栈自动回收。逃逸分析是 编译期完成的，零运行时开销。

📚 Goroutine 栈：可增长的协程栈

🪜 类比：OS 线程的栈是固定尺寸的"房间"（1-8MB），goroutine 的栈是可伸缩的"帐篷"——初始 2KB，不够了自动扩容，甚至可以缩回去。扩容方式不是简单的追加，而是整块复制到新位置（copy stack）。

栈指针会变（拷贝后地址变）不变无连续栈概念

📜 Go GC 演进史：十年磨一剑

版本里程碑关键变化STW 时间变化
Go 1.0简单 STW标记+清扫全过程 STW100ms-数秒
Go 1.3并发清扫Sweep 阶段不 STW50-100ms
Go 1.5并发标记Mark 阶段不 STW（Dijkstra 写屏障）10-40ms
Go 1.6GC Assist分配者可帮忙做标记工作~10ms
Go 1.8混合写屏障取消 rescan 栈（Yuasa+Dijkstra 混合）~0.5ms
Go 1.12+GC Pacing自适应调节 GC 频率和 CPU 占比<0.5ms

核心趋势：从"全 STW 几百毫秒"到"两个 ~10μs 的 STW + 并发标记"。Go 1.5 是分水岭（引入并发标记），Go 1.8 是质变（混合写屏障消除栈重扫）。

🎨 三色标记-清扫算法

🎨 类比：GC 是美术馆清场——先把所有画标白（可能是垃圾），从入口（根对象）出发，遇到画标灰→遍历其引用的画标灰→自身标黑。最后还白着的画清走。整个过程和游客参观同时进行（并发）。

三色状态机

三色标记状态转换：白→(发现)→灰→(扫描完子对象)→黑

标记过程示例

// 对象引用关系：Root → A → B → C
// 初始：Root(黑) A(白) B(白) C(白)
// Step1: 扫描 Root → A 标灰
// Step2: 扫描 A → B 标灰, A 标黑
// Step3: 扫描 B → C 标灰, B 标黑
// Step4: 扫描 C → 无子对象, C 标黑
// 结果：A B C 黑(活) | 其他全白 → Sweep 回收

⚠️ 写屏障：并发标记的"安全网"

🛡️ 类比：并发 GC 期间，你的代码（mutator）在同时修改对象引用——可能刚标记完的黑色对象，被你改成指向一个白色对象（漏标！）。写屏障就是在这时候插入的"安全检查"——修改指针时自动把目标标灰，保证不会漏掉活对象。

两种写屏障 + Go 的混合方案

屏障原理优点缺点
Dijkstra 插入屏障黑色对象指向白色时→标白为灰不需要 STW 重扫需要栈上的屏障（Go1.5-1.7需STW重扫栈）
Yuasa 快照屏障白色对象被删除引用时→标白为灰（保护快照时所有可达对象）不需要栈屏障可能保留"实际已死"的对象
Go 1.8 混合屏障1.写操作时旧指针对象标灰(Yuasa) + 2.新指针对象标灰(Dijkstra)不需要STW重扫栈，精度高实现复杂（但已完成）
// Go 1.8 混合写屏障伪代码
func writePointer(slot *unsafe.Pointer, ptr unsafe.Pointer) {
    shade(*slot)          // 1. 旧值标灰 (Yuasa 风格)
    if currentStackIsGrey {
        shade(ptr)        // 2. 新值标灰 (Dijkstra 风格)
    }
    *slot = ptr           // 3. 执行实际写入
}
// 关键：不需要在栈上插屏障（栈的写入不触发 barrier）
// 原因：Yuasa 的快照保证了栈上"旧引用"的对象存活

为什么混合屏障是 Go GC 的里程碑？Go 1.5-1.7 用纯 Dijkstra 屏障，每次并发标记完了还要 STW 重扫所有 goroutine 的栈——因为栈上没有写屏障。Go 1.8 的混合方案消除这一步，STW 降到 ~100μs。这使 Go GC 延迟和 Java ZGC/Shenandoah 相当。

💡 Go 为什么不分代？

Go 的设计者认为：1) Go 的逃逸分析已经让很多短期对象留在栈上，不需要分代来快速回收；2) 分代 GC 增加了复杂度（需要 write barrier 追踪跨代引用，Go 1.5 前不愿引入）；3) Go 的目标是 低延迟而非最大吞吐，并发标记足以满足需求。

但实际上，Go 社区对分代 GC 的讨论从未停止。2019 年有一个提案提议引入"请求分代 GC"，但未落地。Go 的不分代策略在大量短命对象场景下会导致 GC 频率偏高——这时调高 GOGC 或使用 sync.Pool 复用对象是常用对策。

Go GC 的目标：延迟优先于吞吐

设计目标数值含义
GC CPU 占比≤ 25%GC 最多占用 1/4 的 CPU
STW 时间< 500μs每次 STW 不超过 0.5ms（实际常~50μs）
堆内存开销GOGC=100 → GC 触发时堆约 2x 存活对象默认内存开销 = 100% 存活内存
