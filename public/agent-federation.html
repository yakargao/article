🔗 Agent 联邦与跨组织协作

A2A 协议 · 跨域通信 · 身份认证 · 数据共享 · Go 实现

📅 2026-05-22 ⏱️ 阅读约 25 分钟 🏷️ A2A Go 联邦 安全

1. 为什么需要 Agent 联邦

想象你开了一家装修公司。你的 装修报价 Agent 需要查询多个建材商的库存和价格，还需要预约物流公司的送货时间。但现实是：建材商的库存系统在你的组织防火墙之外，物流公司的 API 需要不同的认证方式，每个供应商的数据格式都不一样。

🎓 费曼类比

「装修公司的 Agent 不能直接操作建材商的库存系统，就像你拿着邻居家的钥匙进不了自己家的门——除非两把锁之间建立一套 互信互通的联邦协议。」

在单个组织内部，Agent 可以共享数据库、调用内部 API、使用统一的身份认证。但一旦跨越组织边界，问题就来了：信任怎么建立？数据怎么共享？通信用什么协议？

Agent 联邦（Agent Federation） 正是为解决这些问题而生。它是一套让不同组织、不同平台的 Agent 能够安全通信、协同工作的架构模式。

图1：单组织局限 vs Agent 联邦架构对比

Agent 联邦的核心价值在于：在不牺牲安全性的前提下，实现跨组织的 Agent 协作。它要求三个层面的统一：通信协议层（A2A）、身份认证层（联邦身份）、数据治理层（访问控制）。

2. A2A 协议（Agent-to-Agent）

2025 年，Google 发布了 Agent-to-Agent (A2A) 协议规范，旨在解决不同供应商的 Agent 之间如何通信的问题。如果说 MCP (Model Context Protocol) 是 Agent 与工具之间的通用接口，那么 A2A 就是 Agent 与 Agent 之间的通用语言。

A2A 核心概念

📋 A2A 概念模型
  • Agent Card — Agent 的"名片"，描述其能力、支持的技能、通信端点。类似 REST API 的 OpenAPI 文档。
  • Task — 一个 Agent 向另一个 Agent 发起的可追踪的工作单元。每个 Task 有状态（submitted/working/completed/failed）。
  • Message — Task 过程中的消息交换，可以是文本、结构化数据、文件引用。
  • Skill — Agent 对外暴露的能力单元，每个 Skill 包含输入/输出 schema 和调用方式。

Agent Card 示例：

{
  "name": "建材商 - 库存查询 Agent",
  "version": "2.1.0",
  "agentCard": "https://supplier.example.com/.well-known/agent-card.json",
  "skills": [
    {
      "id": "inventory.query",
      "name": "库存查询",
      "inputSchema": {
        "type": "object",
        "properties": {
          "productIds": { "type": "array", "items": { "type": "string" }},
          "warehouseId": { "type": "string" }
        }
      },
      "outputSchema": { "type": "object" },
      "authentication": {
        "type": "oauth2",
        "flow": "device_code"
      }
    }
  ]
}

A2A 与 MCP 的关系

很多人容易混淆 A2A 和 MCP。简单理解：

  • MCP (Model Context Protocol) — Agent 内部的工具调用协议。Agent 用它来调用搜索引擎、数据库、计算器等工具。是 纵向 的。
  • A2A (Agent-to-Agent) — Agent 之间的通信协议。你的 Agent 和我的 Agent 怎么对话。是 横向 的。
图2：MCP vs A2A 协议栈对比 — 纵 vs 横

🔧 MCP

  • Agent ↔ Tool
  • 同一组织/进程
  • 同步调用
  • 工具发现 + 调用
  • 类比：RPC / gRPC

🤝 A2A

  • Agent ↔ Agent
  • 跨组织/跨平台
  • 支持异步 Task
  • 技能发现 + 协商
  • 类比：REST API + 事件

3. 跨域通信模式

跨组织 Agent 通信主要有三种模式，各有优劣。选择哪种取决于你的场景对延迟、可靠性和耦合度的要求。

图3：三种跨域通信模式对比

选择建议：

  • 同组织内部 Agent 通信 → 直接调用（gRPC 最佳）
  • 跨组织但松耦合场景 → 消息队列（Kafka / RabbitMQ / Pulsar）
  • 跨组织且需要统一安全策略 → API 网关（Kong / Envoy / APISIX）

4. 身份认证与权限委派

跨组织 Agent 通信最大的挑战是 信任。Agent A 怎么证明自己是它声称的 Agent？Agent B 凭什么相信 Agent A 发来的请求是经过用户授权的？

四种联邦身份方案

🔐 身份认证方案对比
  • OAuth2 Device Flow — 适合无浏览器的 Agent 场景。Agent 获取 device_code，用户通过浏览器确认授权。这是装修 Agent 获取用户授权的典型方式。
  • JWT 委派（Token Exchange） — 用户授权装修 Agent 后，Agent 用 JWT 令牌代表用户访问其他系统。可以链式传递委派令牌。
  • SPIFFE（SPIRE） — 工作负载身份标准。每个 Agent 获得一个 SPIFFE ID（如 spiffe://supplier.example.com/agent/inventory），通过双向 TLS 验证身份。
  • OAuth2 Client Credentials — 组织级 Agent 使用自己的身份凭证直接调用。适合服务端 Agent。

场景：用户授权装修 Agent → 搜索建材商库存

图4：跨域身份认证流程 — OAuth2 Device Flow + JWT + SPIFFE

Go 实现：Agent 身份认证中间件

package agentauth

import (
    "context"
    "crypto/ecdsa"
    "crypto/elliptic"
    "crypto/rand"
    "crypto/x509"
    "encoding/pem"
    "errors"
    "fmt"
    "net/http"
    "strings"
    "time"

    "github.com/golang-jwt/jwt/v5"
    "github.com/spiffe/go-spiffe/v2/spiffetls/tlsconfig"
)

// AgentContext 持有已认证的 Agent 身份信息
type AgentContext struct {
    AgentID   string
    OrgID     string
    Permissions []string
    DelegatedFrom string // 委派来源 AgentID
    ExpiresAt time.Time
}

// FederationMiddleware 联邦身份认证中间件
func FederationMiddleware(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        // 1. 提取 JWT 令牌
        tokenStr := extractBearerToken(r)
        if tokenStr == "" {
            http.Error(w, "missing authorization token", http.StatusUnauthorized)
            return
        }

        // 2. 获取联邦公钥（从组织联邦注册中心）
        pubKey, err := fetchFederationPublicKey(r.Context(), tokenStr)
        if err != nil {
            http.Error(w, fmt.Sprintf("cannot fetch federation key: %v", err),
                http.StatusUnauthorized)
            return
        }

        // 3. 验证 JWT 签名和 Claims
        claims := &AgentClaims{}
        token, err := jwt.ParseWithClaims(tokenStr, claims,
            func(token *jwt.Token) (interface{}, error) {
                return pubKey, nil
            })
        if err != nil || !token.Valid {
            http.Error(w, "invalid token signature", http.StatusUnauthorized)
            return
        }

        // 4. 检查 SPIFFE ID（双向 TLS 环境）
        if r.TLS != nil {
            spiffeID, err := tlsconfig.AuthorizeMember()
            if err != nil {
                http.Error(w, "SPIFFE verification failed", http.StatusForbidden)
                return
            }
            // 验证 SPIFFE ID 与 JWT 的 AgentID 一致
            if !strings.HasSuffix(spiffeID.String(), claims.AgentID) {
                http.Error(w, "agent id mismatch", http.StatusForbidden)
                return
            }
        }

        // 5. 注入认证上下文
        ctx := context.WithValue(r.Context(), CtxKeyAgent, &AgentContext{
            AgentID:       claims.AgentID,
            OrgID:         claims.OrgID,
            Permissions:   claims.Permissions,
            DelegatedFrom: claims.DelegatedFrom,
            ExpiresAt:     claims.ExpiresAt.Time,
        })
        next.ServeHTTP(w, r.WithContext(ctx))
    })
}

// AgentClaims JWT 标准 Claims + 扩展
type AgentClaims struct {
    AgentID       string   `json:"agent_id"`
    OrgID         string   `json:"org_id"`
    Permissions   []string `json:"permissions"`
    DelegatedFrom string   `json:"delegated_from"`
    jwt.RegisteredClaims
}

// extractBearerToken 从 Authorization Header 提取 Bearer Token
func extractBearerToken(r *http.Request) string {
    auth := r.Header.Get("Authorization")
    if !strings.HasPrefix(auth, "Bearer ") {
        return ""
    }
    return strings.TrimPrefix(auth, "Bearer ")
}

// fetchFederationPublicKey 从联邦注册中心获取公钥
func fetchFederationPublicKey(ctx context.Context, tokenStr string) (
    interface{}, error,
) {
    // 从 JWT header 提取 kid，查询联邦 Key Store
    // 实际实现中应缓存公钥并支持定期轮换
    privKey, err := ecdsa.GenerateKey(elliptic.P256(), rand.Reader)
    if err != nil {
        return nil, err
    }
    return &privKey.PublicKey, nil
}

// CtxKeyAgent 上下文键
type ctxKey string
const CtxKeyAgent ctxKey = "agent"

// FromContext 从上下文获取 Agent 身份
func FromContext(ctx context.Context) (*AgentContext, bool) {
    agent, ok := ctx.Value(CtxKeyAgent).(*AgentContext)
    return agent, ok
}

这个中间件集成了 JWT 验证、SPIFFE 双向 TLS 和 Token Exchange 委派。当装修 Agent 请求建材 Agent 时，它携带的 JWT 中包含用户的授权范围和委派链条，建材 Agent 可以验证整个信任链。

5. 数据隔离与共享

跨组织协作中，数据是最敏感也最有价值的部分。哪些数据可以共享？共享到什么粒度？如何在不泄露隐私的前提下提取价值？

🎓 费曼类比

「Agent 联邦中的数据共享，就像两家医院共享患者的病历——你不需要知道患者的名字（标识符），只需要知道『这类症状用哪种药最有效』（统计特征）。」

三种数据共享策略

图5：数据共享边界与隐私保护技术

数据共享的原则：

  • 最小必要 — 只共享完成任务所需的最小数据集
  • 差分隐私 — 在查询结果中添加校准噪声，使攻击者无法判断单条记录是否存在
  • Data Clean Room — 双方数据进入安全沙箱，只能执行预设的联合查询，不能导出原始数据
  • 联邦学习 — 数据不离开本地，只交换模型参数更新

6. 协商与合约

Agent 之间的协作不仅是技术性的——它们还需要 谈条件。装修 Agent 需要向多个建材商 Agent 询价、比价、谈判价格、最终签订合约。这一切都可以自动完成。

图6：多 Agent 自动协商流程

Go 实现：谈判 Agent

package negotiation

import (
    "context"
    "fmt"
    "math"
    "slices"
    "sync"
    "time"
)

// PriceQuote 来自某个建材商 Agent 的报价
type PriceQuote struct {
    SupplierID        string
    ProductID         string
    UnitPrice         float64
    BulkDiscountPct   float64 // 批量折扣百分比
    DeliveryDays      int
    AvailableQuantity int
    ValidUntil        time.Time
}

// NegotiationOffer 谈判中的一轮出价
type NegotiationOffer struct {
    Round       int
    FromAgentID string
    ToAgentID   string
    ProductID   string
    Quantity    int
    TargetPrice float64
    Message     string
}

// NegotiationResult 谈判结果
type NegotiationResult struct {
    Accepted      bool
    FinalPrice    float64
    TotalAmount   float64
    Terms         string
    ContractRef   string
}

// NegotiationAgent 谈判 Agent 核心
type NegotiationAgent struct {
    AgentID      string
    MaxRounds    int
    BudgetLimit  float64
    minMargin    float64
    mu           sync.Mutex
}

// NewNegotiationAgent 创建谈判 Agent
func NewNegotiationAgent(agentID string, budget float64) *NegotiationAgent {
    return &NegotiationAgent{
        AgentID:     agentID,
        MaxRounds:   5,
        BudgetLimit: budget,
        minMargin:   0.08, // 最少保留 8% 利润
    }
}

// StartNegotiation 对指定商品启动谈判流程
func (na *NegotiationAgent) StartNegotiation(
    ctx context.Context,
    productID string,
    quantity int,
    quotes []PriceQuote,
) (*NegotiationResult, error) {
    // 按单价排序
    slices.SortFunc(quotes, func(a, b PriceQuote) int {
        switch {
        case a.UnitPrice < b.UnitPrice:
            return -1
        case a.UnitPrice > b.UnitPrice:
            return 1
        default:
            return 0
        }
    })

    // 最佳初始报价
    bestQuote := quotes[0]
    targetPrice := bestQuote.UnitPrice * (1 - na.minMargin)

    // 多轮谈判循环
    for round := 1; round <= na.MaxRounds; round++ {
        select {
        case <-ctx.Done():
            return nil, ctx.Err()
        default:
        }

        offer := NegotiationOffer{
            Round:       round,
            FromAgentID: na.AgentID,
            ToAgentID:   bestQuote.SupplierID,
            ProductID:   productID,
            Quantity:    quantity,
            TargetPrice: targetPrice,
            Message:     fmt.Sprintf("Round %d: offer %.2f/unit for %d units",
                round, targetPrice, quantity),
        }

        // 发送报价给供应商 Agent（模拟）
        response, err := na.sendOfferToSupplier(ctx, offer)
        if err != nil {
            continue
        }

        if response.Accepted {
            total := response.FinalPrice * float64(quantity)
            if total <= na.BudgetLimit {
                return &NegotiationResult{
                    Accepted:    true,
                    FinalPrice:  response.FinalPrice,
                    TotalAmount: total,
                    Terms:       fmt.Sprintf("Negotiated %d units at %.2f/unit",
                        quantity, response.FinalPrice),
                    ContractRef: fmt.Sprintf("contract/%s/%s", na.AgentID, productID),
                }, nil
            }
        }

        // 调整策略：逐步降低目标价
        targetPrice *= (1 - float64(round)*0.02)
        targetPrice = math.Max(targetPrice, bestQuote.UnitPrice*0.85)
    }

    return &NegotiationResult{Accepted: false}, nil
}

// sendOfferToSupplier 向供应商发送报价请求（模拟实现）
func (na *NegotiationAgent) sendOfferToSupplier(
    ctx context.Context,
    offer NegotiationOffer,
) (*NegotiationResult, error) {
    // 实际实现中会通过 A2A 协议调用供应商 Agent
    // 这里模拟供应商响应逻辑
    price := offer.TargetPrice

    // 第一轮坚持原价，第二轮开始松动
    if offer.Round == 1 {
        price *= 1.05 // 加价 5% 作为谈判起始
    } else if offer.Round >= 3 {
        price *= 0.97 // 第三轮后让步 3%
    }

    // 大额订单额外优惠
    if offer.Quantity > 100 {
        price *= 0.95
    }

    accepted := price <= offer.TargetPrice*1.02
    return &NegotiationResult{
        Accepted:   accepted,
        FinalPrice: price,
    }, nil
}

// EvaluateQuotes 综合评估多个报价（含非价格因素）
func (na *NegotiationAgent) EvaluateQuotes(quotes []PriceQuote) []PriceQuote {
    type scored struct {
        quote PriceQuote
        score float64
    }

    var scoredList []scored
    for _, q := range quotes {
        // 综合评分：价格权重 0.5，交期权重 0.3，库存权重 0.2
        priceScore := 1 - (q.UnitPrice / 1000)
        deliveryScore := 1 - (float64(q.DeliveryDays) / 30)
        stockScore := math.Min(float64(q.AvailableQuantity)/float64(500), 1.0)

        total := priceScore*0.5 + deliveryScore*0.3 + stockScore*0.2
        scoredList = append(scoredList, scored{quote: q, score: total})
    }

    slices.SortFunc(scoredList, func(a, b scored) int {
        switch {
        case a.score > b.score:
            return -1
        case a.score < b.score:
            return 1
        default:
            return 0
        }
    })

    result := make([]PriceQuote, len(scoredList))
    for i, s := range scoredList {
        result[i] = s.quote
    }
    return result
}

这个谈判 Agent 实现了 询价 → 比价 → 多轮谈判 → 签约 的完整流程。它使用综合评分算法评估报价（价格 50% + 交期 30% + 库存 20%），并通过逐步让步策略与供应商 Agent 自动谈判。

7. 实战：装修 Agent 联邦

让我们把前面所有的概念串联起来，构建一个完整的 装修 Agent 联邦 示例：用户想要装修厨房，三个 Agent 协同工作。

图7：装修 Agent 联邦完整架构

协作流程

  1. 用户下单 — 用户在装修平台提交需求："厨房翻新，需要 30 平米地板 + 橱柜 + 水槽"
  2. 装修 Agent 分解需求 — 把需求拆解为多个子任务，分别寻对应的建材 Agent
  3. 跨域查询 — 通过联邦身份认证，向 3 家建材 Agent 发送询价请求
  4. 自动谈判 — 对每家报价进行比价和多轮谈判，目标是总预算控制在 5 万以内
  5. 物流协调 — 确定供应商后，与物流 Agent 协商配送时间，匹配装修工期
  6. 签约支付 — 生成统一报价单，用户确认后，智能合约自动执行
💡 关键设计
  • 装修 Agent 通过 Agent Card 发现建材商的 Skill
  • 认证使用 OAuth2 Device Flow：用户扫码授权 → 装修 Agent 获取委派令牌
  • 通信通过 A2A Task：每个询价请求都是一个可追踪的异步 Task
  • 数据共享使用 Data Clean Room：只能获取库存量，不能获取客户信息
  • 谈判结果通过 智能合约 固化，避免纠纷

8. 常见问题 QA + 总结

常见问题

❓ A2A 和 MCP 到底有什么区别？我该用哪个？

两者不是二选一的关系，而是互补的。

用一句话区分：MCP 是 Agent 的"手脚"（控制工具），A2A 是 Agent 的"嘴巴"（与其他 Agent 对话）。

同一个 Agent 内部可以同时使用 MCP（调用搜索引擎、数据库）和 A2A（与其他组织的 Agent 通信）。MCP 负责"做事"，A2A 负责"协作"。

在装修场景中：装修 Agent 通过 MCP 调用内部数据库查询用户历史订单，通过 A2A 向建材商 Agent 发送询价。

❓ Agent 之间怎么建立信任？如果对方是恶意的怎么办？

信任建立是多层的：

  1. 身份层 — JWT 签名验证 + SPIFFE 双向 TLS，确保通信双方确实是声称的身份
  2. 授权层 — OAuth2 权限委派，限定 Agent 只能执行授权范围内的操作
  3. 数据层 — Data Clean Room / 差分隐私，即使对方恶意也无法获取敏感原始数据
  4. 审计层 — 所有跨组织通信都记录到不可篡改的审计日志

还需要建立 信誉系统：联邦中的 Agent 行为会被评分，恶意 Agent 会被吊销联邦身份。

❓ 谈判 Agent 怎么确保不被对方"套路"？有最优策略吗？

这是一个经典的 博弈论 问题。我们在实现中采用了几个策略：

  • 信息收集优先 — 先询多家获取基线价格，避免信息不对称
  • 渐进让步 — 每轮让步比例递减，让对方觉得接近底线
  • 非价格因素 — 综合评分包含交期、库存可信度等维度，不只是比价
  • 底线约束 — 硬性预算上限，超过即退出谈判

如果你感兴趣，可以进一步采用 强化学习 训练谈判策略，让 Agent 在模拟环境中学习最优出价策略。

❓ 联邦身份认证中的 JWT 委派链过长怎么办？

确实，在链式委派场景中（用户 → 装修 Agent → 建材 Agent → 物流 Agent）JWT 可能会变得很大。有几种解决方案：

  • Token 链压缩 — 使用 JWT 压缩算法（如 ZSTD）压缩 Claim 集合
  • 链式引用 — 不嵌入完整委派链，而是嵌入一个可追溯的引用 ID，接收方通过联邦追溯 API 查询
  • SPIFFE 辅助 — 用 SPIFFE ID 标识 Agent 身份，JWT 只携带委派意图，身份验证靠 TLS
  • 最短路径原则 — 设计联邦拓扑时控制委派深度 ≤ 3 层
❓ 联邦中的数据隐私怎么保障？数据泄露谁负责？

这是联邦架构中最敏感的问题。我们建议：

  1. 合同层面 — 联邦成员签署 DPA（数据处理协议），明确数据使用边界和赔偿条款
  2. 技术层面 — Data Clean Room 确保只能输出聚合结果，不能导出原始行级数据
  3. 审计层面 — 所有数据访问操作都有加密审计日志，定期第三方审计
  4. 最小化原则 — 设计 API 时只暴露必要字段，例如库存查询只返回 "available: true/false"

一旦发生泄露，审计日志可以精确定位到出问题的 Agent 和操作时间，便于追责。联邦协议中应包含 惩罚机制（罚款、吊销身份、公示）。

❓ 装修 Agent 联邦的代码大概需要多少工作量？

以一个 MVP（最小可行产品）为目标：

  • Agent Card 注册/发现服务 — 约 500 行 Go（含 API + 存储）
  • A2A Task 管理引擎 — 约 800 行 Go（含状态机 + WebSocket）
  • 联邦身份中间件 — 约 300 行 Go（JWT + SPIFFE 集成）
  • 谈判 Agent 核心 — 约 400 行 Go（含策略引擎）
  • 装修 Agent 业务逻辑 — 约 600 行 Go（需求分解 + 协调）
  • 前端展示层 — 另计（React/Vue）

总计约 2600 行 核心代码可实现可演示的联邦原型，约 2-3 人周的工作量。
