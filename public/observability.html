📋 可观测性（Observability）实战

一句话：可观测性 = Metrics（指标）+ Logging（日志）+ Tracing（链路追踪），让你在微服务架构中知道系统为什么挂了。

本质区别：传统监控是"我知道这个指标异常了"，可观测性是"我知道为什么异常，影响谁，从哪里开始"。

📊 为什么可观测性？—— 汽车仪表盘

🚗 类比：汽车仪表盘

开车时，你需要知道：

  • 速度多少？ → Metrics（指标）—— 当前 CPU 使用率、QPS
  • 发动机有没有异响？ → Logging（日志）—— 错误堆栈、业务日志
  • 油路一路经过哪里？ → Tracing（链路追踪）—— 请求从 A → B → C 的完整路径

没有可观测性 = 蒙眼睛开车。系统出问题了你只能靠猜。

传统监控 vs 可观测性

维度传统监控可观测性
关注点我知道哪个指标异常我知道为什么异常
数据预定义指标（CPU、内存）指标 + 日志 + 链路全量数据
架构单体，已知问题模式微服务，未知问题排查
问题定位"CPU 高了""用户下单慢 → 订单服务某接口 p99 延迟 3s → 数据库连接池耗尽"

可观测性的核心价值

  • MTTR 降低：Mean Time To Repair（平均修复时间）从小时级降至分钟级
  • 未知问题发现：不需要预先定义所有报警规则（你不知道你不知道什么）
  • 跨团队协作：统一的数据口径，开发/运维/SRE 用同一种语言沟通

📌 核心追问

问：监控（Monitoring）和可观测性（Observability）是一回事吗？

答：不是。监控是已知问题的告警和可视化的集合（你提前设好了 CPU > 90% 报警）。可观测性是通过系统对外暴露的三种数据（指标、日志、链路），让你能回答任何没预想过的问题。监控是基于假设的，可观测性是基于探索的。

🏛️ 三大支柱（Metrics / Logging / Tracing）

📈

Metrics

指标 —— 数值化、可聚合、可统计

"每秒请求数是多少？"

📝

Logging

日志 —— 离散的、带时间戳的事件记录

"这个请求的报错信息是什么？"

🔗

Tracing

链路追踪 —— 跨服务的请求全貌

"这个慢请求经过哪些服务？"

三者对比

维度MetricsLoggingTracing
数据形态时序数值结构化/非结构化文本Span 树（Trace ID → Span ID）
存储时序数据库 TSDB全文搜索引擎专用存储（Cassandra/ES）
代表工具Prometheus, GraphiteELK, Loki, SplunkJaeger, Zipkin, SkyWalking
查询方式PromQL, Graphite 函数全文搜索（KQL, Lucene）TraceID 搜索 + 服务依赖图
典型用途告警、仪表盘、容量规划排障、审计、合规延迟分析、依赖拓扑、瓶颈定位
数据量级小（按时间聚合）大（每条请求一条/多条）中（按采样比例）

💡 为什么是三根支柱？

任何一个都不够：只有指标，你知道 CPU 飙高但不知道哪个接口引起的；只有日志，你看到报错报但不知道影响范围和路径；只有链路，你知道请求经过哪些服务但没有性能统计。三根支柱互相补充 = 系统的全息视图。

📈 Metrics 指标 — Prometheus + Grafana

什么是 Prometheus？

Prometheus 是开源的系统监控和告警工具，2016 年 CNCF 毕业项目（第二个毕业项目，仅次于 Kubernetes）。核心特性：

  • Pull 模型：Prometheus Server 定期从目标服务"拉取"指标（HTTP /metrics 端点）
  • 时间序列数据库：所有数据带时间戳 + 标签（Label）
  • PromQL：强大的查询语言

四种核心指标类型

类型说明例子
Counter只增不减的计数器http_requests_total{method="GET",status="200"}
Gauge可增可减的瞬时值memory_usage_bytes, goroutines

PromQL 速查

# 每秒请求速率（rate）
rate(http_requests_total[5m])

# P99 延迟
histogram_quantile(0.99, rate(http_request_duration_seconds_bucket[5m]))

# 按路径分组
topk(5, sum by(path) (rate(http_requests_total{status=~"5.."}[5m])))

# 机器 CPU 使用率
100 - (avg by(instance) (rate(node_cpu_seconds_total{mode="idle"}[5m])) * 100)

Grafana 可视化

Grafana 是数据可视化平台——不仅显示 Prometheus 数据，还支持 Loki（日志）、Jaeger（链路）、MySQL、Elasticsearch、CloudWatch 等 40+ 数据源。

Grafana 核心概念

  • Dashboard：一个仪表盘包含多个 Panel（图表）
  • Panel：单个图表（时间序列图、表格、统计数字、热力图）
  • Variables：变量（如 $env, $service）实现动态切换
  • Explore：即席查询模式——临时查指标/日志/链路
  • Annotations：在图表上标注事件（如发版、报警）

⚙️ 实战建议

  • RED 方法论：Rate（每秒请求）+ Errors（错误数）+ Duration（延迟）—— 每个服务必备的三个黄金指标
  • USE 方法论：Utilization（利用率）+ Saturation（饱和度）+ Errors（错误数）—— 基础设施监控
  • 告警不要太多：每个团队维护 5-10 个核心告警规则，多了就是告警疲劳

📝 Logging 日志 — ELK / Loki

为什么日志很重要？

指标告诉你"系统慢了"，链路告诉你"哪个环节慢了"，只有日志告诉你为什么会慢——比如数据库慢查询的 SQL 是什么、请求参数是什么、调用栈在哪里。

结构化日志最佳实践

// ❌ 非结构化
log.Info("用户 123 下单成功，金额 99.9")

// ✅ 结构化（JSON）
{
  "@timestamp": "2024-06-01T10:30:00Z",
  "level": "info",
  "trace_id": "abc123def456",
  "service": "order-service",
  "user_id": 123,
  "action": "order_create",
  "amount": 99.9,
  "duration_ms": 45,
  "message": "订单创建成功"
}

📌 日志四原则

  1. 结构化：JSON 格式，不要纯文本拼接（方便搜索）
  2. 带 TraceID：每条日志关联 TraceID，跨服务关联
  3. 级别准确：ERROR = 需要人工关注，WARN = 可容忍异常，INFO = 业务关键事件，DEBUG = 开发调试
  4. 不打印敏感信息：密码、身份证、信用卡号脱敏

LogQL（Loki 查询语言）

# 查询所有错误日志
{app="order-service"} |= "error"

# 多条件过滤
{app="order-service", env="prod"} |= "timeout" != "context deadline exceeded"

# 解析 JSON 字段再筛选
{app="order-service"} | json | user_id = "123"

# 计算错误率
sum by(service) (rate({app="order-service"} |= "error" [5m]))

🔗 Tracing 链路追踪 — Jaeger

为什么需要链路追踪？

在微服务架构中，一个用户请求可能经过 5-10 个服务：
API Gateway → Auth Service → Order Service → Payment Service → Notification Service
当请求变慢时——哪个服务是瓶颈？传统的单机日志根本看不出来。链路追踪让每个请求有一个 TraceID，串联所有经过的服务。

核心概念

概念说明类比
Trace一次请求的完整路径，由一串 Span 组成一次旅行的完整行程
SpanTrace 中的一个步骤（被一个服务处理的一段逻辑）一个景点
Span Context携带 TraceID、SpanID、Baggage 等元数据的上下文行李牌
Parent-ChildSpan 之间的父子关系大景点包含小景点

代码示例：Go 中接入 OpenTelemetry Tracing

import (
    "go.opentelemetry.io/otel"
    "go.opentelemetry.io/otel/trace"
)

// 创建 Tracer
tracer := otel.Tracer("order-service")

// 创建 Span
ctx, span := tracer.Start(ctx, "processOrder")
defer span.End()

// Span 中设置属性
span.SetAttributes(
    attribute.String("order_id", orderID),
    attribute.Int64("amount", amount),
)

// 记录事件
span.AddEvent("payment_completed", trace.WithAttributes(
    attribute.String("payment_id", paymentID),
))

// 从 context 中取 TraceID（日志用）
traceID := trace.SpanFromContext(ctx).SpanContext().TraceID().String()

💡 采样策略

全量链路追踪可能产生巨大数据量（每秒上万条 Span）。常见策略：

  • 概率采样：随机采样 1%-10%（适合低流量服务）
  • 自适应采样：低流量全量，高流量降采样
  • 头部采样：根据请求头部决定是否采样（慢请求才采样）
  • 尾采样：事后根据规则决定是否保留（推荐——确保错误和慢请求被保留）

黄金法则：错误链路必须全量保留，正常业务按需采样。

🔌 OpenTelemetry — 统一的可观测性标准

🎯 一句话：OpenTelemetry（简称 OTel）是CNCF 毕业项目，提供统一的 API、SDK、Collector，解决三大支柱的工具碎片化问题。让你一套代码同时输出 Metrics + Logging + Tracing。

为什么需要 OTel？

没有 OTel 之前的选择题：

  • Tracing 选 Jaeger SDK 还是 Zipkin SDK 还是 SkyWalking SDK？
  • Metrics 用 Prometheus 的 SDK 还是 StatsD？
  • 三根支柱三套 SDK，埋点代码三份，换后端全部重写。

OTel 的承诺：用同一套 API埋点，通过配置选择输出到哪个后端。

OTel Collector 配置示例

receivers:
  otlp:
    protocols:
      grpc:
        endpoint: 0.0.0.0:4317
      http:
        endpoint: 0.0.0.0:4318

processors:
  batch:
    timeout: 1s
  memory_limiter:
    check_interval: 1s
    limit_mib: 512

exporters:
  prometheus:
    endpoint: 0.0.0.0:8889
  otlp:
    endpoint: jaeger-collector:4317
    tls:
      insecure: true

service:
  pipelines:
    traces:   [otlp, batch, otlp]
    metrics:  [otlp, batch, prometheus]

💡 实战建议：从改造一个服务开始

全量迁移 OTel 很重。推荐策略：

  1. 先在一个核心服务接入 OTel SDK（替换旧的 Metrics/Tracing 库）
  2. 部署 OTel Collector 作为代理
  3. 验证数据正确性（对比旧系统的指标）
  4. 逐步推广到所有服务

OTel 是行业趋势——Docker、Kubernetes、AWS、Google Cloud 都在用。

🎯 SLO / SLI / Error Budget

🏢 类比：公司指标

SLI = 你实际测得的数据（利润率 12%）
SLO = 老板定的目标（利润率 ≥ 15%）
Error Budget = 允许失误的空间（你有 -3% 的缓冲）

三个概念

概念全称定义例子
SLIService Level Indicator服务质量的实际测量值99.9% 的请求在 500ms 内完成
SLOService Level Objective服务质量的目标值（团队承诺的边界）目标：99.95% 请求 ≤ 500ms
Error Budget错误预算SLO 允许的容错空间 = 1 − SLO一个月允许 0.05% 的慢请求（≈ 21.6 分钟）

如何定义 SLI？

好的 SLI 要满足：用户能感知到的指标。常见 SLI：

  • 可用性：成功请求 / 总请求（排除预期内的错误）
  • 延迟：X% 的请求在 Y 毫秒内完成（如 P99 ≤ 500ms）
  • 吞吐量：每秒成功处理的请求数
  • 正确性：数据一致、无脏数据写入

Error Budget 怎么用？

黄金法则：Error Budget 花完了就停止发版。

  • 本月 Error Budget 用完（已慢于 SLO 允许的时间）→ 冻结所有非紧急上线
  • Error Budget 剩余很多 → 允许快速迭代、更多发版
  • 本质是用数据驱动的决策代替"拍脑袋"：系统稳定与否由数据说了算

SLO 必须是一个数字吗？

❌ 坏的 SLO

  • "尽可能不宕机"（不可度量）
  • "我们希望 QPS 高点"（没有边界值）
  • "SLA 是 99.99%"（SLA ≠ SLO，SLA 给客户的，SLO 是自己定的）

✅ 好的 SLO

  • "99.9% 的请求在 30 天内返回 200"
  • "95% 的请求 P95 延迟 ≤ 200ms"
  • "99% 的订单在 60 秒内完成支付流程"

💡 实战建议：从"三个九"开始

新手团队不要一上来定 99.99%（四个九）。99.9%（三个九）是大多数内部业务系统的最佳起点——全年允许 8.7 小时故障，够用了。逐步提高到 99.95%、99.99%。

❓ 实战问答

Q1: Metrics、Logging、Tracing 有什么区别？各自解决什么问题？
Metrics：数值化的聚合数据——回答"系统正在发生什么"（大盘、趋势、告警）。
Logging：离散的事件记录——回答"为什么发生了"（排查根因）。
Tracing：跨服务的请求全貌——回答"哪个环节出了问题"（依赖分析、瓶颈定位）。
关键：三者不是替代关系，是互补关系。最佳实践是做"指标驱动、日志深挖、链路定位"。
Q2: Prometheus 的 Pull 模型和 Push 模型各有什么优缺点？
Pull（Prometheus 标准）：Server 主动抓取目标指标。
优点：容易发现目标挂没挂（抓不到就是挂了）；可控制抓取频率；天然避免指标风暴。
缺点：需要目标在网络可达范围内；服务发现配置复杂。
Push（Prometheus Pushgateway / StatsD）：目标主动推送。
优点：适合短生命周期任务（批处理、CronJob）；防火墙友好。
缺点：无法检测目标是否存活；容易推爆 Server。
实战加分：实际上两者可以混合使用——长驻进程 Pull，短任务 Push 到 Pushgateway，Prometheus 再去拉 Pushgateway。
Q3: 如果你的微服务响应变慢，你如何通过可观测性定位问题？
第一步：看大盘（Metrics）——Grafana 看 QPS、延迟、错误率变化。确认是全局问题还是某个接口/服务。
第二步：看链路（Tracing）——找到慢请求的 Trace，看哪个 Span 耗时最长（哪个服务/方法在拖后腿）。
第三步：看日志（Logging）——对应 TraceID 搜日志，看报错信息或超时细节。
第四步：关联基础设施——CPU、内存、磁盘、网络指标是否异常。
实战亮点："先用 RED 指标缩小范围，再用 Tracing 定位服务，最后用 Logging 深挖根因"。
Q4: OpenTelemetry 如何解决可观测性碎片化问题？
统一 API：一套 SDK 同时产生 Metrics、Logs、Traces，不再需要对接三个不同 SDK。
统一传输格式（OTLP）：所有数据都走 gRPC/HTTP 的 OTLP 协议，后端随意切换。
统一 Collector：OTel Collector 作为代理/网关，接收 OTLP 后可以路由到任意后端（Prometheus、Jaeger、Loki、Datadog 等）。
厂商无关：今天用 Jaeger，明天换 Datadog——你的代码不需要改。
注意：OTel 是标准/框架，不是可观测性后端。你还需要 Prometheus/Jaeger/Loki 来存储和展示。
Q5: 什么是 SLO？百万美金级别的 Error Budget 怎么算？
SLO：团队承诺的服务质量目标，如"99.9% 的请求在 500ms 内返回"。
Error Budget：1 − SLO = 允许失败的时间。99.9% → 每月约 43 分钟、99.99% → 每月约 4.3 分钟。
百万美元级别的计算：假设一个电商平台月收入 1000 万美元，99.9% 可用意味着最多 43 分钟的不可用/慢体验。如果这 43 分钟导致收入损失 ≈ 1000万 × (43 / 43200) ≈ 9950 美元。
但是：如果 SLO 越紧（99.99%），Error Budget 越少（4.3 分钟），意味着团队需要投入更多成本去维持稳定性——这就是稳定性投资回报率的权衡。
Q6: 线上突然发现某接口 P99 延迟从 200ms 飙升到 5s，怎么排查？
排障步骤：
① 确认影响面：所有接口还是单个接口？某台机器还是所有机器？
② 查 Metrics：看该接口的 QPS 是否波动（流量突增？）→ 如果是，扩容；查看上游依赖延迟（数据库、缓存、下游服务）
③ 查 Tracing：找一个慢的 Trace，看哪个 Span 耗时最长（数据库慢查询？Redis 打满？HTTP 调用超时？）
④ 查 Logging：对应 TraceID 搜日志，看是否有错误堆栈或超时信息
⑤ 根据根因处理：DB 慢查询 → 加索引/改写 SQL；依赖超时 → 调超时参数/降级；GC 问题 → 调 JVM/Go GC 参数
实战加分："我还会看这个延迟飙升是否伴随了 Error Budget 的快速消耗，判断是否需要回滚或熔断"。
Q7: 你们公司怎么落地可观测性的？从零怎么开始？
从零开始的四步走：
第一阶段：基础设施可观测（第 1 周）
部署 Prometheus + Node Exporter + Grafana，监控所有机器的 CPU、内存、磁盘、网络。
第二阶段：应用 Metrics（第 2-3 周）
在每个服务中接入 Prometheus SDK，暴露 /metrics 端点，添加 RED 指标（Rate/Errors/Duration）。
第三阶段：集中日志（第 3-4 周）
部署 Loki + Promtail，所有服务统一输出结构化 JSON 日志，接入 Grafana Explore。
第四阶段：链路追踪 + OTel（第 5-6 周）
接入 OpenTelemetry SDK、部署 Jaeger、部署 OTel Collector。
推荐：用 Docker Compose 一键部署全套架构，在开发环境先跑通。

🎙️ 口播总结

这篇文章帮你建立可观测性知识的完整地图。

可观测性 = Metrics + Logging + Tracing 三大支柱。监控是知道什么指标异常，可观测性是知道为什么异常。

Prometheus 用 Pull 模型拉取指标，四种核心类型：Counter、Gauge、Histogram、Summary。Grafana 统一可视化，支持 40+ 数据源。ELK 是传统日志方案但成本高，Loki 轻量只索引标签，内容压缩存储。

链路追踪 Jaeger 通过 TraceID 串联跨服务的请求路径，Span 树完美展示调用关系。OpenTelemetry 是行业统一标准，一套 API 同时输出三大支柱，通过 OTel Collector 路由到任意后端。

SLO 是服务目标，SLI 是实际测量，Error Budget 是容错空间——用数据决定能不能发版。99.9% 是新团队的合理起点。

如果只能记住一件事：先用 Prometheus + Grafana 监控 RED 指标，再逐步加上日志和链路。小步快跑，不要试图一步到位。

💬 苏格拉底式提问 —— 检验真理解

以下问题用来逼你追问本质。先自己想 30 秒，再点开验证。

Prometheus 为什么选择 Pull 而不是 Push？什么场景 Push 更适合？Q1
核心原因：Pull 模型能检测目标存活状态。如果 Prometheus 拉不到数据，说明服务挂了（或网络不通）。Push 模型里，服务挂了就没人推数据——你根本不知道它是正常没数据还是挂了。
Push 更适合的场景：① 短生命周期任务（CronJob、批处理）——任务结束后就没有进程让 Prometheus 来拉了。② 防火墙后的服务（外网 Prometheus 不能直接访问）。③ 上传指标到 Pushgateway 统一暴露给 Prometheus。
为什么需要把 Metrics、Logging、Tracing 关联起来？怎么实现？Q2
因为单一数据源不够。比如你看到 Metrics 显示某个接口 P99 延迟飙高——你知道慢了但不知道为什么慢。你需要找到具体 Trace 看哪个 Span 耗时最长。看到慢 Span 后，你还要看对应日志里的具体 SQL/参数/错误信息。
关联方式：Logging 中携带 TraceID（从 Tracing Context 中提取），在日志查询时直接按 TraceID 搜索到该请求的所有日志。Grafana 已经支持在 Metrics 和 Logs 之间按 Label/TraceID 跳转。
一句话：Metrics 做第一层预警，Tracing 做第二层定位，Logging 做第三层根因分析。
99.9% 可用性 = 一年允许宕机 8.76 小时——这个数字可靠吗？什么情况下会误导？Q3
可靠但不全面。99.9% 指"请求成功率"，不等于"所有请求 99.9% 都正常"。它对流量模型敏感：
① 低流量服务：一天 1000 个请求，99.9% 意味着一年允许 8760 次失败——但 8760/365 ≈ 24 次/天。对于一个日请求 1000 的服务，一天失败 24 次 = 97.6% 成功率，远超 99.9% 的上限！
② 请求不等于用户：一个失败的请求可能导致所有重试也失败——统计上 1 次失败被放大为 N 次。
③ 分片可用性：99.9% 整体可用 ≠ 每个分片 99.9%。比如 100 个 Redis 分片，每个 99.9% → 整体可用性 = 0.999^100 ≈ 90.4%。
所以：SLO 要定义清楚（"99.9% 的用户 - 请求成功"），并且要关注分片和流量分布的差异性。
TraceID 是怎么在服务之间传递的？如果用的是 gRPC，怎么自动传播？Q4
传递方式：TraceID/SpanID 在 HTTP 请求中通过 Header 传递（traceparent / tracestate），在 gRPC 中通过 Metadata 传递。
核心机制：每次服务 A 调用服务 B 时，A 的 SDK 把当前的 SpanContext 序列化到请求的 Metadata 中。B 的 SDK 在收到请求时反序列化 SpanContext，创建新的 Child Span 挂到同一个 Trace 下。
gRPC 自动传播：OpenTelemetry 提供了 gRPC Interceptor（拦截器），对代码零侵入——注册一个 ClientInterceptor 和一个 ServerInterceptor 就自动处理了 TraceID 的注入和提取。
HTTP 同理：通过 HTTP middleware/Filter 自动注入 traceparent header（W3C Trace Context 标准）。
如果公司没钱买昂贵的 APM 工具，怎么用开源方案搭建一套可观测性体系？Q5
完全可行！推荐技术栈：
• Metrics：Prometheus（开源）+ Grafana（开源社区版）
• Logging：Loki（Grafana 出品，开源）+ Promtail
• Tracing：Jaeger（CNCF 毕业项目，开源）或 Tempo（Grafana 出品，更轻量）
• 统一标准：OpenTelemetry Collector（开源）
部署方式：全部可以跑在 Docker Compose 或 Kubernetes 上。一台 4C8G 的机器就能跑通全家桶。
对比 SaaS APM：功能上不如 Datadog / New Relic 丰富（缺少 RUM、自动发现等），但核心排障能力完全够用。成本上一年省几十万美金。
