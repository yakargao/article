🧠 操作系统·内存管理

一句话：内存管理是操作系统最核心的模块——它让每个程序员写 int *p = malloc(100) 时，不用管物理内存在哪、够不够大、有没有碎片。它通过 虚拟内存 做隔离和扩容，通过 分页 做细粒度管理，通过 页面置换 在内存不足时做最优决策。本文从直觉出发，逐层深入。

层次核心问题一句话解答
① 直觉为什么需要内存管理？没有它，程序 A 能读写程序 B 的数据，大程序装不进内存
② 虚拟内存怎么让每个程序独享地址空间？MMU + 页表做地址翻译，进程以为自己独占 4GB
③ 分页怎么切分内存才不会浪费？固定 4KB 一页，多级页表按需分配，TLB 加速翻译
④ 置换算法内存满了该踢谁出去？LRU / Clock 近似最优，淘汰以后最不常用的页
⑤ 分配器malloc/new 底层怎么工作？brk 管小块、mmap 管大块、Go 三级缓存无锁分配

💡 第一层直觉：为什么需要内存管理？

🪑 类比：共享办公室桌面

想象你和同事共享一张大办公桌。没有内存管理就像——没有桌位划分、没有工位号、谁的东西放哪全凭自觉。后果很可怕：

  • 你写了一半的报告，被同事的咖啡杯压住了（地址冲突）
  • 同事拿走你的便签纸以为是公用的（数据泄露）
  • 桌上有大片空地方但东一块西一块，你想放一张 A3 图纸却放不下（外部碎片）

操作系统内存管理就是给每个进程划分工位（虚拟地址空间），用工位号（虚拟地址）代替实际物理位置。每个进程安心在自己的工位工作，互相不干扰。

没有虚拟内存会怎样？—— 一个思想实验

假设你在 32 位 x86 裸机上写程序，没有 MMU 没有虚拟内存：

// ❌ 没有内存管理——所有程序直接读写物理地址
// 程序 A 在地址 0x1000 存了用户密码
// 程序 B 不小心也把数据写到 0x1000...
mov [0x1000], eax   // 程序 A
mov ecx, [0x1000]   // 程序 B —— 直接读取了程序 A 的数据！

// 更严重：程序 A 包含了完整的地址映射
// 链接器必须在编译时就确定每个变量在物理内存中的位置
// 换个程序就崩，换个硬件也崩

🔬 用 Go 看看内存长什么样

写一个 Go 程序，直观感受变量地址、指针和内存分配：

package main

import (
    "fmt"
    "unsafe"
)

// Stack 局部变量
func stackDemo() {
    a := 42                // 栈上分配
    fmt.Printf("栈变量 a = %d, 地址 = %p\n", a, &a)
}

// Heap 上的内存
func heapDemo() {
    p := new(int)          // new 返回指针 → 堆上分配
    *p = 100
    fmt.Printf("堆变量 *p = %d, 地址 = %p\n", *p, p)

    slice := make([]byte, 1024*1024)  // 1MB → 堆分配
    fmt.Printf("大对象地址 = %p, 长度 = %d\n", slice, len(slice))
}

func main() {
    // 观察地址空间布局
    stackDemo()
    heapDemo()

    // 打印指针大小——64位系统=8字节
    var ptr *int
    fmt.Printf("指针自身大小 = %d 字节\n", unsafe.Sizeof(ptr))
}

输出示例：

栈变量 a = 42, 地址 = 0xc000046728
堆变量 *p = 100, 地址 = 0xc00001a098
大对象地址 = 0xc000100000, 长度 = 1048576
指针自身大小 = 8 字节

注意看：栈地址和堆地址离得很远——这就是虚拟地址空间的布局。Go 帮你管理了这一切，你只管用 new 和 make。

为什么不能直接操作物理内存？

问题无虚拟内存有虚拟内存
地址空间隔离进程 A 可读写进程 B 的内存每个进程独立页表，互不可见
外部碎片加载/卸载后产生大量空洞分页消除外部碎片
程序容量程序必须 ≤ 物理内存程序可比物理内存大（swap 交换）
共享内存困难，地址冲突风险不同虚拟页映射到同一物理页即可
链接/加载编译时必须固定物理地址地址无关（PIE），动态链接轻松

🧠 第二层：虚拟内存——每个程序都以为自己是唯一用户

🏨 类比：酒店房间的房卡系统

物理内存是酒店的房间（物理帧），虚拟内存是房卡。每个住客（进程）拿到一张房卡，卡上写着自己的"房间号"（虚拟地址），从 101 到 999 连续排列。但实际上：

  • 你的 201 号房间可能实际上是 301 房（地址翻译）
  • 别人的卡刷不开你的门（隔离）
  • 你可以订一个比酒店实际房间还大的套间——因为酒店可以把不用的家具存到仓库（磁盘 swap）

前台服务员 MMU（Memory Management Unit） 负责把你的房卡号翻译成真正的房间号。

进程的虚拟地址空间布局

每个进程看到的地址空间长这样（以 x86-64 Linux 为例）：

图1：进程虚拟地址空间布局 — 从低地址到高地址依次为：代码段 → 数据段 → 堆(向上增长) → mmap区 → 栈(向下增长) → 内核空间

虚拟地址 → 物理地址：MMU 的翻译过程

当 CPU 执行 mov eax, [0x401000]，这条指令里的地址是虚拟地址。MMU 负责把它翻译成真正的物理地址才能去内存取数据。

图2：虚拟地址→物理地址翻译流程 — CPU发出VA → MMU查TLB/页表 → 得到PA或触发缺页中断

📬 类比：翻译官的工作

虚拟地址就像"请找 3 号楼的 502 室"，MMU 就是翻译官，把它翻译成实际的门牌号"B 栋 12 楼 03 室"。如果这个房间根本不在楼里（缺页），翻译官就打电话给后勤（OS）从仓库（磁盘）搬家具过来，然后重新找。

📄 第三层：分页机制——把内存切成小格子

🍫 类比：巧克力排的格子

为什么不整块分配内存？想象一个巧克力排（物理内存），你掰下一块（分配一块空间）。如果用"整块掰"，掰了几次后剩下的形状乱七八糟，可能有一大块但形状不对，放不下下一块巧克力（外部碎片）。分页就是事先把巧克力排切成统一大小的方格（4KB 一页），每次分配按格给，永远不会因为形状问题浪费空间。

代价是：如果只想吃一小口（只用了 1 字节），也得给你一整格（4KB）——这就是内部碎片。

为什么用分页而不是整块分配？

分配方式比喻碎片问题灵活性
动态分区（可变长度）按需切蛋糕外部碎片严重一般
分页（固定4KB）切好的巧克力格仅内部碎片（≤4KB）极高
分段定制包装箱外部碎片低

多级页表——为什么必须分级？

📂 类比：图书馆索引系统

单级页表就像一本书只有目录——如果书有 10 亿页（64位地址空间），光目录就要 32PB。多级页表像图书馆的索引系统：大楼 → 楼层 → 书架 → 书。你只去你需要的书架，不需要给整栋楼建索引。这就是"按需分配"的思想——64位系统用 4 级页表，只需要为实际使用的地址区间分配页表，典型进程只要几十 KB。

图3：Linux 四级页表结构 — CR3→P4→P3→P2→P1→PFN，仅分配使用路径，未用页表不分配

🧮 单级 vs 多级页表计算

单级页表：64位地址空间，页4KB=2¹²，页表项8B → 2⁶⁴/2¹² = 2⁵² 项 × 8B = 2⁵⁵ B ≈ 32 PB → ❌ 完全不可行

四级页表：每级512项(9bit索引)，只为使用中的VA区间分配页表。典型进程只需 几十 KB 页表空间。

ARMv8-A：4KB页用4级，64KB页用2级

TLB 快表——翻译缓存

TLB (Translation Lookaside Buffer) 是 MMU 内部的硬件缓存，缓存最近使用的 VPN→PFN 映射。没有 TLB，每次内存访问都要遍历 4 级页表（4 次内存访问），性能直接崩盘。

情况过程耗时概率
TLB Hit直接从 TLB 拿 PFN~1 CPU cycle99.9%
TLB Miss (页表命中)硬件遍历 4 级页表 → 更新 TLB~几十 cycle~0.1%
TLB Miss + 缺页页表项 Present=0 → 触发 #PF 异常~几 ms（磁盘 I/O）极低

缺页中断（Page Fault）完整流程

图4：缺页中断完整处理流程 — 从 CPU 访问无效地址 → #PF 异常 → 内核处理 → 分配页 → 返回重试
// Linux 缺页中断处理伪代码 (fs/do_page_fault.c 简化)
void do_page_fault(struct pt_regs *regs, unsigned long error_code) {
    unsigned long addr = read_cr2();  // CR2 存了出错的虚拟地址

    // 1. 检查地址是否在进程的 VMA（虚拟内存区域）中
    struct vm_area_struct *vma = find_vma(current->mm, addr);

    if (!vma) {
        // 地址不在任何 VMA → 非法访问
        force_sig_fault(SIGSEGV, ...);
        return;
    }

    // 2. 处理各种缺页场景
    if (vma->vm_ops && vma->vm_ops->fault) {
        // 文件映射缺页 → 从磁盘读
        vma->vm_ops->fault(vma, &fault);
    } else if (vma->vm_flags & VM_GROWSDOWN) {
        // 栈自动增长
        expand_stack(vma, addr);
    } else {
        // 匿名页 → 分配零页 (COW)
        do_anonymous_page(vma, addr);
    }

    // 3. 更新页表，返回用户态重新执行指令
    update_page_table(addr, pfn);
}

🔄 第四层：页面置换算法——内存满了该踢谁？

🧳 类比：整理旅行箱

想象你在打包旅行箱（物理内存），箱子满了，但你要装一件新东西。你必须扔掉一件——问题来了：扔掉哪件最不亏？

  • FIFO：扔掉最早装进去的那件。但最早装的可能是你每天都要用的牙刷！
  • LRU：扔掉最久没碰过的那件。比如在箱底放了一周的游泳裤——大概率这几天也不会穿。
  • Clock：在箱子里画个表盘，指针一圈圈转。每件行李有个"最近用过"的标签：标签还在的清零继续转，标签没了的就扔出去。

理想情况（OPT）：你能预知未来，扔掉以后最不会用到的那件。可惜现实世界没有时光机，所以只能用各种近似策略。

⚙️ Linux 实际用的是什么？

Linux 内核使用 双链回收策略（近似 LRU）：

  • active_list —— 活跃页链表（经常被访问）
  • inactive_list —— 不活跃页链表
  • 页面在 active 中时不会被回收，只有被移到 inactive 后才可能被回收
  • kswapd 内核线程在内存低于 watermark 时唤醒回收

$ cat /proc/sys/vm/swappiness
60

# 值越高 (0-100) → 越倾向于 swap 匿名页而不是回收文件缓存
# 服务器通常设为 10-30，桌面可设 60

🗂️ 第五层：malloc 底层秘密 & Go 内存分配器

🏗️ 类比：malloc 是包工头

malloc(32) 就像包工头需要一小块木板——他去工地的仓库（堆，brk 管理）拿一块，如果仓库不够大就把墙往外推（sbrk 扩展堆）。

malloc(1_000_000) 就像需要一整辆卡车——仓库里放不下，包工头直接给供应商打电话下单（mmap 独立映射），用完直接退货（munmap 归还 OS）。

这就是 malloc 的两条路径：小请求 → brk 管理（快，但不归还 OS），大请求 → mmap 分配（独立，free 即归还）。

brk / sbrk vs mmap 两条路径

  • brk(addr) — 设置堆的结束地址（program break），相当于"墙推到这儿"
  • sbrk(incr) — 增量版 brk，返回旧 break 位置，正数扩展堆，负数收缩
  • mmap(NULL, size, ...) — 在 mmap 区创建独立映射，对齐到页粒度，free 时 munmap 立即归还
// glibc malloc 的分配策略（简化）
void *malloc(size_t size) {
    if (size == 0) return NULL;

    // 小对象 (< 128KB) → 从堆（brk 区域）分配
    if (size <= MMAP_THRESHOLD) {  // 默认 128KB
        // 从 main_arena 的空闲链表找合适 chunk
        // 如果没有 → sbrk() 扩展堆
        return arena_alloc(size);
    }

    // 大对象 (≥ 128KB) → mmap 匿名映射
    return mmap(NULL, size, PROT_READ|PROT_WRITE,
                     MAP_PRIVATE|MAP_ANONYMOUS, -1, 0);
}

void free(void *ptr) {
    if (is_mmap(ptr)) {
        // mmap 分配的 → munmap 立即归还给 OS
        munmap(ptr, chunk_size(ptr));
    } else {
        // brk 分配的 → 标记为空闲，尝试合并相邻空闲块
        // 但不归还 OS（缓存复用，减少系统调用）
        mark_free(ptr);
    }
}

伙伴系统（Buddy System）—— 内核的物理页分配器

Linux 内核用伙伴系统管理物理页，解决外部碎片：

🔧 伙伴系统工作流程

// 伙伴系统：将内存按 2^n 分块管理
// free_area[0] = 1页块, free_area[1] = 2页块, ... free_area[9] = 512页块
//
// 分配 3 页：
//   1. 需要 2²=4 页块 → 检查 free_area[2] (order=2)
//   2. 如果 order2 空 → 从 order3 拿 8 页，分裂为 [A,B] 各 4 页
//   3. 返回 A (4页)，B 挂在 free_area[2] 上
//
// 释放：检查 buddy（伙伴块）是否空闲 → 是则合并回更大块

Slab 分配器——解决内部碎片 + 对象缓存

伙伴系统的粒度是"页"（4KB），但内核经常要分配很小的对象（task_struct ~2KB, inode ~500B）。如果直接用伙伴系统，每个小对象都浪费一页（内部碎片）。Slab 分配器解决了这个问题：

层级功能类比
kmem_cache管理特定类型对象的缓存每种对象类型一个工厂
slab一个或多个连续物理页，等分为对象槽工厂中的托盘
object单个内核对象（inode、task_struct等）托盘上的格子
// Slab 使用示例（Linux 内核）
// 为 task_struct 创建专用缓存
struct kmem_cache *task_cache = kmem_cache_create(
    "task_struct",
    sizeof(struct task_struct),  // ~2KB
    0,                           // 对齐
    SLAB_PANIC,
    NULL
);

// 分配一个 task_struct → O(1)，从空闲槽直接拿
struct task_struct *p = kmem_cache_alloc(task_cache, GFP_KERNEL);

// 用完归还 → O(1)，回到空闲槽，对象仍缓存
kmem_cache_free(task_cache, p);

🚀 Go 内存分配器——TCMalloc 风格的三级缓存

🏪 类比：连锁便利店三级体系

Go 的分配器借鉴了 Google TCMalloc 的设计，像一家连锁便利店体系：

  • mcache = 每个门店的收银台（零排队=无锁）——每个 P（逻辑处理器）一个，直接拿货
  • mcentral = 区域仓库——门店缺货时从仓库调货（需要锁）
  • mheap = 总仓/物流中心——从厂家（OS）大批量进货（mmap）

80% 以上的分配在 mcache 完成，无需加锁！这就是 Go 频繁分配内存也很快的秘诀。

图6：Go 内存分配器三级结构 — mcache(无锁) → mcentral(轻量锁) → mheap(大锁+mmap)

Go 分配流程详解

// runtime.mallocgc 简化流程（Go 源码）
func mallocgc(size uintptr, typ *_type, needzero bool) unsafe.Pointer {
    if size == 0 {
        return unsafe.Pointer(&zeroBase)
    }

    // 1️⃣ 小对象 (≤ 32KB) → 从 mcache 分配
    if size <= maxSmallSize {
        c := getMCache()  // 当前 P 的 mcache，无锁

        // 微对象 (≤ 16B) → tiny allocator（合并到同一个 16B 槽）
        if size <= maxTinySize {
            return c.tinyAlloc(size)
        }

        // 根据 size 计算对应的 size class
        span := c.alloc[sizeToClass(size)]
        obj := span.nextFreeIndex()
        return obj
    }

    // 2️⃣ 大对象 (> 32KB) → 直接从 mheap 分配
    // 大对象由 GC 直接追踪，不经过 mcache/mcentral
    s := mheap_.alloc(size)
    return s.base()
}

// 当 mcache 的 span 满时：
// 1. mcache -> mcentral.refill() → 从 nonempty 链取一个 span（加锁）
// 2. mcentral 空了 → mcentral -> mheap.allocSpan()（加锁）
// 3. mheap 需要新内存 → mmap() 向 OS 申请

关键概念总结

组件作用特点
mspan管理一段连续页（通常 8KB），等分为固定大小对象槽最小分配单位，67 种 size class
mcache每个 P 独享的 span 缓存，每种 size class 最多 1 个 span无锁分配（零竞争）
mcentral全局 span 池，每种 size class 一个 mcentral需要锁，nonempty + empty 双链表
mheap全局堆，管理所有 arena，从 OS mmap 内存大锁保护，负责 span 分配/释放
size class从 8B 到 32KB 共 67 种规格（向上取整到最近 class）减少内部碎片

❓ 第六层：实战进阶——高频问答

Q1: 虚拟内存解决了哪些问题？用直觉解释

A: 相当于给每个程序一个私有的"幻觉"——虚拟内存解决四大问题：

  • 隔离：程序 A 不能读程序 B 的数据（各自页表独立）
  • 扩容：程序可以比物理内存大（不用的页 swap 到磁盘）
  • 消灭外部碎片：连续虚拟页可以映射到任意物理页，物理内存里的"洞"不再影响程序
  • 共享：动态库只加载一份物理内存，多个进程的虚拟页指向同一个物理帧

Q2: 多级页表为什么能省内存？画图解释

A: 关键在 按需分配：

  • 64 位系统如果单级页表：2⁵² 项 × 8B ≈ 32 PB，比整个物理内存还大无数倍
  • 多级页表像图书馆索引系统：不需要给所有书建索引，只建你去的那些书架
  • 四级页表每级 512 项（9bit 索引），未映射的页目录根本不存在
  • 典型进程的页表只要 几十 KB——2-3 个顶层目录 + 几个二级/三级/四级页表

Q3: TLB Miss 后完整流程是什么？

A: TLB Miss 分两种情况：

  1. TLB Miss + 页表命中：硬件（或软件）遍历 4 级页表（4 次内存访问）找到 PFN → 更新 TLB → 重新执行指令。~几十 CPU cycles。
  2. TLB Miss + 页表也 Miss（缺页）：页表项 Present=0 → 触发 #PF 缺页异常 → CPU 陷入内核 → do_page_fault 处理：检查 VMA 合法性 → 分配物理页 → 磁盘 I/O 加载数据（如果需要）→ 更新页表 → 返回用户态重新执行。整个过程 ~几毫秒（磁盘瓶颈）。

Q4: LRU 为什么难实现？Clock 怎么近似它？

A: 严格 LRU 需要在每次内存访问时 更新链表顺序——把刚访问的页面移到链表头部。纯硬件实现成本极高（需要专用硬件追踪所有页面的访问时间），纯软件实现的话每次内存访问都要执行额外指令，性能不可接受。

Clock 算法用近似替代精确：每个页维护一个 use bit（硬件自动置 1），缺页时指针循环扫描：遇到 use bit=1 的清零并继续，遇到 use bit=0 的淘汰。只需要 1 个 bit 的硬件支持，效果接近 LRU。

Q5: Go 内存分配器为什么这么快？

A: 核心原因是 三级缓存 + 无锁路径：

  • 每个 P（逻辑处理器）有独立的 mcache，G 在 P 上运行时，小对象分配直接从 mcache 拿 span——不需要任何锁
  • 只有 mcache 不够时才加锁从 mcentral 批量取一批 span
  • mcentral 不够再加锁从 mheap 申请
  • 统计显示 80%+ 的分配在 mcache 命中，所以绝大多数分配完全无锁

Q6: 什么是 Belady 异常？为什么 FIFO 有而 LRU 没有？

A: Belady 异常指 增加物理帧数反而导致缺页率上升的反直觉现象。只有 FIFO 有该问题。

原因：FIFO 只考虑"进入时间"，不考虑"使用频率"。增加帧数后，被淘汰的页面可能正好是需要频繁使用的"老人"，而新来的页面虽然用得少，但因为是"新人"所以被留下来了。

例子：序列 [1,2,3,4,1,2,5,1,2,3,4,5]，3 帧时 FIFO 缺页 9 次，4 帧时缺页 10 次。

LRU、Clock 等基于局部性原理的算法不存在该问题。

Q7: malloc(1) 实际分配了多少内存？

A: 远比 1 字节多：

  • 元数据开销：每个 chunk 需要 ~8-16 字节的 header（prev_size + size + flags）
  • 对齐要求：16 字节对齐（x86-64），所以 malloc(1) 实际分配至少 16-32 字节
  • 最小 chunk 大小：glibc 中最小 chunk 是 32 字节（32 位）或 48 字节（64 位）
  • 所以 malloc(1) 实际用了 ~48 字节，浪费率 98%——这就是为什么 Go 用 size class (8B/16B/32B...) 来减少浪费

Q8: 栈和堆最大的区别是什么？为什么栈这么快？

A: 栈分配就是 一条指令：subq $N, %rsp（减栈顶指针），释放就是 addq $N, %rsp。没有查找、没有锁、没有碎片、LIFO 绝对顺序。堆分配需要：查空闲链表/找合适 chunk/分割/更新元数据/free 时还要合并——远比栈复杂。

栈的代价是大小有限（~8MB/线程）且生命周期严格绑定函数调用。堆灵活但慢。这就是 Go 的 逃逸分析 的动机——能放栈上的绝不放到堆。
