⚡ Go 并发的艺术

一句话：Go 的并发能力源自三个设计——轻量级 goroutine（2KB 栈、百万并发）、GMP 调度器（M:N 用户态调度、工作窃取）、CSP channel（"通过通信共享内存"）。这三者结合，让 Go 成为并发编程最友好的语言之一。

🪶 为什么 Goroutine：轻量到可以开百万个

🐜 类比：OS 线程是大象（创建慢、吃内存多、调度开销大），goroutine 是蚂蚁（2KB 初始栈、M:N 用户态调度、百万只蚂蚁协同工作）。

// 开 100 万个 goroutine —— 现实可行！
for i := 0; i < 1000000; i++ {
    go func(id int) {
        time.Sleep(time.Hour)  // 每个G休眠，不占CPU
    }(i)
}
// 内存 ≈ 1M G × 2KB = 2GB（栈按需增长）
// 对比：100万 Java Thread = 1TB 栈内存（不可能）

🌍 Python 的"并发困境"

Python 线程受 GIL（全局解释器锁）限制，多线程无法并行执行 CPU 密集任务——同一时刻只有一个线程执行 Python 字节码。Go 没有 GIL，goroutine 可以真正并行在多核上。Python 的替代方案：multiprocessing（进程）、asyncio（单线程协程）。

⚙️ GMP 调度模型深度解析

🏭 类比：G = 任务单（goroutine），M = 工人（OS线程），P = 工位（处理器）。工人必须站在工位上才能干活。如果一个工人在某个工位上卡住了（系统调用），工位就换个工人继续干——生产线不停。

G、M、P 的本质

角色含义数量关键字段
G (goroutine)执行体：栈+PC+状态无上限stack, sched(gobuf), atomicstatus
M (machine)OS 线程的抽象≤10000（默认）g0(调度栈), curg, p
P (processor)资源令牌（逻辑CPU）GOMAXPROCS（默认CPU核数，≤256）runq[256], runqhead/tail, mcache
GMP 模型全景：P的LRQ+全局GRQ+工作窃取+M绑定P执行G

findrunnable 的优先级

func findrunnable() *g {
    // 优先级（从高到低）：
    if gp := runqget(_g_.m.p); gp != nil { return gp }  // 1. 本地队列
    if gp := globrunqget(); gp != nil { return gp }      // 2. 全局队列
    if gp := netpoll(false); gp != nil { return gp }     // 3. 网络轮询（epoll就绪的G）
    if gp := stealWork(); gp != nil { return gp }        // 4. 从其他P偷
    stopm()  // 5. 实在找不到 → M休眠
}

🔄 四大调度场景

场景G 的操作P 的操作M 的操作
正常执行完毕G 调用 goexit → 调度器选下一个 GP 不变，LRQ 出队M 继续绑定 P
阻塞系统调用G 进入 syscall → goparkP 解绑 → 被其他 M 接管M 带着 G 进入内核，可能新 M 创建
Channel 阻塞G 加入 sendq/recvq → goparkP 不变，调度下一个 GM 继续用这个 P 跑别的 G
抢占收到 SIGURG → 检查抢占标志 → morestackP 不变M 执行抢占逻辑

系统调用 Handoff 详解

// G 执行 read(fd) → 进入 syscall
// 1. M 调用 entersyscall() → M 和 P 解绑
// 2. P 的状态变为 _Psyscall
// 3. 如果 P 的 LRQ 不为空 且 有空闲 M → P 转给新 M
// 4. 如果 P 的 LRQ 为空 → P 进入空闲 P 池
// 5. syscall 返回 → M 调用 exitsyscall() → 尝试重新绑定 P
//    - 有 P → 继续执行 G
//    - 无 P → G 放回 GRQ，M 休眠

🔀 Channel 并发模式

Select 多路复用

select {
case v := <-ch1:   // ch1 可读
case ch2 <- x:    // ch2 可写
case <-time.After(5*time.Second):  // 超时
case <-ctx.Done(): // 取消
default:          // 非阻塞（都不可用则走 default）
}

五大经典模式

模式用途实现
Fan-in多输入合并到一通道多个 goroutine 写同一个 chan，select 或 merge
Fan-out一输入分发到多 worker一个 goroutine 读取，分发到 worker chan
Pipeline串联处理阶段ch1→stage1→ch2→stage2→ch3→...
Or-Done任一完成即返回select 监听 done chan + 结果 chan
有界并行限制并发数带缓冲 chan 作为信号量
// 信号量模式：限制最多 N 个并发
sem := make(chan struct{}, 10)  // 容量=10
for _, task := range tasks {
    sem <- struct{}{}            // 获取槽位（满了会阻塞）
    go func(t Task) {
        defer func() { <-sem }()  // 释放槽位
        process(t)
    }(task)
}

🧠 Go 内存模型

核心原则：Go 的 happens-before 保证——如果事件 A happens-before 事件 B，则 A 对内存的修改对 B 可见。Channel 通信、Mutex 的 Lock/Unlock、Once 的 Do 都建立了 happens-before 关系。

Happens-Before 关键规则

// 1. Channel: send happens-before recv
var c = make(chan int)
go func() { x = 1; c <- 0 }()  // x=1 在 send 之前 → 对 recv 可见
<-c                               // 这里保证能看到 x=1

// 2. Mutex: Unlock happens-before Lock
var mu sync.Mutex
mu.Lock(); x = 1; mu.Unlock()     // 释放锁
mu.Lock()                          // 获取锁 → 保证看到 x=1

// 3. Once: do() happens-before 其他 Once.Do 返回

容错recover() 手动Let it crash + Supervisor 树try-catch / Result

最佳场景高并发网络服务容错要求极高的系统I/O 密集单机程序

💡 选型建议

选 Go CSP：构建高并发网络服务（HTTP/gRPC）、需要多核并行、团队熟悉同步思维。

选 Actor：分布式容错系统（电信、金融）、需要进程级隔离和自动重启、状态机驱动的业务。

选 async/await：单机 I/O 密集型（爬虫、代理）、已有 Rust/JS/Python 技术栈、不追求多核并行。

CSP 和 Actor 的哲学差异：CSP 认为"共享内存是万恶之源，用 channel 传数据"；Actor 认为"任何共享都是万恶之源，连 channel 都不共享"。Go 更务实，Erlang 更理想主义。
